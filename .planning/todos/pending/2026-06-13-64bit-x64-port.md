## Why (motivation — earned during the 2026-06-13 cantina door-snap investigation)

The 64-bit port is the single change that addresses several known issues at once, and we have
a **proven reference**: SWG Restoration ships and runs a 64-bit D3D9 client (`D:\SWG Restoration\x64`)
that is stable and does NOT exhibit the cantina door snap.

1. **Door snap (and the interior-snap class).** Root-caused to **32-bit float codegen fragility**
   (x87/SSE-mix under MSVC v145) in the cell→world position handoff — our source there is
   byte-identical to pristine SWG-Source, yet Restoration's x64 build of the same source is
   clean. x64 mandates deterministic SSE-only float + ABI-managed MXCSR → removes the fragility.
2. **32-bit address-space OOM crashes.** This session alone hit `MemoryManager.cpp:1224` FATAL
   `b0780503` (~435 MB, skeletal-animation load out in the world) — the chronic 32-bit
   exhaustion/fragmentation class. x64 effectively eliminates it.
3. **Modern toolchain alignment.** We're already on MSVC / C++20 / MSVC-STL (STLport gone). x64
   is the natural next step and matches the live community clients.

## Scope (mechanical port of the SAME renderer — bounded, unlike the D3D11 rewrite)

- **Build:** add `x64` platform to `src/build/win32/swg.sln` and every `.vcxproj`.
- **x87 inline asm → intrinsics:** `__asm fnstcw/fldcw` in `FloatingPointUnit.cpp` is x64-illegal;
  replace with `_control87`/`_controlfp` (also fixes the x87-only FPU-restore gap noted in the
  door-snap investigation). Grep the whole tree for `__asm` — there will be more.
- **Pointer/int truncation:** fix `(int)pointer` casts, `DWORD`-holds-pointer, serialization
  width assumptions. The compiler flags most (C4311/C4312/C4244); treat-warnings-as-errors helps.
- **Third-party libs need x64 builds.** Restoration's `x64/` dir shows which already exist:
  Miles `mss64.dll`, `dpvs.dll` (x64), `binkw64.dll`, `discord-rpc`, `jpeg62`, `libicu*`, `pcre`,
  `libxml2`. Our D3D9 plugins (gl05/06/07) must be rebuilt x64. **D3DX is x64-hostile** — pairs
  with HARD-05 (port to `d3dcompiler_47`, which Restoration did). Mozilla/Vivox: voice chat is
  already disabled in our cfg, so the Vivox/xul x64 gap may be moot.
- **Struct packing / alignment:** audit `#pragma pack` and any hardcoded `sizeof` assumptions,
  especially in IFF/TRE serialization and network message layout.

## Sub-task: Miles 7.2e → 9.x audio port (SDK in hand, 2026-06-14)

The x64 audio dependency is the single third-party lib that needs real code work, not just a relink —
because **our engine is on the Miles 7.2e API and the only x64 Miles is 9.x** (Restoration ships
**Miles 9.3** in `D:/SWG Restoration/x64/mss64.dll`). There is NO commercial license to buy — Miles is
retired (Epic acquired RAD 2021; "does not want new projects using it"). Procurement = the **9.3b
preservation release** (educational/preservation/modding only; Epic retains commercial rights), the
same posture Restoration operates under. See `reference_restoration_binaries_intel` memory.

**SDK already cloned:** `D:/Code/milesss-v9.3b/` — has all link pieces:
- `win/sdk/include/mss.h` — the 9.x API to port TO
- `win/sdk/lib/mss64.lib` — x64 import lib (+ `mss64midi.lib`)
- `win/sdk/redist64/`, `win/mp3/redist64/`, `win/ogg/redist64/` — x64 redist:
  `mss64.dll`, `mss64mp3.asi`, `mss64ogg.asi`, `mss64dsp.flt`, `binkawin64.asi`

**The work (it's an API PORT, not a relink):**
1. Port the Miles call sites in `src/engine/client/library/clientAudio/src/win32/` (`Audio.cpp`,
   `SoundObject3d.cpp`, `FirstClientAudio.h`) from the 7.2e API to the 9.x API. Restoration did this
   delta in their closed source; expect signature/enum/provider-init changes across 7→8→9.
2. Swap the SDK include/lib paths from `src/external/3rd/library/miles-7.2e/` to the 9.3b SDK (vendor
   it into `src/external/3rd/library/miles-9.3b/` for repo self-containment, mirroring the 7.2e layout).
3. **Stage the x64 redist next to the x64 exe** — `mss64.dll` PLUS the provider set
   (`mss64mp3.asi`, `mss64ogg.asi`, `mss64dsp.flt`, `binkawin64.asi`) and the `miles/` provider dir.
   The .asi/.flt providers MUST be present or audio half-dies with a warning flood — this is exactly
   the [[project_audio_fixed_missing_miles_redist]] failure mode (mp3 codec + 3D providers). Mirror
   Restoration's `x64/miles/` provider set as the reference for what to ship.

Clean-but-expensive alternative (only if the 9.x port proves painful): abstract the audio backend and
replace Miles with open middleware (OpenAL Soft / miniaudio). Big rewrite — SWG is deeply coupled to
Miles 2D/3D providers + .asi codecs + .flt DSP/EAX; Restoration kept Miles, signaling replacement is
the harder road.

## Relationship to D3D11

Complementary, not competing. 64-bit keeps BOTH renderers (D3D9 gl05/06/07 + D3D11 gl11) working;
it's a platform port, not a renderer change. The "complete" client is eventually **64-bit + D3D11**:
they solve different halves (Restoration did 64-bit/kept-D3D9; we did D3D11/kept-32-bit).

## Rider: remove CORNERSNAP instrumentation here

The CORNERSNAP `_DEBUG` probes (`CollisionResolve.cpp` + `CellProperty.cpp`) are intentionally
KEPT through v2.3 (Phase 26 strips only D-15) because they are the acceptance harness for the
door snap. Once this x64 port fixes the door snap and it is verified clean against these probes,
**strip the CORNERSNAP instrumentation as part of this work** — that completes the deferred half
of HARD-03.

## Priority

Not v2.3. Large but bounded effort with high payoff (kills the door snap + the OOM crash class +
modernizes the platform). Scope as its own milestone when ready; HARD-05 (D3DCompile) is a natural
prerequisite/companion since x64 can't use legacy D3DX.

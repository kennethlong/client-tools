Miles can run almost everything on the SPU - mixing, DSP filters, resampling, ADPCM decompression, MP3 decompression and more. About the only major feature that isn't yet on the SPU is Ogg decompression which just accesses too much memory randomly (in the future, we hope to get this codec onto the SPU as well).

Rather that supply a pre-built ELF for the SPU decompression module, we instead supply static libraries that you can use to build this ELF. This technique lets us supply multiple SPU support libraries, so that you aren't locked into any particular SPU image format.

So, to make an SPU ELF, you take the Miles SPU libraries (MilesSPU.a and BinkaSPU.a) and link it with an SPU support library (like MssSPU_sputhreads.a, MssSPU_raw.a, or MssSPU_spurs.a) to create an ELF that you load with the MilesStartAsyncThread function. (You might also add optional libraries for MP3 as well).

Then, you link your PPU application with the matching PPU library (like MssPPU_sputhreads.a, MssPPU_raw.a, or MssPPU_spurs.a). So, there are always two halves to the SPU support - one for the SPU and one for the PPU. One you link with MssSPU.a to make an ELF, and one you link with your PPU game to talk to the other SPU half.

Right now, we ship with SPU thread support, RAW support and SPURS task support. You can also write your own SPU support library as well. To do this, you need to implement the small APIs that are prototyped in the radspu.h and radppu.h header files.

Note that if you are doing custom SPU interface work, the r1 register (the stack pointer) must be initialized as the standard SPU ABI dictates. That is, the 2nd word in r1 must contain the amount of remaining stack space free. SPU Threads, SPURS and RAW SPU all handle this correctly, so you only have to worry about it if you are doing something custom.

So, for example, the command to build the Miles SPU elf for SPURS looks something like this: 

spu-lv2-g++.exe -mspurs-task -o mssspurs.elf -g -Xlinker --gc-sections -L%SCE_PS3_ROOT%\target\spu\lib -fno-exceptions -fno-rtti -L -Xlinker -Map -Xlinker mssspurs.map -Xlinker --start-group libdma.a  mssspu.a binkaspu.a mp3spu.a mssspu_spurs.a -Xlinker --end-group

In this command, notice how the mp3spu.a file is in between the mssspu.a and the mssspu_spurs.a file - this order is important so that Miles sees the MP3 support functions.

OK, now we have an SPU elf image! Some people load this image off the disc directly, but most people embed the SPU image into the PPU elf. Note that if you use the spu_elf-to-ppu_obj command to embed your SPU image into your game, then you need to add the switch "--strip-mode=normal" to the end (Miles uses the extra data that the strip-mode=hard option removes).

Next, you need to install the SPU image by calling MilesStartAsyncThread to start the SPU elf. It is then ready to receive audio commands from the Miles PPU library.

Once you finally have the SPU image built and loaded, then you need to tell Miles itself to use it! To do this, you pass in the AIL_OPEN_DIGITAL_USE_SPU flag to AIL_open_digital_driver that matches the SPU index that you passed to MilesStartAsyncThread. Now, whenever Miles has any work to do, it will call down to the SPU image to perform the work in the background.

Everything else is automatic - the exact same Miles API now takes no PPU time!

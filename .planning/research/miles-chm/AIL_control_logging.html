// Function prototype:

AILTRACECB AIL_control_logging(char const * filename,
AILTRACECB callback,
S32 level);
Description

allows you to control Miles' internal logging mechanisms.

Parameters
filename [in]

allows you to pipe all log information to the given filename. You can pass 0 if you don't want to capture to a file.
callback [in]

should be set to the address of an application function prototyped as an AILTRACECB callback.
level [in]

specifies whether only application calls to Miles (1), application calls plus any resulting internal calls to Miles functions (2), or no calls at all (0) are reported to the trace callback.
Return value
return [out]

The address of the previously-installed debug trace function handler.

Discussion

This function is intended primarily for use in diagnostic work. It provides a single convenient hook point that a Miles application can use to inspect its own calls to the system, and (optionally) any further calls to MSS that are made internally in the course of executing a given Miles function.

You can send the data to a file, to a trace callback, or, if you pass 0 for both the filename and callback, then it will be dumped to the platforms standard logging pane. If you are connected through Telemetry, the log data will be forwarded to it as well.

The ASCII text traffic which will be passed to the AILTRACECB callback function may consist of indenting whitespace, timestamps and other marker/decorative characters as well as function names, parameters, and return values.

Most applications that use AIL_control_logging should pass 1 as the level parameter, to avoid excessive output-text clutter due to nested calls.

Background function calls (those made while Miles is executing from its own internal threads) will not be reported to the trace callback.

You can call this function before AIL_startup, if you want to capture a log of the system initializing (useful for tracking down dynamic loading problems).

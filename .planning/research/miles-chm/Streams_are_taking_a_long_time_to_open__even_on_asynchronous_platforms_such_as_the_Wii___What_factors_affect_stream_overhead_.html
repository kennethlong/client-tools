Q: Streams are taking a long time to open, even on asynchronous platforms such as the Wii. What factors affect stream overhead?

A: AIL_open_stream does 3 things and it has to do them synchronously (since it needs to return success or failure):

  • Open the file (which means accessing the DVD directory)
  • Seek to the beginning of the file
  • Read the first chunk of the file to figure out if it's a format that we support

You can avoid the open by pre-opening the file yourself (and using the special filename format described in AIL_open_stream), or by using your own packed file format that caches all the filenames.

You can avoid the seek by doing an asynchronous read of the first 32 bytes of the file (to force the drive head to move) and then not calling AIL_open_stream until that seek succeeds.

To avoid the delay on the first read, you can either open the stream ahead of time (e.g., at level-load time), or overload the file IO callbacks and do something more sophisticated (e.g., always have the first chunk of every sound file in memory, so that you can always instantly return the data that AIL_open_stream needs.)

In general, the streaming system should be used for things like streaming audio in the background, where a hitch at the beginning of a level would never get noticed. If you're finding yourself using AIL_open_stream for in-game sounds, there are usually ways to improve upon the function's default behavior.

For instance, try breaking up sounds into two pieces (either in your game baking system, or in a preprocess step). The first chunk of the sound should contain about 500 ms of sound data, generally enough to seek to the second chunk of the sound and stream it in. You then precache all of the first halves of each possible sound at level-load time. So, when you want to play a sound, you start the sound effect immediately by playing the first piece, and then schedule the second half of the sound to be loaded in the background and then played when it is finally in memory. There are a few other complications - what do you do when there is no memory? What do you do if the second chunk of sound arrives late? - but these are often the same problems that your game already has to handle on the graphics side.

Another tip: avoid using Ogg Vorbis for latency-critical streams. Ogg Vorbis files are front-loaded with data tables and other content that can require over 64 KB of data to be loaded and processed before the first frame can begin playing. Even static HSAMPLEs in the Ogg Vorbis format can take a relatively long time to start playing. Use Bink Audio as an alternative compressed streaming format - it's way faster starting up.

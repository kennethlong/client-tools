Fire and forget has the side effect of divorcing your game data from the sound that is playing, and quite often you need to maintain the assocation - usually for 3D reasons.

The User Buffer is the solution for this. AIL_enqueue_event_by_name doesn't expose this for simplicity, but you can get the same effect via AIL_enqueue_event by passing MILESEVENT_ENQUEUE_BY_NAME in the flags.

There are two ways of associating data with a sound:

  1. Pass a pointer to your game data. That gets passed on directly and in your sound enumeration you can grab it and cast it to your game types. 

    	S32 token = AIL_enqueue_event_start();
	AIL_enqueue_event_buffer(&token, MyPtr, 0, 1);
	AIL_enqueue_event_end_named(token, "Bank/EventName");


        AIL_begin_event_queue_processing();

    HMSSENUM Token = MSS_FIRST;
    MILESEVENTSOUNDINFO Info;
    while (AIL_enumerate_sound_instances(0, &Token, 0, 0, 0, &Info))
    {
        GameData* Data = (GameData*)Info.UserBuffer;

        // Do stuff with Data, which is managed by you. If you delete it before the sound
        // is done then this will crash.
    }
    
    AIL_complete_event_queue_processing();

  2. Pass a bunch of data in the enqueue function. This gets copied and managed by Miles, and you can access it however. 

    	S32 token = AIL_enqueue_event_start();
	AIL_enqueue_event_buffer(&token, &DataToPass, sizeof(DataToPass), 0);
	AIL_enqueue_event_end_named(token, "Bank/EventName");


        AIL_begin_event_queue_processing();

    HMSSENUM Token = MSS_FIRST;
    MILESEVENTSOUNDINFO Info;
    while (AIL_enumerate_sound_instances(0, &Token, 0, 0, 0, &Info))
    {
        GameData* Data = (GameData*)Info.UserBuffer;

        // Do stuff with Data, which is managed by Miles. It remains valid until the sound
        // is completed. Info.UserBufferLen is equal to sizeof(DataToPass).
    }
    
    AIL_complete_event_queue_processing();

This is extremely useful, and in many situations required.

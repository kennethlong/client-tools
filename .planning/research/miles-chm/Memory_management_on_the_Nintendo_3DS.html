On the Nintendo 3DS, you must call the AIL_mem_use_malloc and AIL_mem_use_free functions to set callbacks to allocate memory. Since there is no "global" heap, the callbacks are necessary to give Bink a way to get it's memory. Note that the memory returned from these function must be "device memory"!

Miles creates an IO and mixing thread to run in the background. When you create a thread on the 3DS, Nintendo uses an object called the "Auto Stack Manager" to allocate stack memory.

Normally, you don't have to worry about this, but if you have allocated 100% of the memory for your game, then this object will fail it's allocations. To enable the AutoStackManager to use your game's allocators, you derive your a class from nn::os::Thread::AutoStackManager. You then install this derived object into the system using nn::os::Thread::SetAutoStackManager. (Note that the stack manager is set up in nninitStarup, so if you override this member function, then make sure to call SetupHeapForMemoryBlock first.)

Finally, also note that the main thread's default stack size is only 16KB. Miles uses alloca frequently to cut down on heap allocations, so you need to modify your .RSF file to make the stack larger to handle this. Miles only needs 32 KB or so, but we usually run with 128 KB here at RAD.

// Function prototype:

U64 AIL_enqueue_event(U8 const * event_or_name,
void* user_buffer,
S32 user_buffer_len,
S32 enqueue_flags,
U64 apply_to_ID);
Description

Enqueue an event to be processed by the next AIL_begin_event_queue_processing function.

Parameters
event_or_name [in]

Pointer to the event contents to queue, or the name of the event to find and queue. If an event, the contents must be valid until the next call to AIL_begin_event_queue_processing. If a name, the string is copied internally and does not have any lifetime requirements, and MILES_ENQUEUE_BY_NAME must be present in enqueue_flags.
user_buffer [in]

Pointer to a user buffer. Depending on enqueue_flags parameter to AIL_enqueue_event, this pointer can be saved directly, or its contents copied into the sound instance. This data is then accessible later, when enumerating the instances.
user_buffer_len [in]

Size of the buffer pointed to by user_buffer.
enqueue_flags [in]

Optional MILESEVENTENQUEUEFLAGS logically OR'd together that control how to enqueue this event (default is 0).
apply_to_ID [in]

Optional value that is used for events that affect sound instances. Normally, when Miles triggers one of these event steps, it matches the name and labels stored with the event step. However, if you specify an apply_to_ID value, then event step will only run on sounds that matches this QueuedID,InstanceID,or EventID too. This is how you execute events only specific sound instances. QueuedIDs are returned from each call AIL_enqueue_event. InstanceIDs and EventIDs are returned from AIL_enumerate_sound_instances.
Return value
return [out]

On success, returns QueuedID value that is unique to this queued event for the rest of this program run (you can use this ID to uniquely identify sounds triggered from this event).

Discussion

This function enqueues an event to be triggered - this is how you begin execution of an event. First, you queue it, and then later (usually once a game frame), you call AIL_begin_event_queue_processing to execute an event.

This function is very lightweight. It does nothing more than post the event and data to a command buffer that gets executed via AIL_begin_event_queue_processing.

The user_buffer parameter can be used in different ways. If no flags are passed in, then Miles will copy the data from user_buffer (user_buffer_len bytes long) and store the data with the queued sound - you can then free the user_buffer data completely! This lets Miles keep track of all your sound related memory directly and is the normal way to use the system (it is very convenient once you get used to it).

If you instead pass the MILESEVENT_ENQUEUE_BUFFER_PTR flag, then user_buffer pointer will simply be associated with each sound that this event may start. In this case, user_buffer_len is ignored.

In both cases, when you later enumerate the sound instances, you can access your sound data with the MILESEVENTSOUNDINFO.UserBuffer field.

You can call this function from any number threads - it's designed to be called from anywhere in your game.

If you want events you queue to be captured by Miles Studio, then they have to be passed by name. This can be done by either using the convenience function AIL_enqueue_event_by_name, or by using the MILESEVENT_ENQUEUE_BY_NAME flag and passing the name in event_or_name. For introduction to the auditioning system, see Integrating The Miles Event System.

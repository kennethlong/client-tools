Discussion

The Miles High Level Event System is the programmer side of the Miles Sound Studio. That is, it's the API and system necessary to incorporate SoundBanks and events into your game.

It's probably easiest to start with some definitions:

  • Miles Sound Studio - the application that your sound designer uses to create and audition soundbanks and events.
  • SoundBanks - a packed file (like a zip) that contains several different kind of objects: sound assets, labels, sound presets, environment presets, and events. There are two kinds of soundbanks - master soundbanks and platform soundbanks. The sound designer creates the master soundbanks, and then deploys the platform soundbanks from the master version (which applies platform specific compression, copies specific files and more). The programmer only deals with the platform soundbanks generally. You usually only check the master sound bank and the original sound files into source code control.
  • Sound assets - a wave file, MP3, Bink Audio, Ogg, or ACPCM compressed sample. Sound assets are either referenced by name within a SoundBank, or the data is actually embedded directly in the SoundBank - it's up to the sound designer.
  • Labels - simple text strings that are applied when playing a sound. You can then later control sounds based on their labels (some game systems call labels, "tags"). A sound can have as many labels as you want (and usually will have many).
  • Sound presets - a set of sound properties (like volume, dsp settings, etc). A sound preset can be applied to a sound immediately, or it can persist into the future so that any sound that matches the label with inherit the properties.
  • Environment presets - a set of global properties for the sound system (master volumes, reverb types, listener 3D positions, etc).
  • Events - little programs that the sound designers create. Events can start sounds, apply presets, control already playing sounds (like stop and pause), control the caching of the sounds and more. Events are the engine that let your sound designers create the sound for your game.
  • Event Steps - an event can have many steps in it - each step is a like a statement in a normal programming language.
  • Sound Instances - a sound instance is created when a sound asset is played from an event. There is one instance for every active sound in the system and each instance ID is unique for the run of your application.

OK, now that the definitions are out of the way, let's talk about how you use the event system. First off, the programmer and sound designer agree on the naming schemes to use for the SoundBanks and their contents. There are several things to think about here:

  • How to name the SoundBanks themselves. Usually, you will want a "global" soundbank and then at least one more soundbank per level. Some games may even break the files up finer.
  • How to name the sound assets within the soundbank, so that they follow a logical ordering. Since the programmer will usually be triggering events that refer to the sound assets internally, it's not critical for the programmer, but discussing this ahead of time will keep your system logical.
  • What compression format to use for each platform and whether the sounds are embedded or not. Since this will control how the IO is handled, it's important for the sound programmer to give the sound designer some direction here.
  • How to name the events. The primary job of the programmer using the event system is to trigger events based on game state. This means you must communicate the names of the events so that they are triggered correctly. You might have names like "Level 1 Start", "Fire Gun", or very generic names like "Wind". But you do have to decide together how to handle this.
  • The label scheme to use. Labels are text based tags, that can be queried and controlled at runtime by the sound designer or the programmer. You might want to pause all sound instances with the "dialog" label, or you might want to set the 3D position of any sound labeled "3D". In general, err on the side of too many labels, because you never know what queries you'll need to issue later!

Once you have agreed on a naming scheme, the programmer's job is really simple. First, use AIL_add_soundbank to load all of the soundbanks provided by the designer. Next, just call AIL_enqueue_event everywhere in the game when a sound event should be triggered. This function is very lightweight -- unlike many AIL functions, it can even be invoked from multiple threads.

Then, once each frame, call AIL_begin_event_queue_processing and AIL_complete_event_queue_processing to process all of the events that have been queued up for that frame. All of the actual sound-processing logic takes place between these two function calls.

Over the project's lifetime, you can continue to work with your sound designer to rename events, add new tags, change compression formats, or tweak other properties. But the work on the programming side is done once the event-processing calls are in place!

# Consult: D3D11 needs a VERTEX-shader fallback for legacy `//asm` shaders (interiors + sky not rendering)

Star Wars Galaxies client, D3D11 renderer port (`gl11`). Repo: `D:\Code\swg-client-v2`. **Be skeptical / challenge the framing** — this project has repeatedly converged on wrong root causes; we only trust hard evidence and we want the design pitfalls, not happy-path advice.

## Confirmed root cause (hard evidence)
Building INTERIORS (POB cells: walls/floor/ceiling) and the open-world SKY do not render under D3D11; the legacy D3D9 path (`gl05`) renders them fine. RenderDoc + targeted instrumentation pinned it:
- Runtime skip-audit logged **120 unique shaders** skipped at `Direct3d11_StateCache::resolveShaders()` returning false (→ `applyPreDrawState` returns false → no `DrawIndexed` → pixels show only the RT clear color). Counter `skippedNullVS=38927`/session.
- Named culprits: `stco_cantina_wall_*`, `intr_{hotel,tech,library,medical,commerce,combat}_*`, `despot_*`, `spaceport_*`, `frn_tatt_wall_*_cantina`, `glss_cantina_window`, `gradient_sky`, etc. — i.e. the entire interior/structural/sky surface set.
- `resolveShaders()` (`Direct3d11_StateCache.cpp:1130-1159`) returns false because the per-key VS variant's `d3dVS` is **null**.
- `d3dVS` is null because these shaders use **`//asm` (D3D9 vertex assembly) vertex programs**, which `Direct3d11_VertexShaderData` records but CANNOT compile under D3D11 (`Direct3d11_VertexShaderData.cpp:8-10, 252-260`: "//asm vsh ... cannot compile under D3D11 ... NULL VS bound ... Plan 11-06 will skip pass"). Confirmed NOT an HLSL compile failure (zero X-errors in the debug log).
- The reauthored `//hlsl` asset family (`vertex_program/*_vs20_for_ps20.vsh`) compiles + renders fine (that's why NPCs / exterior buildings draw). The interior/structural/sky set uses unconverted `//asm` VS.

**The asymmetry:** the PIXEL path already has a fallback (Phase 17). When an asset PS is unusable, `Direct3d11_StateCache.cpp:1286-1296` calls `selectFallbackPSForVS`, which via `buildHlslForVSOutputs` (`Direct3d11_PixelShaderProgramData.cpp:489`) GENERATES an HLSL PS that matches the bound VS's reflected OUTPUT signature register-for-register (textured-passthrough "Variant T", else magenta "Variant M"). The VERTEX path has **no** equivalent — a null `d3dVS` just skips the draw. We want to add the symmetric VS fallback so interiors/sky render.

## Why the VS fallback is the inverse (harder) problem
The PS fallback reflects an existing compiled VS and builds a PS to match. For the VS fallback there is **no compiled VS to reflect** (it's `//asm`, produced null). So we must DEFINE a fallback VS, and its constraints are:
1. **Input signature must match the mesh's vertex format.** The input layout is built by `Direct3d11_VertexDeclarationMap::getOrCreate(VBFormat, vsVariant)` from the VBFormat + the VS's input signature. Interior meshes vary (position always; plus some of normal / diffuse color / one-or-more texcoord sets / tangent). A single generic fallback VS must declare an input signature the input-layout system can satisfy across these formats — or we need per-format variants.
2. **Transform:** the fallback VS must apply the object→world→camera→projection matrix. The asset `//asm` VS read it from a D3D9 `c`-register; under D3D11 the engine packs constants into the `SwgVertexConstants` cbuffer at `b0` (the working asset VS reads e.g. `objectWorldCameraProjectionMatrix` there). The fallback must read the correct field/offset.
3. **Output signature must be consumable by the PS fallback.** Since we'd generate the VS, we control its outputs; `selectFallbackPSForVS`/`buildHlslForVSOutputs` would then generate a matching PS. Minimal useful output = SV_Position + one texcoord (for diffuse) + maybe vertex color.
4. **Injection point:** where `d3dVS` is found null (the `//asm` path in `Direct3d11_VertexShaderData`, or at `resolveShaders()`), supply the fallback VS + its input layout instead of skipping.

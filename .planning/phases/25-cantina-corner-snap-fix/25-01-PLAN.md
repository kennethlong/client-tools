<objective>
Stop the cantina corner-snap by adding a re-entrancy / reversal guard to
`CellPropertyNamespace::Notification::positionChanged` so that the second,
reversed same-frame cell transition for the same object (the A->B->A portal
ping-pong) is suppressed, while legitimate fast multi-cell door traversals
still complete.

Purpose: HARD-02. The mechanism is already root-caused (see
.planning/todos/pending/2026-05-15-cantina-corner-snap-engine-improvement.md
and the 2026-06-12 instrumented capture): on 8 frames the player crossed a
portal plane 3-5 times within ONE frame, each segment the exact reversal of the
prior one. The re-entry is driven by the collision-resolution replay (the
rewind-to-resetPos + waypoint replay at CollisionResolve.cpp ~337-339 each
re-fire a position notification with a reversed segment), NOT by the
setParentCell->cellChanged path (see <interfaces> for the corrected, source-
verified chain). The existing `dueToParent` early-out does NOT catch this
re-entry because each re-entrant notification arrives as a real (non-parent)
position change.

Output: A modified `positionChanged` containing a frame-scoped, per-object
reversal guard plus a small separate `_DEBUG`-only APPLIED/SUPPRESSED counter-log
(placed OUTSIDE the two frozen CORNERSNAP `_DEBUG` blocks so the Phase-26
acceptance harness stays byte-for-byte intact). No other files change. The
committed CORNERSNAP `_DEBUG` instrumentation is preserved untouched (it is the
Phase-26 removal harness; removal is Phase 26 / HARD-03).
</objective>

  - early-out `if (dueToParent) return true;`  (does NOT catch the re-entry)

CORRECTED re-entry chain (the previous plan's note was WRONG; this is the
source-verified ground truth — Cursor's CollisionResolve hypothesis is confirmed):
  - `object.setParentCell(targetCell)` (Object.cpp:1387) does:
      detachFromObject / attachToObject_w / then `cellChanged(false)` (Object.cpp:1408).
  - `Object::cellChanged` (Object.cpp:1276) calls
      `m_notificationList->cellChanged(*this, ...)` which dispatches each
      notification's `cellChanged(...)` — it does NOT call `positionChanged`.
    => setParentCell -> cellChanged does NOT re-fire positionChanged. (Plan's old
       "setParentCell -> cellChanged -> attachToObject_w re-fires positionChanged"
       claim is FALSE — corrected here.)
  - The ACTUAL re-entry: after the portal walk reparents the object, the
    collision-resolution replay runs in the same frame and re-fires positionChanged
    with the reversed segment:
      CollisionResolve.cpp ~313-339 (RR_Resolved branch):
        :337  objectA->setPosition_p(resetPos);     // rewind -> Object::positionChanged(false, oldPos)
        :339  moveObjectAlong(objectA, positions);   // replays waypoints -> more positionChanged(false, ...)
    Each of these enters Object::positionChanged (Object.cpp:1200) ->
    NotificationList::positionChanged (Object.cpp:241) ->
    CellProperty::Notification::positionChanged with dueToParent == false. One of
    those re-entrant segments is the exact reversal of the segment that just
    crossed, so the portal walk crosses straight back -> the A->B->A snap.
    (Cursor cites frame 14895 = 5 player CORNERSNAP-PORTAL logs interleaved with
     CORNERSNAP-RESOLVE collision-replay events; CORNERSNAP-CELLJUMP was zero,
     confirming the net cell change is the portal-walk apply, not the collision
     rewind itself.)

Notification-list contract for the suppressed-path `return false` (VERIFIED,
Object.cpp:241-247): `NotificationList::positionChanged` iterates notifications
and `return`s (stops) the moment any notification returns false. The CellProperty
notification has getPriority() == -100 (low priority). Returning false here only
halts dispatch of THIS one position-change event to any lower-priority
notifications after CellProperty in the list — exactly what the EXISTING crossed-
portal path already does (`object.setParentCell(targetCell); return false;`).
The suppressed path returns false WITHOUT calling setParentCell, i.e. it drops
strictly LESS work than the existing crossed path (no reparent + same dispatch
halt). It therefore cannot drop a required side effect that the existing crossed
path preserves. The executor MUST re-confirm this against the live notification
list at execution (verify nothing lower-priority than CellProperty depends on
running for this specific re-entrant segment).

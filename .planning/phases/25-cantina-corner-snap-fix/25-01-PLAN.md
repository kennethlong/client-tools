<objective>
Stop the cantina corner-snap by adding a re-entrancy / reversal guard to
`CellPropertyNamespace::Notification::positionChanged` so that the second,
reversed same-frame cell transition for the same object (the A->B->A portal
ping-pong) is suppressed, while legitimate fast multi-cell door traversals
still complete.

Purpose: HARD-02. The mechanism is already root-caused (see
.planning/todos/pending/2026-05-15-cantina-corner-snap-engine-improvement.md
and the 2026-06-12 instrumented capture): `setParentCell` from the portal
walk re-enters `positionChanged` with a segment that is the exact reversal of
the one just processed, recursively crossing the portal 3-5 times in one frame.
The existing `dueToParent` early-out does NOT catch this re-entry because the
re-entrant notification arrives as a real (non-parent) position change.

Output: A modified `positionChanged` containing a frame-scoped, per-object
reversal guard. No other files change. The committed CORNERSNAP `_DEBUG`
instrumentation is preserved untouched (it is the acceptance harness; removal
is Phase 26 / HARD-03).
</objective>

Current control flow inside positionChanged (READ the file for the full body):
  - early-out `if (dueToParent) return true;`  (does NOT catch the re-entry)
  - computes world-space `start` (from oldPosition_p) and `end` (current pos)
  - `if (start == end) return true;`
  - _DEBUG block records cornersnapStart_w / cornersnapEnd_w (CORNERSNAP — DO NOT TOUCH)
  - shifts start/end up 0.1m, then `while(currentCell)` walks portals,
    setting `targetCell` when a portal is crossed
  - if (targetCell): _DEBUG CORNERSNAP-PORTAL log (DO NOT TOUCH), then
    `object.setParentCell(targetCell); return false;`

The re-entry path (src/.../object/Object.cpp ~line 1387):
  void Object::setParentCell(CellProperty *cellProperty)
    -> detachFromObject / attachToObject_w -> cellChanged(false)
  This second cellChanged re-fires a position notification whose old/new
  segment is the REVERSAL of the segment just processed. That re-entrant
  positionChanged arrives with dueToParent == false, so the existing early-out
  misses it, and the portal walk crosses straight back.

  <action>
    In `CellPropertyNamespace::Notification::positionChanged`, add a per-object,
    frame-scoped reversal guard that suppresses the second transition when it is
    the exact reverse of the transition just applied to the same object in the
    same frame.

    Design (chosen specifically to satisfy criterion #2 — do NOT implement a
    blanket "one transition per frame" cap, which would break fast legitimate
    door traversals where a player legitimately crosses two distinct cells in a
    single frame):

    - Maintain a small static record of the most recent transition per object:
      the object's NetworkId value (object.getNetworkId().getValue(), int64),
      the frame number it occurred on (Os::getNumberOfUpdates()), the
      source-cell index, and the target-cell index (CellProperty::getCellIndex()).
      A std::map keyed on NetworkIdType (already have <vector>/<algorithm> and
      can add <map>) is acceptable given the tiny working set (a handful of
      moving objects per frame). Prune stale entries lazily (drop any entry whose
      recorded frame != current frame when that object is next seen) so the map
      cannot grow unbounded across a play session.
    - AFTER the portal walk resolves a `targetCell` (inside the existing
      `if (targetCell)` block, AFTER the CORNERSNAP-PORTAL _DEBUG log so the log
      still records what WOULD have crossed), compute the candidate transition as
      (fromCellIndex = object.getParentCell()->getCellIndex(),
       toCellIndex   = targetCell->getCellIndex()).
      Look up this object's last record. If the last record is from the CURRENT
      frame (recordedFrame == Os::getNumberOfUpdates()) AND it is the exact
      reverse of the candidate (recorded.from == candidate.to &&
      recorded.to == candidate.from), then this is the ping-pong re-entry:
      do NOT call setParentCell, leave the object in the first frame's result,
      and `return false` (still consumes the notification, matching the existing
      crossed-portal return contract). Do NOT update the record on a suppressed
      reversal (so a later genuine re-cross in a different frame is unaffected).
    - Otherwise (no record, record from a prior frame, or a non-reversing
      transition such as A->B->C fast traversal): record
      (frame, from, to) for this object, then proceed with the existing
      `object.setParentCell(targetCell); return false;`.

    Justification vs criterion #2 (state this reasoning in the SUMMARY): a fast
    legitimate traversal produces forward transitions (A->B then B->C) whose
    (from,to) pairs are NOT reverses of each other, so they pass the guard
    untouched. Only the A->B then B->A reversal within one frame is suppressed.
    A blanket per-frame cap is explicitly rejected because it would also drop the
    legitimate B->C, which the capture shows can happen on the same frame at a
    foyer chain.

    Safety: the guard must not create a stuck-in-cell state. Because the record
    is only consulted for an exact same-frame reversal and is never consulted
    across frames (stale entries are pruned / ignored when recordedFrame !=
    current frame), an object can always transition normally on any subsequent
    frame. There is no path where the guard permanently blocks a transition.

    Do NOT modify the two `#ifdef _DEBUG` CORNERSNAP blocks (cornersnapStart_w /
    cornersnapEnd_w capture at ~139 and the CORNERSNAP-PORTAL DEBUG_REPORT_LOG at
    ~191). They are the acceptance harness for criterion #3 and are removed only
    in Phase 26. The guard logic itself is production code (NOT inside _DEBUG) so
    it ships in both Debug and Release.
  </action>

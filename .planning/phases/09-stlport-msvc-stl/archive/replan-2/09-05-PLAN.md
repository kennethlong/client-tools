<objective>
Wave 5: Flip `Unicode::unicode_char_t` from `unsigned short` to `wchar_t`, drop `/Zc:wchar_t-` from CMAKE_CXX_FLAGS_DEBUG and _RELEASE, sweep cast sites between `wchar_t*` and `unsigned short*`. After this wave, the codebase uses MSVC's default `wchar_t` (built-in 2-byte type on Win32), the unicode library carries its strings as `std::basic_string<wchar_t>`, and there is no longer an external ABI boundary forcing the typedef-to-unsigned-short workaround. Per CONTEXT D-03, this happens in its OWN wave (separate from STL removal) so a regression bisects to either "STL ABI" (Waves 2-4) or "wchar_t bit-width / mangling" (this wave) — never a tangled mess of both.

Purpose: Phase 7 CLEAN-04 removed XPCOM, which was the original constraint forcing `/Zc:wchar_t-` (XPCOM's `PRUnichar` was `unsigned short` ABI-locked). Wave 5 cashes in on that removal. STLPort 4.5.3 is gone (Wave 4); MSVC STL `std::wstring` works natively with `wchar_t`; the only remaining work is updating the typedef + flag and sweeping the few cast sites where `wchar_t*` and `unsigned short*` are interchanged.

Output: Two source files (Unicode.h + StlForwardDeclaration.h) with the typedef flipped in the SAME commit per D-03. One CMake file (root) with `/Zc:wchar_t-` stripped. ~1+ source cast sites updated (CuiNotepad.cpp known + researcher-enumerated others). Tatooine zone-in boot gate at end of wave.

Wave 5 contract: NO change to StlCompat.h. NO change to per-target CMake link refs (Wave 4 territory). NO source sweep beyond cast-site fixups required by the typedef flip. NO `_USE_32BIT_TIME_T=1` strip in this wave's main commit chain — that is research-only and does NOT block the wave (per CONTEXT D-03).

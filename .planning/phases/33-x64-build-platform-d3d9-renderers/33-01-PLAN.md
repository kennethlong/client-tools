<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Restoration x64 DLL -> our staged binary set | Third-party prebuilt binaries (binkw64/libxml2/pcre/jpeg62) from a non-first-party source enter the client's load path |
| import .lib (generated from DLL exports) -> x64 link inputs | The export surface a generated import lib trusts comes from the lifted DLL, not first-party source |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-33-01 | Tampering | Lifted x64 DLLs (binkw64/libxml2/pcre/jpeg62) from `D:\SWG Restoration\x64\` | mitigate | Record size + SHA256 provenance for each lifted DLL in 33-x64-DLL-CHECKLIST.md (V14 control); these are the same vetted DLLs the Restoration client ships; prefer from-source where possible (zlib/dpvs/ui/udplibrary build from source — Plan 04) |
| T-33-02 | Tampering | Import libs generated from DLL exports | accept | Import libs carry only the export name table (no executable code); the actual code is the staged DLL, covered by T-33-01. dumpbin-verify machine x64 on each generated lib (defense against an x86/x64 mismatch slipping a wrong-machine binary into the link) |
| T-33-03 | Spoofing/DLL-hijack | `stage-x64/` DLL search order | accept | Bink uses a bare `LoadLibrary` name — unchanged from the existing Win32 behavior; `stage-x64/` is the controlled load dir next to the exe (same pattern as the shipped 32-bit stage/) |

All other ASVS categories (V2 auth, V3 session, V4 access, V5 input, V6 crypto) are N/A — Phase 33 adds no input/network/auth/session/crypto surface (33-RESEARCH.md Security Domain).
</threat_model>

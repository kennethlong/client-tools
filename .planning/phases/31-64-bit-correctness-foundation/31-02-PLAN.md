<objective>
BITS-01, the hard chunk and the #1 review finding: port the x87 FPU-control inline asm (D-04) and
the SSE math inline asm (D-05) to MSVC intrinsics — CORRECTLY. Three sites: `FloatingPointUnit.cpp`
(`__asm fnstcw`/`fldcw` → a CRT FP API), `SseMath.cpp` (CPUID check + 4 matrix-transform routines,
13 `__asm{}` blocks → `_mm_*`), and `Transform.cpp`'s `__declspec(naked)` `sse_xf_matrix_3x4` (the
LIVE primary matrix-concat path).

Two correctness traps the four reviewers converged on (all HIGH) drive this plan's spec:

1. The FPU module's `status` WORD is in **raw x87 hardware-control-word layout** (`ROUND_MASK=0x0C00`,
   `PRECISION_MASK=0x0300`, `EXCEPTION_ALL=0x003F` — verified in source :24-41). `_control87`/`_controlfp`
   take the MSVC **abstract `_MCW_*`/`_RC_*`/`_PC_*`/`_EM_*`** encoding whose bit positions DIFFER. Handing
   the hardware-layout `status` straight to a CRT API sets the WRONG state and breaks `update()`'s
   `currentStatus != status` comparison (FloatingPointUnit.cpp:104). Separately, every mutator funnels
   through `setControlWord(status)` where `status` carries `PRECISION_24` (set in `install()`:56 and
   `setPrecision`:160-161); naively masking out precision drops the P_24 single-precision force on the
   SHIPPED 32-bit build too — directly on the door-snap causal theory (D-04 / VERIFY-01). P_24 is RETAINED
   on 32-bit (CONTEXT D-04 review-refinement #1b, LOCKED).

2. The naked-SSE routines have non-obvious lane semantics a faithful rewrite MUST reproduce, and the
   `movaps`/`_mm_load_ps` 16-byte alignment on an embedded `Transform` is a RUNTIME x64 fault the
   compile-only harness is structurally blind to. Boot smoke cannot detect a transposed lane.

Purpose: x64 forbids inline asm (always error C4235). Intrinsics compile on BOTH bitnesses, so the
fixes land in shared source with no `#ifdef` fork for the SSE path (D-05); the FPU path has exactly ONE
justified `#if defined(_M_X64)` guard for the precision write (the only x64-specific behavior). The
committed 32-bit build inherits the same code and is re-validated by the Wave-3 plan-06 dual-renderer
boot smoke PLUS the new `_DEBUG` numeric self-tests.

SCOPE NOTE (checker / reviews): this plan VALIDATES per-TU only — scratch x64 `cl /c` of each touched
file + the `_DEBUG` oracle. It does NOT run the full canonical 5-target 32-bit build or the dual-renderer
boot smoke; those are the Wave-3 plan-06 gate, run ONCE after all wave-2 source edits merge. Running the
full build mid-wave (while plans 03/04/05 are concurrently editing source + writing to the same
`stage/`/`src/compile/win32/` output dirs) would false-green against only this plan's subset of edits.

Output: Three intrinsic-ported files that compile x64-clean in the scratch harness (0 C4235), preserve
32-bit FP precision behavior, and pass a `_DEBUG` numeric-equivalence oracle for every SSE routine.

The cross-AI review (#3, all four reviewers) sharpened the sort-key sub-problem, which the original plan
under-scoped:
- `getShaderTemplateSortKey()` returns `int` and is stored in `ShaderPrimitiveSorter::Entry.shaderTemplateSortKey`
  (an `int` field + comparators in `ShaderPrimitiveSorter.cpp:62/201/223` — NOT previously in files_modified).
- `getSortKey()` is a `virtual int` in shared headers consumed by all four renderer plugins; `getShaderImplementationSortKey()`
  is an inline `int` in `ShaderEffect.h:93`. Widening the RETURN type cascades into the virtual interface, the
  Entry struct, the comparators, and every plugin impl — a shared-header ABI event.
- Cursor found 3 LIVE D3D9 boot-path truncations missing from the plan: `Direct3d9_StaticVertexBufferData.cpp:131`,
  `Direct3d9_DynamicVertexBufferData.cpp:262`, `Direct3d9_StaticShaderData.cpp:388` — `reinterpret_cast<int>(ptr)`,
  live boot-path code, not link-time residue.

CONTRACT DECISION (pre-decided per review #3, recommended option): use a STABLE HASH-TO-INT for every
pointer-derived sort key — `static_cast<int>(std::hash<uintptr_t>{}(reinterpret_cast<uintptr_t>(ptr)))`.
The return type stays `int`, so the virtual interface, the `Entry` struct, the comparators, and the plugin
impls are ALL unchanged — NO ABI cascade, the change stays inside each owned function. Collisions only cost
sort STABILITY (occasional adjacent-draw reorder), never correctness — acceptable for a render sort key.
This is why `ShaderPrimitiveSorter.cpp` is in files_modified only to RECEIVE the hashed key (its `int` field
is correct as-is); the file is listed so the executor reads the comparator and confirms no widening is needed.

Purpose: x64 pointers are 64-bit; a pointer stored/hashed/diffed through `int` drops the high 32 bits
(a latent memory-safety class for non-sort-key uses; a collision/mis-sort class for sort keys). The scratch
harness (plan 01) enumerates IMPLICIT truncations; the explicit-cast-audit.log (plan 01 Task 3c) enumerates
the EXPLICIT casts /we4311 cannot see (review #4). Fix every site in this plan's files_modified surfaced by
EITHER list.

Output: Truncation fixes in shared source (the 32-bit build inherits them) that compile x64-clean and keep
sort keys / pointer math / Win32-boundary returns functional in the 32-bit dual-renderer build.
</objective>

<objective>
BITS-03: audit + correct the IFF/TRE and network-message data-layout / serialization-width assumptions
that move on Windows LLP64 x64 — and lock them with compile-time guards. Per the verified audit (D-07),
the IFF reader is fixed-width by design (SAFE), `int32`/`uint32` typedef to `long` = 32-bit on LLP64
(SAFE), and the ONE real hazard is the `Archive` `std::map` get/put using raw `size_t numKeys`.

THREAT NARRATIVE CORRECTION (review #5 — Opus + Sonnet source-verified): the raw `size_t numKeys` map
hazard does NOT silently widen the wire format. On x64 `size_t` is `unsigned __int64`, and there is NO
8-byte `Archive::get`/`put` overload — so the call FAILS OVERLOAD RESOLUTION at compile time (a hard
error the scratch harness catches), it does not silently ship a corrupted 8-byte count. This is GOOD news
(it cannot silently corrupt the wire), but the prior plan/threat-model wording ("silently widens / changes
wire format") was factually wrong and is corrected here so Phase 33 does not mis-prioritize. The `uint32_t`
pin is still the correct wire-stable fix (it binds the existing 4-byte `unsigned int` overload on both
bitnesses). The two `#pragma pack` structs (TargaFormat, AssetCustomizationManager) are fixed-width/
layout-safe — guard them with `static_assert(sizeof==N)` (D-08; adding asserts does NOT change layout, so
no shared-header ABI cascade).

Because zero `Archive::get/put(std::map<...>)` instantiations exist under `src/engine` (Cursor verified),
boot smoke would NOT exercise the fix — so this plan adds an explicit scratch-only instantiation TU that
forces the map (de)serialization to compile, proving the `uint32_t` binding resolves on x64.

Output: A wire-stable Archive map serializer (with an exercising instantiation TU) + permanent layout
guards, compiling green in the 32-bit build and x64-clean in the scratch harness.
</objective>

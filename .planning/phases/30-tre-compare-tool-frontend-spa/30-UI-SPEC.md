# Phase 30 — UI Design Contract

> Visual and interaction contract for the TRE Compare Tool frontend SPA. Generated by gsd-ui-researcher, verified by gsd-ui-checker.
>
> **Posture:** single-user, localhost-only, read-only diagnostic tool. Bias toward **information density and speed over polish** (CONTEXT.md §Specific Ideas). Not a consumer product.
>
> **Backend contract is FROZEN** (Phase 29). This phase is UI-only. Field names below are quoted verbatim from `tre_compare/api.py` + `diff.py` — the UI consumes them unchanged. If the UI needs data a route does not expose, that is a planning flag, never a silent backend edit.

## Visual Hierarchy

The screen has one unambiguous focal point and a clear top-down scan order — important for a dense data tool where the user must not hunt for the primary object.

| Order | Element | Why it commands attention |
|-------|---------|---------------------------|
| **Focal point (primary anchor)** | The **merged file-tree center pane** | It is the largest surface, the tool's primary information object, and where every diff status badge lives. The eye should land here first after a compare runs. |
| 2 | Top **install-picker bar + "Compare Installations" button** | The entry action; commands attention *before* a compare via the accent-colored primary button, then recedes. |
| 3 | The collapsible **set-delta summary strip** | A secondary lens that scopes the focal tree; collapsed by default so it does not compete with the tree. |
| 4 | The **detail `Sheet`** (slide-over, right) | On-demand; only present after a file is selected, deliberately overlaying rather than sharing the focal plane. |

**Status encoding carries the visual weight inside the focal pane** — colored status badges (semantic palette below) are the brightest elements in the otherwise-neutral zinc tree, drawing the eye to *what differs*. Folder rows and chevrons stay neutral so the badges pop.

**Icon accessibility:** all `lucide-react` icons are either paired with a text label OR, where icon-only (e.g. expand/collapse chevrons, the hide-identical toggle, status-glyph affordances), wrapped in the shadcn **`tooltip`** component so the meaning is reachable. No icon-only control ships without a `tooltip` label. Status is never conveyed by **color or icon alone** — every status badge carries a text label (`added`, `≈ metadata`, `= content`, etc.) so color-blind users and screen readers get the same signal.

## Spacing Scale

Declared values (multiples of 4). This is a **dense** tool — default to the lower end of the scale; reserve `lg`+ for page-level structure only.

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Badge inner padding, icon-to-text gap, tree-row indent step is a multiple of this |
| sm | 8px | Compact element spacing, virtualized tree-row vertical padding, between badges |
| md | 16px | Default gap between layout panels, detail-panel field rows, picker-to-content gap |
| lg | 24px | Detail `Sheet` content padding, set-delta strip internal padding |
| xl | 32px | Top picker bar height region / major vertical section break |
| 2xl | 48px | Reserved — page-level empty-state vertical centering |
| 3xl | 64px | Reserved — not expected in this dense layout |

**Exceptions:**
- **Virtualized tree row height is a FIXED 28px** (`estimateSize: () => 28` — 30-RESEARCH.md Pattern 1). This is a hard constraint of the TanStack Virtual fixed-height model, NOT a spacing token; it is intentionally off-scale (28 ≠ a multiple of 4 in the token sense) because it is a row-virtualization measurement, not padding. Do not "round" it to 24 or 32 — the value is locked by the virtualization pattern. *(Checker: acknowledged FLAG — deliberate, justified, informational only.)*
- **Tree indent step: 16px per depth level** (multiple of 4) — applied as left padding on the row content, not as a spacing gap.

## Typography

Three roles + a dedicated mono data role. Two weights only (400 regular, 600 semibold).

| Role | Size | Weight | Line Height | Color / Treatment |
|------|------|--------|-------------|-------------------|
| Body / data (default) | 14px | 400 | 1.5 | `--foreground` (zinc near-white), mixed-case |
| Label / UI chrome | 13px | 400 | 1.4 | `--muted-foreground` (dimmed zinc), **UPPERCASE + letter-spacing** for picker/column/field labels |
| Section heading (panel titles, set-delta strip header, detail-panel title) | 16px | 600 | 1.2 | `--foreground` |
| Display (app title / empty-state heading) | 20px | 600 | 1.2 | `--foreground` |

**Body vs Label legibility (REQUIRED secondary differentiator — addresses checker FLAG):** Body (14px) and Label (13px) are only 1px apart, so size alone does NOT separate them. The distinction is carried by **color + casing**: Label is `--muted-foreground`, UPPERCASE with letter-spacing; Body is `--foreground`, mixed-case. Never rely on the 1px size delta alone to tell the two roles apart.

**Mono usage (not a separate size — same 14px/13px):** paths, `len`/`clen` sizes, hash hexdigests, archive basenames, priorities render in the monospace font so diagnostic columns align. This is a *font-family* distinction layered on the Body/Label roles, not a fifth type size.

**Rationale for 13–14px base (below the usual 16px body):** a dense, single-user diagnostic table tool where vertical density lets the user scan more of a 10k–100k-row diff per screen. CONTEXT.md explicitly directs "consistent and dense... information density and speed over polish."

## Color

Dark theme, shadcn `zinc` base. Values are the shadcn dark-mode `zinc` tokens. The 60/30/10 split maps onto the three-pane layout.

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `--background` `oklch(0.141 0.005 285.823)` (~`#09090b`) | App background, the merged file-tree center pane (the largest surface) |
| Secondary (30%) | `--card` / `--secondary` `oklch(0.21 0.006 285.885)` (~`#18181b`) | Top picker bar, collapsible set-delta strip, the right detail `Sheet`, table header rows |
| Accent (10%) | `--primary` `oklch(0.92 0.004 286.32)` (zinc near-white) used sparingly as the *focus/selection* accent | See reserved-for list below |
| Destructive | `--destructive` `oklch(0.704 0.191 22.216)` (red) | Fault/error rows ONLY (corrupt-archive `fault` row, `node_errors`, `content-indeterminate` read errors). There are NO user-destructive actions in this read-only tool. |

**Accent reserved for (explicit — never "all interactive elements"):**
- The **selected tree row** highlight (the file whose detail is open in the `Sheet`).
- The **active install-picker** selection state and the primary "Compare Installations" action button.
- The **cross-filter active** indicator when a set-delta archive row is scoping the tree.

Everything else (default buttons, tree chevrons, folder rows, badges other than status) stays on neutral zinc surfaces. Status badges use the **semantic status palette below**, which is distinct from the single UI accent.

### Status / Verdict Badge Palette (the diagnostic core — Kenny's #1 pitfall lives here)

Badge color is **semantic data encoding**, not decoration. These are reserved exclusively for diff status and content verdict and must be mutually distinguishable. **Every badge carries a text label** (never color-only) so the signal survives color-blindness and screen readers.

| Concept | Source field | Badge style | Color intent |
|---------|--------------|-------------|--------------|
| `added` | file `status` / set `status` | solid badge | green (`oklch(0.7 0.15 150)` family) |
| `removed` | file/set `status` | solid badge | red (`--destructive`) |
| `changed` | file/set `status` | solid badge | amber/orange (`oklch(0.75 0.16 70)` family) |
| `identical-by-metadata` | **file** `status` | **muted/neutral outline** badge, label "≈ metadata", **with a dim "not content-verified" affordance** (tooltip + dim icon) | grey/neutral — **must NOT read as a confident green "identical"** |
| `content-confirmed-identical` | **detail** `verdict` (post-xxhash) | **solid green** "= content" badge | green — the ONLY earned confident "identical" |
| `content-changed` | detail `verdict` | "≠ content" badge | red/amber |
| `content-indeterminate` | detail `verdict` (a side's `hash.error` non-null) | **warning/amber** "? content — read error" badge; show the `error` string | amber + show error text |
| `tombstoned` | file `status` / `qualifier` | distinct outline badge | violet/neutral — a deliberate engine removal, not a diff add/remove |
| `fault` | set `status`, `fault_left/right` | red fault row, never dropped | red — corrupt archive surfaced, never silently skipped |
| set `identical` | **set** `status` | neutral | grey (set-level enum, distinct from file-level — keep separate TS union + badge map) |
| `qualifier[]` flags | file `qualifier` (`tombstoned_left/right`, `rejected_left/right`, `error_left/right`) | small inline flag chips | muted, secondary to the main status badge |

**HARD RULE (the honesty distinction — D-04 / D-06):** `identical-by-metadata` (cheap `(len,clen)` match, NOT hash-verified) and `content-confirmed-identical` (drill-in xxhash matched) **must look different on screen**. Reserve the confident solid-green "identical" visual exclusively for the post-hash verdict. Never present a metadata match as a content guarantee.

**Set-level vs file-level statuses are DIFFERENT enums** — `/compare/set` uses `identical`/`fault`; `/compare/files` uses `identical-by-metadata`/`tombstoned`. Two separate TypeScript unions (`SetStatus`, `FileStatus`), two separate badge maps. Never share one map (30-RESEARCH.md Pitfall 3).

## Copywriting Contract

Read-only diagnostic tool — the "primary action" is running a compare, not creating/saving data. There are **no destructive actions** (no extract/repack/delete in v2.3 scope).

| Element | Copy |
|---------|------|
| Primary CTA | **"Compare Installations"** (verb + noun object; button enabled once both left + right installs are picked — the single primary action). A compact **"Compare"** label is acceptable only inside the picker bar where the "installations" object is already visible above it. |
| Empty state — no installs configured | Heading: **"No installations configured."** Body: **"Add installs to `installs.toml` (see `installs.toml.example`), then reload."** *(GET /installs returned `[]` — this is empty, NOT an error.)* |
| Empty state — installs picked, not yet compared | Heading: **"Pick two installations and Compare."** Body: **"Choose a left and right install above, then Compare to diff their TRE sets."** |
| Empty state — compare ran, zero differences after filters | Heading: **"No differing files."** Body: **"Every file matched by metadata. Turn off 'Hide identical' to see all entries, or clear your search."** |
| Loading — `/compare/files` (the heavy call) | **"Comparing installations…"** (with a spinner; this is the one call that can take a moment on a large pair) |
| Loading — `/file/detail` (auto on select) | **"Hashing file contents…"** (inline in the detail `Sheet`; the on-demand xxhash) |
| Error state — `/file/detail` 404 | **"File not found in either merged tree."** (path echoed) — backend returns `{detail:{error:"not found", path}}` |
| Error state — `/file/detail` 400 | **"Rejected path — this path failed safe-virtual-path validation and cannot be inspected."** — backend returns `{detail:{error:"rejected path", path}}` |
| Error state — corrupt archive (set-delta) | Render the row as a **fault row** with `fault_left`/`fault_right` text: **"Could not read archive: {fault message}"** — never drop the row (backend never 500s; it returns faults). |
| Error state — content read degraded | In the detail panel verdict: **"Content indeterminate — a side's bytes could not be read: {hash.error}"** |
| CRC field label (the trap) | **"Path-CRC — not a content signal."** Always adjacent to any CRC/checksum language so it can never be mistaken for the verdict. **See field-gap flag below.** |
| Hide-identical toggle | **"Hide identical"** |
| Search placeholder | **"Filter by name or path…"** |
| Destructive confirmation | **None** — read-only tool, no destructive actions exist in this phase. |

### CRC display — field-gap flag (planning decision, NOT a backend edit)

D-04 / SC#3 ask for a "path-CRC — not content" display, but **the four frozen routes expose no `crc` field** (verified in 30-RESEARCH.md §Pattern 3 / Open Question 1). `diff.py` deliberately never emits the TOC `crc` (it is a path-CRC, never a change signal). Therefore the UI **cannot render a numeric path-CRC value without a backend change, which the frozen contract forbids.**

**Recommended resolution for the planner (Option 1 — contract-preserving):** render an explanatory labeled-absence note — **"Path-CRC is intentionally not shown: it is a checksum of the file *path*, not its contents, so it cannot indicate a content change. See the content verdict above."** This honors D-04's honesty intent fully and satisfies SC#3's "CRC display" as a deliberate, labeled absence. If Kenny wants the literal numeric path-CRC value shown, that is a separate explicitly-scoped backend task (a new field on `_node_brief`/detail) — raise it in planning, do not let a task silently add it.

## Interaction Contract (phase-specific — beyond the standard token dimensions)

Locked from CONTEXT.md D-01..D-04. Listed so the executor implements the layout without re-deciding.

| Region | Behavior | Source |
|--------|----------|--------|
| Layout | Single-page master-detail, **linked lenses** (not tabs). Pickers top → set-delta collapsible strip → merged file-tree center → detail `Sheet` slides over from the right. | D-01 |
| Set-delta ↔ tree cross-filter | Clicking an archive row in the set-delta strip **scopes the file-tree to that archive's files**. Active scope shows the accent indicator + a clear-scope affordance. | D-01 |
| File-tree | Nested collapsible folders, **collapsed to top level by default**, folder roll-up badge counts (added/removed/changed/identical). Virtualized via TanStack Virtual over flattened-visible rows (fixed 28px). | D-03 |
| Filter / search / hide-identical | All in-browser over the one `/compare/files` payload — no server round-trips. Hide-identical must **suppress now-empty folders** (use roll-up counts). | D-03 / Pitfall 6 |
| Detail panel | Auto-fires `/file/detail` on file select (slide-over `Sheet`, `side="right"`, ~440px). Shows both-side `len`/`clen`, winning archive per side, shadowed copies, content verdict badge, CRC honesty label. Re-select is instant (server-memoized). | D-04 |
| Selection race | Rapid file selection must not show a stale prior file's detail. Key the request on `[left_cfg, right_cfg, path]` (TanStack Query, recommended) or an abort/seq guard. | Pitfall 4 |
| Serve model | One localhost process: FastAPI `StaticFiles` mounts `web/build/` at `/`, registered LAST (after the 4 API routes). Relative-path fetches (no hardcoded host) so dev-proxy + prod-same-origin share one code path. | D-02 |

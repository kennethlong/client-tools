## Cross-AI consult on Task 1 â€” synthesized fix path

`.planning/research/CONSULT-vs-ps-interpolator-mismatch-SYNTHESIS.md` documents the Codex + Cursor cross-AI consult convergence:

- **D3D11 stage linkage is register-position strict.** Semantic name + register slot + mask must match between VS output and PS input. Semantic-only matching was a D3D9 assumption that no longer holds. `id=342 = 0` (semantic names match) + `id=343 = 24K` (register positions don't) is the diagnostic signature of independently-compiled VS and PS programs producing incompatible signatures even from correctly-paired asset HLSL sources.
- **VS recompile lane already exists.** `Direct3d11_VertexShaderData.cpp` already retains VSRC and recompiles `//hlsl` VS via `D3DCompile`. Phase 11's Iter-3 added `buildHlslForVSOutputs()` (at `Direct3d11_PixelShaderProgramData.cpp:467`) that builds a PS whose input struct mirrors VS reflection register-for-register â€” and this PS is used via `selectFallbackPSForVS`. The bind-time logic at `Direct3d11_StateCache.cpp:1107-1122` already prefers asset PS over the VS-matched fallback when `ms_currentPSData->getPixelShader()` is non-null. Plan 17-02's `tryCompilePixelShaderFromHlslNoFatal` made `getPixelShader()` non-null for ALL HLSL asset PSes, so the bind-time gate ALWAYS chooses the (incompatible) asset PS, and the VS-matched fallback never runs.
- **Fix is bind-time pair validation, not a new VS recompile lane.** Validate compatibility before letting the asset PS win.

Approach #5 (per-pass interpolator ABI normalization via PSRC source rewrite) is the fallback layer if pair validation fails for too many shaders (i.e., asset PS source IS the problem, not just D3D11's register strictness). Out of scope for Plan 17-04 â€” add only if the asset-PS bind rate post-Plan-17-04 is unacceptably low.

## Tasks 1 + 2 â€” implementation guidance

### Task 1 â€” VSâ†”PS signature pair validation (Direct3d11_StateCache.cpp + PixelShaderProgramData)

**Add validator helper.** Either as a free function inside `Direct3d11_PixelShaderProgramData.cpp` (where the PS-side reflected signature lives) or a method on `Direct3d11_PixelShaderProgramData`. Signature: `static bool isCompatibleWithVS(Direct3d11_VertexShaderData const *vs, Direct3d11_PixelShaderProgramData const *ps)` â€” returns `true` when every PS input semantic has a matching VS output semantic at the SAME register slot, with compatible mask (PS read mask âŠ† VS write mask) and matching component type. SV_Position is a system-value â€” exclude from user-varying comparison.

The VS reflected output signature: need to expose a getter on `Direct3d11_VertexShaderData` (`getReflectedVSOutputs()` returning `std::vector<Direct3d11_ReflectedVSOutput> const &` or similar). Phase 11's reflect-once mechanism for VS should already cache this; if not, add the same `D3DReflect`-once pattern Plan 17-02 added for PS at construct/load time.

**Gate insertion in `applyPreDrawState`.** At `Direct3d11_StateCache.cpp:1107-1122` (Cursor cited), the current logic:
```cpp
if (ms_currentPSData && ms_currentPSData->getPixelShader())
{
    ctx->PSSetShader(ms_currentPSData->getPixelShader(), nullptr, 0);
}
else if (ID3D11PixelShader *fallback = selectFallbackPSForVS(ms_currentVSData))
{
    ctx->PSSetShader(fallback, nullptr, 0);
}
```
Becomes:
```cpp
ID3D11PixelShader *psToBind = nullptr;
if (ms_currentPSData && ms_currentPSData->getPixelShader()
    && Direct3d11_PixelShaderProgramData::isCompatibleWithVS(ms_currentVSData, ms_currentPSData))
{
    psToBind = ms_currentPSData->getPixelShader();
}
else if (ID3D11PixelShader *fallback = selectFallbackPSForVS(ms_currentVSData))
{
    psToBind = fallback;
}
if (psToBind) ctx->PSSetShader(psToBind, nullptr, 0);
```

**Per-cache-entry logging â€” once per (vs, ps) pair, not per draw.** Maintain a `static std::set<std::pair<void*, void*>>` of logged pairs inside the validator helper. On first encounter of a new (vs, ps) pair, log via `DEBUG_REPORT_LOG_PRINT` + `ID3D11InfoQueue::AddApplicationMessage` (R3-02b dual-route) showing:
- VS template name (or first reflected output's owning DLL/file name)
- PS template name
- Result: `compatible` OR `incompatible: reason` (which semantic mismatched, expected register vs actual register)

**Validator gotchas from Codex consult:**
- Compare `D3D11_SIGNATURE_PARAMETER_DESC::Mask` (VS write mask) against PS `Mask` (PS read mask). Compatible means PS read mask is a SUBSET of VS write mask. A VS writing `xy` (mask 0x3) is compatible with a PS reading `xy` (mask 0x3), NOT with a PS reading `xyzw` (mask 0xF).
- Component type matters (float/uint/int) â€” `D3D_REGISTER_COMPONENT_TYPE`. PS reading float when VS writes int = incompatible.
- Interpolation modifiers (`linear`, `centroid`, `nointerpolation`, `sample`) â€” strictly D3D11 allows mismatched interpolation between stages without ID=343, BUT it can produce wrong output. For now: comparison ignored. Add later if a shader misbehaves.
- SV_Position is a system value â€” present in VS output but NOT a user varying the PS consumes via register comparison. Filter on `SystemValueType == D3D_NAME_UNDEFINED` (user semantics only).
- TEXCOORD packing: two semantics can share a register with complementary masks (e.g., `TEXCOORD0 mask=0x3` + `TEXCOORD1 mask=0xC` both at register 2). The comparator must compare each semantic independently and consider partial-register packing.

### Task 2 â€” writeVarByName for SwgVertexConstants schema (Direct3d11_StaticShaderData.cpp)

**The schema problem.** Plan 17-03 hardcoded names: `materialDiffuse`, `materialSpecular`, `materialEmissive`, `textureFactor`, `textureFactor2`. These don't exist in `SwgVertexConstants`. The reflected cbuffer has 9 variables (per D3DReflect dump at `PS-cbuffer: ... totalSize=400 vars=9`). The actual names are likely (per `vertex_shader_constants.inc` referenced in Plan 17-03's executor note): `objectWorldCameraProjectionMatrix`, `material[N]` array (5 elements), `lightData[N]` array, etc.

**Discover, don't assume.** At the START of Task 2 implementation, add a ONE-SHOT diagnostic dump (mirror R3-03g pattern, fire once for sul_eye and once for sul_m_head) that logs ALL 9 variable names + their offsets + sizes from the FIRST reflected layout seen for these shaders. Trace target: `stage/plan-17-04-cbuffer-vars-discovery.txt` (cwd resolves to `stage/stage/`). Use this evidence to write the correct mapping.

**Likely mapping (subject to discovery confirmation):**
- `material[0]` (XMFLOAT4) â†’ `pm.m_diffuse`
- `material[1]` (XMFLOAT4) â†’ `pm.m_specular`
- `material[2]` (XMFLOAT4) â†’ `pm.m_emissive`
- `material[3]` (XMFLOAT4) â†’ `pm.m_textureFactor` (or unused)
- `material[4]` (XMFLOAT4) â†’ `pm.m_textureFactor2` (or `pm.m_power` as `(power, 0, 0, 0)`)

Confirm the mapping by reading `vertex_shader_constants.inc` directly (find with grep) AND the D3DReflect discovery dump.

**Update writeVarByName lookups** to use the discovered names. Keep the current hardcoded-name lookups too â€” if a future shader DOES declare `materialDiffuse` (e.g., a PS that doesn't `#include` the VS header), the writes still land. Multiple candidate names per source value:
```cpp

    bool d = writeVarByName("materialDiffuse",  diff) || writeVarByName("material[0]", diff);
    bool s = writeVarByName("materialSpecular", spec) || writeVarByName("material[1]", spec);
    bool e = writeVarByName("materialEmissive", emi)  || writeVarByName("material[2]", emi);
    return {d, s, e};
};
```

**Keep R3-03g flush log active.** It's the proof signal Kenny will read post-boot. The `wroteDiffuse/Specular/Emissive` flags should now report `1/1/1` instead of `0/0/0`.

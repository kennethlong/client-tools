Key excerpts from the existing buildHlslForVSOutputs (READ-ONLY reference):
- :477-492 â€” validates each reflected VS output's SemanticName is a valid HLSL identifier
- :487-492 â€” sorts the VS outputs by Register; ties broken by SemanticIndex; this is the canonical declaration order to mirror on the PS side
- :561-583 â€” emits `struct PSIn { float4 _pos : SV_POSITION; ... };` with one field per VS output in register order, naming fields `_v0`, `_v1`, etc., and binding each to `SemanticName + SemanticIndex` (e.g. `COLOR0`, `TEXCOORD0`)

This plan's `rewritePsInputSignatureForVSOutputs` operates on the asset PSRC HLSL text:
1. Parses the existing input-struct declaration in the PSRC (the struct passed to `main(...)` â€” typically named `PSInput` or `VSOut` or similar)
2. For each existing input-struct field, identifies its `SemanticName : SemanticIndex` binding (the colon-bound semantic)
3. Re-emits the struct with the SAME field types + bindings but reordered to match the bound VS's reflected output sort order (sorted by VS Register; ties broken by SemanticIndex)
4. Substitutes the rewritten struct into the source text in place of the original
5. Returns the rewritten HLSL text to the caller; `D3DCompile` then assigns PS input registers in declaration order, and v0..vN line up with VS o0..oN

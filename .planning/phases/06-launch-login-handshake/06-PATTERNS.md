### Cache Variable Declaration (`SWG_INSTALL_DIR`)

**Source:** `src/CMakeLists.txt:62` (`option()` precedent for user-overridable build flags)

**Existing pattern:**
```cmake
option(WHITENGOLD_USE_STLPORT_HEADERS "Place vendored STLPort headers before MSVC headers" ON)
```

**Apply to:** Top-level `src/CMakeLists.txt` (added near the existing `option()` call around line 62, before `add_subdirectory(game)` at line 78)

**Phase 6 adaptation (CACHE PATH instead of option BOOL, per RESEARCH.md Pattern 2):**
```cmake
# Phase 6 D-02: SWG retail/SWGSource install root for runtime TRE archives.
# configure_file() substitutes @SWG_INSTALL_DIR@ in client.cfg.in. User overrides
# at configure time with -DSWG_INSTALL_DIR=<path>; survives reconfigures.
set(SWG_INSTALL_DIR "D:/Code/SWGSource Client v3.0" CACHE PATH
    "Path to SWGSource retail client install (TRE archives root).")

if(NOT EXISTS "${SWG_INSTALL_DIR}")
    message(WARNING
        "SWG_INSTALL_DIR='${SWG_INSTALL_DIR}' does not exist. "
        "The generated client.cfg will reference a missing tree. "
        "Set -DSWG_INSTALL_DIR=<path> to your SWG install root.")
endif()
```

Notes:
- `option()` is BOOL-typed; for a path the project should use `set(... CACHE PATH ...)` which gets a folder-picker in cmake-gui
- `CACHE` keyword ensures the value persists across reconfigures and is overridable via `-D`
- Surrounding `option(WHITENGOLD_USE_STLPORT_HEADERS ...)` at line 62 establishes "user-tunable knob" comment + decision-tag style

### INI File Conventions (for `client.cfg.in` content)

**Sources:** `D:/Code/SWGSource Client v3.0/{client,login,live,user}.cfg` (verified-working production cfgs)

**Apply to:** `client.cfg.in`

| Convention | Example | Notes |
|------------|---------|-------|
| Section header | `[SharedFile]` | Unindented; brackets only |
| Key indent | `\tkey=value` (single tab) | Consistent across all 5 SWGSource cfg files |
| Comment | `# foo` or `## foo` | Both accepted; `##` typically used for headed comment blocks |
| Boolean true | `true` or `1` | User's reference mixes both freely |

| Empty section break | blank line between sections | Optional but conventional |

### Project File Layout for Templates

**Source:** Existing template directory layout

```
src/cmake/
├── stubs/
│   ├── libMozilla_stub.cpp        (Phase 3, source stub)
│   ├── SwgVideoCapture_stub.cpp   (Phase 3, source stub)
│   └── time_t_probe.cpp.in        (Phase 1, configure_file template)
└── win32/                          (12 Find modules)
```

**Apply to:** `src/cmake/config/client.cfg.in` (NEW)

The Phase 6 plan introduces a new `config/` sibling to `stubs/` because:
- `stubs/` is for source-code stubs (`.cpp`/`.cpp.in`) the linker consumes
- `config/` is for runtime configuration templates the build copies alongside the exe
- Keeping them separate avoids confusing "build-time source replacement" with "runtime data templating"

This is a **net-new directory**; no analog exists for it in the project. The pattern is borrowed from common CMake convention (`cmake/config/` is widely used for `pkg-config`-style templates and INI configs).

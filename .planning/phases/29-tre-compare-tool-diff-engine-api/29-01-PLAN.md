<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| diff caller ā†’ filesystem | `scanL`/`scanR` nodes carry cfg-derived abspaths the diff opens (headers, payloads); `vpath` controls which entry is read/hashed |
| archive bytes ā†’ decompressor | `read_tre_payload` zlib-decompresses untrusted archive content during drill-in hash |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-29-01-01 | Tampering | `hash_virtual_file` vpathā†’filesystem (loose `path` node) | mitigate | Canonicalize the vpath through `fix_up_file_name`/`safe_virtual_key`; a `safe_virtual_key`ā†’None vpath is never in `entries`, so drill-in of it is a not-found, not a join. Loose-file open joins only the already-`safe_virtual_key`-vetted key (rejects interior `..`/drive/UNC, Phase-28 hardening). |
| T-29-01-02 | DoS | `read_tre_payload` decompress of hostile/oversized archive bytes | mitigate | Reuse the Phase-28 `_preflight_tre`/`_preflight_toc` bounds + `MAX_ENTRIES` (entries already pass preflight to be in the tree); wrap the drill-in hash read in `_NODE_FAULTS` try/except and degrade to `hash_error` ā€” never abort. |
| T-29-01-03 | Info disclosure | TOC payload `.tre` resolution via `toc_tree_path + entry.tre_file` | mitigate | `entry.tre_file` is a bare filename joined to the cfg's own `TOCTreePath` (same install root the user chose); no caller-supplied path component enters the join. |
| T-29-01-04 | DoS | re-parsing a 193k-entry `.toc` per drill-in | accept | Single on-demand drill-in is acceptable cost here; the cache layer (Plan 02) serves `tre_file` from sqlite on the hot path. Low severity for the diff layer. |
</threat_model>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| archive bytes → sqlite | per-archive parse decompresses untrusted archive content before persisting rows |
| concurrent requests → one sqlite connection | FastAPI (Plan 03) runs sync routes in a threadpool sharing this connection |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-29-02-01 | DoS | per-archive parse of a hostile/oversized archive on a cache MISS | mitigate | Reuse `iter_node_entries` which runs the Phase-28 `_preflight_tre`/`_preflight_toc` bounds + `MAX_ENTRIES`; wrap the MISS parse in `_NODE_FAULTS` try/except so a bad archive records+skips, never aborts the build. |
| T-29-02-02 | Tampering | stale/poisoned cache row served as fresh | mitigate | The PK includes `(mtime, size)`; a rewritten archive yields a different key so a stale row is never matched (implicit invalidation). No content trusted across an mtime/size change. |
| T-29-02-03 | DoS | sqlite cross-thread misuse under the FastAPI threadpool | mitigate | `check_same_thread=False` + WAL + a process-wide write `threading.Lock` (Pitfall 3) — concurrent readers + single writer, correct for a single-user localhost tool. |
| T-29-02-04 | Info disclosure | the `.sqlite` cache committed to git | mitigate | `.cache/` added to `.gitignore`; the DB is local dev state, never tracked. |
</threat_model>

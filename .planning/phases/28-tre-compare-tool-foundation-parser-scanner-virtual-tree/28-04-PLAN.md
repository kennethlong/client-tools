  NOTE (review finding #2b — VERIFIED vs read_search_toc_header/read_search_toc_entries
  tre_reader.py:335-413): SearchTOC is TWO-PHASE:
    [0..36)             header (above)
    [36..toc_offset)    tree-name block: numberOfTreeFiles NUL-terminated names; toc_offset =
                        36 + len(all tree names + their NULs)
    [toc_offset..+sizeOfTOC)            TOC block: numberOfFiles × 24-byte rows
    [toc_offset+sizeOfTOC..+sizeOfNameBlock)  name block: per-entry path bytes
  The entry name is NOT located by an absolute offset. read_search_toc_entries WALKS the name
  block: fn_offset starts at 0, and for each row (in order) the path = names[fn_offset : NUL],
  then `fn_offset += fn_field + 1`. So set each row's `fn_field = len(name)` and lay names out
  in row order, each followed by a single NUL. uncompSizeOfNameBlock must be <= len(name block)
  (the reader checks `len(names) < uncomp_size_of_name_block` → set uncompSizeOfNameBlock =
  sizeOfNameBlock for the uncompressed case). (UNLIKE COT2000, read_search_toc_entries does NOT
  assert fn_offset == sizeOfNameBlock at the end — only the short-block check — but build it
  exactly so it round-trips cleanly.)

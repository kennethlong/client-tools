<objective>
Wire the multi-stream `VertexBufferVector` factory + bind slots in the D3D11 plugin so the engine's skeletal-animation rendering path can proceed past `VertexBufferVectorGraphicsData::createVertexBufferVectorData`. Currently both slots are `STUB(createVertexBufferVectorData)` / `STUB(setVertexBufferVector)` at `Direct3d11.cpp:688` and `:690` — deferred at Plan 11-04 wiring time because no earlier smoke reached the SoftwareBlendSkeletalShaderPrimitive code path.

Plan 11-09.6 close smoke surfaced this as the next blocker: with vs_4_0/ps_4_0 in place, the engine successfully compiled all shaders for char-customization avatar composition (boots, underwear, hair_base, Qui-Gon hair mesh) and proceeded to skeletal-mesh rendering — which constructs `VertexBufferVector` objects to bundle position-normal + blend-indices/weights into a multi-stream IASetVertexBuffers call.

Implementation approach: faithful port of `Direct3d9_VertexBufferVectorData` shape, with input-layout cache substitution for the D3D9 cached `IDirect3DVertexDeclaration9`. D3D9 caches a VS-agnostic vertex declaration at VBVector construction time; D3D11 input layouts REQUIRE VS bytecode at creation time so we can't cache at construction — defer to apply-time via the existing `Direct3d11_VertexDeclarationMap` (Plan 11-06 Pitfall 6 cache, already keyed by VBFormat + VS-bytecode-hash).

Scope ceiling: 1 iteration unless the multi-stream input-layout key surfaces non-trivial cache complications. If Iter-1 successfully constructs + binds VBVectors but produces incorrect rendering (geometry corruption, wrong stream offsets, etc.), that's Iter-2 territory — diagnose visible-symptom-driven per the established Plan 11-07/09 pattern.

This plan does NOT touch `D3D11_REWRITE_VERSION` — no shader-cache invalidation needed.
</objective>

### Task 1 — Author `Direct3d11_VertexBufferVectorData` class

**1a. New header `Direct3d11_VertexBufferVectorData.h`.** Mirrors Direct3d9_VertexBufferVectorData.h shape:
- Inherits `VertexBufferVectorGraphicsData` (engine-side base).
- Constructor: accepts `VertexBufferVector const &`, captures the per-stream VBFormat pointers needed for the composite input-layout cache key.
- Destructor: cleanup (likely no-op if no resources cached at construction time — D3D9's caches a vertex declaration; D3D11 defers to apply-time).
- Private state: per-stream `VertexBufferFormat const *` array (up to MAX_VERTEX_BUFFERS=2 per the D3D9 ceiling).
- Public accessor for the per-stream format array (consumed by `applyPreDrawState`'s input-layout cache lookup).

**1b. New impl `Direct3d11_VertexBufferVectorData.cpp`.**
- Constructor: iterate `vertexBufferVector.m_vertexBufferList`, capture per-stream format pointers into the private array. DEBUG_FATAL if more than MAX_VERTEX_BUFFERS streams.
- Destructor: no-op (no D3D11-specific resource cached here).

**1c. Add to `Direct3d11.vcxproj` + `.vcxproj.filters`** following the Plan 11-04 / 11-05 / 11-06 source-add precedent.

<risks>
1. **Input-layout multi-stream cache complications** — composing a cache key across multiple VBFormats is the main novel work. If the existing `Direct3d11_VertexDeclarationMap` API isn't a natural fit for multi-stream lookups, may need to extend it (new method `getOrCreateMultiStream`). Mitigation: if API extension surfaces during implementation, that's an Iter-2 trigger or scope split.

2. **D3D9-style `SetStreamSource` per slot vs D3D11 `IASetVertexBuffers` batch** — D3D9 sets one stream at a time; D3D11 batches all streams in one call. Mitigation: trivial -- collect the per-stream pointers/offsets/strides in arrays, single batch call at the end. Standard D3D11 idiom.

3. **`ms_lastVertexBufferCount` parity** — D3D9 tracks last-bound stream count to clear unused streams (Direct3d9.cpp:3879). D3D11 has no equivalent "stream slot clearing" concept (IASetVertexBuffers with the count argument handles which slots are bound). Mitigation: no-op vs D3D9; document in code comment.

4. **VBVector with mixed static + dynamic buffers** — D3D9 supports this (Direct3d9.cpp:3829-3863). D3D11 ID3D11Buffer is bind-flag-typed but the SetVertexBuffers call accepts mixed bind types. Should JUST WORK. Mitigation: verify with the actual engine call pattern (likely static-VB-stream-0 + dynamic-VB-stream-1 for skeletal blend weights/indices computed per-frame).

5. **Skeletal-shader user-constants drift** — the SoftwareBlendSkeletalShaderPrimitive pushes per-bone constants. Plan 11-09 Fix C wired user-constant routing for VS; this should JUST WORK if the skeletal-mesh path uses setVertexShaderUserConstants. If it uses a direct cbuffer push that we haven't wired, that's Iter-2 territory.

6. **D-05 invariant** — single-line friend addition to engine header is the ONLY engine-side change (Rule-3 deviation; precedent: Plan 11-04 Texture.h). Risk: zero.

7. **BSOD risk** — pure CPU-side wiring (factory + bind). No new GPU resource allocations. No new state changes. Risk: zero.
</risks>

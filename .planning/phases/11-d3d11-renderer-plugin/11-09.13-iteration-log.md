## Iter-1: Replace magenta hardcode with `t.Sample(s, uv)` HLSL body

**Date:** 2026-05-22 (post-Plan-11-09.12 close `6100e469a`)

### Symptom (carried from Plan 11-09.12 close smoke)

Kenny visual confirmation: "All I saw was magenta triangles, no real terrain was rendered." Plan 11-09.12 audit + fix confirmed the all-magenta state is now ISOLATED from any other artifacts — pure Plan 11-05 PEXE rejection (engine ships pre-compiled D3D9 PEXE bytecode in `.psh` IFF chunks; D3D11 `CreatePixelShader` rejects D3D9 bytecode → `m_d3dPS = NULL` for 100% of asset PSes → Plan 11-09 Iter-2's magenta fallback covers every draw).

Plan 11-05 Known Stubs (11-05-SUMMARY.md) explicitly deferred this to "future Phase 12 asset re-author (re-author `.psh` assets as HLSL source)." Plan 11-09.13 takes a tactical middle path: rather than re-authoring assets, swap the fallback PS body from hardcoded-magenta to textured-passthrough.

### Hypothesis

The fallback PS at `Direct3d11_PixelShaderProgramData.cpp:295-300` is a string literal compiled at install time via `compilePixelShaderFromHlsl`. Replacing the body with `Texture2D.Sample` + appropriate TEXCOORD0 PS input should:
1. Sample whatever's bound at SRV slot 0 / sampler slot 0 per the engine's normal flow.
2. For draws with diffuse texture bound at slot 0 (the typical case for SWG geometry): result = correct texture. Specular / per-pass effects missing but world visually recognizable.
3. For draws without slot 0 binding OR with non-diffuse content at slot 0: sample returns either black (no texture bound) or whatever else is there. Iter-1 accepted trade.
4. For VSes that don't output TEXCOORD0: PS input undefined; sampling at undefined UV produces undefined color (commonly 0,0 → top-left texel). Debug-layer may warn (PS input larger than VS output subset). Iter-1 monitors; mitigation deferred to Iter-2 if warning count is high.

Plan 11-06's state cache already wires SRV slot 0 + sampler slot 0 via the engine's setStaticShader / per-pass texture stages (`Direct3d11_StateCache::applyPreDrawState` handles this).

### Investigation

Iter-1 applied per plan: modified `kFallbackHlsl` at Direct3d11_PixelShaderProgramData.cpp:295-300 to declare `Texture2D t : register(t0); SamplerState s : register(s0);` and return `t.Sample(s, uv)` from `float2 uv : TEXCOORD0` PS input.

Built 5-target MSBuild EXIT=0 / 0 warn / 0 err. gl11_d.dll deployed at 08:03.

**Kenny Iter-1 smoke (~08:05):** "All black, audio eventually fires indicating I'm in the world."

**Post-Iter-1 debug-log audit:** session-begin marker at line 778,837 (post-Iter-1 session). Session ERROR count: **483,451** (up from 2 baseline). Zero warnings.

**ERROR category distribution:** 100% `id=342` "ID3D11DeviceContext::Draw: Vertex Shader - Pixel Shader linkage error: Signatures between stages are incompatible. The input stage requires Semantic/Index (TEXCOORD,0) as input, but it is not provided by the output stage."

### Root cause (Iter-1 finding)

PS-input / VS-output signature subset rule is HARD (not a debug-layer warning — D3D11 REJECTS the Draw call entirely, returning early before issuing the GPU command). Every engine VS in the smoke session output a signature that lacks `TEXCOORD0` at the exact semantic-index match. 483K rejected draws = effectively zero rendering = "all black, just clear color" matches Kenny's observation.

Plan's risk #1 ("PS-input / VS-output mismatch warnings") UNDERESTIMATED — these are not warnings, they're hard rejections. Iter-1 hypothesis required a phantom-buffer-equivalent mitigation BEFORE landing the textured body. Lesson: signature subset rule is the load-bearing D3D11 contract for fallback shaders; can't be skipped.

## Iter-3: Reflection-driven fallback-variant selection (CODEX-endorsed Bucket 1)

**Date:** 2026-05-22 (post-CODEX consult, scaffolded pre-implementation)

### CODEX verdict (summary; full text in `11-09.13-CODEX-CONSULT-reflection-driven-fallback-variant-selection.md`)

Bucket 1 (Compile-2-pick-1) is the right architecture but with two corrections:

1. **Selection by full reflected signature**, not just a bool. Component type + mask + semantic name/index must all line up for D3D11 stage-linkage compatibility (Microsoft Learn `D3D11_SIGNATURE_PARAMETER_DESC`).
2. **One stale premise caught:** `Direct3d11_StaticShaderData::apply()` does NOT yet bind per-pass diffuse textures (only VS/PS pointers). Even a perfectly-selected Variant T samples whatever SRV0 holds from earlier engine state. Acceptance bar must include an SRV0 diagnostic to distinguish "linkage works" from "linkage works AND texture pipeline reaches the PS."

CODEX recommended Iter-3 shape (6 steps): reflect VS outputs once per VS → cache on `Direct3d11_VertexShaderData` (mirroring `m_reflectedInputs` from Plan 11-09.8) → compile magenta + conservative textured variant at install → bind textured only when reflection proves linkage compatibility → magenta as universal fallback → one-shot SRV0/sampler0 validity diagnostic when textured variant is selected.

## Iter-4: Per-VS dynamic PS compile (Bucket 2 override)

**Date:** 2026-05-22 (post-Iter-3 regression diagnosis)

### Architecture summary

For each VS, at first encounter (lazy first-draw lookup keyed by VS bytecode hash):
1. Reflected output signature `m_reflectedOutputs` (already populated in Iter-3) feeds an HLSL string generator that mirrors the VS's output struct register-for-register.
2. PS input declarations walk outputs in `Register`-sorted order; HLSL compiler assigns PS input registers in declaration order, so the resulting PS bytecode has v0 == o0, v1 == o1, etc.
3. PS body: if VS has TEXCOORD0 with FLOAT32 + xy declared → samples `t.Sample(s, uv)` from register(t0)/register(s0); else returns magenta.
4. Compile via the existing `compilePixelShaderFromHlsl` helper; cache the result in `std::unordered_map<uint64_t, ComPtr<ID3D11PixelShader>>` keyed by VS bytecode hash. Compile failure → tombstone (null entry) to avoid retry storm.
5. Selection at draw-time: `getOrCompilePSForVS(vsData)` → fall back to Variant M magenta on null.

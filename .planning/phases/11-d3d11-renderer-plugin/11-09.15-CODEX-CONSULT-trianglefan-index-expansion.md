## What we're trying to decide

Plan 11-09.14's SRV0 per-pass texture binding landed the visible-textured-Tatooine milestone — but Kenny's post-close smoke surfaced a half-screen render artifact that was previously invisible under the all-magenta state. Visual signature: textures only on the upper-right TRIANGULAR section of the screen for sky/sun/moons/planets/sprites/beams; lower-left half exposes whatever was there before drawing (clear color or stale framebuffer).

The root cause is in `Direct3d11_StateCache::drawTriangleFan` and its three sibling variants (`drawIndexedTriangleFan`, `drawPartialTriangleFan`, `drawPartialIndexedTriangleFan`). Plan 11-06's STUB-shape implementation:

```cpp
// StateCache.cpp:1342-1353
void Direct3d11_StateCache::drawTriangleFan()
{
    // D3D11 has no native TRIANGLEFAN. Emit a single draw with TRIANGLELIST
    // topology; the engine's triangle-fan asset would normally be
    // triangulated during VB build (D3D9 used D3DPT_TRIANGLEFAN directly).
    // Plan 11-07 smoke will reveal whether any subsystem actually feeds
    // us un-triangulated fans; if so, server-side re-emit lands as a
    // Rule-1 deviation in Plan 11-07.
    if (!applyPreDrawState(D3D11_PRIMITIVE_TOPOLOGY_TRIANGLELIST))
        return;
    Direct3d11_Device::getContext()->Draw(static_cast<UINT>(ms_currentVBVertexCount), 0);
}
```

This submits fan-shaped vertex data with TRIANGLELIST topology — catastrophically misinterprets the geometry. For the canonical 4-vert fullscreen quad fan (SkyBox6Sided, sun, moon, planet, sprite, beam, marker, etc.):

- **As FAN:** triangles `(0,1,2)` + `(0,2,3)` → full quad (2 triangles)
- **As LIST (current behavior):** triangle `(0,1,2)` only → upper-right half (TL/TR/BR); vertex 3 (BL) **discarded**

This matches Kenny's "upper-right hand triangle section" observation exactly.

The Plan 11-06 comment "the engine's triangle-fan asset would normally be triangulated during VB build" was a hypothesis the smoke was supposed to verify — Plan 11-09.14's close smoke proves the hypothesis WRONG. The engine submits genuine fan-shape vertex data; the plugin is responsible for the fan-to-list conversion.

void Direct3d9Namespace::drawQuadList()

    int const numberOfQuads = ms_sliceNumberOfVertices / 4;
    int const numberOfTriangles = numberOfQuads * 2;

## Four architectural buckets

### Bucket A: Reusable static fan-to-list IB (minimal, D3D9-sibling-mirror)

Mirror D3D9's `drawQuadList` pattern: pre-allocate a single ID3D11Buffer at install time with the fan-to-list index expansion pattern `(0, i, i+1) for i = 1..N-1`. Sized to handle typical max fan-vertex-count (likely 1024 verts is plenty for SWG — most fans are 4..32 verts). Lazy resize on demand if a fan exceeds capacity. `drawTriangleFan` binds this IB after `applyPreDrawState` and calls `DrawIndexed(triCount * 3, 0, 0)`. `drawPartialTriangleFan(startVertex, primitiveCount)` uses the same IB with `BaseVertexLocation = startVertex`.

**Pros:** smallest surface; one ID3D11Buffer for all fan draws; zero per-draw allocation cost; D3D9-sibling-mirror; low risk.

**Cons:** doesn't handle indexed-fan variants — `drawIndexedTriangleFan` + `drawPartialIndexedTriangleFan` need separate treatment (see Q3).

### Bucket B: Per-draw dynamic IB

Allocate a small `ID3D11Buffer` (USAGE_DYNAMIC) per `drawTriangleFan` call; map, write the expansion indices for THIS exact fan size, unmap, bind, draw, release/recycle.

**Pros:** trivially handles any fan size; no install-time capacity sizing required.

**Cons:** per-draw allocation cost (Map/Unmap with WRITE_DISCARD); higher draw-call overhead than Bucket A; engine can call drawTriangleFan many times per frame (sky, sun, multiple sprites, multiple beams, etc.) — Map per draw adds up. D3D9 sibling didn't do this; precedent argues against.

### Bucket C: Dynamic IB pool (Bucket B optimized)

Variant of Bucket B with a small pool of pre-allocated dynamic IBs to amortize Map cost. Round-robin or capacity-tier selection.

**Pros:** middle ground between A's static + B's per-call allocation.

**Cons:** complexity for no obvious gain over A; A's "rebuild on capacity miss" already handles the rare large-fan case.

### Bucket D: Geometry shader passthrough

Use a D3D11 geometry shader to expand fan input to list output on-GPU. The GS would consume 3 verts at a time in adjacency-list mode (or accept the engine's vertex buffer as TRIANGLELIST_ADJ topology and expand).

**Pros:** zero CPU cost; no IB management at all.

**Cons:** complete architectural departure from the rest of the plugin (no other path uses GS); D3D11 feature-level requirement may not be met (rasterMajor=11 currently requires no GS); HLSL authoring + compile + cache + binding logic for a one-off use case; way more code than Buckets A/B/C; would invalidate every existing input-layout cache entry (VS sig changes). Far too heavy.

## Our current hypothesis

**Bucket A (reusable static fan-to-list IB)** is almost certainly the right move:

- **Mirrors the D3D9 sibling pattern.** D3D9 already proved this pattern works in `drawQuadList`; the fan-to-list pattern is structurally identical (just a different index pattern). Zero novel D3D11 territory.
- **Smallest surface.** One buffer; one helper; two function rewrites (drawTriangleFan + drawPartialTriangleFan).
- **No per-draw allocation cost.** Static IMMUTABLE buffer; rebuild only on rare capacity misses.
- **State-cache friendly.** Lives entirely in the StateCache namespace alongside the other reusable resources (default sampler, phantom zero buffer, etc.); no engine-side coupling.

Risk profile: low. Additive change; doesn't touch any draw path other than the four fan variants.

## Questions for CODEX

### Q1 — Architecture choice

Bucket A vs B vs C vs D — does our hypothesis hold? Specifically:

- Is there value in Bucket B/C's dynamic IB shape that we're missing — e.g. a frequency-of-large-fans pattern that would make Bucket A's "rebuild on capacity miss" actually expensive in practice? The Plan 11-06 STUB never collected real engine fan-size data; we'd be sizing Bucket A's initial capacity blind. Acceptable to start at 1024 verts and rely on the rebuild-on-overflow path?
- Is Bucket D worth considering at all, or is the GS-passthrough cost / complexity / feature-level concern correctly disqualifying it? (We've avoided GS everywhere else in this plugin per Plan 11-05 simplicity.)

### Q2 — Index format + capacity sizing

- For initial capacity: 1024 verts seems generous (1022 triangles × 3 = 3066 indices × 2 bytes = 6132 bytes). SkyBox fans are 4 verts; even a complex sun-flare or beam-trail fan is probably ≤ 32 verts. Initial=1024 OK?
- For the index format: at R16_UINT we max out at 65535 verts per fan, which any realistic SWG fan will never exceed. Stick with R16_UINT? Or hedge with R32_UINT just in case (doubles IB memory)?
- Growth policy: doubling (1024 → 2048 → 4096) or just-enough (cap = max(cap, requested))? Doubling avoids thrash on intermediate-size fans; cap-and-stop saves memory if a one-off large fan never recurs.

### Q3 — Indexed-fan variants

`drawIndexedTriangleFan` and `drawPartialIndexedTriangleFan` are also broken (same TRIANGLELIST topology applied to fan-shaped indexed data). Fix in this plan or defer?

The structural difference: indexed fans means the ENGINE already supplies an IB with the fan vertex order. To convert to TRIANGLELIST, we'd need to:

- **Option Q3-fix:** CPU-readback the engine's IB at draw time, expand indices into a temp IB, draw indexed-list. Heavy: one Map+readback per indexed-fan draw.
- **Option Q3-defer:** Leave both as currently-broken with a one-shot DEBUG_REPORT_LOG_PRINT diagnostic identifying each caller; if smoke surfaces real callers, Plan 11-09.16 retrofits.

Our hypothesis: DEFER. The Plan 11-06 STUB comment "Plan 11-07 smoke will reveal whether any subsystem actually feeds us un-triangulated fans" applies more to the INDEXED variants than the non-indexed (which are clearly used — SkyBox proves it). For indexed fans, engine code would typically pre-triangulate at IB-build time (especially for fan-shaped meshes where the IB is static anyway). Defer + diagnostic is honest scope management.

What's CODEX's call?

### Q4 — IB-binding state-cache interaction

After `drawTriangleFan` calls `IASetIndexBuffer(ms_triangleFanIB)`, the device IB state is sticky. The next non-fan draw goes through `applyPreDrawState`, which:

```cpp
// 4. IB (only for indexed draws -- caller passes topology that
// implies indexed; we always bind if valid to avoid driver state).
if (ms_currentIBValid && ms_currentIB)
    ctx->IASetIndexBuffer(ms_currentIB, ms_currentIBFormat, ms_currentIBOffset);
```

If `ms_currentIBValid` is true (engine called `setIndexBuffer` before drawTriangleFan), the next draw re-binds the engine's IB correctly. But if the engine's draw protocol calls drawTriangleFan immediately after a non-indexed draw (no prior setIndexBuffer call between), `ms_currentIBValid` might be false, and the next indexed draw wouldn't rebind — using our stale fan IB.

Two mitigation options:
- **Mitigation Q4-a:** Save + restore device IB state in drawTriangleFan (read prior state via PSGetIndexBuffer? Or just save the engine's `ms_currentIB`/`ms_currentIBValid` shadow, bind fan, draw, restore).
- **Mitigation Q4-b:** After our fan draw, explicitly invalidate the device IB state OR set `ms_currentIBValid = true` (forcing applyPreDrawState to rebind even if engine didn't change anything).

OR: is the concern unfounded — does the engine ALWAYS call setIndexBuffer before any indexed draw, making this a non-issue? (D3D9 reference code suggests so — every indexed draw site in Direct3d9.cpp is preceded by setIndexBuffer or has a stable IB binding.)

### Q5 — Topology cache thrash

Our setting `D3D11_PRIMITIVE_TOPOLOGY_TRIANGLELIST` inside drawTriangleFan means consecutive drawTriangleFan calls both set TRIANGLELIST (no change between them) — but if the engine draws TRIANGLESTRIP between, then TRIANGLELIST for the fan, then TRIANGLESTRIP again, we'd churn the topology state.

`IASetPrimitiveTopology` is documented as cheap (~50ns). Is this concern unfounded? Or is there a Plan 11-06 lazy-topology-set pattern we should mirror (set only if changed)?

### Q6 — Performance baseline

Plan 11-09.14 left `ms_drawsWithSRV0Bound` and three sister counters in place. Should Plan 11-09.15 add counters for:

<objective>
Eliminate the smoke-protocol friction Kenny called out at the close of Plan 11-09: `tools/d3d11-smoke/show-window.ps1` is currently required to externally force `ShowWindow(SW_SHOW) + SetWindowPos + SetForegroundWindow` after launching `stage/SwgClient_d.exe`, because the engine's `Os::install` (sharedFoundation/src/win32/Os.cpp:142) creates the application window with `WS_POPUP` style but no `WS_VISIBLE` and no `ShowWindow` call. The render plugin is the established place to reveal the engine window once size/position are configured -- the D3D9 plugin does this at `Direct3d9.cpp:1535` via `updateWindowSettings()` which terminates in `SetWindowPos(..., SWP_NOCOPYBITS | SWP_SHOWWINDOW)` (line 1913 for windowed; line 1927 for fullscreen). The D3D11 plugin shipped through Plan 11-09 close lacks any equivalent; `setWindowedMode` is even explicitly stubbed at `Direct3d11.cpp:583` (`STUB(setWindowedMode);`).

Purpose: structural parity with the established D3D9 plugin pattern for window-show responsibility, and removal of the smoke-protocol workaround that has been carried forward since Plan 11-07 Iter-17.

This is a TACTICAL sub-plan inserted between Plan 11-09 (CLOSED 2026-05-21 at visible-magenta milestone, commit `f3b27054b`) and Plan 11-10 (DPVS re-measurement). Rationale for the slot: (a) the fix is small enough that doing it after 11-10/11-11 means carrying the workaround through two more plans of smoke validation; (b) the fix is architecturally clean (mirror existing D3D9 pattern), not a hack; (c) zero BSOD risk -- pure Win32 USER calls with no GPU path; (d) reduces noise in every subsequent smoke iteration for the rest of Phase 11.

Scope ceiling: ONE iteration unless the post-fix smoke surfaces a regression. The fix is well-understood (mirror D3D9), so the iteration count target is exactly 1.

<diagnosis_summary>
Verified during session 2026-05-21 (post-Plan-11-09 close):

1. **Engine creates window hidden by design.** `Os.cpp:142` calls `CreateWindow(... WS_POPUP, 0, 0, 640, 480, NULL, NULL, instance, NULL)`. No `WS_VISIBLE` style bit. No `ShowWindow` anywhere in `Os::install` or `Os::installCommon`. The engine's `Os` subsystem does not own window-show responsibility -- it owns window-create.

2. **D3D9 plugin shows the window via `updateWindowSettings()`.**
   - Called from `Direct3d9.cpp:1535` during `Direct3d9::install`, right after the device-create loop succeeds.
   - Function body at `Direct3d9.cpp:1870-1930`: sets window style (windowed = `WS_CAPTION | WS_SYSMENU | WS_MINIMIZEBOX | WS_BORDER | WS_CLIPCHILDREN | WS_CLIPSIBLINGS`; fullscreen = `WS_POPUP | WS_CLIPCHILDREN | WS_CLIPSIBLINGS`); adjusts client-rect for chrome via `AdjustWindowRect`; gets adapter monitor info via `GetAdapterMonitor` + `GetMonitorInfo`; centers the window on the monitor if `ms_windowX == INT_MAX`; calls `SetWindowPos` with `SWP_NOCOPYBITS | SWP_SHOWWINDOW` -- the `SWP_SHOWWINDOW` flag is what reveals the window.
   - The `engineOwnsWindow` guard (Direct3d9.cpp:1872) is true on the SwgClient launch path (Os.cpp:155 `ms_engineOwnsWindow = true`).

3. **D3D11 plugin has no equivalent.** `grep -rn "updateWindowSettings\|SetWindowPos\|SWP_SHOWWINDOW\|ShowWindow" src/engine/client/application/Direct3d11/` returns **0 matches** as of HEAD `077b133d6`. `Direct3d11::install` (Direct3d11.cpp:460) calls `Direct3d11_Device::create` then directly proceeds to resource-class installs without ever showing the window. `setWindowedMode` is explicitly stubbed at line 583.

4. **State available to mirror is already there.** `Direct3d11_Device.cpp:51-54` already declares `ms_window` (HWND from Gl_install), `ms_width`, `ms_height`, `ms_windowed`. Missing vs D3D9: `ms_windowX/Y` (window position) -- can default to INT_MAX-style center-on-monitor; `ms_borderlessWindow` -- can default to false for Iter-1; `ms_adapter` -- already in ConfigDirect3d11.cpp:19. Monitor info via DXGI: `swapChain->GetContainingOutput` returns `IDXGIOutput*` which has `GetDesc().DesktopCoordinates` -- equivalent to D3D9's `IDirect3D9::GetAdapterMonitor` + `MonitorFromX` + `GetMonitorInfo`.
</diagnosis_summary>

<risks>
1. **DXGI flip-model + SetFullscreenState interaction.** Flip-discard swap chains have stricter rules around fullscreen state changes than D3D9's blit-model swaps. Iter-1's setWindowedMode_impl may need to skip the SetFullscreenState call entirely if the only goal is to make the window visible on the windowed-mode boot path (the smoke-friction fix). Mitigation: narrow scope per Task 1d's guardrail -- handle only the windowed=true boot case; FATAL on fullscreen toggle with a clear message; defer real toggle to a future plan.

2. **SetWindowPos before swap chain present.** D3D9's `updateWindowSettings` happens AFTER device creation but BEFORE the first present. D3D11 mirror should follow the same ordering. Risk: if SetWindowPos triggers a WM_SIZE that the engine's WindowProc handles by calling something that ultimately calls `swapChain->ResizeBuffers`, we could enter a transient state. Mitigation: the size we're setting matches the swap chain's existing size (ms_width / ms_height), so WM_SIZE handler's early-out at `Direct3d11_Device.cpp:601` (`if (newWidth == ms_width && newHeight == ms_height) return;`) protects against this.

3. **`GetContainingOutput` may return null on multi-monitor edge cases.** D3D9 used `IDirect3D9::GetAdapterMonitor` which is always-valid. DXGI's `GetContainingOutput` can return E_FAIL if the swap chain isn't currently on any output (e.g. before the first present). Mitigation: if it returns null, fall back to `MonitorFromWindow(ms_window, MONITOR_DEFAULTTOPRIMARY)` + `GetMonitorInfo` -- standard Win32 path.

4. **Backwards-compatibility with smoke harness.** `tools/d3d11-smoke/show-window.ps1` currently runs ShowWindow + SetWindowPos + SetForegroundWindow on an already-running process. Post-fix, the window will already be visible when the script runs -- the script's effect becomes a no-op (the ShowWindow + SetWindowPos calls are idempotent and SetForegroundWindow re-foregrounds an already-foregrounded window). Mitigation: NONE NEEDED. The script's preamble already documents it as a workaround for the engine-init root cause being investigated; post-fix the script becomes a diagnostic helper rather than a required smoke step. Update the script's preamble in a follow-up doc commit (not in the Iter-1 atomic).

5. **D-05 invariant.** This plan touches the D3D11 plugin only; no D3D9 path edits expected. Risk: accidentally adding `#include` in a shared engine header that the D3D9 plugin transitively pulls in. Mitigation: scoped header additions inside `src/engine/client/application/Direct3d11/` only.
</risks>

<objective>
Fix the half-screen render artifact Kenny surfaced in the Plan 11-09.14 close smoke. `Direct3d11_StateCache::drawTriangleFan` and its three sibling variants currently submit triangle-FAN vertex data with `D3D11_PRIMITIVE_TOPOLOGY_TRIANGLELIST` topology ‚Äî D3D11 has no native fan topology and the Plan 11-06 STUB never expanded the conversion. For a canonical 4-vert fullscreen-quad fan (SkyBox6Sided face, sun, moon, planet, sprite, beam, marker, etc.):

- **As fan:** triangles `(0,1,2)` + `(0,2,3)` ‚Üí full quad (2 triangles)
- **As list (current STUB behavior):** triangle `(0,1,2)` only ‚Üí upper-right half (TL/TR/BR); vertex 3 (BL) discarded

This produced "magenta upper-right triangle" pre-Plan-11-09.14 (everything was fallback-magenta) and "textured upper-right triangle / black or background lower-left" post-Plan-11-09.14 once SRV0 binding wired real textures to the visible half.

**Architecture decision pending CODEX consult** at `.planning/phases/11-d3d11-renderer-plugin/11-09.15-CODEX-CONSULT-trianglefan-index-expansion.md`. Working hypothesis: **Bucket A ‚Äî reusable static fan-to-list IB at install time** (mirrors D3D9 plugin's `drawQuadList` pattern at `Direct3d9.cpp:2380` / `:4277` ‚Äî pre-allocated `StaticIndexBuffer` with fan-to-list expansion pattern; lazy resize on demand). Alternatives B/C/D presented in the consult; CODEX verdict drives implementation.

**Scope ceiling:** 1 iteration if Bucket A; 2 if CODEX surfaces a non-obvious state-cache interaction.

**Risk profile:** low.
- Additive change in plugin-only code (no engine-side edits; D-05 zero-risk).
- D3D9 reference pattern is well-defined and exercised continuously by the live D3D9 path.
- IASetIndexBuffer + DrawIndexed are bog-standard D3D11 API surfaces.
- The fan-to-list conversion `(0, 1, 2, 0, 2, 3, 0, 3, 4, ..., 0, N-2, N-1)` is the canonical mathematical triangulation; correct by construction.

**Why this surfaced NOW:** Plan 11-09.14's SRV0 binding wired actual textures to the (until-now-magenta) half-rendered geometry. The half-render itself has existed since Plan 11-06's STUB-shape was authored. Pre-11-09.14, the entire screen was magenta, so the half-render was invisible ‚Äî every draw was producing magenta, so "half drew correctly with magenta + half drew nothing" looked identical to "all magenta." Plan 11-09.14 unmasked the bug visually.

**Note on indexed-fan variants:** `drawIndexedTriangleFan` and `drawPartialIndexedTriangleFan` are also broken in the same way ‚Äî but they require CPU-readback of the engine's IB and conversion, which is structurally heavier. CODEX Q3 decides whether to fix them in this plan OR defer with a DEBUG_REPORT_LOG_PRINT diagnostic to surface real callers (if any) first.

Plan 11-09.15 unblocks the final visual-parity verdict (Plan 11-11) ‚Äî the half-screen artifact is the most prominent remaining visual issue blocking the SPEC R6 comparison.

  - `CreateBuffer` with `D3D11_SUBRESOURCE_DATA` pointing at the staging array.
  - Doubling growth (`ms_triangleFanIBCapacityVerts = max(initial, ms_triangleFanIBCapacityVerts*2)` until ‚â• vertCount).
- Release previous `ms_triangleFanIB` ComPtr (auto-managed via ComPtr reassign).

**1c.** `install()` hook: pre-allocate `ms_triangleFanIB` at initial capacity (avoids first-draw stall). `remove()` hook: clear `ms_triangleFanIB` ComPtr (auto-managed but explicit for symmetry with other caches).

### Task 2 ‚Äî drawTriangleFan + drawPartialTriangleFan reimplementation

(Bucket A working hypothesis)

**2a.** Replace `drawTriangleFan()` (currently StateCache.cpp:1342-1353):
```cpp
void Direct3d11_StateCache::drawTriangleFan()
{
    int const vertCount = ms_currentVBVertexCount;
    if (vertCount < 3)
        return;                       // need >= 3 verts for >= 1 triangle
    int const triCount = vertCount - 2;

    ensureTriangleFanIB(vertCount);
    if (!ms_triangleFanIB)
        return;                       // create failure -> no draw

    if (!applyPreDrawState(D3D11_PRIMITIVE_TOPOLOGY_TRIANGLELIST))
        return;

    // Override applyPreDrawState's IB binding (which would have bound the
    // engine's IB if ms_currentIBValid; we want our fan IB for this draw).
    Direct3d11_Device::getContext()->IASetIndexBuffer(
        ms_triangleFanIB.Get(),
        DXGI_FORMAT_R16_UINT,         // or R32_UINT per CODEX Q2 cap decision
        0);

    Direct3d11_Device::getContext()->DrawIndexed(
        static_cast<UINT>(triCount * 3),
        0,
        0);

    // Engine's IB is still in ms_currentIB; next indexed-draw's
    // applyPreDrawState will rebind. No explicit restore needed.
}
```

**2b.** Replace `drawPartialTriangleFan(startVertex, primitiveCount)` (currently StateCache.cpp:1446-1451):
```cpp
void Direct3d11_StateCache::drawPartialTriangleFan(int startVertex, int primitiveCount)
{
    if (primitiveCount < 1)
        return;
    int const vertCount = primitiveCount + 2;
    ensureTriangleFanIB(vertCount);
    if (!ms_triangleFanIB)
        return;
    if (!applyPreDrawState(D3D11_PRIMITIVE_TOPOLOGY_TRIANGLELIST))
        return;
    Direct3d11_Device::getContext()->IASetIndexBuffer(
        ms_triangleFanIB.Get(),
        DXGI_FORMAT_R16_UINT,
        0);
    // BaseVertexLocation = startVertex shifts our static (0,i,i+1) pattern
    // to (startVertex, startVertex+i, startVertex+i+1) automatically.
    Direct3d11_Device::getContext()->DrawIndexed(
        static_cast<UINT>(primitiveCount * 3),
        0,
        static_cast<INT>(startVertex));
}
```

<risks>
1. **IB binding state-cache interaction.** After `drawTriangleFan` binds our fan IB via `IASetIndexBuffer`, the device IB state is sticky until the next `IASetIndexBuffer` call. The next non-fan indexed draw goes through `applyPreDrawState` which re-binds `ms_currentIB` if `ms_currentIBValid` is true. Risk: if a fan draw is immediately followed by an indexed draw WITHOUT the engine having called `setIndexBuffer` in between, the engine-bound IB may not match the actual device IB. Mitigation: verify the engine's draw protocol always calls `setIndexBuffer` before indexed draws (it does per Direct3d9 reference); if not, store/restore device IB or invalidate `ms_currentIBValid` explicitly after fan draw. CODEX Q4.

2. **16-bit index overflow.** R16_UINT max is 65535. For a fan with > 65535 verts we'd need R32_UINT. SWG fan sizes are typically tiny (4..32 verts for sky / sprites / beams); 65535 should be unreachable. Mitigation: assert / runtime check; CODEX Q2 verdict on whether to use R16 always, R32 always, or branch.

3. **Wasted memory if max-fan is small.** Initial capacity (1024 verts) is ~6KB at R16. Acceptable. Doubling growth caps at the largest fan ever submitted; never shrinks. Probably fine for SWG fan sizes.

4. **Indexed-fan-variant deferral risk.** If CODEX Q3 = DEFER and the engine actively uses drawIndexedTriangleFan for visible geometry, the half-render bug PERSISTS for those callers. The DEBUG_REPORT_LOG_PRINT diagnostic should surface this quickly; Plan 11-09.16 retrofits in the same iteration cadence.

5. **Engine `m_quadListIndexBuffer` precedent (D3D9 sibling) uses engine `StaticIndexBuffer` class.** We're creating a raw `ID3D11Buffer` instead. Mismatch from sibling shape; mitigation: the difference is justified because the IB lifetime is plugin-internal (never exposed to engine), so the engine wrapper class adds no value.

6. **Topology cache miss.** We always set TRIANGLELIST inside applyPreDrawState. If the engine's draw protocol relies on a topology-was-already-set assumption between consecutive `drawTriangleFan` calls, we'd churn the topology state. Mitigation: `IASetPrimitiveTopology` is cheap (~50ns per call); no real cost. CODEX Q5.

7. **Multi-stream VB compatibility.** Plan 11-09.7 added multi-stream VB binding (`ms_currentVBVectorActive`). drawTriangleFan goes through the same `applyPreDrawState` path; multi-stream binding still works. Risk: zero (additive IB binding doesn't interact with VB binding).

8. **D-05 invariant.** D3D11 plugin only. Risk: zero.
</risks>

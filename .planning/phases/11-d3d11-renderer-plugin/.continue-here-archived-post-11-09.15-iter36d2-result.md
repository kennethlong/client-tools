## CODEX + Cursor consult (just landed)

`.planning/phases/11-d3d11-renderer-plugin/11-09.15-CODEX-CONSULT-world-mesh-world-matrix-identity.md`

Both advisors converged tightly this time ŌĆö Cursor enumerated the engine code paths (`SoftwareBlendSkeletalShaderPrimitive.cpp:455`, `FullGeometrySkeletalAppearanceBatchRenderer.cpp:290`, etc.), CODEX provided a parallel review with the same conclusions. **Major adjustments from priors:**

- **Iter-35's "98% identity calls" was misleading** ŌĆö many subsystems intentionally pass identity (batch skeletal, terrain, particle systems, etc.). The 50-call cap on `setObjectToWorldTransformAndScale` may have captured startup noise (sky, debug viz) before the real per-mesh skeletal calls.
- **Stronger signal is Iter-34's late-session c4..c7 = identity at flush#87800 paired with non-identity WVP** ŌĆö that pattern specifically implicates `s_cachedWorld` for the draws that should have non-identity W.

**Bonus from CODEX:** Real scale-conversion bug at `Direct3d11_StateCache.cpp:1448` ŌĆö only scales diagonal `_11/_22/_33`, where D3D9's `convertScaleAndTransformToMatrix` scales all three components of each basis row. Doesn't explain identity-W (uniform scale 1.0 = no effect on either implementation), but is a real bug for rotated+non-uniformly-scaled objects. Tag as `Iter-36F` follow-up after the W investigation completes.

**Three hypothesis verdicts from the consult:**
- A (engine expects W=identity, verts pre-transformed): True for batch + terrain. False for our captured `a_specmap_bump_vs20_for_ps20.vsh` draws (standard skeletal ŌĆö requires non-identity W).
- B (missing setter): Ruled out.
- C (upstream passes identity incorrectly): **Most likely for in-world player.** Or character preview is genuinely centered at origin and the visible symptom is camera-far-from-origin + model-space verts collapsing.

<objective>
Replace the hardcoded-magenta fallback PS (Plan 11-09 Iter-2) with a textured-passthrough PS so the world becomes visually recognizable. Plan 11-05 documented PEXE rejection as a "Phase 12 asset re-author" punt: engine ships pre-compiled D3D9 PEXE bytecode in `.psh` IFF chunks (`DWORD * m_exe`), D3D11 `CreatePixelShader` rejects D3D9 bytecode, so 100% of asset shaders end up with `m_d3dPS = null` and Plan 11-09 Iter-2's fallback covers every draw.

Plan 11-09.12's debug-layer audit + Kenny visual confirmation established that the magenta-over-black state is now ISOLATED from any other artifacts (alignment bug fixed, geometry correctly rasterized). The only remaining question between "engine runs cleanly" and "engine actually shows the world" is: what does the fallback PS output?

Two real options:
1. **Constant magenta** (current Plan 11-09 Iter-2 design) — visible debug marker; clear "we're missing real PS"; world is unrecognizable.
2. **Textured passthrough** (Plan 11-09.13) — sample diffuse-texture-slot-0 and output it; world becomes recognizable (terrain textures, character skins, building walls visible); specular / normal-mapped / glow / per-pass-shader effects still missing (would need real per-asset PS compile, which is Phase 12).

Plan 11-09.13 lands option 2. Implementation: modify the HLSL source string at `Direct3d11_PixelShaderProgramData.cpp:295-300` to:
- Declare `Texture2D t : register(t0);` + `SamplerState s : register(s0);`
- Accept `float2 uv : TEXCOORD0` PS input
- Return `t.Sample(s, uv)`

Plan 11-06's state cache already binds SRV slot 0 + sampler slot 0 per the engine's setStaticShader / per-pass texture stages (Direct3d11_StateCache::applyPreDrawState wires this). For draws where slot 0 has a valid diffuse texture, the result is correctly-textured geometry. For draws where slot 0 isn't bound or VS doesn't output TEXCOORD0, the result is black or undefined-but-non-crashing color.

Risk surface:
- **D3D11 debug-layer warnings for PS-input-larger-than-VS-output mismatch.** Some VSes may not output TEXCOORD0; their PS input becomes undefined. D3D11 typically emits a debug-layer warning rather than failing the draw. Plan 11-09.13 Iter-1 monitors this; if warning count is high, mitigation options for Iter-2 include (a) phantom TEXCOORD0 input similar to Plan 11-09.8's slot-15 zero buffer, (b) reflection-driven VS output augmentation, (c) accept the noise.
- **Visual correctness for non-diffuse SRV slot 0 cases.** If the engine binds something other than the diffuse texture at slot 0 (e.g. shadow map, light map, environment cube), the fallback PS samples that instead. Should be rare; accepted as Iter-1 trade.

The D3D9 plugin doesn't need this change — D3D9 successfully consumes its own PEXE bytecode at runtime. D-05 invariant maintained.

Scope ceiling: ONE iteration. Pure HLSL string edit; no structural changes. Zero structural risk.
</objective>

<risks>
1. **PS-input / VS-output mismatch warnings** — Some VSes may not output TEXCOORD0. D3D11 spec says PS input must be a subset of VS output; debug-layer typically warns. Mitigation: monitor in Iter-1 smoke; if warning count is high, evaluate phantom-TEXCOORD0 mitigation similar to Plan 11-09.8's slot-15 phantom buffer.

2. **Non-diffuse SRV slot 0 binding** — If a draw has shadow map / light map / env cube at slot 0, the fallback samples that. Visually wrong but non-crashing. Accepted Iter-1 trade.

3. **Sampling at undefined UVs** — VSes that don't output TEXCOORD0 leave PS input undefined; sampling at undefined UV may produce any color. Non-crashing. Mitigation: accept Iter-1 noise.

4. **Texture format compat** — Texture2D.Sample works on most DXGI texture formats. BC-compressed (DXT1/3/5) work. Float formats work. Edge: array textures / 3D textures need different declarations; if any are at slot 0, sampling fails silently to black or undefined. Accepted Iter-1 trade.

5. **D-05 invariant** — D3D11 plugin only. Zero engine-side edits. Risk: zero.

6. **BSOD risk** — pure HLSL source string change + runtime D3DCompile at install time (existing path). Zero new GPU resources, zero topology changes. Risk: zero.
</risks>

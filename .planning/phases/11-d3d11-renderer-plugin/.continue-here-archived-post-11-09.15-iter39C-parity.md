## The two breakthroughs that landed it

### Iter-38B — matrix TRANSPOSE at cbuffer upload (the universal in-world skewing fix)

The "radial-fan stretched-sliver" symptom that affected ALL in-world content (terrain, buildings, characters) was a row-vector vs column-vector matrix convention mismatch:
- Engine builds matrices in **column-vector convention** (translation lives in last *column*, e.g. `W._14/_24/_34`)
- VS bytecode does **row-vector left-multiply** (`pos_row * M`)

These are duals. To make `(pos_row * stored)` compute `(M * pos_col)`, the cbuffer must hold `transpose(M)`. Fix is 4 lines in `composeSlot0Shadow` calling `XMMatrixTranspose` on `wvp` and `W` before storing into `s_slot0Shadow`.

Hand-walked verification on captured terrain WVP for an in-front-of-camera vertex `(3449.8, 6, -4875, 1)`:
- Before transpose: `clip.w ≈ −44 million` → catastrophic perspective divide → "radial fan from origin"
- After transpose: `clip.w ≈ +9.97` → vertex correctly in front of camera, visible

Both CODEX and Cursor advisors had previously closed the row/col-major packing hypothesis citing `D3DCOMPILE_PACK_MATRIX_ROW_MAJOR` + `row_major float4x4` declarations in the bytecode. They were correct that the COMPILER PACKING was right; what they missed was that the matrix CONTENT (translation in last column) is column-vector convention while the bytecode does row-vector multiply. The empirical Iter-38A2 instrumentation forced the conclusion.

### Iter-39A/B/C — per-pass alpha-blend wiring (the alpha-washout fix)

With geometry now correct after Iter-38B, the Iter-32A workaround of globally forcing `BlendEnable=TRUE` became visible as world content rendering washed/translucent. The proper fix:

- **Iter-39A**: flipped blend default `TRUE` → `FALSE`. World opaque correct, UI/particles regressed (expected).
- **Iter-39B (false start)**: wired per-pass blend to `Direct3d11_ShaderImplementationData::apply()` → discovered that's dead code (base class `ShaderImplementationGraphicsData` has no `apply()` virtual; nothing in the engine ever calls it).
- **Iter-39C (real fix)**: moved the wiring to `Direct3d11_StaticShaderData::apply(passNumber)` at line 497, which IS called per-draw from `StateCache::draw*`. Reads `engPass->m_alphaBlendEnable` from `m_implementation->m_pass[idx]` and calls `Direct3d11_StateCache::setAlphaBlendEnable(bool)` (new setter). Now the engine controls blend per shader: UI canvas / particles / glow get blend ON, opaque world content gets blend OFF.

## Outstanding — only one bug left in the log

The d3d11-debug.log has **exactly one distinct error type** firing 7 times across a full session:

```
id=281: ID3D11DeviceContext::CopySubresourceRegion: Cannot invoke 
CopySubresourceRegion when the Formats of each Resource are not the 
same or at least castable to each other...
```

**Where:** `Direct3d11_TextureData.cpp` has three CopySubresourceRegion call sites at lines 811, 899, 947. Context around each error shows the responsible draws use `texren_alphablend_c1a1.sht` / `texren_copy_c1a1.vsh` — the engine's TEXREN render-to-texture system (post-FX / depth-of-field / motion-blur / color-grading chain). Likely a format mismatch between source and dest in the inter-RT copy.

**Why it matters:** under the D3D11 SDK debug layer, this is severity=ERROR. The debug layer queues each error message internally; when its message queue hits a threshold it raises an exception code 4000001f at the next D3D11 call, which crashes the process. That's the intermittent "crash during world loading" Kenny has seen twice. Crash repro is non-deterministic (depends on whether the queue accumulates enough before a flush).

**Iter-40 candidate:** find which of the three CopySubresourceRegion sites the TEXREN code path hits and fix the format match. Likely either (a) staging texture should match the dest texture format that TEXREN allocated for the post-FX RT, or (b) the engine's TEXREN expects a format conversion that isn't legal under D3D11 CopySubresourceRegion (would need a render-pass-based blit instead).

### Hypothesis

D3D11 / DXGI has no 24bpp packed surface format (D3D9 has D3DFMT_R8G8B8; dropped from DXGI). Plan 11-04 runtimeFormats promotion at `Direct3d11_TextureData.cpp:310-329` correctly picks a 32bpp DXGI format (e.g. DXGI_FORMAT_B8G8R8X8_UNORM = TF_XRGB_8888) at create time — `m_destFormat` stores it.

But at lock time (~line 477), the code uses `lockData.getFormat()` (the engine's expected FILL format = TF_RGB_888 = 24bpp) to look up the DXGI format directly via the translation table → UNKNOWN → FATAL.

The engine fills the texture at 24bpp stride; the GPU surface is 32bpp. Fix: detect the mismatch, allocate a CPU shadow buffer at 24bpp engine stride, return that to the engine for the lock duration, then expand row-by-row 3→4 bytes at unlock with the right channel-order swap (RGB → BGRX or RGBA depending on m_destFormat).

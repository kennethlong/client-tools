## Iter-1: Lock-bridge buffer + channel-swap expansion at unlock

### Hypothesis

D3D11 / DXGI has no 24bpp packed surface format (D3D9 has D3DFMT_R8G8B8; dropped from DXGI). Plan 11-04 runtimeFormats promotion at `Direct3d11_TextureData.cpp:310-329` correctly picks a 32bpp DXGI format (e.g. DXGI_FORMAT_B8G8R8X8_UNORM = TF_XRGB_8888) at create time — `m_destFormat` stores it.

But at lock time (~line 477), the code uses `lockData.getFormat()` (the engine's expected FILL format = TF_RGB_888 = 24bpp) to look up the DXGI format directly via the translation table → UNKNOWN → FATAL.

The engine fills the texture at 24bpp stride; the GPU surface is 32bpp. Fix: detect the mismatch, allocate a CPU shadow buffer at 24bpp engine stride, return that to the engine for the lock duration, then expand row-by-row 3→4 bytes at unlock with the right channel-order swap (RGB → BGRX or RGBA depending on m_destFormat).

### Investigation

Confirmed source byte order by inspecting `TextureFormatInfo.cpp:20` for TF_RGB_888: `rMask=0x00ff0000, gMask=0x0000ff00, bMask=0x000000ff, pixelByteCount=3`. The R mask in the high byte of the 32-bit dword means R lands at byte index 2 in little-endian memory — so memory layout is byte 0=B, byte 1=G, byte 2=R (the D3D9-era "R8G8B8" name preserves the high→low bit-order naming, not the in-memory byte order).

That matches DXGI byte order for `B8G8R8X8_UNORM` (B,G,R,X) and `B8G8R8A8_UNORM` (B,G,R,A) — both 32bpp BGR(+X|A). So for the typical TF_RGB_888 → TF_XRGB_8888 / TF_ARGB_8888 promotion path, the expansion is a **straight 3→4 byte copy** with `dst[3]=0xFF`. No channel swap needed. The plan's hypothesis envisioned a swap; investigation eliminated it.

For RGBA-ordered DXGI promotion targets (DXGI_FORMAT_R8G8B8A8_UNORM) the swap *would* be required, but the engine's runtimeFormats list for TF_RGB_888 sources prefers the BGR-ordered targets first, so RGBA promotion is not hit in Iter-1's smoke. Iter-1 FATALs defensively if m_destFormat ∉ {TF_XRGB_8888, TF_ARGB_8888} with a clear "Plan 11-09.10 territory" message — the bridge is extensible per-target without disturbing the working path.

### Root cause

Confirms hypothesis verbatim: D3D11/DXGI has no 24bpp packed format. Create-time runtimeFormats walk picks a 32bpp DXGI format (TF_XRGB_8888 / TF_ARGB_8888 are the BGR-ordered defaults), but lock-time path went straight through the translation table on `lockData.getFormat()` (the engine's narrow fill format) and FATALed on UNKNOWN. The 32bpp surface was already there; we just needed a shadow buffer + row-by-row expansion to bridge the stride mismatch on unlock.

### Fix

Per `11-09.9-PLAN.md` Tasks 1-2:

- **Task 2a:** New header state in `Direct3d11_TextureData.h` — m_lockBridgeBuffer (std::vector<uint8_t> CPU shadow), m_lockBridgeFormat (TextureFormat), m_lockBridgePitch (UINT), m_lockBridgeWidth/Height, m_lockBridgeActive (bool). Constructor initialized.
- **Task 2b:** `lock()` UNKNOWN-mapping FATAL replaced with bridge detector. Guards: m_isCubeMap/m_isVolumeMap → FATAL "Plan 11-09.10 territory"; lockData.isReadOnly() → FATAL "Plan 11-09.10 territory"; lockData format != TF_RGB_888 || m_destFormat ∉ {TF_XRGB_8888, TF_ARGB_8888} → FATAL "Plan 11-09.10 territory". On match: allocates `m_lockBridgeBuffer.assign(width*3*height, 0)`, returns its `.data()` to engine with pitch=width*3.
- **Task 2c:** `unlock()` checks `m_lockBridgeActive` FIRST (before the existing m_reserved branches). On bridge: allocates a fresh `D3D11_USAGE_STAGING` texture at `m_destFormat`'s DXGI format + width × height, Maps WRITE, copies row-by-row with `dst[x*4+0..2] = src[x*3+0..2]; dst[x*4+3] = 0xFF` honoring `mappedResource.RowPitch`, Unmaps, `CopySubresourceRegion` to destination subresource at lockData.getX/Y/Z + level, releases staging, clears bridge state. Existing dynamic/staging branches unchanged.

D-05 preserved: only `Direct3d11_TextureData.h` + `.cpp` touched; D3D9 source diff is Koogie's pre-existing 135-line working-tree patch (untouched).

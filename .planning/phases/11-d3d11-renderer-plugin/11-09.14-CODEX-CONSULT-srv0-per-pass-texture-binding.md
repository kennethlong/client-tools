## Surface evidence

**The Iter-4 first-dynamic-PS HLSL is exactly what we want, structurally** — the generator produces:

```hlsl
Texture2D    t : register(t0);
SamplerState s : register(s0);
struct PSIn { float4 _pos : SV_POSITION; float2 _v0 : TEXCOORD0; };
float4 main(PSIn input) : SV_TARGET { return t.Sample(s, input._v0.xy); }
```

It expects an SRV at slot 0 + a sampler at slot 0. Currently both are null at draw time → sampling returns zero (D3D11 spec) → black. Bind them, and you get textured world.

**`Direct3d9_StaticShaderData::apply()` reference** (this is what we're missing). The D3D9 sibling reaches per-pass texture binding via:

- `Direct3d9_StaticShaderData::apply(pass)` → `m_pass[pass].apply()` (`Direct3d9_StaticShaderData.cpp:1056`)
- `Pass::apply()` (`.cpp:820+`) handles VS bind + material → cbuffer constants + textureFactor + textureScroll + alpha test + stencil + fog + … and finally iterates `Stages`, calling `Stage::apply(stage_index)` on each.
- `Stage::apply(int stage)` (`.cpp:397`) calls `Direct3d9_StateCache::setTexture(stage, *m_texture)` and applies sampler state per-stage.
- `Stage::construct(StaticShader const &, ShaderImplementation::Pass::PixelShader::TextureSampler const &)` (`.cpp:297+`) extracts `textureSampler.m_textureTag` → `shader.getTextureData(m_textureTag, &textureData)` → `textureData.texture->getGraphicsDataAddress()` → `Direct3d9_TextureData const * const *`. Sampler state (anisotropy, address-mode, filters) comes from the same `TextureData` fields, falling back to `TextureSampler` defaults when the per-instance `textureData.X == TA_invalid` / `TF_invalid`.

So the engine-side data we need is fully exposed via `ShaderImplementation::Pass::PixelShader::TextureSamplers` + `shader.getTextureData(tag, &out)`. The plugin-side data we need is the `Direct3d11_TextureData` SRV (already owned by Plan 11-04's resource-layer work) and an `ID3D11SamplerState` derived from the address/filter/anisotropy combination (Plan 11-06's `Direct3d11_StateCache::getOrCreateSampler(D3D11_SAMPLER_DESC const &)` already exists and is keyed by hash — same pattern as Plan 11-09's RS/BS/DSS caches).

**Plan 11-09.13 dynamic PS reads only slot 0.** The generator's body is `t.Sample(s, input._v0.xy)` — single-texture sample. Multi-texture PSes (specular + normal + glow) require either:
- Phase 12 real per-asset PS compile (the proper path), OR
- An extended generator that knows how to combine multiple SRV slots into a single output color — which it can't, without knowing the asset's per-pass-shader logic. This is the structural limit CODEX called out when ranking Bucket 2 in the Iter-3 consult.

So even if we wire ALL `TextureSamplers` (Bucket B below), the dynamic PS won't sample them. Bucket A binds only what the dynamic PS actually reads.

**Current D3D11 `StaticShaderData::apply()` body** (cpp:208-248) is unchanged from Plan 11-09 Iter-1. It sets `ms_currentVSData` + `ms_currentPSData` via `Direct3d11_StateCache::setCurrentVSData` / `setCurrentPSData` and returns. Zero SRV/sampler/material/textureFactor/textureScroll/alphaTest/stencil/fog handling. The deferral comment quoted above is verbatim from this site.

**No prior SRV binding path exists in the D3D11 plugin.** Grepping `PSSetShaderResources` and `ms_boundSRV` shows the slot-management plumbing from Plan 11-06 (`Direct3d11_StateCache::ms_boundSRV[16] + ms_boundSampler[16]` with per-slot bind tracking) is present and wired into `applyPreDrawState`, but **no caller writes to `ms_boundSRV`/`ms_boundSampler` for asset-shader paths.** The fields exist as scaffolding; Plan 11-09.14 is the first real caller.

## Three architectural buckets

### Bucket A: Diffuse-only at SRV0/sampler0 (minimal)

Mirror just the first `TextureSampler` from `ShaderImplementation::Pass::PixelShader::TextureSamplers` per pass. Bind one SRV at slot 0 + one sampler at slot 0. Match exactly what the Plan 11-09.13 dynamic PS reads.

Surface changes:
- New nested `Stage` struct on `Direct3d11_StaticShaderData` (mirrors `Direct3d9_StaticShaderData::Stage` but minimal — `Direct3d11_TextureData const * const *` + cached `D3D11_SAMPLER_DESC` only).
- `construct()` extracts the FIRST TextureSampler from each pass; calls `shader.getTextureData(tag, &out)`; builds the sampler desc from address/filter/anisotropy fields (fall back to TextureSampler defaults when textureData fields are TA_invalid / TF_invalid per D3D9 sibling line 336-352); records the `Direct3d11_TextureData const * const *` pointer-to-pointer.
- `apply()` adds, after the existing VS/PS pointer set: `PSSetShaderResources(0, 1, &srv)` for the resolved Direct3d11_TextureData's SRV + `PSSetSamplers(0, 1, &sampler)` for the cached ID3D11SamplerState (via `Direct3d11_StateCache::getOrCreateSampler(desc)`).
- Optional but consistent: also handle the "no texture" path the D3D9 sibling has (line 416-417 `setTexture(stage, NULL)`) — unbind SRV0/sampler0 to avoid stale state.

**Pros:** smallest surface; unblocks visible diffuse-textured world; matches what the dynamic PS actually consumes; D-05-safe.

**Cons:** no specular / normal-mapped / glow / per-pass-shader effects (those are Phase 12). UI shaders that use vertex colors instead of textures still render as magenta (which is correct — no diffuse to sample). Multi-texture passes still look "diffuse-only."

### Bucket B: Multi-stage SRV0..SRV7 mirror

Mirror D3D9's full texture-stage cascade. Bind SRV0..SRV{N-1} + sampler0..sampler{N-1} per the count of TextureSamplers in each pass.

Surface changes: same as Bucket A but iterate ALL TextureSamplers, not just the first. Per-stage SRV + sampler binding loop in apply().

**Pros:** full data parity with engine asset metadata; future-proof for Phase 12 when real per-asset PS compile starts consuming multi-slot data.

**Cons:** the Plan 11-09.13 dynamic PS only samples slot 0. Slots 1..7 would be bound but unread by the fallback PS → no visible effect, just extra bind calls. Extra cost for zero current visible benefit. Phase 12 will rebind these anyway with real per-asset PSes; doing it now is premature.

### Bucket C: Hold for Phase 12 asset re-author

Don't do SRV binding in 11-09.14. Accept Plan 11-09.13 close state ("0 id=342 + 0 id=343 + world mostly black/noise") as the architecture-clean Phase 11 endpoint. Move to Plan 11-10 (DPVS remeasurement) and Plan 11-11 (visual-parity verdict + Phase 11 SUMMARY). Phase 12 handles real per-asset PS + multi-texture binding together as one re-author pass.

**Pros:** zero risk of mis-mirroring D3D9 Stage construction quirks; cleanest scope close; per-pass texture binding work isn't wasted retrofit when Phase 12 needs to redo the binding pattern for real PSes.

**Cons:** SPEC R6 "substantially similar to D3D9 baseline" almost certainly fails the visual-parity verdict in Plan 11-11 if world stays black. Plan 11-11 may need to either re-open 11-09.14 OR pre-accept "mostly black" as the Phase 11 close state.

## Our current hypothesis

**Bucket A (diffuse-only at SRV0/sampler0)** is likely the right Iter-4 move for Plan 11-09.14:

- Matches what the Plan 11-09.13 dynamic PS actually reads — no wasted binds.
- Smallest D3D9-sibling-mirror scope, lowest risk of mis-translating the Stage construction quirks (address-mode invalid-fallback, filter-mode invalid-fallback, anisotropy clamp, global-texture short-circuit).
- Unblocks visible diffuse-textured world for SPEC R6 verdict in Plan 11-11.
- Bucket B's slots 1..7 are dead bind cost until Phase 12; the cost is wasted retrofit, not a forward investment.
- Bucket C punts visible-parity entirely to Phase 12, which may not run for months — Plan 11-11 wants a verdict NOW.

Plan 11-09.14 Iter-1 hypothesis: ~150-300 lines across `Direct3d11_StaticShaderData.h/.cpp` (new nested Stage struct + construct extraction + apply binding) + zero new infrastructure (Plan 11-06's `getOrCreateSampler` already exists; Plan 11-04's `Direct3d11_TextureData::getShaderResourceView()` already exists).

## Questions for CODEX

### Q1 — Architecture choice

Bucket A vs B vs C — does our hypothesis hold? Specifically:

- Is Bucket A's "bind only slot 0" the right scope, OR is there value in Bucket B's multi-stage mirror as forward investment (Phase 12 would need it eventually)?
- Is Bucket C's "hold for Phase 12" the safest because mid-stride retrofits often miss invariants — OR is the SPEC R6 visual-parity acceptance bar load-bearing enough that we have to land Bucket A even at the cost of Phase-12 rework?

### Q2 — Engine surface (TextureSamplers iteration)

The D3D9 sibling at `Direct3d9_StaticShaderData.cpp:787-788` iterates `pass.m_pixelShader->m_textureSamplers` directly. For Bucket A, we'd take just the first entry. Is "first entry" the right diffuse channel, or does the engine encode diffuse vs specular vs normal positionally (slot 0 = diffuse by convention) — i.e. should we check the `TextureSampler.m_textureTag` for a "diffuse" tag pattern instead of position 0?

If diffuse is positional (slot 0 = diffuse), Bucket A's simple "first iterator" works. If it's tag-keyed (e.g. `TAG('M','A','I','N')` or similar), we should match by tag, not by position — and a defensive fallback to "first sampler" if the diffuse tag isn't present.

### Q3 — Sampler desc derivation

D3D9 sibling builds sampler state from a mix of `textureData.X` fields (per-instance) and `textureSampler.X` fields (per-shader default), with `TA_invalid` / `TF_invalid` meaning "use the other source" (cpp:336-352). The D3D11 analog needs to build a `D3D11_SAMPLER_DESC` from the same data and pass to `Direct3d11_StateCache::getOrCreateSampler(desc)`. The mapping is:

- `StaticShaderTemplate::TextureAddress` enum → `D3D11_TEXTURE_ADDRESS_MODE` (TA_wrap → WRAP, TA_clamp → CLAMP, TA_mirror → MIRROR, TA_border → BORDER, TA_mirror_once → MIRROR_ONCE — likely needs a small translation table)
- `StaticShaderTemplate::TextureFilter` enum (`TF_point`, `TF_linear`, `TF_anisotropic`) × min/mag/mip × anisotropy clamp → `D3D11_FILTER` (which combines all three into a single bitfield value)

Q3a: Is there an existing D3D11-equivalent of `Direct3d9::getMaxAnisotropy()` (used in D3D9 sibling line 354)? If not, hardcode 16 (D3D11 hardware max) or query from device caps?

Q3b: The D3D11_FILTER combinatorial space is 16 values for non-comparison-non-anisotropic + 8 for anisotropic. Build a translation function `chooseD3D11Filter(min, mag, mip, anisotropy)` mirroring D3D9 sibling line 348-350 → return the right bitfield. Existing precedent in the codebase to mirror, or build from scratch?

### Q4 — Lifecycle / unbind discipline

When should SRV0 / sampler0 be UNBOUND between draws?
- Per-shader-apply: at the start of each `StaticShaderData::apply()`, before binding the new pass's texture, unbind the prior?
- Per-frame: at `beginFrame()` / `endFrame()`, clear slot 0?
- Sticky until replaced: bind only when a new pass has one; leave bound otherwise?

D3D9 sibling at Direct3d9 line 416-417 does explicit `setTexture(stage, NULL)` when a Stage has no texture — suggesting "unbind when absent" is the discipline. The Plan 11-06 `ms_boundSRV[16]` tracking should already help avoid redundant binds; question is whether unbind-on-absence matters for correctness or just hygiene.

### Q5 — Material / textureFactor / textureScroll handling

D3D9 `Pass::apply()` (cpp:835-924) does extensive cbuffer constant writes for material, textureFactor, textureScroll. Plan 11-09 Iter-2's CODEX consult ("dead-static user-constants") established that the engine writes these via `setVSConstants`/`setPSConstants` which we route through `Direct3d11_StateCache::setVSConstants`/cbuffer slot 0. So material/textureFactor/textureScroll ARE already flowing — just into a cbuffer the dynamic-fallback-PS doesn't read.

Does Plan 11-09.14 need to do anything material-side? Or is it strictly "bind SRV0 + sampler0; everything else already flows through the cbuffer pipeline that Plan 11-08/09 wired"?

### Q6 — Stale-premise / scope check

Is there a bigger architectural gap we're missing? Specifically:

- Q6a: Plan 11-05 SUMMARY's `Known Stubs` section deferred BOTH the PS compile path AND the per-pass texture binding to Phase 12. We're partially un-deferring with 11-09.13 (per-VS dynamic PS compile) and 11-09.14 (SRV0 binding). Is this a coherent middle path, or are we accumulating retrofits that Phase 12 will throw away?
- Q6b: Are there rendering paths (UI, debug-line, terrain decal, particles) that DON'T flow through `StaticShaderData::apply()` and therefore wouldn't pick up Plan 11-09.14's binding? If so, those would remain magenta-or-black after 11-09.14 lands — we should know in advance.
- Q6c: The Iter-4 first-dynamic-PS log line came from `bytecode hash 0xB8E97E313573A257` with a single-`float2 TEXCOORD0` output. Is this a single early character-select VS, or representative of the SWG VS population? If most engine VSes have multiple non-SV outputs (BLEND/COLOR/NORMAL + multiple TEXCOORDs), the generated PSes get bigger declarations but only sample TEXCOORD0 — that's fine, just confirming our mental model.

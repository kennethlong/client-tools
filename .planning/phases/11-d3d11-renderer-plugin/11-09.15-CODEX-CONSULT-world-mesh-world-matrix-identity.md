# CODEX + Cursor Consult — D3D11 Plugin: World Meshes Render Collapsed Because cb0.objectWorldMatrix Stays Identity

**Context date:** 2026-05-23 (post-Plan-11-09.15 Iter-32A/33B/34/35)

**Status:** UI rendering FULLY WORKING after Iter-32A (alpha-blend default flip). World-content geometry shows characteristic "radial-fan collapse to single vanishing point" pattern (screenshot attached separately by Kenny). Diagnostic chain Iter-26/33B → Iter-34 → Iter-35 has localized the bug to: **the cbuffer slot `objectWorldMatrix` (c4..c7) stays IDENTITY throughout the entire session, and the engine setter `setObjectToWorldTransformAndScale` fires almost always with identity input.** We need direction on **which engine code path is supposed to set per-mesh world matrices for static + skeletal meshes**, and why we're not seeing those calls.

This consult is deliberately **fact-first, hypothesis-light** — the previous two consults had both advisors over-rotate on the cross-format hypothesis that turned out to be wrong (font atlas was genuinely DXT5; the bug was missing default alpha-blend). We're asking the advisors to do the architectural localization rather than confirm our guesses.

The Iter-35 setter-call loggers reveal who calls the matrix entry points:

| Setter | Calls captured | Sample data |
|---|---|---|
| `setObjectToWorldTransformAndScale` | 50 (hit cap) | **Call #1: rotation matrix (`row0=[0.934 -0.347 0.094 0.208] ...`).  Calls #2..#50: all identity.** |
| `setWorldToCameraTransform` | 30 (hit cap) | All real-looking view matrices with camera-position translation columns |
| `setProjectionMatrix` | 30 (hit cap) | All real-looking perspective matrices (fov ~5.027, ~6.741) |

So:
- View + projection setters fire fine
- The per-mesh setter (`setObjectToWorldTransformAndScale`) fires plenty, but the engine passes identity for ~98% of calls
- The D3D9-register-push path (`setVSConstantsRaw`) NEVER pushes anything other than `c9` viewportData — no user constants, no bone matrices, no per-shader values

We expected one of these setters (or a yet-undiscovered one) to push a non-identity per-mesh world matrix for terrain tiles, static meshes, and skeletal characters. **None of them are.** The cbuffer's `objectWorldMatrix` field remains identity throughout. Every world VS (which reads `cb0[4..7]` for `objectWorldMatrix` and multiplies vertex-local position by it) effectively renders with W=identity → all object-space vertices end up at object-space coords interpreted as world space → degenerate clip-space collapse → the radial-fan symptom Kenny sees.

We want CODEX + Cursor to identify **which engine code path is supposed to populate the cbuffer's `objectWorldMatrix` for static + skeletal mesh rendering** — and why we're not seeing those calls.

## D3D11 plugin architecture (the load-bearing context)

### Cbuffer layout (matches the engine VS expectations)

Looking at the disassembled `ui.vsh` (and consistent with the D3D9-era engine convention):

```
cbuffer SwgVertexConstants
{
    row_major float4x4 objectWorldCameraProjectionMatrix;  // c0..c3  -- WVP
    row_major float4x4 objectWorldMatrix;                  // c4..c7  -- W (per-mesh)
    float3 cameraPosition_w;                                // c8
    float4 viewportData;                                    // c9
    float4 fog;                                             // c10
    struct material { ... };                                // c11..c14
    struct lightData { ... };                               // c15..c43 (lights)
    float4 textureFactor;                                   // c44
    float4 textureFactor2;                                  // c45
    float4 textureScroll;                                   // c47
    float currentTime;                                      // c48
    float3 unitX / unitY / unitZ;                           // c49..c51
    float4 userConstant0..7;                                // c52..c59
    // ...extendedLightData and pad at c60..c71
};
```

### Setter wiring (Gl_api callbacks, see `Direct3d11.cpp:715..735`)

```
setWorldToCameraTransform         → Direct3d11_StateCache::setWorldToCameraTransform
setProjectionMatrix               → Direct3d11_StateCache::setProjectionMatrix
setObjectToWorldTransformAndScale → Direct3d11_StateCache::setObjectToWorldTransformAndScale
setVertexShaderUserConstants      → setVertexShaderUserConstants_impl (→ setVSConstants → setVSConstantsRaw at reg=52+idx)
setLights                         → Direct3d11_LightManager::setLights
```

Each matrix setter writes its respective input into a CPU shadow (`s_cachedView`, `s_cachedProj`, `s_cachedWorld`), then calls `composeSlot0Shadow()` which:

```cpp
DirectX::XMMATRIX V   = XMLoadFloat4x4(&s_cachedView);
DirectX::XMMATRIX P   = XMLoadFloat4x4(&s_cachedProj);
DirectX::XMMATRIX W   = XMLoadFloat4x4(&s_cachedWorld);
DirectX::XMMATRIX wtp = XMMatrixMultiply(P, V);     // P * V (CODEX Q1 from Iter-2.7)
DirectX::XMMATRIX wvp = XMMatrixMultiply(wtp, W);   // (P * V) * W
XMStoreFloat4x4(&s_slot0Shadow.objectWorldCameraProjectionMatrix, wvp);  // c0..c3
XMStoreFloat4x4(&s_slot0Shadow.objectWorldMatrix,                 W);    // c4..c7
s_slot0Dirty = true;
```

Then `flushSlot0IfDirty()` runs at the top of `applyPreDrawState` (every draw) and uploads the entire 1152-byte struct to slot 0 via `Direct3d11_ConstantBuffer::updateVS(0, &s_slot0Shadow, sizeof(s_slot0Shadow))`.

D3D11 shader-cache invalidates with `D3D11_REWRITE_VERSION=18` (current). Matrix packing is `D3DCOMPILE_PACK_MATRIX_ROW_MAJOR` (Plan 11-08 Iter-1.5 retrofit; matches the `row_major float4x4` cbuffer declarations).

### What we explicitly DID NOT find in the Gl_api surface

Diffing `Direct3d11.cpp` vs `Direct3d9.cpp` (Koogie's working D3D9 sibling): every `ms_glApi.xxx` slot wired in D3D9 is also wired in D3D11 (or marked STUB for slots like `screenShot`/`writeImage`/etc. that are unrelated to rendering). **There is no `setBoneMatrices`-shaped slot we forgot to wire** — at least not one named that way.

`setObjectToWorldTransformAndScale` calls #1..#5:

```
#1 row0=[0.934 -0.347 0.094 0.208] row1=[0.116 0.044 -0.993 0.837] row2=[0.340 0.938 0.081 0.077] scale=[1.000 1.000 1.000]

```

`setVSConstantsRaw` calls #1..#100: every single one is `reg=9 count=1 first4=[0.002 -0.003 -1.000 1.000]`. That's the viewportData computed at engine width=1024 height=768. Nothing pushed at any other register, ever.

### What that says

For the 500 captured WORLD draws (character body parts + clothing + equipment, all on `a_*_vs20_for_ps20.vsh` family), the engine submits geometry data correctly (real vertCounts 26..703, real SRVs bound for textures, no per-VS PS fallback / tombstone fires) — but **per-mesh world transforms are uniformly identity** in the engine's input to our plugin. Either:

- (A) **The engine expects W=identity for these meshes** because vertex data is supposed to be pre-transformed to world-space CPU-side. In that case, the bug is somewhere else — maybe the CPU-skinning path that's supposed to be writing world-space vertices into a dynamic VB before drawing.
- (B) **The engine pushes per-mesh transforms via a code path we haven't hooked** — some sibling-of-`setObjectToWorldTransformAndScale` setter that exists in the engine's Gl_api but isn't named obviously. (Our diff showed no obvious missing slot, but we may have missed something.)
- (C) **`setObjectToWorldTransformAndScale` IS the right setter, but the engine code that *calls* it for skeletal/static meshes is broken upstream** — passing identity instead of the real world matrix. Would mean we need to find the call sites in `clientGraphics` / `clientObject` / `clientSkeletalAnimation` and audit them.

We can't distinguish A/B/C from inside the D3D11 plugin alone. Need engine-side code-path analysis.

## Questions for CODEX + Cursor

1. **Who calls `Graphics::setObjectToWorldTransformAndScale` for skeletal-mesh rendering in the SWG engine?** Specifically tracing the call site that fires for a character body part like `appearance/skeletal/sul_m_body.sat` (Sullustan male body skeletal mesh) — does the engine call this setter with the player's world position, or does it pass identity because the character's mesh data is already pre-transformed into world space CPU-side?

2. **If skeletal rendering pre-transforms vertices CPU-side** (option A above), where in `clientSkeletalAnimation/` does that happen, and what writes the transformed vertex data into the GPU VB the D3D11 plugin draws from? If our D3D11 dynamic-VB lock/unlock path is missing the format conversion that pre-skinning needs (similar in shape to the font-atlas bug we just fixed), that'd produce the symptom.

3. **What about static (non-skeletal) world meshes** like terrain tiles or props? Do they push per-mesh world matrices through `setObjectToWorldTransformAndScale`, or some other path? In the engine, is there a `Graphics::setVertexShaderConstants(register, data, count)` D3D9-style entry point that we wire on the D3D11 side via `setVSConstantsRaw` — and if so, which engine subsystems use it for per-mesh transforms vs the dedicated setObjectToWorld... API?

4. **In the D3D9 plugin, what's the calling pattern that ends up writing the per-mesh world matrix into the GPU?** If the D3D9 sibling does it via a different path than we're hooking on D3D11 (e.g., `IDirect3DDevice9::SetVertexShaderConstantF` pushed from inside `Direct3d9_StateCache::applyPreDrawState`, called per-mesh by something we haven't ported), that'd identify the gap.

5. **Architecture sanity check on `composeSlot0Shadow`** (`Direct3d11_StateCache.cpp:315`): we multiply `wtp = P * V; wvp = wtp * W`. With row-major matrix packing (`D3DCOMPILE_PACK_MATRIX_ROW_MAJOR`) and the engine VS source presumably using `mul(position, modelViewProjection)` convention (D3D9-era row-vector style), is this the correct multiplication order? The cbuffer fields are declared `row_major float4x4`, and Iter-34 shows WVP composes to sensible-looking values when V+P are set, so this seems fine — but worth a sanity check given the cross-format bias both advisors had in the last consult.

6. **Fastest next diagnostic** — what's the highest-information-density next iter to confirm the right hypothesis? Candidates we've considered:
   - Iter-36A: log `setVSConstants` calls with the full register/count/data (not just first4), capped at first 200 — would surface any hidden bulk push to non-c9 registers we haven't seen
   - Iter-36B: log every Gl_api callback signature that's ever fired (using `STUB`-trampoline-style instrumentation), to confirm we're not missing a callback the engine is silently failing on
   - Iter-36C: temporarily force-write a non-identity W matrix in `composeSlot0Shadow` (e.g., scale=2) to confirm the cbuffer path IS receiving values when we explicitly write them — if force-write doesn't change rendering, the bug is downstream of the cbuffer

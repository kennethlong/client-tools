**Pitfalls to expect surfacing in this plan** (per RESEARCH §Common Pitfalls — quick-reference):

1. **SV_POSITION corner cases** — if asset HLSL uses POSITION as a vertex INPUT semantic AND output semantic, the macro substitution corrupts the input. Watch for: vertex shader compiles but pixel shader fails OR solid-black output. Fix: pre-process source text to substitute only output-position semantics, not macro-blanket.

2. **cbuffer alignment errors** — wrong values for specific fields after the first one. static_asserts catch struct-size mismatches at compile, but mismatch BETWEEN HLSL and C++ struct (e.g., HLSL `float3 a; float4 b;` C++ `XMFLOAT3 a; XMFLOAT4 b;` — HLSL packs `a` to 12 bytes then 4-byte pad to align `b`; C++ packs `a` to 12 bytes then `b` at 12, causing 4-byte offset). Fix: match HLSL packing exactly; add `XMFLOAT4 _pad` after problematic fields.

3. **Forgetting OMSetRenderTargets after Present** — first frame renders, subsequent frames black. Plan 03 already enforces unconditional rebind in beginScene; verify it's there if you see this symptom.

4. **SRV/sampler split** — textures appear correctly first-bound, wrong when state dirty. Plan 06 StateCache enforces independent tracking; verify the bind sequence (PSSetShaderResources + PSSetSamplers in applyPreDrawState).

5. **Map DISCARD/NO_OVERWRITE confusion** — dynamic geometry tearing or ghost data. Plan 04 Dynamic*BufferData enforces the wrap/append distinction; if you see this, audit those .cpp files.

6. **Input-layout VS-bytecode mismatch** — vertex attributes scrambled. Plan 06 VertexDeclarationMap keys by VS bytecode SHA1; if you see this, audit the cache key construction.

7. **DLL base-address collision** — none expected (Plan 02 set 0x60A00000 ≠ existing gl05_* at 0x62A00000); only cosmetic if it happens.

8. **Graphics Tools optional feature absent** — Plan 03 already detect-and-fallbacks; if you see DXGI_ERROR_SDK_COMPONENT_MISSING, the fallback isn't installed properly.

## Iteration 3: Cantina interior, black NPCs

- **Date:** 2026-MM-DD
- **Symptom:** NPCs in cantina interior render as solid-black silhouettes. Terrain + cantina structure + HUD all visible and correct.
- **Hypothesis:** Skeletal mesh pixel shader cbuffer alignment broken (Pitfall 2). NPCs use a different shader path than static geometry; the alignment issue affects only the skeletal-shader cbuffer.
- **Investigation:**
  - D3D11 debug-layer shows no validation errors → not API misuse, must be data
  - Grep'd `Direct3d11_LightManager.cpp` for the cbuffer struct → found `XMFLOAT3 directionalDir; XMFLOAT4 directionalColor;`
  - HLSL `cbuffer Lighting` declares `float3 directionalDir; float4 directionalColor;`
  - HLSL packs `directionalDir` to 12 bytes + 4-byte pad → `directionalColor` at offset 16
  - C++ struct: `XMFLOAT3` is 12 bytes, `XMFLOAT4` follows immediately at offset 12 → C++ thinks `directionalColor` at offset 12, HLSL reads it at offset 16. Misalignment.
- **Root cause:** Pitfall 2 (cbuffer 16-byte alignment).
- **Fix:** Changed `XMFLOAT3 directionalDir` → `XMFLOAT4 directionalDir_pad` (xyz = dir, w = unused) in Direct3d11_LightManager.cpp `LightingCB` struct + matching `float4` in shader. Static_assert(sizeof(LightingCB) % 16 == 0) still passes; runtime now correct.
- **Verification:** Cantina NPCs now render with proper lighting; terrain + structure unchanged; commit a1b2c3d.

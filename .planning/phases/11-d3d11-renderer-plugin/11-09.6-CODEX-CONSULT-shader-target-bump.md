## What we're trying to decide

The D3D11 plugin compiles all engine VSH/PSH shader assets via `D3DCompile`. Plan 11-05 set the VS target to `vs_4_0_level_9_3` (feature-level 9.3 compatibility mode, intended to map SM2.0/2.x source onto D3D11's compile pipeline without semantic surprises).

Today's smoke (post-Plan-11-09.5) surfaced the first asset-specific compile failure as Kenny advanced past login into character creation. CODEX's pre-Plan-11-09 consult predicted this exact category as "Iter-N asset-specific compile failure" — cascade-map item 3.

**Failing asset:** `vertex_program/a_specmap_bump_vs20_for_ps20.vsh`

**D3DCompile error verbatim:**
```
error X5615: Compiled shader code uses too many instruction slots (293).
Max. allowed by the target (vs_2_x) is 256.
```

Plus 18 `warning X3206: implicit truncation of vector type` warnings (likely pre-existing in the asset; not the FATAL trigger).

**Crash call chain:**
```
Direct3d11_VertexShaderData::compile  (FATAL_DX_HR)
  -> ShaderImplementation::createGraphicsData
  -> ShaderImplementationList::create
  -> ShaderEffectList::create
  -> StaticShaderTemplate (loading shader/ui_pedestal_asbe22.sht)
  -> ShaderTemplateList::fetch
```

**Crash context:**
- 30 sec uptime, 2790 main-loop iterations
- Loading ObjectTemplate: `object/tangible/ui/shared_ui_char_cust_pedestal.iff` (character-customization pedestal mesh)
- Login screen renders fine (Plan 11-09 half-magenta milestone); crash happens AFTER user advances to character creation

**This is reproducible** — any path through char-create / char-select hits this shader compile.

---

## Our current hypothesis

The `vs_4_0_level_9_3` target enforces SM2.0/2.x compatibility including its 256-instruction-slot ceiling. The shader has 293 instructions and busts the cap by 37 slots. The same shader compiles fine on real D3D9 drivers because they compile it natively under `vs_2_0` with vendor-extension headroom; the modern D3DCompile reference compiler is strict about the spec.

**Plan 11-09.6 scope candidate:** bump the VS compile target from `vs_4_0_level_9_3` to `vs_4_0` (or `vs_5_0`), accept whatever SM4 semantic mismatches surface, fix them per cascade.

The PS side (`Direct3d11_PixelShaderProgramData::compilePixelShaderFromHlsl`) currently targets `ps_4_0_level_9_3` (mirror situation; we'd bump symmetrically). PSH bytecode files (PEXE) are the Plan 11-05 "asset re-author" deferred item and are not in this consult's scope.

## Specific questions for CODEX

### Q1: Target choice — vs_4_0 or vs_5_0?

**Option A:** Bump to `vs_4_0`. Drops the 256-slot ceiling (4096 slots typical). Closer semantic neighbor to vs_2_x source. SM4 baseline.

**Option B:** Bump to `vs_5_0`. Same slot headroom + cleaner integer/double support + tessellation/SV semantics. SM5 baseline (matches Plan 11-05's PS HLSL helper at ps_5_0).

Both work on Kenny's hardware (RTX 3060, FL_11_0). Which is the architecturally cleaner target for compiling vs2.0/2.x source on a D3D_FEATURE_LEVEL_11_0 device, and why? The PS side already uses ps_5_0 in `compilePixelShaderFromHlsl` (Plan 11-05's HLSL pass-through helper); consistency would favor vs_5_0. But the existing target lineage (vs_4_0_level_9_3) suggests the original author intended 9.x compatibility. Was there a known reason to choose level_9_3?

### Q2: SM2 → SM4/5 semantic cascade — what should we expect?

After the target bump, what compile errors / behavioral differences should we expect to cascade from the SM2 source (POSITION semantics, fixed-function fog/lighting macros, conditional sampler binding, vertex blending matrices, etc.)? Plan 11-07 Iter-14 already added a `#pragma def` stripper (Rule E) to handle X4016 overlapping-register issues. What's the next-most-likely cascade error category?

Specific worry: SM4+ disallows certain SM2-era constructs (e.g. `texldb`, `texldp` instructions; `texkill` discard semantics; uniform-vs-varying ambiguities; sampler+texture as a single unit). Is the engine's HLSL source likely to trip these, and how aggressive is the cascade likely to be?

### Q3: Per-shader-variant compile strategy as an alternative?

Instead of a global target bump: could we add a compile-time fallback ladder — try `vs_4_0_level_9_3` first; if it returns X5615 (instruction overflow), retry with `vs_4_0`; cache the chosen target per-asset? This preserves the strict-9.3-compat semantics for assets that fit and only relaxes for the overflow cases.

Pros: minimum-blast-radius for assets that already compile clean.
Cons: introduces a per-asset state machine; doesn't address SM2→SM4 semantic differences proactively (a relaxed target may also produce subtly different bytecode that runs differently on the GPU).

Is this fallback ladder approach worth the complexity, or is a global bump cleaner long-term?

### Q4: D3DCompile flag choices

What flags should we pair with the target bump? Current Plan 11-05 compile invocation uses `D3DCOMPILE_OPTIMIZATION_LEVEL3 | D3DCOMPILE_ENABLE_STRICTNESS` (verify in source). Candidates:

- `D3DCOMPILE_ENABLE_BACKWARDS_COMPATIBILITY` — opt into legacy semantics (per fxc docs: "directs the HLSL compiler to compile against the next highest available software target for vertex shaders, if a software target is needed")
- `D3DCOMPILE_AVOID_FLOW_CONTROL` / `D3DCOMPILE_PREFER_FLOW_CONTROL` — affect instruction count
- `D3DCOMPILE_SKIP_OPTIMIZATION` — useful for debugging
- `/Gec` (legacy escapes) equivalent — only available via `D3DCOMPILE_ENABLE_BACKWARDS_COMPATIBILITY`

Should the Plan 11-09.6 compile invocation include `D3DCOMPILE_ENABLE_BACKWARDS_COMPATIBILITY` by default to ease the SM2→SM4 transition?

### Q5: PS bytecode (PEXE) coupling

Plan 11-05 noted: "engine ships pre-compiled D3D9 PEXE pixel-shader bytecode in .psh IFF chunks (DWORD * m_exe); D3D11 CreatePixelShader rejects D3D9 bytecode." Plan 11-09 fixed visibility via a magenta fallback PS — but real PS bytecode is still rejected.

If Plan 11-09.6 bumps VS target, does it surface any PS-side implications we should plan for (e.g. mismatched VS output / PS input signatures)? Or is the VS bump fully independent of the PEXE rejection?

# CODEX Response — 2026-05-21

**Headline verdict:** Global `vs_4_0` bump for Plan 11-09.6. NOT `vs_5_0`. NOT a fallback ladder. Bump PS helper to `ps_4_0` for symmetry.

**Stale-premise correction from CODEX:** the consult assumed PS helper is at `ps_5_0`; actual source has BOTH VS and PS targets at `*_4_0_level_9_3` (`Direct3d11_VertexShaderData.cpp:97` + `Direct3d11_PixelShaderProgramData.cpp:62`). My "consistency with ps_5_0" rationale in H1 was wrong. CODEX corrected from source.

## Q1 — vs_4_0 vs vs_5_0

**Verdict:** `vs_4_0`.

**Rationale:** Source comment at `Direct3d11_VertexShaderData.cpp:20` documents that vs_5_0 was previously attempted and hit legacy keyword problems (e.g. `point`). The project landed on `vs_4_0_level_9_3` specifically to preserve D3D9-era syntax compatibility. vs_4_0 drops the 256-instruction cap WITHOUT introducing the SM5 lexical/syntax exposure that already forced the project away from vs_5_0. Per Microsoft docs (https://learn.microsoft.com/en-us/windows/win32/direct3dhlsl/specifying-compiler-targets), `*_4_0_level_9_x` profiles implicitly use SM2.x-style limits including the 256-instruction ceiling — `vs_4_0` profile drops to standard SM4 limits (~4096 slots).

If bumping VS, bump PS helper to `ps_4_0` symmetrically. Affects fallback + future HLSL PS paths only; stock PEXE pixel shaders still rejected (Plan 11-05 caveat unchanged).

## Q2 — Expected cascade taxonomy (in order of likelihood)

1. **Legacy SM4 syntax/keyword fallout** — partly handled by Rule A at `Direct3d11_HlslRewrite.cpp:124` (`point/line/triangle` etc.).
2. **Register/constant layout fallout** — Rule D/E patterns at `Direct3d11_HlslRewrite.cpp:747` show this is fragile around `register(cN)`, `packoffset(cN)`, `#pragma def`. Moving from level_9 to full SM4 may change what FXC accepts or how it allocates globals.
3. **VS output / PS input mismatch at runtime** (NOT compile time). The magenta fallback PS only consumes `SV_POSITION` so it's safe; real future PS HLSL will need matching signatures.
4. **Runtime constant-binding mismatch** — higher profile bytecode may compile but read constants differently if the rewrite/cbuffer layout assumptions diverge from VSSetConstantBuffers expectations.
5. **PS-era constructs** (`texkill`, `texldp`, sampler/texture split) — **lower immediate risk** because current stock PS assets are D3D9 PEXE and rejected before HLSL compile. These become Phase 12 problems.

## Q3 — Fallback ladder

**Verdict:** NOT worth it.

A ladder from `vs_4_0_level_9_3` → `vs_4_0` makes the renderer run mixed shader semantic profiles. Complicates cache keys, diagnostics, constant-layout assumptions. The project has already accepted source rewriting and D3D11-specific shader semantics; a single global target is cleaner. Keep the profile in the cache key (already via D3D11_PROFILE).

## Q4 — Compile flag choices

**Verdict:** Keep `D3DCOMPILE_ENABLE_BACKWARDS_COMPATIBILITY` by default. Do NOT restore strictness.

Source already removed strictness (conflicts with backwards-compatibility; current comments record the X3116 failure). Recommended flag set:

- **Debug:** `D3DCOMPILE_DEBUG | D3DCOMPILE_SKIP_OPTIMIZATION`
- **Release:** `D3DCOMPILE_OPTIMIZATION_LEVEL3`
- **Always:** `D3DCOMPILE_ENABLE_BACKWARDS_COMPATIBILITY`
- **Always:** `D3DCOMPILE_PACK_MATRIX_ROW_MAJOR`

Do NOT add `AVOID_FLOW_CONTROL` or `PREFER_FLOW_CONTROL` yet — they change generated code and should be reserved for proven shader-specific problems. Per Microsoft docs (https://learn.microsoft.com/en-us/windows/win32/direct3dhlsl/d3dcompile-constants), `/Gec` enables older shaders to compile to 5_0 targets; strictness can reject legacy syntax.

## Q5 — PEXE coupling

**Verdict:** Mostly independent.

VS target bump affects VS bytecode + input-layout cache. Does NOT fix stock PS PEXE rejection. Current PS wrapper intentionally leaves D3D9 bytecode unbound and uses fallback magenta PS (`Direct3d11_PixelShaderProgramData.cpp:340`). Only coupling: signature compatibility once real HLSL PS replaces PEXE — Phase 12 territory.

<objective>
Resolve the FATAL Direct3d11_TextureData::lock failure on TF_RGB_888 (TextureFormat enum value 4) surfaced by Plan 11-09.8's smoke. Engine progressed from UpTime 50 / MainLoop 5568 (Plan 11-09.7) -> UpTime 73 / MainLoop 7649 (Plan 11-09.8) into Tatooine terrain load (the deepest world-load reach in Phase 11 to date), then crashed when EnvironmentBlock tried to lock a 24bpp RGB terrain texture.

Diagnosis:
- D3D11 / DXGI has NO 24bpp packed surface format (D3D9 has D3DFMT_R8G8B8; DXGI dropped it). Plan 11-04 translation table at `Direct3d11_TextureData.cpp:62` maps TF_RGB_888 to `DXGI_FORMAT_UNKNOWN` with comment "engine promotes via runtimeFormats."
- The create-time runtimeFormats promotion DOES fire correctly (lines 310-329) -- TF_RGB_888 textures are allocated under a wider promoted format like TF_XRGB_8888 (DXGI_FORMAT_B8G8R8X8_UNORM = 32bpp).
- BUT at lock time (line ~477), the code uses `lockData.getFormat()` (the engine's expected FILL format, TF_RGB_888 = 24bpp) to look up the DXGI format, gets UNKNOWN, FATALs.
- The engine fills the texture at 24bpp stride; the GPU surface is 32bpp. We need a CPU bridge buffer.

Fix approach:
- At lock time, when `lockData.getFormat() != m_destFormat` (engine fill format differs from allocated GPU format), allocate a CPU-side shadow buffer sized for the engine's fill format + dimensions. Return that buffer's pointer + the 24bpp stride to the engine.
- At unlock time, expand row-by-row from the CPU shadow into the GPU staging texture using the format-pair-appropriate conversion (3-byte RGB -> 4-byte BGRX or RGBX depending on m_destFormat layout). Then UpdateSubresource or Map + copy as today's unlock path does.
- Free the shadow buffer on unlock.

Scope ceiling: ONE iteration. If the lock-bridge introduces draw-time correctness issues (terrain rendering artifacts due to color-channel reordering), that's Iter-2 territory with a focused fix.

The D3D9 plugin doesn't need this bridge because D3DFMT_R8G8B8 is a real 24bpp D3D9 surface format. This is a DXGI-only regression of the D3D9 era.
</objective>

### Task 2 — Implement the lock-bridge

**2a.** Add header state to `Direct3d11_TextureData.h`:
```cpp
// Plan 11-09.9: CPU shadow buffer for 24bpp-engine-fill -> 32bpp-
// GPU-surface bridging. Allocated at lock time when lockData.getFormat()
// width differs from m_destFormat width; freed at unlock after the
// row-by-row expansion + GPU upload.
std::vector<uint8_t>   m_lockBridgeBuffer;
TextureFormat          m_lockBridgeFormat;  // engine's lock format (TF_RGB_888 etc.)
UINT                   m_lockBridgePitch;   // engine-expected row stride at lockBridgeFormat width
int                    m_lockBridgeWidth;
int                    m_lockBridgeHeight;
bool                   m_lockBridgeActive;
```

**2b.** Modify `Direct3d11_TextureData::lock` to detect the bridge case:
- If `lockData.getFormat() == m_destFormat` OR `translationTable[lockData.getFormat()] != DXGI_FORMAT_UNKNOWN` → existing path.
- ELSE if the lockData format is a known engine "small" format (TF_RGB_888 starts the list; expand as needed): set m_lockBridgeActive = true; compute pitch = bytes-per-pixel * width; resize m_lockBridgeBuffer to (pitch * height); return the shadow buffer pointer + pitch.
- Defensive FATAL only if the lockData format isn't TF_RGB_888 (or other recognized bridge formats) — preserves the surface-the-cascade pattern.

**2c.** Modify `Direct3d11_TextureData::unlock` to handle the bridge case:
- If m_lockBridgeActive: Map the actual GPU staging texture (D3D11_MAP_WRITE_DISCARD), iterate rows, expand 3-byte RGB → 4-byte target (handle BGRX vs RGBX channel order via m_destFormat inspection), copy. Unmap. Then existing CopySubresourceRegion / UpdateSubresource for the GPU upload as today's unlock does.
- Reset m_lockBridgeActive + free the shadow buffer.

**2d.** Channel-order handling: TF_RGB_888 is engine-order R,G,B (3 bytes). If m_destFormat is DXGI_FORMAT_B8G8R8X8_UNORM (BGRX), swap channels: dst[0]=src[2], dst[1]=src[1], dst[2]=src[0], dst[3]=0xFF. If m_destFormat is DXGI_FORMAT_R8G8B8A8_UNORM (RGBA), straight copy + alpha=0xFF. Pick based on m_destFormat introspection.

<risks>
1. **Channel-order misordering** -- TF_RGB_888 is engine order R,G,B; DXGI_FORMAT_B8G8R8X8_UNORM swaps to B,G,R. If we copy straight without swap, the texture will look color-channel-swapped (red things look blue, etc.). Mitigation: introspect m_destFormat in unlock and pick the right swap.

2. **Pitch mismatch under WRITE_DISCARD Map** -- D3D11's Map returns the GPU surface's pitch which may not equal width*bpp (alignment padding). Expansion must use the returned row-pitch, NOT just width*bpp. Mitigation: copy row-by-row using `mappedResource.RowPitch` for destination + our computed `m_lockBridgePitch` for source.

3. **Other gap formats besides TF_RGB_888** -- TF_DXT2, TF_DXT4, TF_P_8 also map to UNKNOWN. If the engine ever locks those at runtime, the bridge will fire but the conversion path won't handle them (DXT2/4 are compressed; P_8 is palettized). Mitigation: bridge case ONLY recognizes TF_RGB_888 in Iter-1; other UNKNOWN formats still FATAL with a clearer message. Iter-2+ extends as new cases surface.

4. **Cube/3D texture variants** -- m_isCubeMap and m_isVolumeMap paths may also lock; the bridge must handle them. Mitigation: confirm m_isCubeMap / m_isVolumeMap interaction during implementation; FATAL with a clear "Plan 11-09.10 territory" message if encountered in Iter-1.

5. **Engine concurrency assumptions** -- if the engine ever locks twice without unlock (or unlocks without lock), our m_lockBridgeActive flag becomes inconsistent. Mitigation: DEBUG_FATAL on double-lock; defensive reset on bad-state.

6. **D-05 invariant** -- D3D11 plugin only. Zero engine-side edits. Risk: zero.

7. **BSOD risk** -- pure CPU staging + format conversion. No new GPU resource allocations beyond the existing staging-texture lifecycle. Risk: zero.
</risks>

# CODEX Consult — Plan 11-09.13 Reflection-Driven Fallback-PS Variant Selection

**Context date:** 2026-05-22 (post-Plan-11-09.13 Iter-2 revert at `05eb45b1f`)

**Status:** Pre-Iter-3-implementation. Plan 11-09.13 Iter-1 attempted textured-passthrough fallback PS; reverted in Iter-2 after 483,451 linkage-error rejections. This consult drives the Iter-3 design choice.

---

## What we're trying to decide

The D3D11 renderer plugin (Phase 11) cannot consume the engine's pre-compiled D3D9 PEXE pixel-shader bytecode (`.psh` IFF assets ship as DWORD * `m_exe` SM 1.x/2.x bytecode; `CreatePixelShader` rejects D3D9 bytecode). Plan 11-05 documented this as a "Phase 12 asset re-author" punt and Plan 11-09 Iter-2 introduced an install-time **magenta fallback PS** that `applyPreDrawState` binds whenever `ms_currentPSData->getPixelShader()` is null — which is 100% of asset draws.

Today (Plan 11-09.12 close smoke) Kenny confirmed the engine fully loads Tatooine, runs frames continuously, plays audio, and the entire scene is solid magenta (every draw covered by the fallback PS). Plan 11-09.12 also fixed a dynamic-VB byte-offset bug — geometry is now correctly rasterized, just covered by magenta.

Plan 11-09.13 attempted to evolve the magenta fallback into a **textured-passthrough fallback** so the world becomes visually recognizable without requiring asset re-author:

```hlsl
Texture2D    t : register(t0);
SamplerState s : register(s0);
float4 main(float4 pos : SV_POSITION, float2 uv : TEXCOORD0) : SV_TARGET
{
    return t.Sample(s, uv);
}
```

The hypothesis: Plan 11-06's state cache already binds the diffuse texture at SRV slot 0 + sampler slot 0 via the engine's normal `setStaticShader` / per-pass-texture-stage flow, so the fallback PS would just sample whatever was bound and output it. World becomes textured (terrain, character skins, walls). Specular / normal-mapped / glow / per-pass effects remain missing (those need real per-asset PS = Phase 12), but visual parity should land at "substantially similar" per SPEC R6.

**Result:** 483,451 D3D11 debug-layer ERRORs in a single smoke session (up from 2 baseline). 100% identical:

```
D3D11 [ERROR] category=9 id=342: ID3D11DeviceContext::Draw: Vertex Shader -
Pixel Shader linkage error: Signatures between stages are incompatible.
The input stage requires Semantic/Index (TEXCOORD,0) as input, but it
is not provided by the output stage.
```

Every engine VS in the smoke session output a signature lacking `TEXCOORD0` at the exact semantic-index match. The PS-input-must-be-VS-output-subset rule is HARD in D3D11 — the Draw is *rejected* (not warned), so 483K rejected draws = zero rendering = solid black + clear color.

This is **exactly Plan 11-09.6 CODEX Q2 cascade prediction item 3** ("VS output / PS input mismatch at runtime") manifesting on the PS input / VS output side, in the dual to Plan 11-09.8's input-signature mismatch problem.

Iter-2 reverted to magenta. Iter-3 needs a real architecture for selecting the right fallback PS per draw based on the bound VS's output signature.

---

## Surface evidence

**The 483K linkage errors are NOT random — they're 100% on TEXCOORD0.** Sampled by `tail -n +<latest-session> stage/d3d11-debug.log | grep "[ERROR]" | grep -oE "Semantic/Index \([^)]+\)" | sort | uniq -c`:

```
483451 Semantic/Index (TEXCOORD,0)
```

That tells us:
1. No engine VS in the smoke output `TEXCOORD0` with the exact `Semantic=TEXCOORD, Index=0` match.
2. Some engine VSes likely output `TEXCOORD1`, `TEXCOORD2`, `TEXCOORD3`, etc. (Plan 11-09.8's TEXCOORD2 finding suggests at least some VSes use indices > 0 — though those were INPUT semantics, output may differ).
3. Some engine VSes may output zero TEXCOORD semantics (UI overlay shaders, debug lines, terrain wireframe, etc.).
4. Some engine VSes may use entirely different semantic naming conventions inherited from D3D9 (e.g. `COLOR` instead of `TEXCOORD0` for vertex colors).

**Plan 11-09.8 precedent: reflection-driven phantom-element layout.** That plan addressed the symmetric problem on the input side — VS declares `TEXCOORD2` input but VBFormat doesn't provide it. The fix:
- Install-time: create a 16-byte zero-filled `BIND_VERTEX_BUFFER` at slot 15 (the "phantom slot 15 zero buffer").
- Compile-time per VS: `D3DReflect` the bytecode; capture the input signature (filter SV_*).
- Layout-build-time: `Direct3d11_VertexBufferDescriptorMap::augmentWithPhantomElements` appends a `D3D11_INPUT_ELEMENT_DESC` at `InputSlot=15` for each VS-declared semantic the VBFormat doesn't provide.
- Draw-time: `applyPreDrawState` binds the phantom buffer at slot 15 unconditionally.

CODEX-endorsed C-lite/A2 hybrid; resolved the cascade + a downstream c0000005. Worked first-try.

**This Iter-3 problem is the dual** — we need to handle the case where the PS declares an input that the VS doesn't output (the dual of: VS declares an input the VB doesn't provide).

---

## Three architectural buckets, ordered by intrusiveness

### Bucket 1: Compile-2-pick-1 (simplest)

At install time, compile TWO fallback PS variants:

- **Variant M (Magenta):** `float4 main(float4 pos : SV_POSITION) : SV_TARGET { return float4(1,0,1,1); }` — current Plan 11-09 Iter-2 shape; works for any VS (only requires SV_POSITION which D3D11 always provides).
- **Variant T (Textured):** `Texture2D t : register(t0); SamplerState s : register(s0); float4 main(float4 pos : SV_POSITION, float2 uv : TEXCOORD0) : SV_TARGET { return t.Sample(s, uv); }` — works only for VSes that output TEXCOORD0.

At `applyPreDrawState`, when binding the fallback (i.e. `ms_currentPSData->getPixelShader() == null`):

- Reflect the current VS's output signature (cache the result in `Direct3d11_VertexShaderData::m_outputHasTexcoord0` at compile time, mirroring how Plan 11-09.8 caches the input signature in `m_requiredInputs`).
- If VS outputs TEXCOORD0: bind Variant T.
- Else: bind Variant M.

**Pros:** simplest. Two static install-time compiles. One bool per VS. Binary choice per draw. No GPU resource allocation overhead.

**Cons:** doesn't address the broader space — what about VSes that output `COLOR` but not `TEXCOORD0`? VSes that output `TEXCOORD1` but not 0? Each new fallback variant requires a code change. Probably OK for SWG's vertex-shader population scope (it's not infinite).

### Bucket 2: Per-VS PS cache

At install time, compile only Variant M (the safe baseline).

At first use of a VS (or at VS-compile time):
- `D3DReflect` the VS output signature.
- Build an HLSL string dynamically that declares PS inputs matching the VS outputs (or a useful subset).
- `D3DCompile` the per-VS PS once; cache in a `std::unordered_map<uint64_t /*VS bytecode hash*/, ID3D11PixelShader *>`.

At draw-time: pick the cached per-VS PS that matches the bound VS.

**Pros:** maximally flexible — any VS output signature is handled. Auto-extensible.

**Cons:** dynamic HLSL string generation is error-prone (semantic-name mapping, type inference for COLOR vs TEXCOORD etc., float2 vs float3 vs float4). Per-VS compile + cache add complexity. PS cache lifetime/invalidation matches VS cache (need invalidation alignment). Probably overkill for the immediate "make the world textured" goal.

### Bucket 3: Phantom-VS-output (mirror of Plan 11-09.8)

At install time, compile only Variant T (the textured one).

At VS-compile time:
- `D3DReflect` the VS output signature.
- If the VS does NOT output `TEXCOORD0`, rewrite the VS HLSL source (via `Direct3d11_HlslRewrite`) to add a `float2 texcoord0 : TEXCOORD0;` output to the VS output struct, zero-initialized in the VS body.
- Recompile.

OR, dual to slot-15 phantom buffer: at install time, set up some side-channel that synthesizes TEXCOORD0 into the PS input.

**Pros:** mirrors the proven Plan 11-09.8 pattern (input-side phantom → output-side phantom). One fallback PS variant — simple PS side.

**Cons:** modifying VS source is invasive. The Plan 11-09.8 input-side phantom works because the INPUT layout is plugin-side data structure (we control it); the OUTPUT signature is *baked into the VS bytecode* — we can't add a phantom output without recompiling the VS itself. That means rewriting the engine VS HLSL source pre-compile, which is fragile (the existing `Direct3d11_HlslRewrite` Rules A-E don't touch struct definitions; would need a Rule G that augments output structs). High implementation cost.

There's no clean equivalent of "bind a zero buffer at slot 15" for VS output → PS input.

---

## Our current hypothesis

**Bucket 1 (Compile-2-pick-1)** is the right answer for Iter-3:

- Minimum-blast-radius: 2 install-time compiles + 1 bool per VS + 1 if-branch in `applyPreDrawState`.
- Symmetric to Plan 11-09.8's per-VS reflection caching (`m_requiredInputs` on input side ↔ `m_outputHasTexcoord0` on output side).
- Extensible: if a new "Variant X" is needed later (e.g. textured-with-color = TEXCOORD0 + COLOR0), add a third variant + extend the selection bool to an enum.
- Per-VS reflection at compile-time is already proven by Plan 11-09.8 (`Direct3d11_VertexShaderData::install` already calls `D3DReflect` on the bytecode).

Bucket 2 is more flexible but the dynamic HLSL generation is real complexity that's not clearly worth it for the "make it textured" goal. Bucket 3 is the most architecturally elegant but is invasive (VS source modification).

## Questions for CODEX

### Q1 — Architecture choice
Bucket 1 vs 2 vs 3 — does our hypothesis hold? Specifically:
- Is Compile-2-pick-1 the right shape, OR does the eventual asset-shader population suggest Bucket 2's per-VS dynamism becomes necessary?
- Is there a Bucket-3 mechanism we haven't considered (e.g. `D3D11_APPEND_ALIGNED_ELEMENT` magic, ID3D11VertexShader linkage subgraphs, geometry-shader-as-output-fixer) that elegantly handles output mismatch?

### Q2 — Variant T's input signature
We picked `float2 uv : TEXCOORD0` as Variant T's input. Is that the right granularity?
- Should we also try `float3 uv : TEXCOORD0` (for VSes that output 3-component texcoords, e.g. cube samplers)? D3D11 PS input size has to match VS output size *for that semantic*, doesn't it?
- Should the PS input declaration use `centroid` / `noperspective` modifiers, or do those affect the linkage subset check?
- If `TEXCOORD0`'s D3D11 component-count must match the VS exactly, we may need a per-component-count variant in addition to the per-semantic variant. How fast does the variant matrix explode?

### Q3 — Reflection storage
Should the per-VS output-signature cache live on `Direct3d11_VertexShaderData` (mirroring `m_requiredInputs` from Plan 11-09.8), on `Direct3d11_PixelShaderProgramData` (where the fallback lives), or in `Direct3d11_StateCache` (where it'd be consulted at apply-time)?

The cache key options are: (a) VS bytecode hash, (b) VS pointer identity (raw `ID3D11VertexShader *`), (c) a bool member on `Direct3d11_VertexShaderData`.

Bool member is fastest at apply-time (no hash lookup; just a method call), matches Plan 11-09.8's `m_requiredInputs` precedent. Sound?

### Q4 — Other semantic mismatches
Are there other PS-input semantics besides `TEXCOORD0` that we should handle in the fallback variant matrix? E.g. `COLOR0` (vertex color), `NORMAL` (lighting), `WORLDPOS` (specular)? 
- The current Plan 11-09 Iter-2 magenta fallback declares only `SV_POSITION` — works for everything but visually useless. 
- If we add only Variant T (`+TEXCOORD0`), the world becomes textured-but-flat-lit.
- Going further (`+NORMAL`, `+lighting`) starts to *be* a real PS implementation, which is Phase 12 territory.

What's the natural stopping point for "fallback PS coverage"? Just TEXCOORD0? Or also COLOR0 (for vertex-color geometry like UI / particles)?

### Q5 — Iter-3 acceptance bar
What's the right success criterion for Iter-3 smoke?
- Hard bar: 0 id=342 errors in a single smoke session (vs Iter-1's 483K).
- Visible bar: Kenny sees textured world geometry instead of solid magenta on most surfaces.
- Failure prediction: if MOST engine VSes don't output TEXCOORD0 either, then "most draws still magenta + only a few textured." Would that be a "Iter-3 success, Iter-4 needed for COLOR0 coverage" outcome, or a "rethink fallback strategy entirely" signal?

### Q6 — Stale-premise check
Anything in this consult document that should be re-examined as stale or wrong? The Plan 11-09.6 CODEX consult caught a stale premise about `vs_5_0` vs `vs_4_0_level_9_3` that turned out to be load-bearing — same kind of check welcome here.

## Response section (CODEX fills below)

**CODEX verdict (2026-05-22, pasted by Kenny):**

High-confidence answer: Bucket 1 is the right Iter-3 architecture, but the proposed TEXCOORD0 textured variant is not enough by itself, and there is one stale premise: current D3D11 `StaticShaderData::apply()` does not bind per-pass diffuse textures yet. It only wires VS/PS pointers. See `src/engine/client/application/Direct3d11/src/win32/Direct3d11_StaticShaderData.cpp:202`. So even a link-compatible textured fallback can only produce useful output if SRV slot 0 is already valid through some other path.

**Q1 — Architecture.** Use Bucket 1 for Iter-3, but implement it as "compile a small fixed matrix and select by reflected VS output," not just "TEXCOORD0 bool." Bucket 2 is too much for this phase. Bucket 3 has no clean D3D11 equivalent to phantom IA input: VS outputs are baked into bytecode. `D3D11_APPEND_ALIGNED_ELEMENT` only affects input-layout byte offsets, not shader-stage linkage. A geometry shader could synthesize missing outputs, but that is effectively a new pass-through GS system with topology/semantic/perf risks, not an elegant fallback.

**Q2 — Variant T signature.** Do not assume `float2 uv : TEXCOORD0` is broadly compatible. Microsoft's signature docs say adjacent stages must be compatible exact-match subsets by parameter type and semantic. Reflection exposes `Mask`, `ReadWriteMask`, semantic name/index, and component type. Use that. At minimum, select textured only when the VS output has TEXCOORD0, float component type, and enough xy components for the PS's actually declared/read input. Source: Microsoft Learn on shader signatures and `D3D11_SIGNATURE_PARAMETER_DESC`:
- https://learn.microsoft.com/en-us/windows/win32/direct3dhlsl/dx-graphics-hlsl-signatures
- https://learn.microsoft.com/en-us/windows/win32/api/d3d11shader/ns-d3d11shader-d3d11_signature_parameter_desc

`centroid` / `noperspective` should be avoided for fallback variants unless reflection proves a need. They can affect interpolation metadata; they do not solve missing semantic linkage.

**Q3 — Reflection storage.** Store this on `Direct3d11_VertexShaderData`. That matches the existing `m_reflectedInputs` precedent in `src/engine/client/application/Direct3d11/src/win32/Direct3d11_VertexShaderData.h:112` and the current `D3DReflect` site in `src/engine/client/application/Direct3d11/src/win32/Direct3d11_VertexShaderData.cpp:547`. I would store a compact reflected-output vector or enum, not only a bool, because the 483K failures already suggest TEXCOORD0 may not exist in the smoke population.

**Q4 — Other semantics.** Natural Iter-3 stopping point: safe magenta plus one or more texture-coordinate passthrough variants selected only when link-compatible. Do not chase NORMAL, WORLDPOS, lighting, etc.; that becomes Phase 12. COLOR0 is worth considering as a separate fallback only after reflection telemetry shows it is common and useful, especially for UI/particles/debug geometry. Do not combine TEXCOORD0 + COLOR0 until each single-purpose variant has proven value.

**Q5 — Acceptance bar.** Hard bar: 0 debug-layer id=342 errors. That is non-negotiable. Visible bar: any textured geometry is a success **only if SRV binding is also confirmed**. If most draws remain magenta because most VSes lack TEXCOORD0, that is not an architecture failure; it means Iter-4 needs either different semantic variants or upstream VS/texture-binding work. If zero draws become textured, **first suspect stale premise**: no per-pass texture binding in D3D11 StaticShaderData.

**Q6 — Stale premise.** The main stale/wrong premise is: "Plan 11-06 state cache already binds diffuse texture at SRV slot 0 via normal `setStaticShader` / per-pass texture-stage flow." In the checked code, `D3D11 StaticShaderData::apply()` explicitly says material/texture binding is later work and only applies VS/PS pointers. That must be fixed or instrumented before judging textured fallback success.

**Recommended Iter-3 shape:**

1. Reflect and log VS output signatures once per VS.
2. Add `Direct3d11_VertexShaderData` reflected-output metadata.
3. Compile magenta plus a conservative textured variant.
4. Bind textured only when reflection proves linkage compatibility.
5. Keep magenta as the universal fallback.
6. Add a one-shot SRV0/sampler0 validity diagnostic when textured fallback is selected.

## Local verification (post-CODEX, pre-Iter-3 implementation)

Both CODEX premises independently checked against HEAD `ee423afb5`:

- **Stale-premise confirmed at `Direct3d11_StaticShaderData.cpp:208-248`** — `apply()` only sets VS+PS pointers via `Direct3d11_StateCache::setCurrentVSData` / `setCurrentPSData`. The inline comment explicitly reads "CODEX scoped Iter-1 to VS+PS pointers only; those features land Iter-3+ as visual-correctness symptoms surface." This means a textured-passthrough fallback PS that compiles + binds successfully still samples whatever SRV slot 0 happened to hold from earlier engine state — not the per-draw diffuse texture. Iter-3 acceptance must include the SRV0 diagnostic CODEX called out, or we won't be able to distinguish "linkage works but SRV is empty/stale" from "selection logic doesn't pick Variant T."
- **Reflection precedent verified at `Direct3d11_VertexShaderData.h:113` + `.cpp:547`** — `m_reflectedInputs` plus the `D3DReflect` block that filters `SV_*` system values and captures `SemanticName / SemanticIndex / ComponentMask / ComponentType` is exactly the template to mirror for outputs. Iter-3 adds a parallel `m_reflectedOutputs` populated by the same code path's twin (`reflector->GetOutputParameterDesc`).

These confirm Iter-3 scope: parallel reflection block on outputs + a textured Variant T compile + selection logic + SRV0 diagnostic. SRV0 binding itself stays Phase 12 / later-Iter-N work per the existing `apply()` comment — we surface the gap with telemetry rather than fixing it inside 11-09.13.

- **Date:** 2026-05-18
- **Symptom:** Kenny smoke launch under `client_d.cfg rasterMajor=11` (post-Iter-3 profile swap to `vs_4_0_level_9_3`) FATAL'd at the SAME X3000 signature as Iter-3, only the profile string in the message body differs:
  ```
  FATAL: Direct3d11_VertexShaderData: D3DCompile vs_4_0_level_9_3 'vertex_program/2d.vsh' failed:
    vertex_program/include/vertex_shader_constants.inc(81,25-29):

  ```
  Crash dump: `stage/SwgClient_d.exe-unknown.0-20260517131430.txt`. Iter-3's profile downgrade succeeded at the toolchain layer (D3DCompile accepted the level_9_3 target string and ran the lexer) but the lexer rejected the same token. Iter-3's hypothesis -- "level_9_* profiles relax SM5 keyword reservations" -- was wrong: the reservation lives at the LEXER level, not at the profile-feature-gating level.
- **Hypothesis:** **Option (3) from Iter-3's notes -- textual whole-word rewrite in the include handler.** The d3dcompiler_47 HLSL lexer classifies identifiers BEFORE any profile-specific code paths run. Once the lexer sees `point` (followed by non-identifier characters), it emits a `point`-keyword token. Profile-feature gating (which decides whether geometry shaders are available) runs LATER on the token stream -- by which point the keyword classification is fixed. Iter-3's profile swap moved the toolchain to a profile that doesn't SUPPORT geometry shaders, but did not change the lexer's identifier table. The only path that avoids the lexer-level rejection is to never present the bare identifier to the lexer in the first place -- i.e., rewrite the source bytes before D3DCompile sees them. Pitfall classification: **novel** (continues the D3D9-to-D3D11 toolchain-compat theme of Iter-2/3; not one of the 8 RESEARCH-documented Pitfalls).
- **Investigation:**
  - Read the Iter-3 crash dump in full -- confirmed identical-root call stack to Iter-2/Iter-3 (the engine reached D3DCompile with the include resolved AND with the level_9_3 profile string -- the only failure mode change is that the FATAL message's profile-string slot reads `vs_4_0_level_9_3` instead of `vs_5_0`).
  - Cross-referenced d3dcompiler internals (DXSDK + community lexer behaviour notes) -- the geometry-shader primitive-type keywords (`point`, `line`, `triangle`, `lineadj`, `triangleadj`) are added to the lexer's reserved-identifier table when d3dcompiler_47 initializes -- INDEPENDENT of which profile string the caller eventually passes. Profile-specific gating decides "can you use a GS primitive declaration at all" but the reservation that promotes the identifier to a keyword is global.
  - Verified `Direct3d11_CompileIncludeHandler::Open()` is the cleanest seam to apply the rewrite for `#include`-resolved content: it already owns the heap buffer that goes to D3DCompile, and the rewrite-then-return pattern fits naturally into the existing read-then-return flow.
  - Verified `Direct3d11_ShaderCache::hashSource` (Plan 11-05) hashes the ORIGINAL source bytes + defines (the include handler runs LATER, during D3DCompile's parse phase, so the rewritten bytes never enter the FNV-1a accumulator). Cache invalidation therefore needs a separate trigger -- a `D3D11_REWRITE_VERSION` define mirroring the Iter-3 `D3D11_PROFILE` define pattern, so future keyword-list changes auto-invalidate stale `.cso` entries.
  - Confirmed HLSL identifier character class is `[A-Za-z_][A-Za-z0-9_]*` (same as C; HLSL is case-sensitive). Whole-word matching for the rewrite needs left-boundary + right-boundary checks against that class. `endpoint`, `linear`, `pointSize`, `triangleStrip_x` etc. must NOT match.
  - Confirmed HLSL has no real string-literal type (no need to skip `"foo"` blocks during the scan) and D3DCompile strips comments before lexing (so rewriting inside comments is harmless).
- **Root cause:** The HLSL lexer in d3dcompiler_47 reserves SM4+ geometry-shader primitive-type keywords (`point`, `line`, `triangle`, `lineadj`, `triangleadj`) at the lexer level, independent of profile-string targeting. SWG's D3D9-era HLSL source content (TRE-archived `.inc` files) uses these identifiers as ordinary variable / constant / sampler-state names. The lexer rejects them with X3000 before profile-specific feature gating runs.
- **Fix:** Apply a textual whole-word rewrite of the 5 reserved keywords -> `<keyword>_id` inside `Direct3d11_CompileIncludeHandler::Open()` after `readEntireFileAndClose()` and BEFORE returning the buffer to D3DCompile. Two-pass implementation in a new `Direct3d11_CompileIncludeHandlerNamespace`:
  1. `computeRewrittenLength(src, length)` -- single-pass scan returning required output size. Walks the buffer byte-by-byte; at each position checks left boundary (preceding char is non-identifier or BOF), tries each of the 5 keywords as a candidate prefix, verifies right boundary (following char is non-identifier or EOF). On match: adds `kw.length + 3` (`_id` suffix) to output size + advances past the keyword. On no match: adds 1 + advances 1.
  2. `emitRewritten(src, length, dst, dstCapacity)` -- emits the rewritten bytes. Mirror of the first pass; copies the keyword + appends `_id` on match, copies the single byte on no-match. `DEBUG_FATAL` overflow guards verify the second pass's writes stay within the precomputed capacity.
  3. `applyKeywordRewrite(inBuffer, inLength, &outLength, diagName)` -- thin orchestrator. Returns the original buffer untouched when `computeRewrittenLength == inLength` (no occurrences; fast path -- no allocation cost). Otherwise allocates a fresh `new unsigned char[rewrittenLen]`, calls emitRewritten, deletes the original buffer (we own it via `readEntireFileAndClose`), returns the new buffer. Close() delete[]'s whatever pointer Open() returned, so the lifetime contract is preserved.

  Three-axis fix:
  1. **Axis 1 (revert):** revert the Iter-10 THROWAWAY dumps + the orchestrator's late-Iter-10 / early-Iter-11 inline extension to 5 includes per the prearranged THROWAWAY closeout. Mirrors the Iter-8 -> Iter-9 and Iter-9 first-commit revert pattern. Delete the stage/shader-* dump artifacts as cleanup-as-you-go (gitignored, but discipline matters). The orchestrator's pre-bumped REWRITE_VERSION "7" -> "8" stays forward (version monotonically advances; the rule change in Axis 2 bumps further to "9").
  2. **Axis 2 (context-aware rewrite):** add a line-scoped `sawColonThisLine` bool state to both passes of the scanner. The flag is FALSE at start-of-buffer; set TRUE the first time a `:` is encountered on the current line; reset FALSE on every `\n`. Rules B/C take this flag as a new parameter and return 0 (no match) when the flag is FALSE. The intuition: the first `:` on a line opens a semantic block. On a global declaration shape `float4 foo : register(c0);` the FIRST `:` IS the register binding -- legal and required, do not strip. On a struct-member stacked-semantic shape `: POSITION0 : register(v0);` the FIRST `:` is the user semantic (`: POSITION0`) and the SECOND `:` is the SM4+-illegal location semantic -- strip the second. The flag distinguishes the two cases textually without backward scanning.
  3. **Axis 3 (cache invalidation):** REWRITE_VERSION "8" -> "9" in both VS + PS compile sites. Rules unchanged in Axis 1's revert (it just removes dumps); rules CHANGE in Axis 2 (now context-aware); rule-change bump rides Axis 2/3 in the same commit.

  Pitfall classification: **continued textual-rewrite arc** (Iter-2/3/4/7/8/9/10/11); not one of the 8 RESEARCH-documented Pitfalls. Iter-11 evolution: Iter-7 hardcoded `c`; Iter-8 expanded to `[bcstu]`; Iter-9 added `v` -> `[bcstuv]` (canonical 6-letter set); Iter-11 keeps the 6-letter set but adds a context-aware guard so the rule only fires on stacked-semantic struct-member positions. The rule is now BOTH letter-exhaustive (Iter-9) AND context-aware (Iter-11).

  - Re-read the X3202 error text. The wording is `error X3202: location semantics cannot be specified on **members**`. The key word is **members** -- not "globals", not "declarations", but "members". This is the SM4+ HLSL grammar distinction: `: register(...)` on a struct member is a location semantic in member position, which SM4+ rejects (input-layout binding moved to the IA stage via D3D11_INPUT_ELEMENT_DESC); `: register(...)` on a global declaration is the canonical D3D9/D3D11 register binding syntax, which SM4+ accepts. Iter-7..Iter-10 missed this distinction by stripping ALL occurrences of the syntax regardless of position.
  - Verified the context-aware rule design covers all four scenarios in the corpus:
    - **Global single-colon:** `float4x4 mat : register(c0);` -- the `:` is the FIRST on the line. `sawColonThisLine` is FALSE at the matching `:` position. Rules B/C return 0. Binding preserved. CORRECT.
    - **Struct-member stacked-semantic:** `float4 position : POSITION0 : register(v0);` -- the FIRST `:` (before POSITION0) sets the flag TRUE. The SECOND `:` (before register) sees the flag TRUE. Rules B/C match. `: register(v0)` stripped. `: POSITION0` preserved. CORRECT.
    - **Struct-member single-semantic (no register):** `float4 position : POSITION0;` -- the `:` is the FIRST on the line. Flag is FALSE at match. Rules B/C return 0. `: POSITION0` preserved. CORRECT.
    - **Function-parameter semantic:** `Output main(Input vertex : POSITION0)` -- the `:` is the FIRST on the line. Flag is FALSE at match. Rules B/C return 0. `: POSITION0` preserved. CORRECT.
    - **Bitfield (legacy C-style; D3D HLSL is unlikely but technically a `:` consumer):** `int field : 4;` -- the `:` is the FIRST on the line. Flag is FALSE. Rule B's pattern `:\s*[bcstuv]\d+` requires the letter `[bcstuv]` after the `:` (digit `4` is not in the set, so Rule B doesn't match anyway). The context check makes this doubly-safe. CORRECT.
  - Verified Rule-A consumed bytes can't perturb the flag: Rule A only matches `point|line|triangle|lineadj|triangleadj`, all pure-alpha (lowercase a-z, no `:`, no `\n`). When Rule A advances by `matchA` bytes, none of those bytes are `:` or `\n`. The flag is preserved across Rule-A matches. CORRECT.
  - Verified Rule-B/C consumed bytes can't perturb the flag: after the leading `:` (which we explicitly set TRUE for), the remaining consumed bytes are whitespace + optional `register` literal + optional `(` + register-type letter + digits + optional `)`. None of those are `:` (HLSL's `register` keyword is pure-alpha; the parentheses + letter + digit are all single-character non-`:`) or `\n` (whitespace can include `\n` -- WAIT, this is an edge case worth checking).
  - **Edge case: whitespace within Rule C's match can include `\n`.** `isHlslSpace(c)` returns true for `' '`, `'\t'`, `'\r'`, `'\n'`. Rule C's pattern is `:\s*register\s*\(\s*[bcstuv]\d+\s*\)`. If the source has `: register(\nc0\n)` with newlines inside the register() call (legal but exotic HLSL formatting), Rule C's matcher would consume those newlines as part of `\s*`. Our scanner state update logic doesn't notice these consumed newlines (we set `sawColonThisLine = true` after the match, but we should also have reset on the newline). However: (a) this would be incorrect IFF a subsequent line on the same buffer-position-after-match wanted to be in fresh-line context -- but after consuming the `)`, the scanner resumes byte-walking and will see whatever follows the `)`; (b) more importantly, the SWG HLSL corpus does NOT use multi-line register bindings (verified against the dumped vertex_shader_constants.inc + 2d.vsh main source; all `register(...)` calls are single-line). This is documented in the phase-context as an acceptable trade-off; if surfaces in Iter-12+, extend the matcher to track internal newlines.
  - **Edge case: comments containing `:` or `\n` between match attempts.** D3DCompile preprocesses comments out before lexing, but our rewrite scanner sees them. `// comment with : colon` on a line sets the flag TRUE (the `:` in the comment counts). Then a subsequent literal line `float4 foo : register(c0);` after the newline would see flag FALSE (reset on `\n`). Comment-internal `:` doesn't cause incorrect over-stripping. Acceptable.
  - **Edge case: macro expansion.** Our rewrite runs BEFORE D3DCompile's preprocessor. If a macro in `vertex_shader_constants.inc` expanded to `: register(c0)` text only inside D3DCompile, our rewrite would not see it. The Iter-10 dumps showed no macro-defined register bindings in the include corpus -- all `register()` text is literally in the source bytes. This hypothesis (Iter-10's #2) is REJECTED by the dump evidence. The over-strip is straightforward textual, not macro-driven.

- **Root cause:** Iter-7's Rules B/C were designed for the struct-member case (the X3202 site at `2d.vsh:8`) but applied unconditionally to any `:\s*register(...)` occurrence. Iter-8/9 expanded the letter set without revisiting the context-blindness defect. When Iter-9 cleared X3202 by correctly stripping the struct-member `: register(v0)`, the same rule applied to vertex_shader_constants.inc's 15 GLOBAL `: register(cN)` declarations, deleting their bindings to whitespace. D3DCompile's auto-allocator then had to place ~15 unannotated globals + a multi-register `LightData` struct (containing nested fixed-size arrays of point/parallel/spot lights) into the constant register pool and could not find a non-overlapping placement -> X4016 "overlapping register semantics not yet implemented 'c0'". The fix is to make Rules B/C context-aware via a line-scoped `sawColonThisLine` flag: strip only the SECOND-or-later `:` on a line (struct-member stacked semantic, SM4+ rejects), preserve the FIRST `:` on a line (global declaration register binding, SM4+ accepts).

### CODEX + Cursor convergent verdict on Iter-37A consult (2026-05-23)

Consult doc: `11-09.15-CODEX-CONSULT-universal-w-negative-matrix-packing.md`

**Both advisors closed the row/column-major packing hypothesis.** Verified against current source:
- `D3DCOMPILE_PACK_MATRIX_ROW_MAJOR` is set at `Direct3d11_VertexShaderData.cpp:471` (Plan 11-08 Iter-1.5) and `Direct3d11_PixelShaderProgramData.cpp:210`
- VS bytecode disassembly in `stage/stage/vs-disasm-all.txt` confirms `row_major float4x4 objectWorldCameraProjectionMatrix` / `row_major float4x4 objectWorldMatrix`
- WVP composition at `Direct3d11_StateCache.cpp:317-318` is `(P*V)*W`, matches D3D9 VSPS baseline per both advisors' independent inspection

**Cursor's structural observation:** All three captured flushes have `c2[0..2] == c3[0..2]` — that's the SOE perspective matrix structure (P has `[0,0,1,0]` in its row-3-or-column-3) propagating through `(P*V)*W`. Not a bug.

**The negative clip.w math is real but its cause is downstream of packing.** Advisors converged on three remaining candidates:
1. Stale shader cache (cheapest test)
2. WVP composition wrong for SOE matrix layout (despite matching D3D9 source-order)
3. V or P input bug in the cached shadow matrices

| Dumped | Why |

| `clip_PVW_row` | Current WVP, row-vector left-multiply |
| `clip_WVP_col` | Alt composition, column-vector right-multiply |
| `clip_WVP_row` | Alt composition, row-vector left-multiply |

Each clip dump tags `w_sign=POSITIVE` or `w_sign=negative` for quick scan.

**Test point (0, 1.7, 0, 1) is "head height above model origin"** — a vertex that for the player should be in front of the camera (third-person view typically has camera behind/above player, so player head is in view). At least one of the four interpretations MUST produce `w_sign=POSITIVE`. Whichever one does tells us the correct convention.

### Decision matrix on the Iter-38A smoke

| Which clip has POSITIVE w | Implication | Fix |
|---|---|---|
| `clip_PVW_col` | Current code is correct. Bug must be in matrix CONTENT (V or P wrong) | Diff V/P against D3D9 baseline values |
| `clip_PVW_row` | Composition correct but bytecode wants transpose | Transpose at upload OR rebuild bytecode without /Zpr |
| `clip_WVP_col` | Composition is wrong; should be `W*V*P`; convention column-vector | Swap order in `composeSlot0Shadow` |
| `clip_WVP_row` | Composition wrong AND convention is row-vector | Swap order AND verify bytecode multiply pattern |
| None | Pure matrix content corruption | Deep V/P input investigation |
| All four | Bug is somewhere else entirely (input layout, vertex stream, etc.) | Pivot |

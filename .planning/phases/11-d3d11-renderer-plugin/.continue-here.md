## The c0000005 crash — root cause (CONFIRMED, fix pending)

**It is NOT the async .mgn loader.** It's a **use-after-free in the main-thread mouse-over/targeting collision test**, and it's **D3D11-specific by mechanism**.

- **Crash site:** `SoftwareBlendSkeletalShaderPrimitive::collide` (`SoftwareBlendSkeletalShaderPrimitive.cpp:817`), reached from `SwgCuiHud::update → findAllTargettableObjects → ClientWorld::collide → SkeletalAppearance2::collide`. The `.mgn` line in the dump is a misleading breadcrumb.
- **Bug:** `collide` reads a `m_systemStream` vertex buffer that an intervening LOD rebuild/eviction (`SkeletalAppearance2::rebuildMesh` :2260 / `unloadUnusedResources` :4405) already freed. cdb proved the iterator's vftable is valid but all 3 data members hold coordinate-float bit patterns (freed memory reused) → `getPosition()` returns a `Vector&` at a garbage address → AV.
- **Why D3D11 only:** `SoftwareBlendSkeletalShaderPrimitive::install` branches on `ms_useMultiStreamVertexBuffers`. Our D3D11 plugin reports `supportsStreamOffsets=false` / `maxStreamCount=1` → flag FALSE; D3D9 on real HW → TRUE. Opposite branches; the free-window only opens on D3D11's single-stream path.
- **EMPIRICALLY CONFIRMED:** D3D9 repro (rasterMajor=5) ran the SAME NPC-heavy cantina mouse-over loop with **no crash, no visual anomalies** (Kenny, 2026-05-24). D3D11 crashed 5× at the same activity.
- **Separate/out-of-scope:** `0x087a` @ `77454984` (always `armor_marauder…belt.mgn`) is a DIFFERENT, cross-client crash — ignore it for this fix.

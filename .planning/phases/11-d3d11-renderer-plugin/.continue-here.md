## Diagnosis summary

**Iter-4 Test A (CPU-side VB snapshot at unlock-time):** surfaced TWO distinct vertex format patterns flowing through `drawTriangleFan`:

| Pattern | Caller | Position layout | Stride 24 layout | Frequency |
|---|---|---|---|---|
| 1 | UI/2D (e.g., CuiWidgetGroundRadar, splash) | **Pixel coords** (-0.5..1023.5, -0.5..767.5, 1.0) | `XYZRHW + UV` (xyzw float4 + uv float2) | First ~2000 fans/session |
| 2 | In-world UI overlays | **Clip space** (-0.65..0.65, 0..0.65, 0) | `XYZ + COLOR + UV` (xyz float3 + BGRA u32 + uv float2) | Ongoing gameplay |

Pattern 1's "color"-byte = `0x3F800000` = IEEE-754 float 1.0 bit pattern = the RHW (w) component, NOT a BGRA color. Pattern 2's "color"-byte = real BGRA u32 with varying alpha.

**Iter-4 Test B (fan-IB bypass):** with `kBypassFanIBForTestB=true`, drawTriangleFan reverts to pre-Plan-11-09.15 `Draw(vertCount, 0)` TRIANGLELIST. Visual confirmed:
- Pattern 1 (XYZRHW) callers STILL produce stretched radial triangles → bug is upstream of fan-list expansion
- Pattern 2 (clip-space) callers render as half-quads → the pre-Plan-11-09.15 half-screen artifact, working as expected

Eliminates H4 from the Iter-3 hypothesis matrix (Plan 11-09.15 = unmasker, not cause).

**Mechanism:** D3D9's `D3DFVF_XYZRHW` bypasses the VS entirely under FFP and uses `D3DDECLUSAGE_POSITIONT` for FFP / `D3DDECLUSAGE_POSITION` for VSPS. D3D11 has no XYZRHW bypass — every vertex goes through the VS. Our `Direct3d11_VertexBufferDescriptorMap` correctly identifies the format (emits `POSITION float4`), but downstream the VS bytecode is multiplying pixel-coord inputs through worldViewProj, producing nonsense clip-space.

The radial signature emerges because pixel `(-0.5, -0.5)` × cameraWorldViewProj ≈ world origin in clip space — the "world anchor" you see on screen.

## Fix paths under consult

1. **Path A — Shader-level fix** in `Direct3d11_VertexShaderData::compile` HLSL injection (scope the `POSITION → SV_POSITION` macro to output struct only), or modify SWG's stock UI `.vsh` bodies to handle transformed input correctly.
2. **Path B — Engine-side CPU conversion** — structurally hard to do cleanly given the lock/unlock lifecycle. Likely not viable.

Our read: **Path A or C**. Awaiting CODEX validation + recommendation.

## Adjacent latent-bug noted (NOT addressed in this iter)

`Direct3d11_DynamicVertexBufferData::beginFrame()`, `Direct3d11_DynamicIndexBufferData::beginFrame()`, and `Direct3d11_StaticShaderData::beginFrame()` are defined but never called by `Direct3d11_Device::beginScene`. D3D9 calls all of them per-frame (`Direct3d9.cpp:2484-2487`).

**Mitigating:** D3D9 ships with `discardDynamicBuffersAtBeginningOfFrame=false` default → D3D9's `beginFrame` is effectively a no-op too. So this isn't the radial cause.

Worth fixing in a follow-up plan for parity hygiene; not blocking on this consult.

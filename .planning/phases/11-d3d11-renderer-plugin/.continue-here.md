- **No D3D11 validation errors** in the InfoQueue (no draw rejections, no IB-format mismatches).

Per-call diagnostic from Iter-2 (first 5 fan draws):
```
fan-draw#1..#5: vertCount=4 triCount=2 VB=0x1F4DF1EC stride=24 offset=0 VBValid=1
                IB=0x00000000 IBValid=0 IBFormat=57 IBOffset=0 IBIndexCount=0
                VBVectorActive=0 VBVectorStreamCount=0
```

ALL 5 captured fans use the SAME VB at offset=0. VB=0x1F4DF1EC is the shared dynamic ring buffer (`Direct3d11_DynamicVertexBufferData::getSharedRingBuffer()`). Stride=24 = position(12) + color(4) + UV(8), matches the position+color+UV format used by CelestialAppearance, GameBitmap, BinkVideo, etc.

**Interpretation:** Iter-2 captured 5 startup-frame fan draws (intro/loading state). Each is the first lock of a new frame → discard path → `ms_used=0` → m_offset=0. NOT a sharing bug; just startup noise. Iter-3's sampling extends capture into in-world frames.

## What we ruled out

1. **IB binding leak** (CODEX Q4 said this shouldn't happen). Iter-2's defensive IB rebind after fan `DrawIndexed` didn't fix the radial pattern. Eliminates the leak hypothesis.

2. **Indexed-fan path** (CODEX Q3 caveat about `ShaderPrimitiveSetTemplate::SPSPT_indexedTriangleFan`). Shutdown counters confirm 0 indexed-fan calls. Not relevant for this session.

3. **Dynamic VB ring overflow per-lock** (would force every lock to discard → offset=0). Ring size is 256KB-2MB (set per video memory tier at `Direct3d11_DynamicVertexBufferData.cpp:79-82`); each 4-vert lock at 24-byte stride = 96 bytes. No way every lock overflows. Theory dead.

4. **Fan IB content corruption** — math verified, generator produces `(0, i, i+1)` pattern correctly for `i = 1..N-2`. For 4-vert fan: first 6 indices = `(0,1,2, 0,2,3)`. Correct fan triangulation.

5. **D-05 issue / engine quirk** — D3D9 renders the same scene correctly at same coords. Bug is in our D3D11 path.

## Hypotheses still on the table

**H1. Many engine objects collapse to a single world position.** If many fan-rendered objects (sprites, beams, glows, etc.) have their world transform broken or all pointing to the same world coord, their fan-apex (vert 0) all project to the same screen point. The "moves with world" signature fits this.

Possible cause: WVP cbuffer (Plan 11-08) not flushing per-object for some draw path; setObjectToWorldTransformAndScale not propagating to slot-0 shadow; matrix shadow corrupted between draws.

Test: capture WVP matrix at fan draw time and compare values across 5+ draws. If WVP is identical for multiple fans of different "objects," world transform isn't flowing.

**H2. Plan 11-09.13's per-VS dynamic PS misinterprets a non-TEXCOORD-style VS output as UV.** The dynamic PS generator (Plan 11-09.13 Iter-4) walks the VS reflected output struct and generates a matching PS. For VSes whose first non-SV output ISN'T a UV (e.g., COLOR0, NORMAL, TANGENT), the PS would sample texture using non-UV values, producing stretched/wrong textures.

But: the radial pattern is a GEOMETRY signature (verts at specific clip-space positions), not a texture-coords signature. UV misinterpretation produces wrong COLORS but doesn't change vertex positions.

Lower probability. Possibly contributing to "stretched textures" appearance but not the radial geometry.

**H3. Vertex 0 of many fan-callers is genuinely at world (0,0,0) or another shared world point.** Some engine pattern not yet identified — maybe a debug/test geometry, particle system, or HUD billboard with apex at a shared anchor.

Test: Iter-3 sampled diagnostic will surface fan-callers' VB addresses across the session. If many in-world fans use the SAME VB address (NOT just the dynamic ring), it's evidence of shared geometry.

**H4. Some unrelated state corruption introduced by Plan 11-09.15.** Less specific — but maybe my `IASetIndexBuffer` calls disturb something I haven't anticipated.

Test: temporarily skip the fan IB binding for ONE draw to confirm the half-screen state returns. If half-screen returns when I bypass fan IB, Plan 11-09.15 IS the regression source as expected. If radial pattern persists even without fan IB, the bug is somewhere else entirely.

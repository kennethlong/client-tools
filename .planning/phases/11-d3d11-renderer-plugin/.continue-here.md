## TL;DR — UI fully rendering, world textures still skewed

Three-stage fix landed since 2026-05-22 closed Iter-26:

1. **Iter-27** (commit `13ad39ba5`) — implemented `Direct3d11_StateCache::drawQuadList` stub. CODEX+Cursor convergent root-cause identification. Surfaced UI quads to the D3D11 plugin for the first time.
2. **Iter-28** (in `800796d3a`) — per-VS PS detects `COLOR0` VS output and modulates by it: `tex * input._vColor`. Fixed UI panel tinting and login backdrop coloration.
3. **Iter-32A** (in `800796d3a`) — flipped `Direct3d11_StateCache::install()` BlendState default from `BlendEnable=FALSE / One,Zero` to `BlendEnable=TRUE / SrcAlpha,InvSrcAlpha` (standard "over" composite). **THE breakthrough for fonts**: SWG UI font atlases are DXT5 with white RGB + alpha-mask glyphs; without alpha-blend, the mask values write directly to the RT as solid colored rectangles. With the standard composite, they composite correctly onto background. Opaque draws unchanged (alpha=1 yields source).

Smoke confirms: login, char-select, in-world UI (skills + chat) all render with correct fonts. **No magenta tombstones** in normal flow.

**Open frontier: world geometry has skewed textures** (terrain, models). Separate issue — likely Plan 11-10/11-11 territory, possibly related to D3D9-era texture-coordinate transform, world VS path, or the per-VS PS generator not honoring some texture-stage transform.

## Open question on resume — world geometry skewed textures

Kenny's verification report: "world content still has the skewed textures". Means terrain, models, NPCs, etc. — anything NOT routed through `uicanvas_filtered.sht` UI compositing.

### What we know / suspect (unverified)

- World content uses different VS paths (`a_simple_*`, `a_specmap_*`, `a_emis_*`, `vertex_color_lighting`, etc. — visible in `stage/stage/vs-disasm-all.txt`)
- World content likely uses `drawIndexedTriangleList` / `drawTriangleList`, NOT `drawQuadList` (so Iter-29B drawQuadList logger won't catch it)
- Per-VS PS gen does `t.Sample(s, _vN.xy)` plus optional COLOR0 modulate — this almost certainly drops world-shader features like texture-factor modulation, texCoord transforms, multi-stage texture ops, normal mapping
- D3D9-era world shaders have many texture stages (diffuse, normal, spec, env, etc.); our D3D11 plugin only binds slot 0
- "Skewed" could mean: warped UVs (texture transform missing), wrong texture per stage (we only bind stage 0), or geometry-level vertex-attribute mapping issue

1. **Quick visual reconnaissance** — Kenny runs the client, drops into a known world scene (tatooine cantina or starting zone), takes 2-3 screenshots showing the skewed textures. Helps me localize what "skewed" looks like (warped UV stretching vs. wrong texture content vs. geometry tearing).
2. **Add `drawTriangleList` / `drawIndexedTriangleList` routing diagnostics** mirroring Iter-29B's shape (shader template + VS + SRVs bound). First ~100 entries to localize what world content uses.
3. **Iter-30 texture-creation logger is already live** — world textures will appear there with their filenames and formats. Useful to identify suspicious texture-format/dimension combinations.
4. **Possibly: widen Iter-31B SearchTOC::open filter** from `texture/font/` to `texture/` to confirm which TREs world content loads from.
5. **Consider a fresh CODEX+Cursor consult** for the world-textures hypothesis once we have the routing data. Both advisors had a known bias toward format-mismatch hypotheses in the font case; framing the consult around what the routing data SHOWS (rather than what we suspect) might keep them honest.

### Iter-32B follow-up — per-pass blend wiring

The Iter-32A global-default flip works because most opaque draws output alpha=1. The "correct" form is reading `pass.m_alphaBlendEnable / Source / Destination / Operation` from `ShaderImplementation::Pass` and wiring them through a new public setter on `Direct3d11_StateCache`. Recommend as a follow-up but NOT blocking the world-textures work. The single-line fix is functional.

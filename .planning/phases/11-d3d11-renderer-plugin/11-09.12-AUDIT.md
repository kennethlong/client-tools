### Category 1: `id=366` ID3D11DeviceContext::Draw VB offset alignment (152,869 hits)

**Sample message:**
```
D3D11 [ERROR] category=9 id=366: ID3D11DeviceContext::Draw: Vertex Buffer Offset (2451)
at the input vertex slot 0 is not aligned properly. The current Input Layout imposes
an alignment of (4) because of the Formats used with this slot.
```

**Distribution:**
- **Slot:** 100% on `input vertex slot 0` (152,869 / 152,869)
- **Alignment requirement:** 100% `alignment of (4)` (152,869 / 152,869)
- **Offset values:** hundreds of distinct values across 1,000-36,000 range, all non-4-aligned (mod 4 ∈ {1, 2, 3})

The wide spread of offsets rules out "one stuck draw fires per frame." It's hundreds-of-thousands of distinct submissions, each with a non-aligned VB offset on slot 0.

**Root-cause analysis:**

Tracing the offset flow:
1. `Direct3d11_StateCache::applyPreDrawState` (line ~525-537) calls `IASetVertexBuffers(0, 1, vbs, strides, &ms_currentVBOffset)` for single-stream draws.
2. `ms_currentVBOffset` is populated by `Direct3d11_StateCache::setVertexBuffer` at line 1038: `ms_currentVBOffset = static_cast<UINT>(data->getOffset());` for the dynamic-VB branch.
3. `Direct3d11_DynamicVertexBufferData::getOffset()` returns `m_offset` from line 234 of `Direct3d11_DynamicVertexBufferData.cpp`: `m_offset = ms_used / vertexSize; // VERTEX index (engine sets BaseVertexLocation per draw)`.

So **m_offset is a vertex INDEX**, not a byte offset. The single-stream setVertexBuffer was passing that vertex index directly to IASetVertexBuffers as if it were a byte offset.

**Cross-reference D3D9 reference architecture:**
- D3D9 single-stream path at `Direct3d9.cpp:3695,3740,3752`: passes `byteOffset = 0` to SetStreamSource; captures `data->getOffset()` as `ms_sliceFirstVertex` for use as `BaseVertexIndex` on the subsequent draw call.
- D3D9 multi-stream path at `Direct3d9.cpp:3857`: passes `byteOffset = data->getOffset() * vertexSize` to SetStreamSource (explicit multiplication; mirrors what D3D11 multi-stream does correctly).

**Cross-reference D3D11 multi-stream path:**
- `Direct3d11_VertexBufferVectorData.cpp:145`: `offsets[stream] = static_cast<UINT>(data->getOffset()) * strides[stream];` — correct.

The bug is **single-stream-D3D11 only**. The single-stream path was missing the `* stride` multiplication that the multi-stream path has had since Plan 11-09.7.

**Two architecturally valid fixes:**

**A. Inline multiplication (chosen).** `ms_currentVBOffset = data->getOffset() * ms_currentVBStride;`
- Matches the multi-stream path exactly (consistent invariant).
- Minimum-blast-radius (one line + comment).
- IASetVertexBuffers gets the correct byte offset.
- Draws continue passing `0` as the StartVertexLocation argument.

**B. Pass via Draw's StartVertexLocation.** Keep `ms_currentVBOffset = 0`; modify drawPointList / drawTriangleList / etc. to pass `data->getOffset()` to Draw's StartVertexLocation. Matches D3D9 architectural pattern more closely.
- Touches more files (every draw call).
- More work; no functional advantage over Option A for non-indexed draws.
- For indexed DrawIndexed, `BaseVertexLocation` parameter handles this — but currently the IB-offset path is separate (`ms_currentIBOffset`) and the VB offset would still need to be 0 with Option B.

Picked Option A. (Notes for posterity in the fix comment.)

**Verdict:** `real-bug-fix-now`. Applied as Plan 11-09.12 Iter-1 in this audit pass.

**Expected post-fix behavior:**
- For typical engine vertex sizes (12, 16, 20, 24, 28, 32 bytes — all multiples of 4): `vertex_index * stride` is automatically 4-aligned.
- 152,869 id=366 errors should drop to **0** in the next smoke session.
- No visual change expected as a primary effect — but indirectly, the GPU was reading the wrong bytes for vertex data (offset 2451 instead of 2451*stride). The "flickering triangles" Kenny observed are likely garbage geometry from the misaligned reads. Post-fix the geometry should be CORRECT, even though the magenta fallback PS will still cover it (Plan 11-05 PEXE rejection unchanged).

**This may be a significant visual-correctness fix in addition to the cleanliness fix** — if so, that's a bonus that bridges into Plan 11-11 visual-parity work. Kenny smoke will confirm.

## PEXE-rejection evidence collection (Plan 11-11 hand-off material)

The debug-layer log shows **zero** errors directly attributable to Plan 11-05's PEXE rejection (no `CreatePixelShader failed`, no `PSSetShader with null PS`, no `Invalid shader bytecode`).

This is consistent with how the magenta fallback PS works (Plan 11-09 Iter-2):
- When the engine's asset PEXE bytecode fails to compile, the plugin's `Direct3d11_PixelShaderProgramData` stores a null `m_d3dPS` ComPtr.
- At draw time, `applyPreDrawState` (line ~562 onward) detects the null and falls back to `Direct3d11_PixelShaderProgramData::getFallbackPS()` (the install-time magenta PS).
- PSSetShader is then called with a VALID magenta PS handle — no D3D11 debug-layer error fires.

So PEXE rejection is **invisible to the D3D11 debug layer** by design. Plan 11-11's entry point will need a different signal:
1. **Plugin-side count.** Add a metric: how many compilePixelShaderFromHlsl / equivalent attempts failed at install time? Plan 11-05 / 11-07 may already have counters.
2. **Asset-shader inventory.** Survey the engine's PEXE asset bytecode population to scope the work.
3. **Spike one real PS compile.** Pick one common asset shader, attempt to compile it under the current D3D11 plugin code path, observe the failure mode, fix the closest gap.

**No additional Plan-11-11 entry-point data extractable from this audit.**

**Cascade map from CODEX post-implementation consult** (the 3 next-immediate failures to expect after Iter-1 VS-binding lands):

1. **PS NULL color output** -- D3D11 has no implicit fixed-function pass-through PS. Without a bound PS, geometry rasterizes but no fragment writes occur (invisible). Plan 11-05 documented this caveat; the engine ships D3D9 PEXE pixel-shader bytecode which D3D11 CreatePixelShader rejects, so m_d3dPS stays null. **Iter-2 addresses via minimal fallback PS** (~30 lines of HLSL + 15 lines of compile + ms_fallbackPS singleton in Direct3d11_PixelShaderProgramData; applyPreDrawState binds the fallback when m_currentPSData->getPixelShader() returns null).

2. **Input-layout creation may fail/skip if VS signature mismatches VB format** -- Pitfall 6 territory. Direct3d11_VertexDeclarationMap::getOrCreate calls CreateInputLayout which validates the layout against VS bytecode signature; mismatch returns nullptr. Engine assets may use vertex formats whose layout doesn't match the (post-rewrite) HLSL VS signature. **Iter-3+ surfaces and addresses** by either (a) adjusting buildInputElementDesc semantics, (b) adjusting Plan 11-07's HLSL rewrite rules to match engine vertex format, or (c) adding per-asset VS variant compile.

3. **D3D11 VS compile lacks D3D9's per-static-shader texture-coordinate macro variants** -- the D3D9 plugin compiles VS variants per StaticShader's texture-coordinate-set-tag mapping (Direct3d9_VertexShaderData::getVertexShader(textureCoordinateSetKey)). Direct3d11_VertexShaderData::compileOrLoad currently doesn't variant-compile; line 225 has a stub comment about it. Specific assets may surface compile failures or wrong-attribute reads. **Iter-N (asset-dependent)** addresses via per-StaticShader VS variant cache + appropriate texture-coordinate-set-tag macro injection.

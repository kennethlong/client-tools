### The math problem

Assume c4..c7 store W as ROWS (standard row-major D3D9 convention with translation in last column):
- W = standard rigid transform ✓ — rotation in upper-left 3x3, translation `[3449.8, 4.003, -4884.965]` in last column for the player, plausible for the player's actual world position.

Assume c0..c3 store WVP as ROWS as well (HLSL bytecode reads with `mul(pos, mtx)` row-vector convention).

**For a terrain vertex at the player's feet `(3450, 0, -4885, 1)` under the identity-W terrain flush:**
```
clip.w = 3450 * c0[3] + 0 * c1[3] + (-4885) * c2[3] + 1 * c3[3]
       = 3450 * (-5659.204) + 0 * (-8916.521) + (-4885) * 3384.932 + 1 * 3385.079
       ≈ -3.6 × 10⁷
```
clip.w must be **positive** for a vertex in front of the camera. ~3.6 × 10⁷ negative is catastrophically wrong — the vertex would be deep behind the camera, perspective divide flips sign, NDC coords explode.

**For the player skeletal draw**, at the model's own origin `(0,0,0,1)` (i.e., the rigid-transform origin point):
```
clip = (0 * c0) + (0 * c1) + (0 * c2) + (1 * c3) = c3
     = [-0.006, 0.792, 0.610, -0.084]
```
clip.w = -0.084. Vertices around the model origin straddle w=0, perspective divide blows up to infinity in different directions per vertex → "radial fan stretched-sliver" pattern.

**For the building draw**: same form, same problem.

**The pattern is universal across families.** This is the observed visual symptom (in-world skewing affects terrain + buildings + skeletal characters uniformly) reflected in the cbuffer math.

## Leading hypothesis — row/column-major packing mismatch between engine writer and VS bytecode reader

D3D9 fixed-function and D3D9 HLSL by convention use row-major matrices with row-vector-left-multiply semantics: `out_row_vec = pos_row_vec * matrix`.

D3D11 HLSL `float4x4` defaults to **column-major** unless explicitly declared `row_major` or compiled with `/Zpc` style. The D3D11 plugin in this project translates D3D9 vsh shader-assembly into HLSL at runtime (via `Direct3d11_PixelShaderProgramData.cpp::buildHlslForVSOutputs` and the VS counterpart) and compiles to bytecode. **If the engine packs the cbuffer with rows of the matrix in `c0..c3` (D3D9 row-major convention) but the translated bytecode interprets `c0..c3` as columns (D3D11 column-major default), every `dp4 oPos.x, v0, c0`-style instruction in every VS sees a transposed matrix**, producing exactly the kind of structurally-wrong-but-internally-consistent clip-space output we observe.

This would explain:
- **Universality**: every VS (terrain, static, skeletal) is broken the same way.
- **Internally consistent garbage**: math IS being done with real values, just on a transposed matrix → coherent but wrong topology, not random crashes.
- **W matrix appearing valid in c4..c7** (it's the SAME matrix either way at the byte level — it's only the bytecode's INTERPRETATION that differs).
- **Iter-36D2 showed setter input is correct**: the setter receives the right `Transform&` from the engine; the bug is downstream in how the bytes get packed for the cbuffer or how the bytecode reads them.

CODEX in the prior consult identified a separate scale-conversion bug at `Direct3d11_StateCache.cpp:1448` ("only scales diagonal `_11/_22/_33`, where D3D9's `convertScaleAndTransformToMatrix` scales all three components of each basis row"). That bug lives at the SAME setter and is the SAME class of error (mis-interpreting rows vs columns / mis-mapping field indices). Possibly two symptoms of the same root cause.

## What we're asking the advisors

**Q1 (verification):** Verify the math above is correct. Given the captured c0..c3 / c4..c7 values, does the "row-major-stored, column-major-read" interpretation explain the negative-w outputs? If not, what packing convention DOES make the numbers internally consistent?

**Q2 (audit):** Read `Direct3d11_StateCache.cpp` setter functions and `s_slot0Shadow` layout. What byte order does the engine actually write into c0..c3 and c4..c7? Identify the exact lines that lay down the matrix bytes.

**Q3 (audit):** Read the HLSL translation path (`Direct3d11_PixelShaderProgramData.cpp::buildHlslForVSOutputs` and the VS translation function). Does the emitted HLSL declare matrices as `row_major float4x4` or rely on the column-major default? Is there an explicit `#pragma pack_matrix(row_major)`?

**Q4 (cross-reference):** Compare against the D3D9 baseline at `Direct3d9.cpp:3132`. What's the canonical packing convention the D3D9 path uses for the SAME engine setter inputs?

**Q5 (fix candidates):** If row/column-major mismatch is confirmed, what's the minimum-blast-radius fix — declare matrices `row_major` in the emitted HLSL, transpose at pack time in `Direct3d11_StateCache.cpp`, or both? Are there secondary callers (bone matrices via `setVSConstantsRaw`, light data, etc.) that would also need adjustment?

**Q6 (alternative hypotheses):** If row/column-major is NOT the bug, what other failure mode produces internally-consistent-but-w-negative cbuffer math across multiple unrelated VSes? Candidates to consider: cbuffer offset/alignment, perspective-divide handedness, view matrix sign, or position-attribute interpretation in the input layout.

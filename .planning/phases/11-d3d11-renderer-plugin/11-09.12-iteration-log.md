### Investigation

Located latest session-begin marker at line 566,984 of stage/d3d11-debug.log (749,572 lines total; ~198 MB). Post-marker range: 182,588 lines covering the Plan 11-09.11 close smoke (2026-05-22T07:33:09).

Session-only `[ERROR]` count: **152,870**. `[WARNING]` count: 0.

Categorization via `grep -oE "id=[0-9]+" | sort | uniq -c`:
- `id=366` × 152,869 (99.9994% of errors) — DRAW alignment failure
- `id=281` × 1 — RESOURCE format mismatch

For id=366 (the load-bearing category), distribution analysis showed:
- 100% on `input vertex slot 0`
- 100% require `alignment of (4)`
- Hundreds of distinct VB offset values (1,000-36,000 range), all non-4-aligned

Traced the offset flow:
- `Direct3d11_StateCache::applyPreDrawState` calls `IASetVertexBuffers` with `ms_currentVBOffset`.
- `setVertexBuffer` at line 1038 populates `ms_currentVBOffset = static_cast<UINT>(data->getOffset());` for dynamic VBs.
- `Direct3d11_DynamicVertexBufferData::getOffset()` returns `m_offset = ms_used / vertexSize` — a **VERTEX INDEX**, not a byte offset.

Cross-references confirmed the bug is **single-stream-D3D11 only**:
- D3D9 single-stream (`Direct3d9.cpp:3695,3740,3752`): passes `byteOffset = 0` to SetStreamSource, uses `ms_sliceFirstVertex = data->getOffset()` for the draw's BaseVertexIndex parameter.
- D3D9 multi-stream (`Direct3d9.cpp:3857`): `byteOffset = data->getOffset() * vertexSize`.
- D3D11 multi-stream (`Direct3d11_VertexBufferVectorData.cpp:145`): `offsets[stream] = data->getOffset() * strides[stream]` — already correct (since Plan 11-09.7).
- D3D11 single-stream (`Direct3d11_StateCache.cpp:1038`): **MISSING the multiplication.** Bug.

For id=281 (one-shot), captured as `Plan-11-13+-territory` (background-noise watchpoint).

PEXE-rejection evidence collection: zero direct debug-layer signals (magenta fallback PS is a valid handle; D3D11 sees a successful PSSetShader). Captured as a Plan 11-11 design note: PEXE rejection is invisible to the D3D11 debug layer by design, so Plan 11-11 needs a different entry point (plugin-side compile-failure counters, asset-shader population inventory, or a single-spike fix-the-first-shader approach).

### Root cause

`Direct3d11_StateCache::setVertexBuffer` line 1038 was passing `data->getOffset()` (a VERTEX INDEX per the source comment at `Direct3d11_DynamicVertexBufferData.cpp:234`) directly to `IASetVertexBuffers` as if it were a BYTE OFFSET. Single-stream dynamic-VB path only — static VBs always pass offset=0 (line 1024), and the multi-stream path correctly multiplied by stride.

The wide spread of offsets in the error log (1,000-36,000 range) tracks the dynamic ring buffer's vertex-index growth across the session — every dynamic-VB-backed draw was misinterpreting the offset.

### Fix

One-line change at `Direct3d11_StateCache.cpp:1038`:

```cpp
// OLD:
ms_currentVBOffset = static_cast<UINT>(data->getOffset());
// NEW:
ms_currentVBOffset = static_cast<UINT>(data->getOffset()) * ms_currentVBStride;
```

Plus a multi-paragraph comment explaining the bug + the cross-reference to the multi-stream pattern. After the fix, single-stream and multi-stream paths share the same invariant (`byte_offset = vertex_index * stride`).

D-05 preserved: only `Direct3d11_StateCache.cpp` touched in the D3D11 plugin. D3D9.cpp diff is Koogie's pre-existing 135-line working-tree patch (untouched).

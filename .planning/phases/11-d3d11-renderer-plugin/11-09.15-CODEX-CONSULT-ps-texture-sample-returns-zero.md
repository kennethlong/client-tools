# CODEX Consult — D3D11 PS Texture.Sample Returns Zero Despite Valid SRV Binding

**Context date:** 2026-05-22 (post-Plan-11-09.15 Iter-12/13 isolation diagnostics)

**Status:** Pipeline confirmed reaching PS; texture sampling returns zero RGB. Need direction on D3D9→D3D11 texture migration pitfalls.

---

## What we're trying to decide

The first CODEX consult on this thread (XYZRHW transformed-vertex handling) was diagnostically correct but did not surface the actual bug. After eight more iterations of bisection diagnostics, we have isolated the failure to **PS texture sampling at the `Texture2D.Sample(SamplerState, uv)` call site** for transformed-vert 2D UI quads. The texture sample call returns a value that produces a black-equivalent final framebuffer write, even though every D3D11 binding fact we can directly measure looks clean.

We want CODEX to identify which D3D9→D3D11 texture-migration pitfall most likely causes "SRV bound to valid pointer + sampler bound + UVs in (0..1) + sampling returns zero RGB" — or to direct the next diagnostic step if our framing of the failure space is incomplete.

   ```
   vs_4_0
   dcl_constantbuffer CB0[10], immediateIndexed
   dcl_input v0.xy
   dcl_input v1.xy
   dcl_output_siv o0.xyzw, position
   dcl_output o1.xy
   dcl_temps 2
   mov r0.xy, v0.xyxx
   mul r0.x, r0.x, cb0[9].x       // pixel.x * viewportData.x
   add r1.x, r0.x, cb0[9].z       // + viewportData.z = ndc.x
   mul r0.x, r0.y, cb0[9].y       // pixel.y * viewportData.y
   add r1.y, r0.x, cb0[9].w       // + viewportData.w = ndc.y
   ```

## The narrow remaining failure mode

`t.Sample(s, input._v0.xy)` returns zero RGB despite:
- `Texture2D t : register(t0)` matching the bound SRV slot 0
- `SamplerState s : register(s0)` matching the bound sampler slot 0 (default sampler pre-installed)
- `ms_boundSRV[0]` non-null per the draw counter
- `ms_boundSampler[0]` pre-filled with `ms_defaultSampler` at StateCache install (lines 854-855)
- UV coordinates that should land inside (0, 1) for fullscreen-quad rendering
- No D3D11 validation warnings about texture/sampler binding

The bound SRV must point to something. The sampler must do something. But the sampler reads zero RGB from the SRV.

## Hypothesis space (CODEX, please rank/correct)

**H1 — SRV format mismatch.** The texture is created/uploaded as one DXGI_FORMAT (e.g., `DXGI_FORMAT_BC1_UNORM` or `DXGI_FORMAT_BC3_UNORM` for compressed UI bitmaps; or `DXGI_FORMAT_B8G8R8A8_UNORM` for raw BGRA from D3D9-era pipeline), but the SRV's view format doesn't match. D3D11 silently zeroes the sample when the view format is incompatible with the resource format on some drivers; on others it sample-fails entirely. Pre-D3D11.1 drivers (we're targeting feature level 11.0) can be particularly silent about this.

**H2 — SRV points at a STAGING or other non-shader-visible resource.** Direct3d11 SRVs require the underlying ID3D11Resource to have been created with `D3D11_BIND_SHADER_RESOURCE`. If the SWG texture-loading path created the resource with `D3D11_USAGE_STAGING` or omitted the bind flag (D3D9 didn't require an explicit bind flag), sampling that resource returns zero.

**H3 — Resource was created but ID3D11DeviceContext::UpdateSubresource (or Map+Unmap for upload) was never called.** SWG's texture-loader unpacks the bitmap from TRE archives into a CPU-side buffer; if the D3D11 plugin's port forgot to upload that buffer to the GPU resource (or uploaded with wrong row pitch and the driver silently zeroes the texture region), sampling returns the cleared-on-create zero state.

**H4 — Sampler descriptor problem.** The default sampler at install may have `D3D11_FILTER_MIN_MAG_MIP_POINT` + `D3D11_TEXTURE_ADDRESS_BORDER` with BorderColor `(0,0,0,0)`. UI quads sample at corner UV coords `(0,0), (1,0), (1,1), (0,1)` — exactly at the texture edge. With BORDER addressing and zero border color, the rasterizer's centroid-sample may land at the edge boundary and sample the border = (0,0,0,0).

**H5 — SWG-side texture lookup mismatch.** The path is `engine sets static-shader pass with texture tag → StaticShaderData::apply() resolves tag to a TextureData → Direct3d11_TextureData::getShaderResourceView() returns the SRV → setPixelShaderResource(slot, srv) writes ms_boundSRV[slot]`. If `getShaderResourceView()` returns the WRONG SRV (e.g., a CACHED SRV pointing at a freed texture from a prior pass), the sample fails silently.

**H6 — Plan 11-09.13's per-VS PS register linkage drift.** The PS declares `float2 _v0 : TEXCOORD0;`. The VS outputs `TEXCOORD0 at register o1`. The PS's FXC auto-allocator might assign `_v0` to v0 (rather than v1). D3D11 stage linkage matches by SEMANTIC NAME + INDEX (not register position) — per MSDN, this should work — but if any driver / debug-layer interpretation differs, the PS would receive zero for TEXCOORD0 input. Then `t.Sample(s, zeroUV)` samples at (0, 0) — typically the texture's corner pixel. For a splash bitmap with a black border at the corner (very common for UI assets), the corner sample is black.

H6 is the most "feels like the answer" to us:
- It's the only hypothesis that explains "consistent zero across multiple texture-bearing draws"
- It's consistent with magenta lines for position-only shaders working (no TEXCOORD0 issue)
- It would also explain Iter-4's original radial pattern (different VS, different output register layout, different zero-fold pattern)
- And it would explain why Iter-12 green hack works (no Sample call → no UV needed)

But we cannot directly verify H6 without disassembling the COMPILED per-VS PS bytecode (we currently only disassemble VS). And per MSDN, D3D11 stage linkage SHOULD match by name + index. So H6 either points at a real driver bug, or at our incomplete understanding of D3D11 input-signature matching.

## What we want from CODEX

1. **Rank H1–H6 (or propose H7).** Which is the most likely cause of "SRV bound, sampler bound, t.Sample(s, uv) returns zero RGB"?

2. **What's the cheapest diagnostic to disambiguate?**
   - For H1: dump the SRV's view-format vs the underlying ID3D11Resource's format
   - For H2: dump the underlying resource's Usage + BindFlags
   - For H3: try `Map + Read` on a staging-copy of the bound texture and check if pixels are zero
   - For H4: dump the sampler's filter + addressing mode + borderColor
   - For H5: dump the SRV pointer's lifetime — is it the same SRV each draw, or thrashing
   - For H6: D3DDisassemble the per-VS PS and check if `dcl_input_ps v0.xy` matches the VS output register

3. **Is there a known D3D11 pattern for the "PS receives zero TEXCOORD when VS output register doesn't match PS input register" hazard?** Plan 11-09.13 Iter-3's design explicitly tried to handle this by sorting VS outputs by register; but if the PS's FXC auto-allocator doesn't match the VS register, we still have linkage drift even though semantic names match. Has CODEX seen this pattern in D3D11 porting work?

4. **What's the cleanest fix surface?** If H6 is the cause, fix is at the PS HLSL generator (declare per-VS PS input registers explicitly via `: TEXCOORDn` + matching the VS output structure literally — or even pass a `: TEXCOORDn` register hint inline). If H1–H5 is the cause, fix is downstream in texture creation / sampler config / SRV cache lookup.

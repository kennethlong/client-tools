## Plan 11-09.9 status — fully scaffolded, ready to implement

**Diagnosis (from Plan 11-09.8 close smoke):**
- FATAL on `Direct3d11_TextureData::lock` for `TextureFormat 4` = `TF_RGB_888` (24bpp RGB)
- D3D11 / DXGI has NO 24bpp packed surface format (D3D9 has `D3DFMT_R8G8B8`; DXGI dropped it)
- Plan 11-04 runtimeFormats promotion at create time DOES fire (`Direct3d11_TextureData.cpp:310-329`) — TF_RGB_888 surfaces get allocated under a wider 32bpp DXGI format like `B8G8R8X8_UNORM`. `m_destFormat` stores it.
- BUT at lock time (~line 477), the code uses `lockData.getFormat()` (engine's expected FILL format = TF_RGB_888 = 24bpp) for the translationTable lookup → UNKNOWN → FATAL.

**Fix approach (per `11-09.9-PLAN.md`):**
1. Add CPU shadow buffer state to `Direct3d11_TextureData.h` (`m_lockBridgeBuffer/Format/Pitch/Width/Height/Active`).
2. `lock()`: detect format-width mismatch via `lockData.getFormat() != m_destFormat AND translationTable[lockData.getFormat()] == UNKNOWN`. For recognized engine 24bpp formats (TF_RGB_888 first), allocate shadow + return to engine at 24bpp stride.
3. `unlock()`: if bridge active, Map GPU staging WRITE_DISCARD, expand row-by-row 3→4 bytes respecting `mappedResource.RowPitch`, channel swap based on `m_destFormat` (RGB→BGRX vs RGB→RGBA), Unmap, proceed with existing CopySubresourceRegion. Free shadow.
4. Defensive FATAL on other `DXGI_FORMAT_UNKNOWN` cases (TF_DXT2/4, TF_P_8) with clear "Plan 11-09.10 territory" message.

**Crash dump (preserved for diagnosis reference):** `stage/SwgClient_d.exe-unknown.0-20260522022933.{mdmp,txt}` (4481-byte txt, full 31-deep stack down through GameNetwork transport).

**Iteration target:** 1. **Risk profile:** zero BSOD (pure CPU staging + format conversion). **CODEX consult?** Optional but probably not needed — the architecture is straightforward CPU bridge; channel-order swap is the only nuance and it's introspectable from `m_destFormat`.

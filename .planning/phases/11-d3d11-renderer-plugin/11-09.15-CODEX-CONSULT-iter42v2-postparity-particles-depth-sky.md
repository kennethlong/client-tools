# CODEX + Cursor Consult — D3D11 Post-Parity: Particle Billboard Rendering, Per-Pass Depth State, Sky/Gamma Washout

**Context date:** 2026-05-23 (post-Plan-11-09.15 Iter-39C visual parity, Iter-40 baked-RT format fix, Iter-41 screenshot impl, Iter-42v2 row-scale fix)

**Status:** D3D11 plugin reaches visual parity with D3D9 for the dominant content categories (UI, characters with proper-scaled basis, terrain, opaque buildings, fonts). The Mos Eisley screenshot smoke (`stage/screenshots/screenShot0001.jpg` = D3D9 reference, `screenShot0003.jpg` = D3D11 current) surfaced THREE remaining visual deltas that the work-in-progress hypotheses partially addressed but did not close. We've burned the obvious candidates for the particle issue — the consult is to validate that the remaining lines of investigation are pointed in the right direction.

This consult is **fact-first, hypothesis-light** (same posture as the prior Iter-37 consults that converged correctly). The screenshots are in the screenshots/ directory; we've also included verbal descriptions of each symptom because the math we want validated is text-only.

## What's been validated so far this iteration

### Iter-42v1 (column-scale) - REJECTED by smoke

First attempt at fixing the diagonal-only scale composition at `Direct3d11_StateCache::setObjectToWorldTransformAndScale` (StateCache.cpp:1666). Pre-Iter-42 code only scaled diagonal elements:

```cpp
m[0][0] * scale.x, m[0][1],           m[0][2],           m[0][3],
m[1][0],           m[1][1] * scale.y, m[1][2],           m[1][3],
m[2][0],           m[2][1],           m[2][2] * scale.z, m[2][3],
```

Iter-42v1 scaled basis COLUMNS (mathematically pure object-space scale composition):

```cpp
m[0][0]*sx, m[0][1]*sy, m[0][2]*sz, m[0][3],
m[1][0]*sx, m[1][1]*sy, m[1][2]*sz, m[1][3],
m[2][0]*sx, m[2][1]*sy, m[2][2]*sz, m[2][3],
```

**Smoke result:** no visible improvement on the symptom (`screenShot0002.jpg`). Character head still flat, stretched ribbons unchanged.

### Iter-42v2 (row-scale, matching D3D9 sibling) - ACCEPTED

`Direct3d9.cpp:3155-3168` is the D3D9 reference, which scales ROWS:

```cpp
matrix._11 = transform.matrix[0][0] * scale.x;
matrix._12 = transform.matrix[0][1] * scale.x;
matrix._13 = transform.matrix[0][2] * scale.x;
matrix._14 = transform.matrix[0][3];   // NOT scaled
matrix._21 = transform.matrix[1][0] * scale.y;
...
```

Iter-42v2 mirrors this row-scale pattern (`StateCache.cpp:1681-1684`):

```cpp
m[0][0]*sx, m[0][1]*sx, m[0][2]*sx, m[0][3],
m[1][0]*sy, m[1][1]*sy, m[1][2]*sy, m[1][3],
m[2][0]*sz, m[2][1]*sz, m[2][2]*sz, m[2][3],
```

**Smoke result (`screenShot0003.jpg`):** character head shape restored (Ithorian hammerhead now visible). Mountain colors back to correct gray-brown. PARTIAL win — ribbons + trim + particles unchanged.

### Iter-43 (per-pass blend factors) - REJECTED by smoke, reverted

Iter-39C wired only `m_alphaBlendEnable` per-pass; `m_alphaBlendSource/Destination/Operation` remained at install() defaults (`SrcAlpha`/`InvSrcAlpha`/`Add`). Iter-43 added per-pass propagation via direct enum-to-enum translation:

```cpp
D3D11_BLEND translateBlend(ShaderImplementationPass::Blend b) {
    switch (b) {
    case SIP::B_Zero:                    return D3D11_BLEND_ZERO;
    case SIP::B_One:                     return D3D11_BLEND_ONE;
    case SIP::B_SourceAlpha:             return D3D11_BLEND_SRC_ALPHA;
    case SIP::B_InverseSourceAlpha:      return D3D11_BLEND_INV_SRC_ALPHA;
    // ... all 11 engine enum values mapped 1-to-1 to D3D11_BLEND
    }
}
// Plus equivalent translateBlendOp for BlendOperation enum
// Plus Direct3d11_StateCache::setAlphaBlendFactors that pushes
// SrcBlend/DestBlend/BlendOp into RT[0] of the BlendState descriptor
```

The D3D9 sibling does this via `RSM(D3DRS_BLENDOP/SRCBLEND/DESTBLEND, ...)` at `Direct3d9_ShaderImplementationData.cpp:259-261`.

**Smoke result (`screenShot0002.jpg`):** WORSE. New large white square appeared mid-frame. Mountain acquired snow-like patches. Existing amber square got larger. Reverted to Iter-39C state (BlendEnable only).

## Remaining visual deltas

### Delta 1: "Horizontal ribbon" structures crossing mid-frame

**Visible in:** `screenShot0000.jpg` (original D3D11), `screenShot0003.jpg` (current D3D11 with Iter-42v2). NOT visible in `screenShot0001.jpg` (D3D9 reference at same position).

**Morphology:** multiple long horizontal banded structures crossing the mid-frame at the height of the building tops. Brown/amber/orange gradient color, soft-edged. They span ~50-70% of screen width. Same shape and position across all D3D11 captures.

**Iter-42v2 did NOT close this** — the ribbons are unchanged before and after the row-scale fix. This proves the affected objects do not go through `setObjectToWorldTransformAndScale`. They go through a different transform path.

**Hypotheses:**

- (A) **Particle billboard trails** — smoke/dust/sand particles that emit trailing billboard quads. The "trail" geometry is typically a stream of quads oriented along a velocity axis, and if the orientation math is off (e.g., billboard-axis computation), they stretch into "ribbons" along one world axis. This is the same family as the amber/white square particle bug.
- (B) **Transformed vertices (pre-projected)** — XYZRHW-style quads that bypass world/view/projection matrices entirely. The engine's UI canvas, sun, moon, sprite system use these. The Iter-26 era investigation surfaced that some content uses transformed vertices.
- (C) **Per-instance matrices in vertex buffers** — static-mesh decorations (banners, awnings, building trim) using DrawInstanced or per-instance VB streams. The D3D11 plugin's per-instance handling may not match D3D9.

### Delta 2: Solid colored square "particles"

**Visible in:** all D3D11 captures. Most prominent in `screenShot0003.jpg` as a ~150px white square mid-left and a smaller amber square mid-frame.

**Morphology:** opaque solid-color rectangle of the particle's tint color (white for smoke / "blink amber" yellow for the amber light). Edges hard, not soft. Size suggests a billboard quad at full opacity instead of a soft alpha-blended sprite.

**Engine has the right textures loaded** — `texture/light_blink_glow.dds` (DXT5, has alpha), `texture/pt_smoke_twisting.dds` (DXT3, has alpha), `texture/light_blink_glow_add.dds` (DXT5, has alpha) — Iter-30 texture-create log confirmed all SRVs exist.

**Iter-39C wired `m_alphaBlendEnable` per-pass.** The amber/blink texture's pass has `m_alphaBlendEnable=true`, so we're at least in additive territory. But the BlendOp/SrcBlend/DestBlend stay at the install() default (`SrcAlpha`/`InvSrcAlpha`/`Add`).

**Iter-43 attempt** (per-pass factor wiring with direct enum translation) made it WORSE. New larger white square appeared. This is the key data point we want consultee insight on.

**Hypothesis for why Iter-43 went wrong:**

The D3D9 path applies `m_alphaBlendSource/Destination/Operation` via `RSM` macro which only-conditionally pushes the value (looking at the macro: `m_renderState.push_back(RenderState(rs, m[pass.v]))` where `m` is a lookup table). The D3D9 render-state engine likely applies these values only when `alphaBlendEnable=true` AND/OR resets to defaults when the pass ends. Our D3D11 wiring applied them UNCONDITIONALLY per-pass, which means many passes with `m_alphaBlendEnable=false` and uninitialized/zero `m_alphaBlendSource/Destination` fields wrote `(Zero,Zero,Add)` to the BlendState. While `BlendEnable=false` short-circuits at the hardware level, the BlendState DESCRIPTOR with `SrcBlend=Zero, DestBlend=Zero` is still cached and could be picked up by a subsequent pass that sets `BlendEnable=true` without re-setting factors.

**Open question:** is that the right diagnosis? Or is there a missing translation step (e.g., D3D9's `RSM` lookup table has a NON-IDENTITY mapping from engine enum to D3D9 enum)?

### Delta 3: Alpha bleed-through on character head (eyes visible through back-of-head)

**Visible in:** D3D11 captures with character facing away from camera. When the character "blinks" (eye animation), the eye texture renders THROUGH the back of the head — they appear over the back-of-head fragment in screen space.

**Hypothesis:** depth-write or depth-test state is incorrectly configured on the character's multi-pass shader. Either:

- (a) The back-of-head pass has `m_writeEnable=false` for depth, so doesn't write to the depth buffer
- (b) The eye pass has `m_zCompare=Always` instead of `LessEqual`, so passes regardless of depth
- (c) The Iter-39C deferral comment in `Direct3d11_StaticShaderData.cpp:539-542` is the smoking gun:

> *"Still-deferred (small follow-on): m_alphaBlendSource/Dest, m_alphaBlendOperation, **m_writeEnable, m_zEnable/Write/Compare**. Existing install() defaults (..., DepthEnable=TRUE, LESSEQUAL, write-all) match what typical UI/particle/opaque shaders need."*

This comment explicitly mentions `m_writeEnable` and `m_zEnable/Write/Compare` as still-deferred. Skeletal-character multi-pass shaders likely DO override these per-pass and we're not honoring it.

### Delta 4: Mini-map clip region (circular in D3D9, square in D3D11)

**Visible in:** upper-right corner of the HUD in `screenShot0000.jpg` + `screenShot0003.jpg` (D3D11) vs `screenShot0001.jpg` (D3D9). D3D9 shows a circular clipped radar view; D3D11 shows the full square radar texture without the circular alpha mask applied.

**Hypothesis:** the mini-map widget composites a square radar render-target through an alpha mask (circular hole texture) to produce the round look. The masking is either an alpha-test path or a multi-pass alpha-blend chain. Since this is a UI canvas effect, it likely uses the same per-pass state wiring that Iter-39C touched. May be the SAME family of bug as the particle squares (per-pass blend/alpha state not fully wired) or may be a specific stencil/scissor rect issue.

This is interesting because the mini-map is normally one of the cleanest UI surfaces in the engine. Whatever's missing here is likely a small, specific state-cache wiring that the rest of the UI didn't depend on but the mini-map composite does.

### Delta 5: Sky color washed out + overall brightness/gamma elevated

**Visible in:** `screenShot0003.jpg` shows a flat cream/beige sky vs D3D9's gorgeous pink-orange-purple Tatooine sunset gradient. Overall scene is dramatically brighter and lower-contrast in D3D11.

**Caveat:** the D3D11 capture was at in-game time 15:48 and the D3D9 capture at 15:19, so Tatooine sunset cycle adds SOME natural difference. But the washed-out flat-color sky in D3D11 is too dramatic to be only time-of-day.

**Hypotheses:**
- (A) Skybox shader chain — the SkyBox6Sided geometry uses specific shaders; one of them might be missing a sample or rendering with default tint
- (B) Gamma correction — D3D11 plugin may be missing the gamma ramp setup that D3D9 had via `IDirect3DDevice9::SetGammaRamp`
- (C) Tone mapping / post-FX color grade pass missing or routed wrong
- (D) Fog application different (fog usually DARKENS, not BRIGHTENS, so this is unlikely)

This delta is the lowest priority for us right now — closing the particle + depth deltas first is the priority.

## Specific consult questions

### Q1 [HIGH] - particle billboard squares

What's the right model for porting D3D9's `RSM`-based per-pass render-state propagation to D3D11? Specifically:

- Are `m_alphaBlendSource`, `m_alphaBlendDestination`, `m_alphaBlendOperation` directly authoritative on a per-pass basis, or does the D3D9 engine only apply them under specific conditions?
- Does D3D9 reset blend factors to defaults at frame/pass boundaries, masking the "uninitialized blend field" problem?
- Should the D3D11 path only push per-pass factors when `m_alphaBlendEnable=true`, leaving the install() default in place when blend is off?

### Q2 [HIGH] - ribbon-stretched geometry

What transform path does Tatooine's outdoor "draped cloth / banner / awning" content use? Specifically the Mos Eisley building trim - is it:

- (A) Static-mesh appearance with per-instance scale via setObjectToWorldTransformAndScale (Iter-42v2 should reach it)
- (B) ParticleSystem trail emitter (billboard quads along a velocity axis)
- (C) Transformed-vertex pre-projected geometry (UI-canvas-like)
- (D) Other

The visual signature is "long horizontal stretched bands" — strongly suggests an orientation-axis bug in a billboard or strip-quad path.

### Q3 [MEDIUM] - per-pass depth state

D3D9 reference at `Direct3d9_ShaderImplementationData.cpp:259-300+` shows `RSM` writes for `D3DRS_ZFUNC`, `D3DRS_ZWRITEENABLE`, etc. The D3D11 plugin's `Direct3d11_StaticShaderData::apply` (line 543-549) currently only honors `m_alphaBlendEnable`. Is per-pass depth state the right fix for the "eyes through back-of-head" symptom, and is the right port to add `setDepthStencilEnable/setDepthWriteEnable/setDepthCompareFunc` to the StateCache the way Iter-39C added `setAlphaBlendEnable`?

### Q4 [LOW] - skybox / gamma washout

Lowest priority for this consult — flag if you see an obvious cause from the screenshots, but we can investigate after Q1-Q3 land.

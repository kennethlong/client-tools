<risks>
1. **D3D9 sibling Stage construction quirks** — TA_invalid / TF_invalid fallback to TextureSampler defaults; global-texture short-circuit via `Direct3d9_TextureData::isGlobalTexture(tag)`. Mis-translating any of these silently produces wrong-but-non-crashing renders. Mitigation: read D3D9 sibling line-by-line; defensive log when invalid-fallback fires.

2. **TextureData pointer-to-pointer lifetime** — `Direct3d9_TextureData::const * const *m_texture` is a pointer to the engine `Texture`'s `m_graphicsData` field which can be re-assigned if the texture is reloaded. D3D9 sibling explicitly handles this via the double indirection (`*m_texture` at apply time, not at construct time). Plan 11-09.14 must mirror this pattern.

3. **Sampler cache key collision risk** — `Direct3d11_StateCache::getOrCreateSampler(D3D11_SAMPLER_DESC const &)` keys via FNV-1a on the desc struct. If the desc has padding bytes, hash key may vary across compiles. Mitigation: zero-initialize the desc before fill (existing pattern in Plan 11-06).

4. **Multi-pass shaders** — Plan 11-09.14 binds per-pass. If a multi-pass shader has different textures per pass, each pass's apply() rebinds. Should already work via the existing per-pass dispatch in `Direct3d11_StateCache::applyPreDrawState` → `StaticShaderData::apply(passNumber)`.

5. **Cache-thrash if binds aren't tracked** — frequent SRV/sampler changes per draw can be expensive. Plan 11-06's `ms_boundSRV[16] + ms_boundSampler[16]` per-slot tracking should already short-circuit redundant binds; verify in implementation.

6. **D-05 invariant** — D3D11 plugin only. Zero engine-side edits expected (engine `TextureSampler` data is already exposed; we just consume). Risk: zero.

7. **BSOD / GPU hang** — SRV + sampler binding is the most well-trodden D3D11 path imaginable. Risk: zero.
</risks>

## What Iter-40 fixed

The 7-per-session `D3D11 [ERROR] id=281 CopySubresourceRegion: Formats not castable` was firing in `Direct3d11_RenderTarget::copyRenderTargetToNonRenderTargetTexture` (line 284 ‚Äî the "bake to texture" path used by `BlueprintTextureRendererTemplate` + similar render-into-engine-texture flows).

**Root cause:** the persistent 512√ó512 offscreen RT (`ms_renderTargetTexture` at `Direct3d11_RenderTarget.cpp:93`) was created at `DXGI_FORMAT_B8G8R8X8_UNORM` (BGRX) to mirror the D3D9 sibling's `D3DFMT_X8R8G8B8`. D3D11's `CopySubresourceRegion` strict-rejects when source and dest formats live in different typeless families. Engine bake destinations are predominantly `TF_ARGB_8888` ‚Üí BGRA8 (`TextureRendererTemplate.cpp:74` confirms `singleTextureFormat = TF_ARGB_8888`), and BGRX8 ‚Üî BGRA8 live in SEPARATE typeless families (`B8G8R8X8_TYPELESS` vs `B8G8R8A8_TYPELESS`) ‚Äî so every bake-back failed.

**Fix:** one-line format change BGRX ‚Üí BGRA at install. Plus a one-shot diagnostic in `copyRenderTargetToNonRenderTargetTexture` that warns if any non-BGRA8 dest slips through (`Iter-40 BAKE_FMT_MISMATCH`).

**Smoke verification (2026-05-23 14:52 + 14:55 sessions):**
- `grep -c "id=281" stage\d3d11-debug.log` ‚Üí 0 in both post-patch sessions (was 7)
- `grep -c "[ERROR]" stage\d3d11-debug.log` for post-patch range ‚Üí 0 (clean stream)
- `grep -c "BAKE_FMT_MISMATCH" stage\d3d11-debug.log` ‚Üí 0 ‚Üí BGRA8 is correct for every observed bake dest; no Iter-40b needed

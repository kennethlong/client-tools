## Iter-1: Bump VS to vs_4_0, PS helper to ps_4_0, REWRITE_VERSION 14 → 15

**Date:** 2026-05-21 (late afternoon session; CODEX consult returned mid-session)

### Symptom (carried from Plan 11-09.5 close smoke)

Post-Plan-11-09.5 D3D11 smoke (Kenny advanced past login into char-create) produced a reproducible FATAL during world-load:

```
D3DCompile vs_4_0_level_9_3 'vertex_program/a_specmap_bump_vs20_for_ps20.vsh' failed:
error X5615: Compiled shader code uses too many instruction slots (293).
Max. allowed by the target (vs_2_x) is 256.
```

Crash chain:

Crash dump preserved: `stage/SwgClient_d.exe-unknown.0-20260521202203.{mdmp,txt}` (139KB mdmp / 6470-byte txt).

### Hypothesis

Plan 11-07 Iter-3 chose `vs_4_0_level_9_3` profile to dodge vs_5_0's reserved-keyword problem (`point`, `line`, etc. — surfaced as X3000 syntax errors on engine HLSL). level_9_3 maps to SM3-equivalent feature constraints + 256-instruction-slot ceiling. The `a_specmap_bump_vs20_for_ps20.vsh` shader has 293 instructions and busts the cap by 37 slots.

**Fix:** bump VS profile from `vs_4_0_level_9_3` to plain `vs_4_0`. Drops the level_9 instruction-slot ceiling while preserving the D3D9-era syntax compatibility (vs_5_0 keyword issue still avoided). Mirror PS helper to `ps_4_0`. Bump REWRITE_VERSION to invalidate `.cso` cache.

CODEX consult (`11-09.6-CODEX-CONSULT-shader-target-bump.md`) confirmed `vs_4_0` over `vs_5_0` + rejected fallback-ladder pattern + identified expected cascade order: legacy SM4 syntax fallout > register/constant layout > VS/PS signature mismatch > runtime constant-binding > PS-era constructs (low risk because PEXE rejected pre-compile).

### Investigation

CODEX consult drafted at `.planning/phases/11-d3d11-renderer-plugin/11-09.6-CODEX-CONSULT-shader-target-bump.md` (5 questions: target choice; cascade prediction; fallback-ladder viability; flag set; PEXE coupling; acceptance bar). CODEX response endorsed `vs_4_0` over `vs_5_0` (corrected my consult's stale premise that PS helper was at ps_5_0 — actually at ps_4_0_level_9_3 same as VS), rejected fallback-ladder pattern, identified the cascade order (legacy SM4 keyword fallout most likely next), kept the flag set unchanged.

Source-tree audit confirmed the rationale-block at `Direct3d11_VertexShaderData.cpp:28-55` accurately recorded the Plan 11-07 Iter-3 reasoning for choosing `vs_4_0_level_9_3` (vs_5_0 reserves D3D9-era HLSL keywords like `point`). The level_9_3 trade-off inherited the SM2.x-style 256-instruction-slot cap; that's what X5615 surfaces.

### Root cause

Plan 11-07 Iter-3's profile choice (`vs_4_0_level_9_3`) was correct for its time (defended against vs_5_0's keyword reservation) but the level_9 profile bucket carries SM2.x feature-level constraints — including the 256-instruction-slot ceiling per fxc target docs (https://learn.microsoft.com/en-us/windows/win32/direct3dhlsl/specifying-compiler-targets). The `vertex_program/a_specmap_bump_vs20_for_ps20.vsh` shader compiles to 293 instructions, busting the 256 cap by 37 slots. Real D3D9 drivers compile the same source under vs_2_0 with vendor-extension headroom; D3DCompile is strict.

Hypothesis confirmed verbatim. Profile bump to `vs_4_0` drops the cap (~4096-slot envelope) while keeping the relaxed legacy-keyword treatment.

### Fix

Per `11-09.6-PLAN.md` Task 1:

- **1a.** `Direct3d11_VertexShaderData.cpp:97`: `kVertexShaderProfile = "vs_4_0"` (was `"vs_4_0_level_9_3"`).
- **1b.** `Direct3d11_PixelShaderProgramData.cpp:62`: `kPixelShaderProfile = "ps_4_0"` (was `"ps_4_0_level_9_3"`).
- **1c.** `D3D11_REWRITE_VERSION` bump from `14` → `15` in both files (Direct3d11_VertexShaderData.cpp:338 + Direct3d11_PixelShaderProgramData.cpp:139). Forces `.cso` cache regeneration on next launch.
- **1d.** Append rationale-block comment update in both files documenting the Plan 11-09.6 bump (level_9_3 → vs_4_0; X5615 trigger; vs_5_0 still avoided).

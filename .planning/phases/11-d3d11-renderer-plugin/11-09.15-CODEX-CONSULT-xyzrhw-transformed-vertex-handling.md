# CODEX Consult — D3D11 Handling of D3D9 `D3DFVF_XYZRHW` (Pre-Transformed) Vertices

**Context date:** 2026-05-22 (post-Plan-11-09.15-Iter-4 A+B test smoke at commit `36534c488`)

**Status:** Diagnosis complete with HIGH confidence; consulting on fix-path selection.

---

## What we're trying to decide

We've identified the root cause of the "radial-from-world-anchored-point" rendering artifact that has dominated D3D11 visual parity for the back half of Plan 11-09.15. The cause is **D3D9 `D3DFVF_XYZRHW` (pre-transformed) vertex handling** — not the fan-list IB path (Plan 11-09.15) that originally seemed implicated.

**The question:** Which of three fix paths should we take to make D3D11 render XYZRHW callers correctly, matching D3D9 VSPS behavior?

1. **Path A — Shader-level fix:** Modify SWG's UI/2D `.vsh` HLSL bodies (or our HLSL injection layer in `Direct3d11_VertexShaderData::compile`) to detect transformed input and skip the world-view-proj multiply.
2. **Path B — Engine-side CPU conversion:** In `Direct3d11_StateCache::setVertexBuffer`, when `VBFormat.isTransformed()`, run a CPU-side pixel-to-NDC conversion on the dynamic VB slice before submission. The downstream VS then receives clip-space data and passes it through.
3. **Path C — Plugin-local passthrough VS substitution:** When `VBFormat.isTransformed()`, the D3D11 plugin substitutes a generic passthrough VS that does pixel→NDC conversion using current viewport dimensions, ignoring the engine's per-pass VS bytecode.

Our read: **Path A is closest to the D3D9 VSPS shape and likely the right answer** — but we want CODEX to validate the diagnosis, sanity-check whether SWG's stock `.vsh` files already contain a transformed-vert variant we're missing, and confirm path selection.

## Iter-4 Test A — CPU-side VB snapshot (sampled at unlock-time, before Unmap)

`Direct3d11_DynamicVertexBufferData::unlock(int)` was patched to read the slice CPU memory before `Unmap` and dump the first 4 verts to InfoQueue, filtered to fan-shaped slices (`numberOfVertices==4 && vertexSize==24`). The mapped pointer was stashed at `lock()` time so unlock can read it before the driver releases/rotates the backing memory.

Iter-4 smoke (Mos Eisley, ~3 min session, 28065 fan calls, ~19000+ fan-shaped unlocks captured):

**Pattern 1 — Startup loading-screen fullscreen quads (unlocks #1–#2000):**

```
unlock#1 vert#0 pos=(-0.500, -0.500, 1.000) color=0x3F800000 uv=(0.000, 0.000) offsetVerts=0
unlock#1 vert#1 pos=(1023.500, -0.500, 1.000) color=0x3F800000 uv=(1.000, 0.000) offsetVerts=0
unlock#1 vert#2 pos=(1023.500, 767.500, 1.000) color=0x3F800000 uv=(1.000, 1.000) offsetVerts=0
unlock#1 vert#3 pos=(-0.500, 767.500, 1.000) color=0x3F800000 uv=(0.000, 1.000) offsetVerts=0
```

Reading the bytes labeled "color":

- `0x3F800000` is the IEEE-754 bit pattern for **float 1.0**, not a BGRA color.
- Positions are in **pixel coordinates matching the 1024×768 screen resolution** (with D3D9's half-pixel offset convention `-0.5`).
- UVs span (0,0) to (1,1) corner-to-corner.

This is **`D3DFVF_XYZRHW + D3DFVF_TEX1`** — pre-transformed vertex format. Stride = 16 (xyzw float4) + 8 (uv float2) = 24 bytes. Our snapshot misinterpreted bytes 12..15 as "color" because the diagnostic was written assuming the more common `XYZ + COLOR + UV` layout.

The startup splash screen ran ~2000 fan calls of this format pattern. All went to `offsetVerts=0` because the splash is the first dynamic VB user of every frame (per-frame discard cycle, `ms_used=0`).

**Pattern 2 — In-world UI overlays (unlocks #2500+):**

```
unlock#2500 vert#0 pos=(-0.650, 0.650, 0.000) color=0x1AFFFFFF uv=(0.000, 0.000) offsetVerts=2412
unlock#3000 vert#0 pos=(-0.550, 0.550, 0.000) color=0x2CFFFFFF uv=(0.000, 0.000) offsetVerts=2534
unlock#5000 vert#0 pos=(-0.550, 0.550, 0.000) color=0x2CFFFFFF uv=(0.000, 0.000) offsetVerts=2730
```

- Positions in **clip space** (range -0.65..0.65).
- "color" bytes 12..15 = `0x1AFFFFFF` / `0x2CFFFFFF` / `0x00FFFFFF` → genuine BGRA u32 colors with alpha varying (typical UI fade-in/fade-out).
- This is **`D3DFVF_XYZ + D3DFVF_DIFFUSE + D3DFVF_TEX1`** format. Stride = 12 + 4 + 8 = 24 bytes. Same stride, different layout.

Different `offsetVerts` values (2412, 2534, 2746, 2730, 2450) confirm the dynamic VB ring is appending NO_OVERWRITE for these in-world callers.

**Quantitative split:** Pattern 1 dominates startup; Pattern 2 dominates ongoing gameplay. Both routes through `Graphics::drawTriangleFan` → `Direct3d11_StateCache::drawTriangleFan`. Pattern 1 IS the source of the radial artifact.

## Why the bug manifests as a radial-from-world-anchored-point pattern

Mechanism:

1. SWG's UI/2D widget code (e.g., `clientUserInterface/CuiWidgetGroundRadar.cpp:157`) builds vertices via:
   ```cpp
   VertexBufferFormat format;
   format.setPosition();
   format.setTransformed();   // ← THE FLAG
   format.setColor0();
   format.setNumberOfTextureCoordinateSets(2);
   ...
   Graphics::setVertexBuffer(vertexBuffer);
   Graphics::drawTriangleFan();
   ```
   Position data is written in **pixel coordinates** (e.g., (0, 0), (1024, 0), (1024, 768), (0, 768)), `.w` = 1.0 (RHW typically = 1.0 for orthographic UI).

2. D3D9 plugin's `Direct3d9_VertexDeclarationMap` emits `D3DDECLUSAGE_POSITION` (VSPS build) or `D3DDECLUSAGE_POSITIONT` (FFP build) with `D3DDECLTYPE_FLOAT4`. Under VSPS, the engine's `.vsh` body presumably handles the pixel→clip conversion (D3D9 VSPS works correctly in Kenny's environment).

3. D3D11 plugin's `Direct3d11_VertexBufferDescriptorMap::buildInputElementDesc` (lines 217–231) correctly emits `POSITION float4` when `isTransformed()` is true:
   ```cpp
   if (format.isTransformed())
       add("POSITION", 0, DXGI_FORMAT_R32G32B32A32_FLOAT, sizeof(float) * 4);
   else
       add("POSITION", 0, DXGI_FORMAT_R32G32B32_FLOAT, sizeof(float) * 3);
   ```

4. The InputLayout cache binds this correctly. So the VS receives `float4 pos = (-0.5, -0.5, 1.0, 1.0)` (etc.) as input. The VS bytecode then executes — and this is where the divergence sits.

5. **Possibility 1:** SWG's stock UI `.vsh` files contain pass-through VS bodies for transformed callers, but our HLSL preprocessor injection (`Direct3d11_VertexShaderData::compile` injects a `POSITION -> SV_POSITION` macro per Pitfall 1) breaks them. If applied globally, `float4 pos : POSITION` becomes `float4 pos : SV_POSITION` on both VS input AND output — invalid as VS input in D3D11; compile would either fail (yielding null VS, counted in our `skippedNullVS=72025` stat) or succeed-wrong.

6. **Possibility 2:** SWG's stock UI VS code does an unconditional `output.pos = mul(input.pos, worldViewProj)`. In D3D9 VSPS, this works because the engine binds an identity `worldViewProj` for transformed callers. In D3D11, we may be binding the actual world-view-proj from the current camera state, so pixel coords get scrambled by a perspective matrix.

The visual signature — "anchor at world-origin projected to screen, moves with camera" — is the dead giveaway that **`pixel_coord × cameraWorldViewProj` is being computed**. World origin `(0,0,0)` projected to screen is exactly what you'd see if pixel `(0,0)` got run through `worldViewProj`.

## Latent adjacent bug (NOT the radial cause, noted for transparency)

While searching for per-frame state reset, we discovered that the D3D11 plugin **does not call**:

- `Direct3d11_DynamicVertexBufferData::beginFrame()`
- `Direct3d11_DynamicIndexBufferData::beginFrame()`
- `Direct3d11_StaticShaderData::beginFrame()`

These are all defined-but-never-called. D3D9's equivalent (`Direct3d9.cpp:2484-2487` from `beginScene`) calls all of them. `Direct3d11_Device::beginScene` does only RTV/DSV rebind + deferred clears.

**Mitigating context:** D3D9 ships with `ConfigDirect3d9::ms_discardDynamicBuffersAtBeginningOfFrame = false` by default, so D3D9's `beginFrame` is also effectively a no-op for the dynamic-VB ring. So this isn't the radial cause. But it's a latent hygiene gap worth fixing in a follow-up plan (currently the D3D11 dynamic VB ring only resets on ring-overflow `ms_used + length > ms_size`, never per-frame).

## Fix-path comparison

### Path A — Shader-level fix (in `Direct3d11_VertexShaderData::compile` or the .vsh sources)

**Concept:** Inject conditional macros or HLSL fragments into the compile flow so transformed-vert VS bodies do pixel→clip conversion correctly.

**Pros:**
- Closest semantic match to D3D9 VSPS shape (engine-authored VS bodies are the source of truth).
- Once the shader-level path is correct, ALL transformed-vert callers work transparently (UI, splash, bitmap, etc.).
- No per-draw runtime overhead.

**Cons:**
- Requires inspecting (or extracting from TRE archives) the engine's stock UI/2D `.vsh` files to understand what they DO with transformed input.
- May require editing the macro injection in `Direct3d11_VertexShaderData::compile` to handle `POSITION -> SV_POSITION` selectively (only on VS output, not input).
- Risk of regressing non-UI shaders.

**Specific questions for CODEX:**
- Are SWG's stock UI/2D `.vsh` files known to use a recognizable "passthrough" pattern (e.g., naming convention, or a particular HLSL idiom we should look for)?
- Should our HLSL macro injection (`POSITION -> SV_POSITION`) be scoped to VS-output structs only? If so, what's the cleanest way to do that for D3DCompile (which sees the full file)?

### Path B — Engine-side CPU conversion (in `Direct3d11_StateCache::setVertexBuffer`)

**Concept:** When `VBFormat.isTransformed()` is true, walk the dynamic VB slice (which we have CPU access to during `setVertexBuffer` — no, we don't, the engine has already unmapped by then) ...

**Issue:** Actually, we don't have CPU access at `setVertexBuffer` time. The caller has already `unlocked` the dynamic VB. To do CPU conversion we'd need to intercept the lock/unlock pair, which is fragile.

**Pros:**
- Doesn't require touching shader compile path.

**Cons:**
- Lifecycle is awkward — the conversion has to happen between `unlock` and `setVertexBuffer`, but those are caller-driven and we don't intercept them.
- Or we'd have to do the conversion via a CPU readback after Unmap, which D3D11 doesn't allow on `USAGE_DYNAMIC` buffers (no `D3D11_CPU_ACCESS_READ`).
- Or we'd add a parallel "transformed ring buffer" that callers write to via a wrapper — major engine refactor.

**Verdict:** Path B is structurally hard to do cleanly. Likely not the right answer.

### Path C — Plugin-local passthrough VS substitution

**Concept:** When `VBFormat.isTransformed()` is true at `applyPreDrawState` time, IGNORE the engine's per-pass VS and bind a single hardcoded passthrough VS that does pixel→NDC conversion using current viewport dimensions:

```hlsl
cbuffer ViewportCB : register(b0) {
    float2 viewportSizeInv;  // (1.0/width, 1.0/height)
};

struct VSInput {
    float4 pos : POSITION;       // xyzrhw pixel coords from XYZRHW
    float4 color : COLOR0;       // optional
    float2 uv : TEXCOORD0;
};

struct VSOutput {
    float4 pos : SV_POSITION;
    float4 color : COLOR0;
    float2 uv : TEXCOORD0;
};

VSOutput main(VSInput input) {
    VSOutput o;
    // pixel (px, py) -> NDC clip space (-1..1, 1..-1)
    o.pos.x = input.pos.x * viewportSizeInv.x * 2.0 - 1.0;
    o.pos.y = 1.0 - input.pos.y * viewportSizeInv.y * 2.0;
    o.pos.z = input.pos.z;
    o.pos.w = input.pos.w;  // RHW from input
    o.color = input.color;
    o.uv = input.uv;
    return o;
}
```

**Pros:**
- Local to D3D11 plugin; no engine touch.
- Matches D3D9 FFP `D3DDECLUSAGE_POSITIONT` semantic (driver-level passthrough).
- One VS substitution covers ALL transformed callers.
- Compiles offline / known-good HLSL.

**Cons:**
- Loses any per-pass VS logic (e.g., color modulation done in the engine's VS body for a specific 2D shader effect). For SWG's UI/2D shaders, the VS bodies are typically very simple (pass-through + simple alpha modulation) so this may be acceptable.
- Requires a viewport-size cbuffer slot (slot 0 or a dedicated slot).
- Color/UV semantics need to match across SWG's various transformed-vert callers. Some may not have COLOR0; some may have multiple texcoord sets (CuiWidgetGroundRadar uses 2 texcoord sets).

**Specific questions for CODEX:**
- Is plugin-local VS substitution a reasonable pattern in D3D11 architecture, given the engine's per-pass shader binding contract? Or does this break some other invariant in `Direct3d11_StaticShaderData`?
- Can a single passthrough VS handle the multi-texcoord-set variants, or do we need a small family (one per texcoord-set count)?

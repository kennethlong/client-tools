# CODEX Consult — Plan 11-09.8 TEXCOORD2 Input-Signature Mismatch Strategy

**Context date:** 2026-05-21 (post-Plan-11-09.7 close at `ef94b6662`)

**Status:** Pre-implementation. Plan 11-09.8 not yet scaffolded; this consult drives scope decision.

---

## What we're trying to decide

Plan 11-09.6 bumped the VS compile profile from `vs_4_0_level_9_3` to plain `vs_4_0`. Plan 11-09.7 wired the multi-stream VertexBufferVector slots, allowing the engine to advance from char-cust-pedestal load (Plan 11-09.6 crash) into character-creation avatar composition. Plan 11-09.7's post-fix smoke surfaced 20+ identical D3D11 debug-layer ERRORs immediately before the engine's c0000005 access violation:

```
D3D11 [ERROR] category=5 id=163:
ID3D11Device::CreateInputLayout: The provided input signature expects to read an
element with SemanticName/Index: 'TEXCOORD'/2, but the declaration doesn't
provide a matching name.
```

The VS bytecode (compiled under our new `vs_4_0` profile) declares an input for `TEXCOORD2`, but the engine `VertexBufferFormat` for the failing mesh doesn't provide a TEXCOORD2 element. SM4 strict signature validation rejects the layout. `CreateInputLayout` returns FAIL → our `Direct3d11_VertexDeclarationMap` (single-stream OR multi-stream variant) caches `nullptr` → `applyPreDrawState` skips the draw → no geometry on those passes.

`vs_4_0_level_9_3` was lenient about this — D3D9-era HLSL routinely declares TEXCOORD0..3 inputs even when the body only reads TEXCOORD0..1, and the older profile dead-stripped or just ignored the unused declarations. `vs_4_0` preserves declared inputs in the signature regardless of usage.

Surface evidence:
- Plan 11-09.7 smoke produced **24,291 CreateInputLayout + IASetVertexBuffers calls** logged. Multi-stream pipeline IS dispatching.
- Kenny observed **partial-magenta draws in the black area** — SOME VS/VBFormat combinations DO produce successful layouts (those VS that don't declare TEXCOORD2), producing visible Plan 11-09 fallback-PS magenta pixels.
- Most draws fail — the bulk of UI + skeletal-mesh + character shaders declare TEXCOORD2 inputs.
- Crash dumps preserved at `stage/SwgClient_d.exe-unknown.0-20260521235916.{mdmp,txt}` + `stage/SwgClient_d.exe-unknown.0-20260522000101.{mdmp,txt}` (both 484 bytes; identical c0000005 access violation on `object/tangible/hair/human/shared_hair_human_male_s40.iff` load with `shader/ui_shader_add.sht`).

This is **exactly CODEX Q2 cascade prediction item 3** from the Plan 11-09.6 consult:
> 3. VS output / PS input mismatch at runtime, not necessarily compile time. The magenta fallback PS only consumes SV_POSITION, so it is safe; real future PS HLSL will need matching signatures.

Except it's manifesting on the VS INPUT side (VBFormat → VS) not the VS-output → PS-input side. Same underlying cause: SM4 strictness on signature equality.

## Our current hypothesis

Engine D3D9-era HLSL routinely declares TEXCOORD0..N input attributes even when the body doesn't read them all. Under `vs_4_0` these get preserved in the input signature. Our `Direct3d11_VertexBufferDescriptorMap::buildInputElementDescForStream` walks the `VertexBufferFormat` and emits a TEXCOORD<n> element ONLY for the texcoord sets the format provides. The VBFormat is authoritative for the mesh's actual vertex stream contents — it can't fabricate a TEXCOORD2 attribute that isn't there.

So the layout describes "what the VB actually contains" and the VS signature describes "what the VS shader declares" — and they're a partial mismatch.

Three rough architectural buckets, in order of intrusiveness:

**Bucket A — Pad the input layout with phantom elements to match the VS signature.**
ID3D11InputLayout permits elements at higher InputSlots than the bound VB count IF those slots are unbound (driver passes zero). Or we could create a dummy 16-byte zero-stride buffer and bind it at a "phantom" slot, then add a `D3D11_INPUT_ELEMENT_DESC` for TEXCOORD2 pointing at that slot. The VS reads zero for the unused attribute, semantics match, validation passes.

Variants:
- A1: Statically bind a global zero-buffer at slot 7 (or whatever); always include a TEXCOORD2 phantom element. Heavy if the engine's slot management collides.
- A2: Use ID3D11ShaderReflection to introspect the VS signature; add phantom elements only when the VS declares a semantic that the VBFormat doesn't provide; bind the global zero-buffer at the corresponding slot when needed.

**Bucket B — Rewrite the HLSL to strip unused TEXCOORD declarations before compile.**
Plan 11-07 already has a `Direct3d11_HlslRewrite.cpp` rule pipeline (Rules A through E). Add a "Rule F" that scans VS source for TEXCOORD declarations and removes any that the bytecode wouldn't reference. Hard problem because we need to know which inputs are unused BEFORE compile — could use a two-pass approach (compile, reflect, recompile with stripped inputs) but that doubles compile cost. Or use heuristics (e.g., scan the function body for sampler reads + lerp uses to infer which TEXCOORDs are referenced).

**Bucket C — Use ID3D11ShaderReflection to drive the input-layout-element set from the VS bytecode, not the VBFormat.**
At input-layout creation time, after compile, query the VS bytecode's input signature via `ID3D11ShaderReflection::GetInputParameterDesc`. For each declared input, find a matching VBFormat element by semantic; if found, emit at the format's stride/offset/format; if not found, emit a phantom element at a sentinel InputSlot pointing at a zero-buffer. This is the "right" architectural answer but is the most invasive — touches input-layout-cache key, descriptor-map, applyPreDrawState.

**Bucket D — Selective fallback to vs_4_0_level_9_3 for shaders that hit this.**
The CODEX 11-09.6 consult rejected the fallback-ladder pattern because mixed-profile renders complicate the codebase. But if the answer to "what to do about the cascade" is "let some VSes be more lenient," this is the simplest path. Cost: re-introduces the 256-instruction-slot cap for those shaders.

## Specific questions for CODEX

### Q1: Is this expected SM4 behavior, or a quirk we should expect to surface again?

Specifically: when the source HLSL `struct VsIn { float4 pos : POSITION; float2 uv0 : TEXCOORD0; float2 uv1 : TEXCOORD1; float2 uv2 : TEXCOORD2; ... }` declares fields that the body doesn't read, does fxc:
- (a) Always preserve them in the bytecode input signature under vs_4_0+, OR
- (b) Strip dead inputs aggressively, OR
- (c) Depend on optimization-level flags (D3DCOMPILE_OPTIMIZATION_LEVEL3 affects this)?

If (a), we'll keep hitting this cascade across most engine shaders. If (b), our compile flags might be wrong. If (c), worth tuning.

### Q2: Best architectural fix — Bucket A/B/C/D?

Rank them. The Plan 11-09.6 consult endorsed `vs_4_0` over `vs_5_0` + rejected the fallback-ladder; that endorsement still stands. But which of A/B/C is the right next step?

I lean A2 (ID3D11ShaderReflection + targeted phantom elements + global zero-buffer at a sentinel slot). It's bounded scope — one or two file changes plus a new sentinel zero-buffer in Direct3d11_Device install — and addresses the root cause without source-rewrite gymnastics.

But the consult should think about: does B (HLSL strip rule) buy us anything A2 doesn't? Does C (full reflection-driven) avoid an architectural mistake A2 makes?

### Q3: ID3D11ShaderReflection integration cost

If A2 or C are the answer: what's the cost / complexity of integrating ID3D11ShaderReflection into the input-layout path?
- New dependency (`d3dcompiler.lib` reflection API — already linked via Plan 11-05).
- Per-VS one-time reflection cost (or per-layout-creation? cache the reflection result).
- Any gotchas with how `GetInputParameterDesc` reports `SystemValueType` (SV_INSTANCEID, SV_VERTEXID) that we should NOT pad?

### Q4: Phantom-buffer slot strategy

If A2 is the answer: best strategy for the phantom buffer?
- A 16-byte zero-buffer bound at a sentinel InputSlot (say, slot 7 — far above the 0/1 slots multi-stream uses) on every IASetVertexBuffers call?
- Per-frame zero-buffer rebound lazily?
- Multiple sentinel slots if multiple distinct phantom-element strides are needed (TEXCOORD2 vs COLOR1 vs etc.)?

### Q5: The c0000005 access violation — same root cause or separate?

The crash dump shows the engine deref'd a NULL pointer during world-load of `shared_hair_human_male_s40.iff`, just after the InputLayout ERRORs. Most likely the engine's draw-dispatch code path assumed an InputLayout (or downstream resource) would be non-null. Is this likely:
- (a) A direct consequence of CreateInputLayout returning FAIL — our `applyPreDrawState` skips the draw cleanly, but a NEXT-stage engine call expected the draw to have happened and accesses NULL state?
- (b) An independent issue (engine asset corruption, dangling pointer, etc.)?

If (a), fixing TEXCOORD2 likely fixes the access violation as a side effect. If (b), we need to investigate the engine call path independently as Iter-2.

### Q6: Plan 11-09.8 acceptance bar

What does "Iter-1 success" look like for this plan?

I'd propose:
- 0 D3D11 debug-layer ERRORs on CreateInputLayout (the InputLayout problem itself is gone)
- Visible geometry on the char-create avatar preview screen (some real-looking pixels, not just magenta — though that depends on PS path which is mostly fallback)

# CODEX Response — 2026-05-21

**Headline verdict:** Treat as real SM4 input-signature/layout mismatch (NOT a D3D11 quirk to work around with compiler flags). Implement reflection-guided input-layout assembly with a bound zero phantom slot. Do NOT revert shader model. Do NOT hardcode only TEXCOORD2. Do NOT rewrite HLSL just to hide unused inputs.

## Q1 — Expected vs quirk + compile flags

**Verdict:** This is **expected SM4 behavior**.

`CreateInputLayout` validates the layout against the compiled VS input signature. Under `vs_4_0` (especially in debug with `D3DCOMPILE_SKIP_OPTIMIZATION`), unused declared inputs can remain in the signature. TEXCOORD2 showing up after the profile bump is plausible and not surprising.

**Do NOT** try to "fix" this by tuning optimization flags — risks debug/release divergence. The compiled VS bytecode signature should be the authority.

Docs:
- Microsoft `CreateInputLayout`: https://learn.microsoft.com/en-us/windows/win32/api/d3d11/nf-d3d11-id3d11device-createinputlayout
- Microsoft `D3DReflect`: https://learn.microsoft.com/en-us/windows/win32/api/d3dcompiler/nf-d3dcompiler-d3dreflect

## Q2 — Architectural bucket ranking

1. **C-lite / A2 hybrid:** Reflect the VS input signature, build the input layout from reflected requirements, match real VB elements where present, supply phantom zero-buffer elements ONLY for missing non-system semantics.
2. **A2 if smallest patch is wanted:** reflection-guided augmentation only. Do NOT hardcode TEXCOORD2.
3. **B (HLSL strip rule)** is risky. Source-level HLSL stripping is hard to prove correct, and `Direct3d11_HlslRewrite.cpp:1` is already doing fragile compatibility rewrites.
4. **D (selective vs_4_0_level_9_3 fallback)** should be rejected. Returning to level_9_3 reintroduces the 256-instruction-slot problem.

## Q3 — ID3D11ShaderReflection integration cost

**Verdict:** Modest cost; worth it.

Best shape:
- Reflect **ONCE per compiled VS bytecode** in `Direct3d11_VertexShaderData.cpp:97`.
- Store a compact list of required input params as a class member.
- Use during layout construction in `Direct3d11_VertexDeclarationMap.cpp:84`.

**Do NOT** reflect inside every draw. Per shader / layout creation is fine.

**Skip system-value inputs** (`SV_VertexID`, `SV_InstanceID`, etc.). For normal attributes, derive format from reflected component type + mask.

## Q4 — Phantom-slot strategy

**Verdict:** Real phantom zero buffer at **slot 15** (NOT slot 7).

Strategy:
- Reserve input slot 15 (better than slot 7 because engine uses low slots for real streams; D3D11 exposes 16 IA slots total).
- Create a small zero-filled buffer (immutable, BIND_VERTEX_BUFFER).
- Bind it with **stride=0**, so every vertex reads the same zero value.
- Add missing reflected semantics to the layout with `InputSlot=15`, `AlignedByteOffset=0`.
- Track whether the cached layout needs the phantom slot, OR initially bind slot 15 unconditionally before draws.

Multiple phantom elements CAN share the same zero buffer/offset — the goal is stable zero input, not meaningful per-vertex data.

## Q5 — c0000005 access violation

**Verdict:** Should NOT be assumed to share root cause.

Current code suggests failed layouts cause skipped draws, NOT direct null deref:
- `Direct3d11_StateCache.cpp:500` returns false when input layout is null BEFORE binding/drawing.
- `Direct3d11_VertexDeclarationMap.cpp:84` returns nullptr on failed layout creation but does NOT cache the nullptr.

**Plan:** Fix the input-layout errors first (real + blocking). If hair/avatar crash remains, treat as Iter-2 with a real call stack / minidump.

## Q6 — Plan 11-09.8 acceptance bar (approved + refined)

- Zero `CreateInputLayout` missing-semantic errors for the failing character/avatar path.
- Formerly failing VS/VB combinations produce non-null input layouts.
- `ms_skippedDrawsNullLayout` no longer climbs for those draws.
- Avatar-preview geometry progresses beyond partial/missing output.
- Magenta fallback pixels acceptable for now (real PS support is separate).
- Existing Plan 11-09 clear/draw/multi-stream behavior does NOT regress.
- The c0000005 is either gone OR explicitly captured with a stack + moved to next investigation.

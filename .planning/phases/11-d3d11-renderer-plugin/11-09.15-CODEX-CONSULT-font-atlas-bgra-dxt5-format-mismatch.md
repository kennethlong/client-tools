# CODEX + Cursor Consult — D3D11 Plugin: Font Atlas BGRA8 Source → DXT5 Destination → No Format Conversion in `unlock()` → Garbage Glyphs

**Context date:** 2026-05-22 (post-Plan-11-09.15 Iter-30 texture-creation diagnostic)

**Status:** Root cause localized cold. Iter-29B routing + Iter-30 texture-creation logging + manual TRE extraction of `texture/font/verdana_14_000.dds` from all four TREs that ship it (`data_texture_02.tre`, `patch_05.tre`, `patch_06.tre`, `patch_12_00.tre`) confirm a format mismatch the D3D11 plugin's lock/unlock path does not handle, and the D3D9 sibling does. We need direction on the cleanest fix.

---

## What we localized

### The font atlas resource
- File: `texture/font/verdana_14_000.dds` (Verdana 14pt — the UI default font, drawn through `shader/uicanvas_filtered.sht`)
- All four shipped versions (base + 3 patches) are **128×128 or 256×256 BGRA8 uncompressed** (DDS_PIXELFORMAT.dwFlags = 0x41 = `DDPF_RGB | DDPF_ALPHAPIXELS`, dwFourCC = 0, 32bpp, BGRA mask layout). First pixel is `FF FF FF 00` = white with α=0 → the canonical "white glyph + alpha mask" font atlas
- Patch 12 (highest priority — `maxSearchPriority=12` in `client_d.cfg`) is the live version: **256×256 BGRA8, 262,144 bytes pixel data + 128 byte header**

### What the engine creates at the GPU
From Iter-30 in-plugin texture-creation logger (one line per `Direct3d11_TextureData` ctor at SRV-create time):
```
Iter-30 tex#200 SRV=0x2093E5BC file='texture/font/verdana_14_000.dds'
                W=256 H=256 Mips=1 destFormat=11 nativeDXGI=77
```
- **`destFormat=11` = `TF_DXT5`**
- **`nativeDXGI=77` = `DXGI_FORMAT_BC3_UNORM`**
- The texture is created as BC3 / DXT5

### Other Verdana atlases — for contrast
```
Iter-30 tex#205 SRV=0x2093FEAC file='texture/font/verdana_bold_13_000.dds' destFormat=0 nativeDXGI=87
Iter-30 tex#206 SRV=0x2093E064 file='texture/font/verdana_bold_12_000.dds' destFormat=0 nativeDXGI=87
```
- `destFormat=0` = `TF_ARGB_8888`, `nativeDXGI=87` = `DXGI_FORMAT_B8G8R8A8_UNORM`
- These get the EXPECTED native BGRA8 destination. Only `verdana_14_000` ends up as BC3

### What happens during data upload
`Texture::loadSurface` (`src/engine/client/library/clientGraphics/src/shared/Texture.cpp:337–426`) reads the DDS bytes and calls `lock(lockData)` then writes pixels. `lockData.getFormat()` is the **source** DDS format (TF_ARGB_8888 here), NOT the runtime destination format.

`Direct3d11_TextureData::lock` (`src/engine/client/application/Direct3d11/src/win32/Direct3d11_TextureData.cpp:521`) at line 521 does:
```cpp
DXGI_FORMAT const native = translationTable[lockData.getFormat()];   // = BGRA8 for our font
...
D3D11_TEXTURE2D_DESC desc = {};
desc.Format = native;                                                 // staging is BGRA8
device->CreateTexture2D(&desc, nullptr, &staging);                   // BGRA8 staging
context->Map(staging, ...);                                          // engine writes BGRA8 bytes
lockData.m_pixelData = mapped.pData;
```

Then `Direct3d11_TextureData::unlock` (around line 781) does:
```cpp
context->Unmap(staging, 0);
context->CopySubresourceRegion(
    m_texture.Get(),                  // BC3 destination
    dstSubresource,
    ..., staging, ...);                // BGRA8 source
```
**Cross-format CopySubresourceRegion — illegal per D3D11 spec.** No format-family compatibility between BGRA8_UNORM (87) and BC3_UNORM (77). Either silently produces garbage or fails.

The D3D11 debug layer is on for this session and we see no `category=3` (RESOURCE_MANIPULATION) errors mentioning the copy. Hypothesis: the runtime allows the copy raw-bytes (no transcoding), and the BGRA8 row-pitch (1024) doesn't match a BC3 row-pitch (256 = 64 blocks × 16 bytes for 256-wide), so only some fraction of the BGRA8 bytes lands in the BC3 layout and decodes as block-uniform colors — consistent with the visual symptom of "solid colored hashmark blocks at glyph quad positions" that Iter-29A1/A2 (alpha-composite PS variants) produced.

### Contrast — how D3D9 handles this exact case
`Direct3d9_TextureData::lock` (`src/engine/client/application/Direct3d9/src/win32/Direct3d9_TextureData.cpp:500–569`) takes the **format-mismatch branch** explicitly:
- Creates an `IDirect3DSurface9` "plain offscreen surface" at `lockData.getFormat()` (BGRA8 for us)
- Hands that to the engine to write pixels
- In `unlock` (`:603`): `D3DXLoadSurfaceFromSurface(destSurface, ..., plainSurface, ..., D3DX_FILTER_NONE, 0)` — D3DX does format conversion + BC3 compression CPU-side and writes valid block-compressed bytes into the dest

The D3D11 plugin has no equivalent code. `unlock` just `CopySubresourceRegion`s the BGRA8 staging into the BC3 destination directly.

## Why does the engine pick TF_DXT5 for this one font but not the others?

Genuinely unknown right now. Two clues:
1. The `Texture::load(filename)` path uses `ms_conversions[sourceFormat]` to build the `runtimeFormats` list passed to `createTextureData`. For sourceFormat = TF_ARGB_8888 (per the BGRA8 DDS header), the allowed conversions are `[TF_ARGB_8888, TF_ARGB_4444]` only (`Texture.cpp:81–82`). **TF_DXT5 is NOT in that list.** So a vanilla `Texture::load("verdana_14_000.dds")` cannot reach `destFormat = TF_DXT5`.
2. The non-load Texture constructors take a caller-supplied `runtimeFormatArray`. Some font asset path must be constructing the font's texture with an explicit `[TF_DXT5, ...]` array, then calling `load()` afterward. Iter-30 only reveals the loaded filename + dimensions + final destFormat; it does NOT show how `createTextureData` was called.

The asset .inc descriptor (`ui/font/verdana_14.inc` — extracted and inspected) is just a glyph-rectangle table — no texture-format opinions, just per-glyph `sourcerect='x1,y1,x2,y2'` references into the atlas. Nothing in the .inc selects DXT5.

Other Verdana fonts (`verdana_bold_13_000`, `verdana_bold_12_000`) end up at `TF_ARGB_8888` — same loader, same UI canvas template, just a different `destFormat` outcome.

## What we want from CODEX + Cursor

1. **WHERE does the engine pick `TF_DXT5` as the runtime format for `verdana_14_000.dds`?** Likely candidates: a SWG font subsystem (`sharedFont`/`UIFontCharacter`/etc) that pre-allocates a font atlas Texture with an explicit `[TF_DXT5, …]` runtime list, then loads the DDS bytes into the pre-allocated texture; or an engine-wide "compress UI textures to DXT5" heuristic keyed on dimensions / power-of-two / format. Pointers to specific call sites preferred.

2. **CLEANEST fix direction — three options we see:**

   **A.** **Format-correct staging in `Direct3d11_TextureData::lock`** — when `lockData.getFormat() != m_destFormat`, create the staging at `lockData.getFormat()` (source format) and add CPU-side conversion in `unlock` before the CopySubresourceRegion. For BGRA8→BC3 specifically, we'd need a BC3 encoder. DirectXTex (`DirectX::Compress`) is the canonical Win32 dep; alternatively, a stub stb_dxt-style block-compressor.
      Pros: full D3D9 parity; future-proof for other format-mismatch cases.
      Cons: adds a dependency (or 200 lines of BC3 encoder); CPU cost at texture load time.

   **B.** **Override `destFormat` when mismatch detected on first lock** — if the engine asks for `TF_DXT5` destination but first `lock()` arrives with `TF_ARGB_8888` data, recreate `m_texture` at BGRA8 and the matching SRV. The font atlas burns 4× the GPU memory (256×256 BGRA8 = 256KB vs BC3 = 64KB) but it just works.
      Pros: trivially small change in `Direct3d11_TextureData`; no engine modification; no new dep.
      Cons: silently violates the engine's stated runtime-format preference; could mask other bugs of the same shape; recreate-mid-load could race lock cycles.

   **C.** **Fix it at the engine level** — track down where the font subsystem requests `TF_DXT5` for BGRA8 source data and either change the runtime-format list to `[TF_ARGB_8888, TF_ARGB_4444]` (matching what Texture::load() would have chosen for the same source), or have the font code load via `Texture::load(filename)` directly rather than the explicit-runtimeFormats path.
      Pros: corrects the actual misconfiguration; no D3D11 plumbing changes; D3D9 might also be running on a wasted DXT5 round-trip we'd save.
      Cons: requires finding the offending caller; might be a deeper engine assumption baked into other places.

   We lean toward C if the offending call site is small and isolated, A if it's diffuse and we want defense-in-depth, B never (smell test fails).

3. **For diagnostic confirmation before committing:** is there a faster way to confirm "this is the bug" than implementing one of the fixes? E.g., a one-line override at the start of `Direct3d11_TextureData` constructor — "if any candidate format is TF_ARGB_8888, prefer it over compressed candidates" — would test the hypothesis in one build cycle. Worth doing as a smoke before deciding A/B/C?

4. **Once fonts render**, what's the cleanest path to extend the same fix-direction to **splash backdrops** (the residual H3 from the previous consult — splash uses `shader/2d_texture.sht` + `2d_texture.vsh`, no COLOR0 output)? Likely a separate problem — splash atlas content may need TF_ARGB_8888 modulation by texture-factor, which our current "TEXCOORD-only → raw `tex.Sample`" PS branch doesn't apply.

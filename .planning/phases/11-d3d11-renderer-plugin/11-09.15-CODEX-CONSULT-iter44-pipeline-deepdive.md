## CODEX consult response - 2026-05-23

### Executive diagnosis

The D3D11 plugin is no longer a "can it render anything?" scaffold. It is a functioning D3D11 draw backend with real device/swap-chain setup, texture/SRV creation, static/dynamic VB/IB plumbing, transform cbuffer upload, most topology dispatch, render-target plumbing, per-pass depth/alpha/blend/color-write state, and late-bound SRV/sampler arrays.

Conceptually, it is still not a semantic D3D9 renderer. The remaining largest gap is not one more missing state setter. It is that D3D9's material model is split across render states from `ShaderImplementation::Pass`, FFP texture-stage cascades from `ShaderImplementation::Pass::Stage`, optional D3D9 pixel shader bytecode, and D3D9 FFP lighting/material/fog/alpha-test behavior.

The D3D11 backend currently converts only part of that model. It binds textures to slots, but the fallback PS only evaluates `sample(t0, uv0) * COLOR0` or direct color. That explains why single-diffuse, vertex-color, and alpha-mask paths can become recognizable, while eyes/head/terrain/detail/reflection/material-specific passes remain wrong even though the draw state and SRVs are bound correctly.

The right Phase 12 shape is:

1. Finish pass-state parity where D3D11 has direct equivalents or small shader-cbuffer equivalents.
2. Add a shader-material evaluator for the dominant SWG FFP texture-stage cases, generated from engine `ShaderImplementation::Pass::Stage`, not guessed from bound SRVs.
3. Separately decide whether old D3D9 PS bytecode assets are to be reauthored/translated, because that is a structural asset-pipeline problem, not an iteration-sized bug.

### Q1 - comprehensive gap inventory

#### A. Still routed to `scaffold_fatal_stub`

Current D3D11 `STUB(...)` slots in `Direct3d11.cpp`:

- `resize`
- `lockBackBuffer`, `unlockBackBuffer`
- `presentToWindow`
- `setMouseCursor`, `showMouseCursor`
- `getOneToOneUVMapping`
- `optimizeIndexBuffer`
- `setBloomEnabled`
- `writeImage`
- debug video-buffer slots under `PRODUCTION == 0`: `createVideoBuffers`, `fillVideoBuffers`, `getVideoBufferData`, `releaseVideoBuffers`

Impact:

- `resize`, `presentToWindow`, cursor slots are user/workflow completeness gaps, not currently the Mos Eisley visual deltas.
- `lockBackBuffer` / `writeImage` overlap screenshots/readback tooling; if screenshots now work elsewhere, these still remain API holes.
- `setBloomEnabled` is part of output/lighting parity and explains missing bloom-side semantics.
- `getOneToOneUVMapping` can become visible in UI/post-process/screen-space texture alignment once those paths exercise it.

#### B. D3D9 pass render-state stream only partially mirrored

D3D9 ground truth is `Direct3d9_ShaderImplementationData::Pass::construct/apply`:

- Always pushes `D3DRS_SHADEMODE`
- `D3DRS_DITHERENABLE`
- `D3DRS_ZENABLE`, `D3DRS_ZWRITEENABLE`, `D3DRS_ZFUNC`
- `D3DRS_BLENDOP`, `D3DRS_SRCBLEND`, `D3DRS_DESTBLEND`
- `D3DRS_ALPHATESTENABLE`, `D3DRS_ALPHAFUNC`
- color write mask
- for FFP passes: `D3DRS_LIGHTING`, `SPECULARENABLE`, `COLORVERTEX`, material-source states
- stencil enable, compare, fail/zfail/pass, reference, mask, write mask
- if supported: two-sided stencil and CCW stencil func/ops

D3D11 currently covers:

- alpha blend enable: yes
- alpha blend src/dst/op: yes after Iter-44C
- depth enable/write/compare: yes
- color write mask: yes
- alpha-test enable/reference: yes, but with incomplete compare semantics
- cull/fill/scissor/viewport/fog: partially present in `StateCache`

Missing or wrong:

- `m_alphaTestFunction` is ignored. D3D11 PS `clip()` currently assumes the dominant `>= ref` style. Any pass using less/equal/always/never/not-equal will be wrong.
- Stencil pass-state from `ShaderImplementation::Pass` is not mirrored into `D3D11_DEPTH_STENCIL_DESC`. Defaults install with stencil disabled and reference `0` at `OMSetDepthStencilState`.
- `D3DRS_SHADEMODE` is not represented. D3D11 has no FFP Gouraud/flat render state; flat shading requires shader interpolation modifiers or geometry/vertex data treatment.
- `D3DRS_DITHERENABLE` has no D3D11 equivalent and can be documented as ignored unless a postprocess dither is wanted.
- FFP material-source states and FFP lighting/specular/colorvertex are not implemented. This is a material/lighting cbuffer plus shader-generation gap, not a simple state object gap.
- Alpha-to-coverage is hardcoded false. D3D9 pass data shown here does not push alpha-to-coverage, so this is probably an acceptable limitation unless engine config elsewhere expects MSAA foliage behavior.
- Point-size/point-sprite states are deliberate no-ops. Stars/point sprites render at hardware default point size unless emulated by CPU/GS expansion.
- Multisample setup exists at device/present parameter level in D3D9; D3D11 rasterizer state uses `MultisampleEnable` but sample mask is always `0xffffffff`. No per-pass multisample mask model is present.

#### C. Texture-stage cascade is the largest semantic hole

D3D9 FFP pass construction maps every stage to:

- `D3DTSS_COLOROP`, `COLORARG0/1/2`
- `D3DTSS_ALPHAOP`, `ALPHAARG0/1/2`
- `D3DTSS_RESULTARG`
- then explicitly disables the next stage with `D3DTOP_DISABLE` to terminate the cascade when no PS is active

The engine enum maps all D3D9 texture ops:

- select, modulate/modulate2x/modulate4x
- add/addsigned/addsmooth/subtract
- blend diffuse/texture/factor/current alpha
- premodulate
- modulate-alpha/color add variants
- bump env map, bump luminance, dotproduct3
- multiplyadd, lerp

D3D11 currently binds slots 0..7 SRV/sampler, but the dynamic fallback PS reads only `t0/s0`, `TEXCOORD0`, and optional `COLOR0`. Therefore multi-stage binding is necessary but not sufficient. Eye/head symptoms are strongly explained by this. Terrain detail blends, spec maps, environment/reflection maps, light shafts, and normal/dot3 paths remain structurally wrong until the pass-stage cascade is evaluated in HLSL.

#### D. Pixel shader asset path is structurally incomplete

The D3D11 `createPixelShaderProgramData` path creates wrappers but cannot use shipped D3D9-era pixel shader bytecode as D3D11 bytecode. The per-VS dynamic PS is a bridge, not a replacement for asset pixel shaders.

- FFP-style passes can be addressed by generating D3D11 HLSL from engine stage ops.
- Real authored D3D9 PS passes need source HLSL recompile, bytecode translation, or replacement shader library. This is Phase 12+ asset pipeline work.

#### E. Vertex transform paths

Canonical static geometry:

- `ShaderPrimitiveSet::prepareToDraw` calls `Graphics::setObjectToWorldTransformAndScale(m_owner.getTransform_w(), m_owner.getScale())`.
- D3D11 row-scale fix applies here.

Identity/world-space dynamic geometry:

- `ParticleEmitter::prepareToDraw` writes world-space particle quad vertices and sets `Transform::identity`.
- `ShaderPrimitiveSorter` screen/world helper path writes a dynamic VB and sets `Transform::identity`.
- Global water paths set identity and render pre-worlded/static large geometry.
- Texture renderer and postprocess screen paths also use identity or 2D transformed-vertex style geometry.

Object-local dynamic ribbon/trail/beam/sprite geometry:

- `RibbonAppearance` builds a dynamic VB in owner-local space after deriving offsets from world-space calculations, then sets owner transform.
- `TrailAppearance` builds owner-local quad/fan data, then sets owner transform.
- `BeamAppearance` builds local quad data and sets a camera-facing rolled transform.
- `SpriteAppearance` sets camera transform for billboard-facing local quad data.
- `SwooshAppearance` and `ParticleEmitter` are dynamic-VB heavy and need separate scrutiny in smoke logs.

Current D3D11 handling:

- Dynamic VB ring-buffer upload exists and `StateCache::setVertexBuffer` handles `DynamicVertexBuffer`.
- Non-indexed fan expansion exists.
- Quad-list expansion exists.
- Triangle strip/list and indexed list/strip paths exist.
- Indexed triangle fan paths are known broken by the code comments: they bind `TRIANGLELIST` against fan-shaped indexed data and only count/log the event. If smoke ever shows nonzero indexed-fan calls, this becomes a real geometry bug.

The "stretched ribbon" symptom is less likely to be "bypasses `setObjectToWorldTransformAndScale` entirely" for `RibbonAppearance`/`TrailAppearance`, because those do call it. More likely candidates:

- dynamic geometry generated in already-world-space coordinates but still transformed again, or generated local but paired with identity
- wrong dynamic VB offset/stride/count in a specific primitive path
- triangle-strip/fan topology mismatch for a trail/swoosh path
- shader input layout mismatch for dynamic vertex formats with color/uv
- PS/state issue making an otherwise correct strip visually appear as a solid stretched band

For Phase 12, add draw-path instrumentation that logs active appearance/shader + vertex format + topology + transform class: identity, owner, camera-facing, XYZRHW. Source search alone is not enough to prove the live path for the Mos Eisley ribbon.

#### F. XYZRHW / transformed vertices

D3D11 has explicit support for transformed-vertex VS handling and dynamic PS generation based on reflected VS outputs. This solved several UI/post-FX linkage problems. Remaining risks:

- Transformed paths depend on matching the correct 2D VS and generated PS signature. Any old path that expects D3D9 FVF `XYZRHW` behavior without a compatible VS can still misrender.
- `getOneToOneUVMapping` is stubbed, so exact screen-space texel mapping can still be off.
- Back-buffer lock/present-to-window/video paths remain stubs.

#### G. Render targets, readback, resolve/present

Implemented enough for baked-RT format work and screenshot progress:

- Texture creation, SRV pairing, render target texture path, `setRenderTarget`, copy to non-RT texture have real D3D11 equivalents.
- BGRX/BGRA format strictness has been handled for the known baked-RT case.

Remaining risks:

- `resize`, `presentToWindow`, `lockBackBuffer`, `unlockBackBuffer`, `writeImage` are still stubs.
- MSAA resolve semantics are not equivalent to D3D9 unless explicitly wired.
- Fullscreen/windowed DXGI behavior is not equivalent to D3D9 reset/present parameter flow.

#### H. Lighting, material, skeletal

Lighting/material:

- D3D9 `Direct3d9_LightManager` selects lights, applies FFP or VS/PS constants, and packs alpha fade/bloom into light data.
- D3D11 slot-0 cbuffer initializes material/light ranges to zero in the default path unless user constants or later code populate them.
- `setLights` exists as a D3D11 API route, but parity with `Direct3d9_LightManager::applyLights` is not established by current smoke.

Skeletal:

- Software blend skeletal path uses dynamic streams and calls `setObjectToWorldTransformAndScale`, so it is compatible with current D3D11 dynamic VB plus transform model.
- Hardware/skinned shader paths that rely on bone matrices in vertex shader constants need explicit confirmation. If source VSes expect D3D9 constant register layouts beyond slot-0's WVP/world/camera/fog/material/light layout, they are not guaranteed by the current dynamic PS work.

### Q2 - the PS generation gap

The minimum viable extension is option **(e): read engine `TextureOperation` data and implement the dominant cases first**, not modulate-everything and not per-asset hand bytecode injection as the first move.

Recommended shape:

1. Generate a per-pass material PS, not only a per-VS PS.
2. Key the cache by VS output signature/hash plus pass-stage signature plus alpha-test function plus relevant shader flags.
3. Preserve the current per-VS reflected input-struct generation. That part is correct and solved D3D11 linkage/register-position issues.
4. Replace the body with an evaluator generated from `ShaderImplementation::Pass::Stage`.
5. Implement a compact but real D3D9 FFP cascade subset:
   - `TO_disable`
   - `TO_selectArg1`, `TO_selectArg2`
   - `TO_modulate`, `TO_modulate2x`, `TO_modulate4x`
   - `TO_add`, `TO_subtract`
   - `TO_blendTextureAlpha`, `TO_blendDiffuseAlpha`, `TO_blendFactorAlpha`, `TO_blendCurrentAlpha`
   - `TO_lerp`
   - alpha op equivalents for the same set
   - `TA_current`, `TA_diffuse`, `TA_texture`, `TA_textureFactor`, and complement/alpha-replicate modifiers
6. Log-and-magenta or log-and-fallback for unimplemented ops like bump env map, dotproduct3, multiplyadd until a live asset requires them.

Why not modulate-everything:

- It will improve some eye/head/diffuse overlay cases but will be actively wrong for additive glow, alpha lerp masks, specular, reflection, and normal/dot3 paths.
- It hides which semantic was needed, making future diagnosis worse.

Why not jump straight to hand-authored per-template bytecode:

- It is necessary for true authored D3D9 PS parity, but it does not solve FFP-stage assets generically.
- It creates per-asset churn before the renderer can even express the engine's common material algebra.

So the pragmatic Phase 12 path is an intermediate generated FFP material PS. Treat real D3D9 pixel shaders as a separate "asset shader reauthor/translation" track.

Alpha test extension:

- Current `clip(col.a - ref)` only models one compare direction. Include alpha compare function in the PS key/cbuffer and generate the right predicate or specialized code for the small enum.

Texture coordinates:

- Current PS always samples all textures with `TEXCOORD0`. The FFP stage generator must honor stage texture-coordinate index and texture transform where available. `setTextureTransform` is currently noted as deferred/partial in D3D11 StateCache comments, so terrain/detail scrolling and projected textures can remain wrong until this is in the material key/cbuffer.

### Q4 - render-state architectural model

The D3D11 hash-cache model is the right D3D11 architecture, but it is only correct if every D3D9 semantic mutates the shadow descriptor before each draw. Current completeness:

Covered:

- blend enable
- blend source/destination/op
- color write mask
- depth enable/write/compare
- alpha-test enable/reference through PS cbuffer
- raster fill/cull/scissor/viewport basics
- SRV/sampler slots are fully flushed, reducing sticky texture leakage

Incomplete:

- stencil enable, read/write mask, reference, front-face ops, back-face/two-sided ops
- alpha-test compare function
- shade mode
- dither
- material-source render states
- FFP lighting/specular/colorvertex
- point size/point scale/point sprite emulation
- alpha-to-coverage/MSAA semantic controls
- texture factor/current/temp/resultarg stage state in PS generation
- texture transform/projected texture-coordinate state

Important architectural warning:

- `StaticShaderData::apply` caches by `(ms_active == this && pass)`. That is safe only if all relevant mutable per-pass/per-static-shader state is either immutable for that pass or independently refreshed elsewhere. For D3D9 parity, pass application should be thought of as "push the complete pass semantic packet." Caching can skip redundant GPU state objects, but it must not skip semantic shadow updates if any source value can vary per `StaticShader` instance, alpha ref tag lookup, texture binding, or global state.

### Q5 - gamma / brightness / contrast / tone mapping

D3D9 builds a 256-entry gamma ramp:

```cpp
result = pow(0.5f + contrast * ((f * brightness) - 0.5f), 1.0f / gamma)
```

Then writes it to all RGB channels with `IDirect3DDevice9::SetGammaRamp`, and also fills `ms_colorCorrectionTable`.

D3D11/DXGI model recommendation:

- Do not use `IDXGIOutput6` as the primary answer. It is output/display/HDR metadata oriented, not a portable per-swap-chain replacement for D3D9 gamma ramp semantics.
- Do not accept total gamma loss if visual parity is a goal. The consult's sky washout is exactly the kind of global output delta this controls.
- Implement a final fullscreen post-process color correction pass or LUT pass before present.

Pragmatic implementation:

1. Store brightness/contrast/gamma in D3D11 state when `setBrightnessContrastGamma_impl` is called.
2. If values are all defaults, skip the pass.
3. Otherwise render the resolved scene/backbuffer source through a fullscreen triangle/quad into the presentable backbuffer using the same formula as D3D9, or a 1D LUT texture generated with the D3D9 formula.
4. Preserve screenshot expectations: decide whether screenshots capture pre- or post-color-correction and document it. D3D9 gamma ramp affected display output, not necessarily CPU backbuffer pixels in the same way.

This is Phase 12 visual parity work, not a blocker for geometry/material debugging, but it should be tracked because it globally changes sky/terrain/UI tone.

### Architectural decisions that now look wrong or temporary

1. **Dynamic PS keyed only by VS output.** Correct as a linkage workaround; wrong as a material model. It must evolve to VS signature + pass material signature.
2. **Treating multi-stage SRV binding as visible progress by itself.** It is necessary plumbing, but without generated PS stage evaluation it cannot affect pixels.
3. **Defaulting unimplemented D3D9 states instead of logging pass usage.** This made early smoke cleaner but now hides whether a visual bug is "state never needed" or "state silently ignored."
4. **Alpha test as enable/ref only.** It fixed important cases, but compare function is part of the D3D9 semantic.
5. **Global texture tag low-nibble slot assignment.** Code comments already call this naive. It should become an explicit engine tag-to-slot mapping before environment maps/screen captures matter.
6. **Indexed fan deferral.** Fine if counters remain zero; wrong if any real asset uses indexed fans. The current draw is knowingly incorrect.

#### Must do for "visual parity baseline"

### Direct answer: slot-0-only dynamic PS

The slot-0-only dynamic PS is an iteration-fixable bug only for the simplest class of multi-texture materials: "multiply a small number of ordinary color textures together with vertex color." That would likely improve some head/eye cases.

For renderer parity, it is a structural material-pipeline issue. The correct unit is not "sample all bound slots." The correct unit is "evaluate this pass's D3D9 texture-stage program, or run this pass's authored pixel shader." Sampling slots 1+ without the stage ops will create new false wins and regressions.

So: do not handwave it as Phase 12 asset reauthoring only, and do not patch it as modulate-everything only. Build a real but scoped FFP stage-code generator first, then handle authored D3D9 PS assets as a separate translation/reauthoring track.

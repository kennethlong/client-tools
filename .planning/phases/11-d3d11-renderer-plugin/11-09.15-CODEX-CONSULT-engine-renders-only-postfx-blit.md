# CODEX + Cursor Consult — D3D11 Plugin: Engine Renders ONLY the Post-FX Blit, No Splash UI Draws Fire

**Context date:** 2026-05-22 (post-Plan-11-09.15 Iter-26 all-draws diagnostic)

**Status:** Direct evidence collected. Engine's UI render path emits ZERO draw calls during the early session (200 consecutive `applyPreDrawState` invocations = 200 post-FX blits, 0 UI elements). Need direction on which engine code path is responsible.

---

## What we're trying to decide

Plan 11-09.15 has now had two prior CODEX/Cursor consults (XYZRHW transformed-vertex handling; PS texture.Sample returns zero) and ~13 follow-on diagnostic iterations. All of them assumed the splash UI was rendering correctly but its texture binding or sampling was broken. The latest iter-26 data reveals that hypothesis is **wrong**: **the splash UI never reaches Direct3d11_StateCache at all during the first ~200 frames of the session**. The only draw that fires is `PostProcessingEffectsManager`'s post-FX blit — every frame, exactly one draw call.

We want CODEX + Cursor to identify why SWG's UI render path emits no draws under D3D11 (`rasterMajor=11`) when it presumably emits many under D3D9 VSPS (`rasterMajor=5`), and recommend the next investigation step.

## What we know HARD now

1. **`PostProcessingEffectsManager::ms_copyShader`** (the post-FX blit shader) is `shader/2d_texture.sht`. Its MAIN texture is bound to `ms_primaryBuffer` via line 246 of `PostProcessingEffectsManager.cpp`:
   ```cpp
   ms_copyShader->setTexture(TAG(M,A,I,N), *ms_primaryBuffer);
   ```

2. **Per-frame postFX flow**:
   ```
   CuiIoWin::draw():
     PostProcessingEffectsManager::preSceneRender() // RT = ms_primaryBuffer (if enabled)
     Bloom::preSceneRender()
     IoWin::draw()                                  // UI renders to wherever RT is bound
     Bloom::postSceneRender()
     PostProcessingEffectsManager::postSceneRender() // RT = backbuffer; blit primaryBuffer → backbuffer
   ```

3. **Iter-15 readback** at the blit's drawTriangleFan call shows `ms_primaryBuffer` content = all zeros. UI didn't write to it.

4. **Iter-26** shows 200 consecutive `applyPreDrawState` invocations = 200 blits. **`IoWin::draw()` is emitting ZERO draws** during the early session, OR those draws are bypassing the D3D11 plugin path entirely.

5. **Iter-22** disabled post-FX → everything black including the magenta-line draws that were visible with post-FX enabled. So even those non-textured draws somehow depend on the post-FX path.

6. **Shutdown stats** show 6276 total fan draws across the session. So UI does eventually render — just not in the first 200 frames.

7. **Plan 11-08 Iter-3a** primed slot-0 cbuffer + slots 1/2/3 with default identity matrices + zeros, so the VS doesn't read undefined data.

8. **Plan 11-09.13 Iter-4** generated per-VS dynamic PS handles SV_POSITION+TEXCOORD0 stage linkage correctly. Cursor verified by recompiling the generated HLSL with FXC.

9. **Plan 11-09.14** wires SRV0 + sampler0 binding per pass via `Direct3d11_StaticShaderData::apply`.

10. **Iter-11** added `PSSetShaderResources(0, 16, kNullSRVs)` before `OMSetRenderTargets` in `beginScene` → dropped 1966 RT/SRV-conflict warnings to 0.

11. **Iter-8** routes `beginScene` viewport setup through `Direct3d11_StateCache::setViewport` so c9 (viewportData) is correct from frame 1.

12. **Iter-9** confirmed `cb0[9] = (0.001953, -0.002604, -1.0, 1.0)` at every flush. The 2d_texture.vsh's `transform2d()` math computes correct clip-space pixel→NDC.

13. **D3D11 InfoQueue** shows zero ERRORs and zero WARNINGs in the latest session (post Iter-11 fix).

## Hypothesis space

**H1 — `IoWin::draw()` is short-circuiting early under D3D11 because of some state check that fails.**
SWG's UI render code might check `Graphics::isDX9()` or similar capability and bail out under D3D11. If true, the UI render loop never enters, and the only draws come from PostProcessingEffectsManager which IS unconditional.

**H2 — UI assets fail to load under D3D11 toolchain.**
SWG's UI textures and shaders are stored as `.dds` + `.sht` + `.psh` + `.vsh`. If the texture-load or shader-load path fails under D3D11 (silently — no `FATAL`s, no warnings), then no UI elements can render. The engine might short-circuit "skip this element if its texture is null" check, producing zero draws.

**H3 — UI runs on a separate code path that doesn't reach `Direct3d11_StateCache::applyPreDrawState`.**
There might be a CPU-side bitmap blit (e.g., GDI / `SetCursorPos`-style direct pixel writes) that the splash uses, which doesn't go through the D3D11 shader pipeline at all.

**H4 — `PostProcessingEffectsManager::preSceneRender`'s setRenderTarget(primaryBuffer) is silently failing under D3D11.**
If the RT switch doesn't actually bind primaryBuffer (e.g., because primaryBuffer's RTV creation fails), UI draws that follow would target whatever RT was previously bound. If that's an invalid state, draws might be silently dropped by D3D11 validation (with messages we missed because we don't filter for the right ID).

**H5 — UI rendering is gated on `Bloom::preSceneRender()` succeeding, which depends on PostProcessing internals.**
If Bloom requires PostProcessing's primaryBuffer to be present, and that's failing in D3D11, Bloom might short-circuit and the UI render frame is skipped.

Our read: H1 and H2 are most likely. H3 is unlikely for SWG (it's not a GDI app). H4 — we already verified `OMGetRenderTargets` returns a valid RTV at draw time (the backbuffer), but only at the BLIT draw time; we haven't verified what RT is bound during the supposed-UI draw time (because no UI draws fire). H5 — Bloom is enabled by default but the code path doesn't obviously gate UI on Bloom.

---

## What we want from CODEX + Cursor

1. **Rank H1–H5 (or propose H6).** Where in SWG's code does UI rendering get gated by a graphics-capability check that might fail under D3D11?

2. **What's the cheapest diagnostic?** Possibilities:
   - Hook into `Direct3d11_Device::beginScene` and `endScene` and count the number of `applyPreDrawState` calls per frame — confirm UI render emits zero draws per frame for the first N frames
   - Add a tracepoint in `IoWin::draw()` and `CuiIoWin::draw()` to see if they're being called at all
   - Search for capability checks like `Graphics::isDX9` or `Graphics::getSupportedFeatures` in the UI render path
   - Check whether `GameBitmap::draw()` (or whatever SWG splash uses) has an early-return path under D3D11

3. **Is the "PostProcessingEffectsManager is the only thing drawing" pattern symptomatic of an ASSET LOAD failure?**
SWG loads `.tre` archives for UI assets. If those loads fail (or assets are in the wrong format for D3D11), `GameBitmap` might render zero quads. The engine's UI tree (CuiManager / CuiSystem) might still iterate widgets but each widget's render produces no draws.

4. **Is there a SWG-side debug log that surfaces "I tried to render but the texture/shader failed"?**
Engine warnings like `DEBUG_WARNING` flow through `DEBUG_REPORT_LOG_PRINT` which is invisible under explorer-launched D3D11 sessions (Plan 11-09.13 Iter-3 documented this). Should we route those to the InfoQueue sink so we see them?

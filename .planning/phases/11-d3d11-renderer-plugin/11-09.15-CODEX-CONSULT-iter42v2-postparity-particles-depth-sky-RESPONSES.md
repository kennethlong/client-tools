## Cursor response

> **Executive Summary**
>
> The remaining deltas share one structural theme: D3D11 only wires `m_alphaBlendEnable` per-pass (Iter-39C) while D3D9 pushes the full pass block every time via `RSM` — blend factors, depth, alpha test, and color-write mask. Iter-42v2 correctly fixed row-scale for transform-driven draws; particles, swoosh trails, and radar are on different paths and were never going to move from that fix.
>
> **Recommended iteration order:**
> 1. **Iter-44A** — per-pass depth (`m_zEnable`, `m_zWrite`, `m_zCompare`) → eyes-through-head
> 2. **Iter-44B** — per-pass alpha test (`m_alphaTestEnable`, func, reference) → mini-map circle + particle soft edges
> 3. **Iter-44C** — per-pass blend factors + `m_writeEnable` color mask (re-attempt Iter-43 only after A+B)
> 4. **Iter-44D** — `setBrightnessContrastGamma` (currently no-op) → sky washout
>
> **Q1 — Particle Billboard Squares & Blend Factor Porting**
>
> Verdict: Consult's Iter-43 failure diagnosis is partially wrong; the right model is simpler than "uninitialized zero fields + cache poisoning"
>
> What D3D9 actually does (`Direct3d9_ShaderImplementationData.cpp:259-261`):
>
> ```cpp
> RSM(D3DRS_BLENDOP,    m_alphaBlendOperation,   BlendOperation);
> RSM(D3DRS_SRCBLEND,   m_alphaBlendSource,      Blend);
> RSM(D3DRS_DESTBLEND,  m_alphaBlendDestination, Blend);
> ```
>
> Pass::apply() then sets enable first, then flushes all render states. The `Blend[]` lookup is 1:1 enum identity — same mapping Iter-43 used.
>
> "Uninitialized zero fields" is ruled out — `ShaderImplementationPass` constructor defaults are NOT zero (`ShaderImplementation.cpp:917-920`):
>
> ```cpp
> m_alphaBlendEnable(true),
> m_alphaBlendOperation(BO_Add),
> m_alphaBlendSource(B_SourceAlpha),
> m_alphaBlendDestination(B_InverseSourceAlpha),
> ```
>
> **Why Iter-43 made things worse (high confidence):** Iter-43 likely applied correct shader-authored factors (e.g. `B_One`/`B_One` additive for `light_blink_glow_add.sht`-class passes) **before the rest of the pass pipeline exists in D3D11**:
>
> | Pass state       | D3D9        | D3D11 today                            |
> |------------------|-------------|----------------------------------------|
> | Blend enable     | ✅ Iter-39C | ✅                                     |
> | Blend factors    | ✅ always   | ❌ install defaults only               |
> | Alpha test       | ✅ per-pass | ❌ not wired at all                    |
> | Depth write/test | ✅ per-pass | ❌ install defaults only               |
> | Color write mask | ✅ per-pass | ❌ not wired                           |
> | Real PS          | ✅ asset PS | ⚠️ dynamic fallback `tex * color0` only |
>
> With Iter-39C only: blend ON + default `SrcAlpha/InvSrcAlpha` accidentally produces wrong but partially masked results for additive content. Iter-43 switched to correct additive factors (`One/One`), which AMPLIFIES the broken output → larger, brighter white/amber squares.
>
> **Should D3D11 gate factor push on m_alphaBlendEnable?** No — D3D9 sets them unconditionally; when BlendEnable=false hardware ignores them. Gating is not the fix.
>
> Re-attempt blend factors AFTER depth + alpha test land, not in isolation.
>

> **Q3 — Per-Pass Depth State**

## CODEX response

> 1. **Q1 blend-state model:** the "D3D9 conditionally applies blend factors" hypothesis is probably wrong. D3D9 constructs `m_alphaBlendSource`, `m_alphaBlendDestination`, and `m_alphaBlendOperation` for every pass, then applies every recorded render state in `Pass::apply()`. The blend enum mapping is direct. Also, `ShaderImplementationPass` initializes blend fields, so "uninitialized/zero blend fields" is unlikely.
>
>    Iter-43 likely got worse because blend factors alone do not solve the particle squares. The square symptom points more strongly at missing shader/texture-stage semantics: D3D11's generated PS only samples TEXCOORD0/COLOR0 from slot 0 and ignores most FFP-style stage behavior, alpha test, multi-texture masks, texture factor, and stage combiners.
>
> 2. **Particle squares:** keep per-pass blend factors, but do not expect them to close this. The stronger suspects are:
>    - particle shader pass not resolving/binding the intended texture/SRV at draw time,
>    - fallback/generated PS not matching the asset's D3D9 texture-stage alpha behavior,
>    - missing alpha-test/alpha-reference handling,
>    - or the particle shader uses more than slot 0/stage 0 semantics.
>
> 3. **Q2 ribbons:** probably either static world cloth/awning geometry or a particle/swoosh-style quad-list path. SwooshAppearance uses `drawQuadList()` with identity object transform → would not be affected by Iter-42's row-scale fix. Exact classification needs runtime draw logging.
>
> 4. **Q3 depth state:** porting per-pass depth state is correct and should be done. D3D9 applies `ZENABLE`, `ZWRITEENABLE`, and `ZFUNC` per pass; D3D11 currently uses global defaults. Add D3D11 state-cache setters for depth enable/write/compare and call them from `Direct3d11_StaticShaderData::apply`.
>
>    **IMPORTANT CAVEAT:** D3D9's `Compare[]` table appears to have `C_GreaterOrEqual` and `C_NotEqual` swapped versus the enum comments. For visual parity, mirror D3D9's actual table unless you intentionally want to fix historical behavior.
>
> 5. **Mini-map:** very likely missing shader semantics, not clipping/scissor. `CuiWidgetGroundRadar` submits transformed verts with two texture coordinate sets and uses `shader/uicanvas_radar.sht`; D3D11's dynamic PS currently samples only one texture coordinate/source. That explains square radar instead of circular mask.
>
> 6. **Sky/gamma:** D3D11's `setBrightnessContrastGamma_impl()` is a no-op, while D3D9 builds a gamma ramp and calls `SetGammaRamp`. So gamma washout is a real known gap. The sky may also involve missing shader-stage/fog semantics, but gamma is an obvious confirmed difference.

## Synthesis

**Convergent verdict:**

1. Iter-43's blend-factor hypothesis was wrong in attribution but right in target — D3D9 IS unconditional; the regression came because we wired blend before depth + alpha-test + full PS exist. Re-land after those.
2. **Iter-44A (per-pass depth) is the right next step** — high confidence on the eyes-through-head symptom. Both agree.
3. **Iter-44B (alpha test) is the right step after that** — high confidence on mini-map round shape. Both agree.
4. **Iter-44C (re-attempt blend factors)** — convergent on this being correct AFTER A+B land.
5. **Iter-44D (gamma)** — convergent diagnosis; `setBrightnessContrastGamma_impl` is a stub.

**Divergence worth noting:**

- **CODEX raises a deeper concern** about particle squares: even after A+B+C, the dynamic fallback PS in D3D11 (`PixelShaderProgramData.cpp:436-443` — only `texture * vertexColor`) doesn't implement FFP-style multi-stage combiners, texture factor, alpha-test logic that asset PSes use. Particles MAY still look wrong after the state-cache port if their `.sht` expects an asset PS with these semantics. That's a Phase 12 PS-authoring concern.
- **CODEX's caveat on Compare[] enum mapping** — D3D9's table has `C_GreaterOrEqual` and `C_NotEqual` swapped vs the enum comments. Mirror D3D9's actual table for parity (don't "fix" the historical bug).

**Recommended iteration order (confirmed):**

| Iter | Scope | Expected win | Risk |
|---|---|---|---|
| 44A | Per-pass depth (zEnable/zWrite/zCompare/writeEnable) | Eyes-through-head; possibly some unrelated depth-state edge cases | LOW |

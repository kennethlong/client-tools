## Per-VS PS architecture (the load-bearing context)

The D3D11 plugin (`src/engine/client/application/Direct3d11/`) faces a hard asset-pipeline constraint: SWG ships **pre-compiled D3D9 PEXE pixel-shader bytecode**, which `ID3D11Device::CreatePixelShader` rejects. So every asset PS's `m_d3dPS = NULL` at install time. The runtime selects a fallback PS per draw:

1. **Per-VS dynamic PS (Plan 11-09.13 Iter-4)** — `Direct3d11_PixelShaderProgramData::getOrCompilePSForVS(vsData)`. Keyed by VS bytecode hash; generated at first encounter from the VS's reflected output signature. Cached.
2. **Variant M (magenta) fallback** — declared minimum (SV_POSITION only), used when per-VS generation returns null or input requirements can't be met.

The per-VS HLSL generator (`buildHlslForVSOutputs`, currently Iter-28) walks reflected outputs and emits:

```hlsl
// (Iter-28 simplified form)
struct PSIn {
    float4 _pos : SV_POSITION;
    float4 _v0  : SV_Position;   // = SV_Position from VS, harmless duplicate
    float4 _v1  : COLOR0;        // only when VS outputs COLOR0 xyzw FLOAT32
    float2 _v2  : TEXCOORD0;     // only when VS outputs TEXCOORD0 xy FLOAT32
    // ...remaining VS outputs in register order
};
Texture2D    t : register(t0);
SamplerState s : register(s0);

float4 main(PSIn input) : SV_TARGET {
    // Iter-28 branches:
    //   has TEXCOORD0 + COLOR0  -> return t.Sample(s, _v2.xy) * _v1;          // modulate (UI panels w/ tint)
    //   has TEXCOORD0 only      -> return t.Sample(s, _v2.xy);                 // untinted texture (2d_texture.vsh — splash?)
    //   has COLOR0 only         -> return _v1;                                  // colored quad, no texture (2d.vsh)
    //   neither                 -> return float4(1,0,1,1);                      // magenta
}
```

The cache key incorporates `D3D11_REWRITE_VERSION` (currently `"16"`); bumping it invalidates `.cso` cache entries for the next session.

Stage 0 texture binding is handled by `Direct3d11_StaticShaderData::construct` (line 316–447) — it walks `pass->m_pixelShader->m_textureSamplers`, finds the entry with `m_textureIndex == 0`, resolves via `shader.getTextureData(tag, ...)` or `Direct3d11_TextureData::getGlobalTexture(tag)`, and stores the SRV in `m_passStage0[passIndex].m_texture`. `apply(passNumber)` binds that SRV to slot 0 and the matching sampler. **Slots 1–7 are NOT bound** — Bucket A scope (Plan 11-09.14) was deliberately slot-0-only.

## Two clean hypotheses, both worth challenge

### H1 — Fonts ride `ui.vsh` (COLOR0 modulate path) but the glyph atlas format is wrong for `tex * color`

Most likely if fonts go through `CuiLayer::EngineCanvas::RenderQuad` like everything else. Three sub-cases:

- **H1a** — Atlas is BGRA8 with white glyph in RGB and mask in A: `tex * vColor` = `(1,1,1,a) * (textR,textG,textB,textA) = (textR,textG,textB,a*textA)`. Should work. If it doesn't, vColor.rgb might actually be (0,0,0) or vColor.a might be 0 for fonts (engine fade-alpha bug? `ms_opacityRelativeMin` interaction in UITextStyle.cpp:815?).
- **H1b** — Atlas is BGRA8 with white glyph in RGB **but mask in R** (alpha channel = 1 throughout): `tex.Sample` returns `(maskR,maskR,maskR,1)` → modulated by vColor gives `(textR*maskR, textG*maskR, textB*maskR, textA)` → would render correctly. If invisible, atlas might have RGB=0 throughout and only useful data in some other channel.
- **H1c** — Atlas is A8 / R8 only (single-channel): `tex.Sample` returns `(r,0,0,1)` or `(r,r,r,r)` depending on SRV format declaration. Either way, the resulting modulation produces wrong colors / wrong alpha.

The original D3D9 font shader almost certainly does **alpha-test + alpha-blend** with a specific channel-selection idiom: `return float4(vColor.rgb, tex.a * vColor.a);` is the canonical "white text" pattern. Our generic `tex * color` modulation does NOT replicate this when the atlas has zero RGB.

### H2 — Fonts ride a different VS or shader template than `ui.vsh`

Possible if SWG uses a font-specific shader template (e.g. `shader/uifont.sht`) which selects a VS with a different output signature. If that VS lacks `COLOR0` or `TEXCOORD0` we'd hit the wrong branch:
- No TEXCOORD0 + has COLOR0 → returns `input._v1` (vertex color only, no atlas sample) → would render as solid color filled quads at glyph positions, not invisible
- Has TEXCOORD0 only → returns `t.Sample(s, uv)` (no color modulation) → for A8 atlas would render as black on UI gray = nearly invisible ← **this matches the symptom**

The "one magenta pixel" in the screenshot is interesting — it suggests at least one quad somewhere uses a VS that hits the Variant M tombstone (no TEXCOORD0, no COLOR0). The text caret? An ornament?

### H3 — Splash page graphics use a different shader template than login UI

Symmetric question for splash. The fact that login backdrop now renders correctly post-Iter-28 means the backdrop's shader template uses a VS with COLOR0 output. If the splash pages use **`shader/2d_texture.sht`** (whose VS is `2d_texture.vsh`, no COLOR0), they'd hit the "TEXCOORD0 only" branch and render the raw texture without vertex-color modulation. If the splash page graphics depend on a vertex tint that was applied in D3D9 via TFACTOR / texture stage modulation, we'd lose that → desaturated appearance.

`shader/2d_texture.sht` is also `PostProcessingEffectsManager::ms_copyShader` (line 246 of `PostProcessingEffectsManager.cpp` binds primaryBuffer to its MAIN tag). If the splash UI Page system happens to instantiate the same `.sht` for its page bitmaps, both flow through the same VS — would explain symmetry.

## What we want from CODEX + Cursor

1. **For fonts** — given the architecture (`buildHlslForVSOutputs` generator + Stage-0-only texture binding + Iter-28 COLOR0 modulate), what's the most plausible reason fonts render invisibly across every UI surface? Specifically: which VS does SWG's UI font path use, and what is its output signature? Is there a known channel-swizzle convention for SWG font atlases (white-in-RGB+alpha-mask vs. mask-in-red vs. A8) that our generic `tex * color` modulation breaks?
2. **For splash** — what is `SwgCuiSplash`'s rendering path, and why would it use a different VS from the login UI? Is it likely using `shader/2d_texture.sht` directly for its page bitmaps, bypassing `ui.vsh`'s COLOR0 path?
3. **Architecture question** — is per-VS PS generation **fundamentally sufficient** for SWG's UI rendering needs, or do we need a small library of hand-written specialized PSes selected by shader template name / asset tag (e.g. one PS for fonts that does `float4(vColor.rgb, tex.a * vColor.a)`, one for blit, one for modulate, etc.)? If yes, what's the cleanest selection mechanism that keeps the per-VS reflection generator as a safety net?
4. **Next diagnostic** — what's the highest-information-density diagnostic to add for one more iter to pin H1/H2/H3 down? Candidates we've considered: log the bound StaticShader's template name + asset path per drawQuadList call, log SRV0 DXGI format + dimensions for each first-N draws, capture a single-frame staging readback of an SRV that we believe is a font atlas, dump UITextStyle.cpp's per-glyph color/canvas chain.

## Resume strategic decision (READ FIRST on resume)

**The tension:** Plan 11-10 is scaffolded as **DPVS re-measurement** (per `11-10-PLAN.md` — SPEC R7 / ROADMAP success criterion #6). But the current visible state is only **half-quad coverage** on the login screen — full UI is not rendering yet (engine submits 4-vert `drawTriangleList` expecting 1 triangle by D3D9 convention; D3D11's strict topology renders only 1 of the 2 triangles per quad). Running DPVS measurements against a half-rendered scene would produce numbers that don't reflect the real Phase 11 final state.

**Three reasonable paths forward (Kenny to choose):**

### Option A — Insert a new Plan 11-09.5 (full-quad coverage) BEFORE Plan 11-10

Smallest-scope work to get full UI visible before DPVS measurement. Likely 1–2 iterations:

- **Iter-1:** Wire `Direct3d11_StateCache::drawQuadList` (currently STUB) — mirror `Direct3d9.cpp:4277` reusable-index-buffer pattern: install-time create a static 16-bit IB with N quad triangulations (0,1,2, 2,3,0 repeated), then at draw time bind it + `DrawIndexed` for `(quadCount * 6)` indices. This unblocks `CuiLayerRenderer` if that's the engine path; need to confirm with a quick instrumentation probe.
- **Iter-2 (if needed):** PS user-constants Fix C mirror. Plan 11-09 fixed only VS user-constants — `setPixelShaderUserConstants_impl` (Direct3d11.cpp:420) currently stores to a function-local-static `s_perMaterialShadow` that nothing reads. Mirror Fix C: create a namespace-scope `Direct3d11_PixelSlot0CB` (or pick the right shadow target — PerMaterialCB seems to be what the code already names) + dirty flag + lazy flush in `applyPreDrawState`. Engine push routes to register 0 (or wherever `PSCR_userConstant` lives — verify against `Direct3d9_PixelShaderConstantRegisters.h`).

After Plan 11-09.5: full UI visible → Plan 11-10 DPVS measurements are meaningful.

### Option B — Execute Plan 11-10 as scaffolded (DPVS) NOW

Accept that DPVS measurements happen on a half-rendered scene; document the caveat in `docs/recon/11-dpvs-d3d11-remeasure.md`. Justification: DPVS `resolveVisibility()` cost is a *CPU* cost driven by scene-graph traversal, not per-draw-call cost — so half-visible-quads shouldn't change the DPVS timing much. PIX/Nsight captures would still show the resolveVisibility() spike in the frame timeline regardless of whether the resulting draws all surface on-screen.

Risk: indoor/outdoor verdict could be sensitive to draw-call cost (since fewer post-PS pixels = different driver overhead). Plan 11-10 might end up needing a re-run after Plan 11-09.5 anyway. Probably not worth it.

### Option C — Tactical: fix `show-window.ps1` engine-init root cause first

Smallest standalone phase (~1 iteration). The script's preamble says: "Originated in Phase 11 Plan 11-07 Iter-17, where the engine's normal `ShowWindow(SW_SHOW)` path didn't fire on launch." Three candidate fixes (from this session's earlier discussion):

1. Investigate `Os::createWindow` / early `Gl_install` flow — find where the engine's intended `ShowWindow` call should fire and why it doesn't. Engine-side fix.
2. Add explicit `ShowWindow(SW_SHOW) + SetForegroundWindow` at end of `Direct3d11_Device::create`. Plugin-side workaround. Pragmatic but masks the engine bug.
3. Patch `Os::displayInitialWindow` (or equivalent) with explicit show + foreground call.

Doesn't unblock Phase 11 progress but eliminates the smoke-protocol friction Kenny asked about this session.

### Recommendation

**Option A first (Plan 11-09.5 full-quad), then Plan 11-10 DPVS.** Rationale: half-quad is a visible incomplete state — fixing it is a small, focused arc that completes the "visible UI" promise of Phase 11 before measurement work. Option C is a nice-to-have that can fit anywhere.

## Critical context / anti-patterns to carry forward

1. **Build slashes are PowerShell-only** — `Bash` mangles them via Git Bash POSIX path conversion. Use `PowerShell` for all MSBuild invocations. The handoff archive at `.continue-here-archived-plan-11-09.md` has the canonical command shapes.

2. **Background command exit codes can mislead** — if you pipe MSBuild through `tail` / `Select-Object`, the pipeline's exit code is the tail process's, not MSBuild's. Always check the actual output file for the real status. (Caught this session: Bash pipeline reported EXIT=0 but the underlying MSBuild had ERR MSB1008 due to slash mangling.)

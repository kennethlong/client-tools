SwgClient.vcproj — what it actually is

src/game/client/application/SwgClient/build/win32/SwgClient.vcproj (Visual Studio 2005, project format 8.00) is a wrapper shell, not the client. It compiles only three C++ source files and inherits the entire engine through linker dependencies in swg.sln.

All three are 32-bit Win32 (<Platform Name="Win32"/>), SubSystem="2" (Windows GUI). The two prebuilt artefacts that ship in the repo are exe/win32/SwgClient_o.exe and exe/win32/SwgClient_r.exe.

Compiler settings worth noting

  • CharacterSet="2" — MBCS, not Unicode.
  • TreatWChar_tAsBuiltInType="FALSE" — wchar_t is treated as unsigned short for XPCOM ABI compatibility. Modern VS defaults TRUE — flipping it breaks the link.
  • WarningLevel="4" + WarnAsError="TRUE" — strict gate. Modern compilers add new warnings → instant build failure.
  • UsePrecompiledHeader="3" through FirstSwgClient.h: 
    #ifndef INCLUDED_FirstSwgClient_H
#define INCLUDED_FirstSwgClient_H
#include "sharedFoundation/FirstSharedFoundation.h"
#include "_precompile.h"
#endif
  • RuntimeTypeInfo="TRUE" — RTTI enabled.
  • Detect64BitPortabilityProblems="FALSE" — explicitly off; client is always 32-bit.
  • _USE_32BIT_TIME_T=1 — pinned for STLPort 4.5.3 compatibility.

WinMain.cpp — the entry shim

From src/game/client/application/SwgClient/src/win32/WinMain.cpp:124:

int WINAPI WinMain(HINSTANCE hInstance, HINSTANCE hPrevInstance,
                   LPSTR lpCmdLine, int nCmdShow) {
  if (!SetUserSelectedMemoryManagerTarget())
    SetDefaultMemoryManagerTargetSize();
  return ClientMain(hInstance, hPrevInstance, lpCmdLine, nCmdShow);
}

SetUserSelectedMemoryManagerTarget reads the env var SWGCLIENT_MEMORY_SIZE_MB. The default size is 3/4 of total physical RAM, clamped to [250 MB, 750 MB] in dev or [250 MB, 2000 MB] in production.

The client looks for misc/override.cfg inside the TRE search path — i.e. servers can ship config overrides via the patcher.

Indirect / runtime-loaded dependencies

The .vcproj doesn't link these — they're loaded via LoadLibrary (or required by the linked DLLs). All ship as DLLs in exe/win32/:

  • Bink — bink/lib/win32/binkw32.dll, binkw32.lib. Loaded via Bink::bindBink() in BinkDLL.h — function pointers, not static link.
  • Vivox runtime — vivox/lib/{vivoxsdk,vivoxplatform,ortp,alut,libsndfile-1,wrap_oal}.dll, plus VivoxVoiceService.exe.
  • Mozilla XPCOM runtime — xpcom.dll, xul.dll, nspr4.dll, plc4.dll, plds4.dll, nss3.dll, nssckbi.dll, smime3.dll, softokn3.dll, ssl3.dll, js3250.dll, gksvggdiplus.dll + exe/win32/mozilla/ chrome tree.
  • MSVC 7.1 CRT — exe/win32/msvcr71.dll. Mismatch: the .vcproj targets VS 2005 (would emit msvcr80-bound binaries), but the prebuilt EXEs use VS 2003's CRT. The original team built with VC++ 7.1; the VS 2005 settings are a later port.

What's missing — what would actually block a build today

#BlockerWhy
1Visual Studio 2005 itself, or a clean upgrade pathFormat Version="8.00" is VS 2005-specific. VS 2022 upgrade routinely fails on sister libs (changed flags, deprecated CRT, removed headers, MFC/ATL relocation).
2Pre-2010 Windows Platform SDKThe DX9 headers reference wtypes.h/unknwn.h from the era's SDK. Newer SDKs break on dinputd.h / ddraw.h.
3The swg.sln project graph must build in orderSwgClient.vcproj has no inline <References> — it relies on the ~70 GUID dependencies in swg.sln. Standalone build = wall of LNK2019.
4STLPort 4.5.3 library naminglib/ has six variants (vc6/vc7/vc71 × release/debug). VS 2005 (vc8) isn't there. Original team built VC++ 7.1.
5TreatWChar_tAsBuiltInType="FALSE" non-defaultFlipping it to TRUE (modern VS default) gives unresolved externals against XPCOM, which was built with wchar_t = unsigned short.
6WarnAsError="TRUE" + level 4 + 2026-era compilerNewer MSVCs add warnings → instant errors. Standard fix is global /wd suppression — meaningful porting effort.
7Sony Station / soePlatform integrationLinkable, but phones home at runtime to dead SOE endpoints.
8client.cfg with valid LoginServerShipped client.cfg targets retired SOE hosts.
9.tre asset archivesSetupSharedFile + TreeFile::open("misc/override.cfg") crashes the boot if no archives exist on the search path.
10Working data/ tree on the search pathBeyond .tre, the engine wants data/sku.0/... paths from a retail install or community asset rebuild.

What is NOT actually missing

Surprisingly, almost every external dependency is physically present. The leak is more complete on the client than on the server:
  • DirectX 9 SDK headers + import libs are in-tree.
  • DPVS (Umbra) source + libs are in-tree.
  • Bink, Miles, Vivox, Logitech LCD, libxml2, PCRE, zlib — all linkable.
  • libMozilla wrapper + XPCOM .lib + DLLs all present.
  • Custom UI engine (external/3rd/library/ui) is full source.
  • SOE UDP library (udplibrary) is full source — has to be, for client/server protocol symmetry.

TL;DR

SwgClient.vcproj is a 3-file Win32 shell that, through the swg.sln graph, links the entire SOE engine plus a stack of vendored 3rd-party SDKs. Realistic blockers:

  1. 2005-era toolchain (VS 2005 + matching Platform SDK + MSVC 7.1 CRT mismatch).
  2. Strict warning gates (WarnAsError, wchar_t ABI).
  3. Building swg.sln as a whole, not just this one project.
  4. Runtime: missing .tre archives, dead SOE auth, no working client.cfg.

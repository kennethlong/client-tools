Gaps in this archive

This is a source archive, not a distribution. The leak intentionally (or as a side-effect of how it was extracted) does not include most of the things a player or operator needs. Below is the practical gap inventory and how the public community has worked around each one.

1. Asset archives (the .tre files)

missing Models, textures, animations, sounds, terrain, UI XML, music, world snapshots — none ship in the repo. publish/*.mft describes the live tree and publish/data_*.txt + publish/patch_*.txt describe the patch tree, but the archives themselves are not here.

Workaround: use a real retail SWG client install (still obtainable from old install media or community archives) and copy its data/ + patch/ trees alongside. SWG-Source maintains a client-assets repo that is the closest thing to a public asset baseline.

2. Database schema and seed data

missing Server expects an Oracle 9i database with login + cluster schemas. Configs name the user (databaseUID=login) and password (login_prod8) but no schema dumps are included. The C++ uses native OCI through sharedDatabaseInterface and serverDatabase.

Workaround: SWG-Source's swg-main/build.xml includes create_database and update_database targets that bring an Oracle instance to a known state from migration scripts. Modern installs use Oracle 12c rather than 9i.

3. Authentication infrastructure

missing LoginServer talks to sdt-session1.station.sony.com:3000. The Station service was retired with the game in 2011. The whole 3rd/library/stationapi + soePlatform tree expects working SOE backends.

Workaround: set validateStationKey=0 in loginServer.cfg. Replace SOE Station with a community auth service — swg-ostrich/site and rohvani/swg-site are node.js / Vue community implementations.

4. Proprietary SDKs

licensed Required to build but cannot be redistributed legitimately:

  • DirectX 9 SDK
  • Autodesk Maya SDK (4 / 5 / 7 / 8)
  • Oracle OCI client
  • RAD Bink + Miles
  • Umbra DPVS
  • Vivox VoIP SDK
  • NaturalPoint TrackIR SDK
  • NxN Alienbrain XDK
  • Logitech LCD UI SDK

Some of these are discoverable in old archives; some are not. The modern fork has been progressively decoupled — e.g. Bink replaced with alternative video, voice often disabled.

5. Java runtime

missing src/BUILDING demands IBM Java 1.3.1 in /usr/java. That JRE is not on any current Linux distro. SWG-Source's port targets a more recent JVM, with corresponding adjustments to the JNI bridge in serverScript.

6. Build toolchain

era-locked gcc 2.95 / 3.2.x, automake 1.6, libtool 1.4, STLPort 4.5.3 are no longer present on modern systems and the codebase relies on quirks of those versions. Modern GCC will reject huge swaths of the code without porting.

Workaround: use a CentOS / RHEL 5–6 era VM, or use the StellaBellum dev VM from SWG-Source which freezes the right toolchain.

7. Pre-built tool binaries are tied to old infra

caution The exe/win32/ editors run, but several reach for SOE infrastructure on first launch (perforce, alienbrain, internal mail relays, internal log endpoints). Disconnect from a network when first running them, or expect long hangs / failed lookups.

8. Per-machine configs

missing Lots of template_*.cfg files exist but tools/configFileBuilder.pl needs to be run with site values (hostnames, ports, paths) to produce real configs. Without that, you'll see "missing config" errors.

9. Internal libraries with stubs only

Some directories in src/external/3rd/library/ appear to contain only headers or partial source — e.g. perforce, vivox, libEverQuestTCG, soePlatform. Real builds require pairing them with the matching internal artifact, which isn't here.

10. Patcher / signing material

missing No code-signing certificates, no patcher signing keys, no per-server crash reporter endpoints. Not strictly required to run — only required to ship to end users.

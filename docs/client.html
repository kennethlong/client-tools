Client architecture

The shipping product (SwgClient.exe) is a single 32-bit Windows DirectX 9 application that pulls every asset out of .tre archives via the same TreeFile layer the server uses. Source: src/game/client/application/SwgClient/.

How rendering binds

clientGraphics is decoupled from the API; the backend lives in src/engine/client/application/Direct3d9/ and is loaded as a DLL (gl06_o.dll / gl07_o.dll etc. in exe/win32/). That's why there are multiple gl0X_*.dll variants — they're alternate renderer builds.

Asset loading

The client opens .tre archives via sharedFile/TreeFile. Inside a .tre are .iff (chunk-based binary, EA Interchange Format), .stf (string tables), .tab (datatables), .dds textures, animations, etc. None of these are in this repo — see Data Pipeline.

The editor / tool clients

Many of the prebuilt *.exe in exe/win32/ are content-creation tools shipped with their own configs (template_*.cfg):

  • AnimationEditor — edit anim graphs.
  • ParticleEditor — VFX authoring.
  • QuestEditor — quest scripting front-end.
  • SwgConversationEditor — branching NPC dialogue.
  • SwgDraftSchematicEditor — crafting schematics.
  • SwgSpaceQuestEditor / SwgSpaceZoneEditor — JTL space content.
  • TerrainEditor — procedural-terrain authoring.
  • TextureBuilder, ShaderBuilder, SoundEditor, NpcEditor, LightningEditor, ClientEffectEditor, ShipComponentEditor, SwooshEditor.
  • Viewer — the in-house model viewer.
  • ViewIff — inspect .iff files (handy if you want to read templates by hand).
  • WorldSnapshotViewer — inspect a .ws world snapshot.
  • SwgGodClient — the in-game admin tool, also runnable standalone.
Useful trick: some of these editors will launch on modern Windows if you copy them next to a real SWG client install (so the .tre files are on the search path). They were the only way SOE designers worked with the data, and many community modders still use them.

The scripting layer

SWG separates the engine (C++) from the gameplay rules (scripts). Scripts under dsrc/sku.0/sys.server/compiled/game/script/ describe how every interaction works: combat dispatch, NPC AI, faction reactions, quest progression, crafting, structure placement, social systems. Script content is what designers and live-team engineers touched daily.

The scripting language is "scriptlib + script", not vanilla Java

Files come in two flavours:

  • *.script — gameplay handlers attached to a specific object kind. They look like Java with extra primitives (obj_id, location) and a handler-style entry-point convention.
  • *.scriptlib / *.java — shared library code consumed by scripts. *.java here are not ordinary Java — they are the SWG dialect with engine-only types — but they go through a build step that converts them to bytecode the embedded JVM can run.

The first lines of a real script (script/base/remote_object.script) look like:

// =====================================================================
//
// remote_object.script
// copyright, Sony Online Entertainment
//
// =====================================================================

//-- cluster wide data object manager
const String REMOTE_OBJECT_MANAGER = "remote_object_manager";

//-- parameter prefix
const String BASE_NAME = "remote_object";
...

The familiar const String, String, package-style organisation, and message-handler patterns (OnAttach, OnTrigger) are why this is often described as "Java-like".

How a script is hooked to an object

Server object templates in dsrc/sku.0/sys.server/compiled/game/object/ list which scripts attach to a given object type. When the engine spawns an object (e.g. a Tusken Raider creature), it consults that template, instantiates the C++ object, then attaches each named script to it. From then on, engine events on that object (movement, combat, conversation) trigger the corresponding OnX handler in the script.

Datatables: the "tabs"

Side-by-side with scripts is a huge body of .tab spreadsheets in dsrc/sku.0/sys.server/compiled/game/datatables/ — flat tables that scripts and engine code read at runtime:

  • ai/ai_combat_profiles.tab — AI combat behaviour matrices.
  • loot/ — drop tables.
  • combat/ — special-attack tables, crit tables.
  • creature_resource/, resource/ — harvestable resource tables.
  • quest/, missions/ — content side data.
  • jedi/, jedi_trials/ — Force / Jedi unlock data.

The .tab format is a tab-separated text spreadsheet plus a header row with column types. The engine compiles them to .iff for runtime, but the authoritative source-of-truth is the .tab. Many tools in tools/ exist to round-trip .tab ↔ .iff.

Compilation / linting

The pipeline is run via tools/swglint.sh + tools/swg-lint.bat / tools/scriptlint/, which:

  1. parses .java / .scriptlib with a SWG-flavoured front-end (uses flex/bison binaries shipped under tools/),
  2. resolves references and engine-bound types,
  3. emits class files / .scriptbin bundles loaded by the embedded JVM.
This pipeline assumes IBM Java 1.3.1. See src/BUILDING: "ibm java2 1.3.1 installed in /usr/java". That JRE is not on modern Linux distros. The SWG-Source fork either ships a vendored binary or has been ported to a more recent Java — that is one of the bigger porting efforts the community has done on top of this code.

Build system

Two parallel build systems live in the tree, and they are not interchangeable: Windows uses Visual Studio 2005 projects; Linux uses GNU autotools. Each subsystem has both a build/win32/*.vcproj and a Linux Makefile.am.

  • Top-level solution: src/build/win32/swg.sln.
  • Roll-up project files (_all_*.vcproj) batch-build whole categories.
  • Each application or library has its own build/win32/<Name>.vcproj next to its source.
  • Old-style .dsp Visual C++ 6 projects also exist as a parallel set — maintained automatically by Perl scripts in tools/ (e.g. BuildVcprojs.btm, BuildAllDsps.btm).
Reality check: opening swg.sln in Visual Studio 2022 will prompt you to "upgrade" the projects. The upgrade often succeeds for tools but fails for the heavyweight client/server projects because of changed compiler flag semantics, MFC ATL paths, removed headers, and the missing DirectX 9 SDK / Maya SDK / Oracle OCI libs. SWG-Source carries a patched VS solution that has been incrementally fixed; this leak does not.

The Linux side

Linux builds use GNU autotools, driven by src/configure.in and a per-directory Makefile.am tree. The full bootstrap from src/BUILDING is: 

./siteSetup    # creates Makefile.in's and runs configure
make install   # builds & installs into --prefix

For the servers specifically, the top-level src/servers.mak recurses into six application build/linux directories: 

all:
    +@make -C engine/server/application/LoginServer/build/linux
    +@make -C engine/server/application/CentralServer/build/linux
    +@make -C game/server/application/SwgGameServer/build/linux
    +@make -C game/server/application/SwgDatabaseServer/build/linux
    +@make -C engine/server/application/ConnectionServer/build/linux
    +@make -C engine/server/application/PlanetServer/build/linux

(TaskManager is commented out — usually built separately.)

Other build-time tools

  • tools/BuildVcprojs.btm — regenerates Windows project files from per-directory additionalFiles.rsp response files.
  • tools/BuildIncludeHeaderFiles.btm — generates the unity-style "FirstX.h" precompiled headers seen throughout the tree.
  • tools/buildtre.btm — packs .tre archives for the patcher.
  • tools/configFileBuilder.pl — generates per-machine cfg files from templates (see exe/win32/template_*.cfg).
  • tools/recompileAllScripts.pl — driver for the dsrc/ compile.
  • tools/swglint.sh + tools/scriptlint/ — script linter / compiler entry points.
  • tools/AutomatedDailyBuild.pl, AutomatedDailyServerBuild.pl — the original CI orchestration.

Modern build alternative (SWG-Source)

The community-maintained SWG-Source/swg-main repository wraps the same C++ tree in an Apache Ant meta-build. Targets like ant swg, ant create_database, ant update_database, ant update_configs drive the underlying autotools, the Oracle migrations, and the cfg generation in one command. This is the path you want if "running locally" matters more to you than "build with the original 2010 toolchain". See Run Locally for the practical walkthrough.

Build artifact layout

After install (Linux), $PREFIX/bin/ contains the server executables and $PREFIX/lib/ the shared libraries. Configs and shell scripts live one level up under exe/linux/ and exe/shared/. Servers are launched indirectly through start_cluster.pl and TaskManager; never run a single binary directly in production.

Client-side runtime middleware

1. DirectX 9 — graphics API

  • Where: includes src/external/3rd/library/directx9/{include,lib}. Render-side call sites: only inside the Direct3d9 plug-in (src/engine/client/application/Direct3d9/, ~22 .cpp / 26 .h, ~450 KB).
  • Source files (selected): Direct3d9.cpp, Direct3d9_DynamicIndexBufferData.cpp, Direct3d9_DynamicVertexBufferData.cpp, Direct3d9_LightManager.cpp, Direct3d9_Metrics.cpp, Direct3d9_PixelShaderProgramData.cpp, Direct3d9_RenderTarget.cpp, Direct3d9_ShaderImplementationData.cpp, Direct3d9_StateCache.cpp, Direct3d9_StaticIndexBufferData.cpp, Direct3d9_StaticShaderData.cpp, Direct3d9_StaticVertexBufferData.cpp, Direct3d9_TextureData.cpp, Direct3d9_VertexBufferDescriptorMap.cpp, Direct3d9_VertexBufferVectorData.cpp, Direct3d9_VertexDeclarationMap.cpp, Direct3d9_VertexShaderData.cpp, plus MemoryManagerHook.cpp, SetupDll.cpp, WriteTga.cpp, ConfigDirect3d9.cpp, FirstDirect3d9.cpp.
  • API surface: IDirect3D9, IDirect3DDevice9, D3DPRESENT_PARAMETERS, D3DPOOL, D3DXMatrix*, vertex/pixel-shader constant tables, dinput8 (input), dsound (sound stub), ddraw (legacy fallback).
  • Linking: Direct3d9.vcproj is ConfigurationType="2" — builds as gl05_d.dll / gl05_o.dll / gl05_r.dll / gl06_*.dll / gl07_*.dll. Statically linked against libjpeg.lib odbc32.lib odbccp32.lib winmm.lib delayimp.lib dxguid.lib d3d9.lib d3dx9.lib ddraw.lib dxerr9.lib.
  • Integration depth: Contained. The rest of the engine talks to Graphics::; the only direct DX9 type leak outside the plug-in is in the launcher (SwgClientSetup/ClientMachine.cpp).
  • Replacement cost: rewrite one DLL against a current API (D3D11, Vulkan).
  • Search evidence: IDirect3D9|IDirect3DDevice9|D3DPRESENT|D3DPOOL|D3DXMatrix appears in 10 files — 8 are the DX9 SDK headers themselves, 1 is the launcher, 1 is a sample under DejaInsight/Demos/TinyEngine/.

2. Bink Video — RAD intro/cinematic codec

  • Where: src/external/3rd/library/bink/{include,lib/win32} (3 headers: bink.h, rad3d.h, radbase.h; binkw32.lib, binkw32.dll, binkw32.pdb). Call sites: 6 files in src/engine/client/library/clientGraphics/src/Bink/.
  • API surface: ~25 Bink::Bink* functions wrapped as inline forwarders in BinkVideo.h — BinkOpen, BinkPause, BinkGoto, BinkDoFrame, BinkCopyToBuffer, BinkCopyToBufferRect, BinkRegisterFrameBuffers, BinkDX9SurfaceType, BinkSetVideoOnOff, BinkSetSoundOnOff, BinkSetVolume, BinkSetPan, BinkGetKeyFrame, BinkNextFrame, BinkWait, BinkShouldSkip, BinkGetRects, BinkGetSummary, BinkGetRealtime, BinkControlBackgroundIO, BinkGetTrackID, BinkClose.
  • Linking: Late-bound. BinkDLL.h: 
    namespace Bink {
    bool bindBink(const char *i_dllname);
    void unbindBink();
    bool isBinkReady();

    namespace RAD { #include "bink.h" }
    using RAD::HBINK; using RAD::BINKFRAMEBUFFERS; ...

    BINK_STORAGE HBINK (BINK_CALL *BinkOpen)(const char *file_name,
                                             BINK_OPEN_FLAGS open_flags);
    BINK_STORAGE void (BINK_CALL *BinkClose)(HBINK bink);
    // ... all functions are pointer typedefs
}
    Bink is loaded via LoadLibrary and resolved through function pointers — not statically linked. SwgClient.vcproj does not list binkw32.lib.
  • Integration depth: Contained. Behind the abstract clientGraphics/Video interface.

3. Miles Sound System — RAD audio middleware

  • Where: src/external/3rd/library/miles/{include,lib/win} (one big Mss.h, 224 KB; Mss32.lib). A second copy at miles-7.2e/ exists but isn't linked. Runtime DLL: exe/win32/Mss32.dll.
  • API surface: 188 raw API tokens (HSAMPLE, H3DSAMPLE, HDIGDRIVER, HPROVIDER, HSTREAM, MSS_MC_SPEC) inside clientAudio. Call-site files:
    • clientAudio/src/win32/Audio.cpp — the manager (182 token occurrences)
    • clientAudio/src/win32/Sample2d.h
    • clientAudio/src/win32/Sample3d.h
    • clientAudio/src/win32/SoundObject3d.h, SoundObject3d.cpp
    • clientAudio/src/win32/SampleStream.h
  • Linking: Statically linked (mss32.lib in SwgClient.vcproj), runtime work in Mss32.dll. Exposes a Miles digital driver to clientGame: 
    VideoList::install(Audio::getMilesDigitalDriver());
  • Integration depth: Contained but not strictly facaded. Miles types do leak into a few public clientAudio headers (Sample2d.h, Sample3d.h), so a swap requires a small header refactor.
  • Search evidence: HSAMPLE|H3DSAMPLE|HDIGDRIVER|HPROVIDER|HSTREAM appears in 7 files — 2 are Miles headers, 5 are inside clientAudio. Zero leakage outside.

5. Vivox — voice chat

  • Where: src/external/3rd/library/vivox/{include,lib} (vendored SDK: vivoxsdk.dll, vivoxplatform.dll, ortp.dll, alut.dll, libsndfile-1.dll, wrap_oal.dll, VivoxVoiceService.exe). Wrapped by src/external/3rd/library/vivoxSharedWrapper/.
  • Vendored Vivox SDK headers: Vxc.h, VxcErrors.h, VxcEvents.h, VxcExports.h, VxcRequests.h, VxcResponses.h, vivox-config.h, vxplatform/VxcPlatform.h, vxcplatformmain.h.
  • API surface: All calls go through vivoxSharedWrapper::Vivox from exactly 5 client files:
    • clientUserInterface/src/shared/core/CuiVoiceChatGlue.h
    • clientUserInterface/src/shared/core/CuiVoiceChatManager.{cpp,h}
    • clientUserInterface/src/shared/core/CuiVoiceChatEventHandler.{cpp,h}
    Vivox SDK type names (Vxc*, vivoxsdk) appear nowhere outside the wrapper itself.
  • Linking: Static link to vivoxSharedWrapper_Debug.lib / _Release.lib.
  • Integration depth: Contained by design. The wrapper was originally for EverQuest 2 (Eq2Vivox.h).
  • Replacement cost: low — gut the wrapper and 5 client files.

6. libMozilla / Mozilla XPCOM — in-game web browser

  • Where: src/external/3rd/library/libMozilla/. Wrapper is full SOE source. Browser runtime is vendored Mozilla 1.x XPCOM under libMozilla/include/private/: xul.dll, xpcom.dll, nspr4.dll, plc4.dll, nss3.dll, js3250.dll, softokn3.dll, ssl3.dll, nssckbi.dll, gksvggdiplus.dll + chrome/ tree.
  • Public header is a redirect: libMozilla/include/public/libMozilla/libMozilla.h is 41 bytes: 
    #include "../../src/win32/libMozilla.h"
  • API surface: libMozilla::init(hwnd, path), libMozilla::release(), page-render and event hooks. Call sites: only 4:
    • ClientMain.cpp (init/release)
    • swgClientUserInterface/src/shared/page/SwgCuiWebBrowserWidget.{cpp,h}
    • swgClientUserInterface/src/shared/core/SwgCuiWebBrowserManager.cpp
  • Init code in ClientMain.cpp: 
    // We want to use the Mozilla that's shipped with the game, not whatever's on the system
char szCWD[_MAX_PATH + 1];
GetCurrentDirectory(_MAX_PATH, szCWD);
std::string sPath(szCWD);
sPath += "\\mozilla";
if (!libMozilla::init(Os::getWindow(), sPath.c_str())) {
    DEBUG_FATAL(true, ("Mozilla init failed.\n"));
}
  • Linking: Static against XPCOM .libs (nspr4.lib plc4.lib xpcom.lib xul.lib profdirserviceprovider_s.lib).
  • Integration depth: Contained — already a thin facade.
  • Replacement cost: low — replace with CEF / a stub / nothing.

7. Logitech LCD UI (lgLcd) — G15 keyboard LCD

  • Where: src/external/3rd/library/lcdui/lib/lgLcd.lib + headers. Full Logitech sample source vendored at src/external/3rd/library/lcdui_src/ (LCDOutput, LCDManager, LCDGfx, EZ_LCD).
  • API surface: lgLcd* C API wrapped behind EZ_LCD. Game-side: 3 files — swgClientUserInterface/SwgCuiG15Lcd.{cpp,h} + a config flag in clientUserInterface/ConfigClientUserInterface.{cpp,h}.
  • Linking: Static lgLcd.lib.
  • Integration depth: Contained. Single feature, gated by enableG15Lcd config switch.
  • Replacement cost: trivial.

8. TrackIR — head tracking

  • Where: src/external/3rd/library/trackIR/include/NPClient.h (header only; no .lib referenced).
  • API surface: Effectively zero in the runtime client. The only file that references TrackIR is src/engine/client/library/clientGame/src/shared/core/ClientHeadTracking.cpp. Other matches (3 files in SwgClientSetup) are launcher-side hardware detection.
  • Linking: Not in SwgClient.vcproj's linker list at all.
  • Integration depth: Vestigial. Likely a feature that was prototyped (since SOE shipped JTL-era space combat that benefited from head tracking) and shelved.
  • Replacement cost: nothing to do.

14. Pegasus PICTools / SOE videocapture — JPEG2000 + screen capture

  • Where: src/external/3rd/library/videocapture/{AudioCapture,VideoCapture,ImageCapture,Smart,SoeUtil,ZlibUtil,PICTools}/lib/win32. Debug + Release variants.
  • API surface: SOE in-house screencapture stack. Pegasus PICTools is the JPEG2000 codec underneath (picn20m.lib / picn20md.lib).
  • Linking: Static. Linked unconditionally, but only used in non-PRODUCTION builds (gated by #if PRODUCTION == 0).
  • Integration depth: Contained. Effectively a development feature.

Server-side runtime middleware

15. Oracle OCI — database client

  • Where: src/external/3rd/library/oracle/, oracle_win32/ora90/oci/. Call sites: all of src/engine/shared/library/sharedDatabaseInterface/src_oci/ — OciQueryImplementation.{cpp,h}, OciSession.{cpp,h}, OciServer.h.
  • API surface: OCIEnv, OCILogon, OCIStmtPrepare, OCIDefineByPos, OCIBindByName, OCIError.
  • Linking: Dynamic OCI client.
  • Integration depth: Contained by deliberate ports-and-adapters split:
    • sharedDatabaseInterface/src/ — vendor-neutral abstraction (Bindable, Query, Session)
    • sharedDatabaseInterface/src_oci/ — Oracle implementation
  • Configuration: databaseProtocol=OCI, DSN=..., databaseUID=..., databasePWD=..., databaseThreads=4.
  • Replacement cost: medium — implement a parallel src_pgsql/ or src_mysql/. The abstraction was clearly designed for it.

16. IBM Java 1.3 JVM — gameplay scripting host

  • Where: src/external/3rd/library/java/ (header stubs). The actual JRE has to be installed at /usr/java per src/BUILDING. Call sites: src/engine/server/library/serverScript/ — JavaLibrary.{cpp,h}, JNIWrappers.{cpp,h}, plus 80 ScriptMethods*.cpp files.
  • API surface: full JNI — jni.h, JavaVM, JNIEnv, jclass, jmethodID, JNI_CreateJavaVM. 382 occurrences across 15 surveyed files alone.
  • JavaLibrary.h excerpt: 
    //========================================================================
//
// JavaLibrary.h - interface to the JVM via JNI.
//
// copyright 2001 Sony Online Entertainment
//
//========================================================================

#include <jni.h>
#include "serverScript/JNIWrappers.h"
#include "serverScript/ScriptDictionary.h"
  • Linking: JVM loaded as a DLL/.so at server startup.
  • Integration depth: Pervasive. Most load-bearing middleware in the entire tree. Every gameplay subsystem dispatches into the JVM via these JNI binders.
  • Replacement cost: very high but doable for IBM Java 1.3 → OpenJDK. Replacing the JVM with a different scripting host (Lua/JS) means rewriting dsrc/.../script/ — many tens of thousands of lines. Don't.

Both-side / cross-cutting middleware

9. PCRE — Perl-compatible regex

  • Where: src/external/3rd/library/pcre/4.1/win32/{include,lib} (libpcre.a).
  • API surface: raw pcre_compile / pcre_exec + a memory-allocator hook. Call sites: 5 files:
    • sharedRegex/src/shared/SetupSharedRegex.cpp — installs the malloc hook
    • serverGame/src/shared/core/RegexList.cpp — server-side chat/word filtering
    • sharedTemplateDefinition/src/shared/core/TemplateDefinitionFile.h — template compiler
    • swgClientUserInterface/src/shared/parser/SwgCuiCommandParserScene.cpp
    • swgClientUserInterface/src/shared/parser/SwgCuiCommandParserMount.cpp
  • Linking: Static, with PCRE_STATIC define.
  • SetupSharedRegex.cpp install: 
    void SetupSharedRegex::install() {
    InstallTimer const installTimer("SetupSharedRegex::install");
    DEBUG_FATAL(s_installed, ("SetupSharedRegex already installed."));

    ConfigSharedRegex::install();

    // Save old malloc and free functions.
    s_oldMalloc = pcre_malloc;
    s_oldFree   = pcre_free;

    // Hook up PCRE malloc and free to our memory manager.
    pcre_malloc = &RegexServices::allocateMemory;
    pcre_free   = &RegexServices::freeMemory;

    s_installed = true;
    ExitChain::add(remove, "SetupSharedRegex");
}
  • Integration depth: Contained-ish. Only 5 callers, but no facade — they #include "pcre/pcre.h" directly.
  • Replacement cost: low. Swap to std::regex or re2.

What about udplibrary?

src/external/3rd/library/udplibrary/ is technically in the third-party tree, but it's SOE in-house source, not licensed middleware. It's the reliable-UDP transport layer the wire protocol depends on. You cannot replace it without breaking compatibility with retail clients. Counts: 7 headers, 3 .cpp.

Bottom line: where rewriting hurts

If anything in this tree ever needs to be modernised aggressively, the cost lives in just two places:
  1. The IBM Java 1.3 JVM dependency — pervasive, unfixable without porting the JNI bridge, unavoidable because all gameplay rules live in the JVM. Plan for this.
  2. The Sony Station / soePlatform / stationapi suite — not pervasive in any single server, but in aggregate it cuts across LoginServer, ChatServer, CustomerServiceServer, TransferServer, MetricsServer. Replacing it means writing community equivalents of five separate services.
Everything else — DirectX, Bink, Miles, DPVS, Vivox, libMozilla, Logitech LCD, Oracle OCI, PCRE, libxml2, zlib, boost, STLPort — is contained behind a facade or behind a single library boundary, and can be swapped or stubbed without touching gameplay code.

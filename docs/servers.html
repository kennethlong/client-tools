Server processes

All servers share the same SharedFoundation stack: SetupSharedFoundation, SetupSharedNetwork, SetupSharedFile, etc. Each main() wires up the subsystems it needs and enters a tick loop. Configuration is layered: each process reads its own *.cfg plus servercommon.cfg via the .include directive.

Look at src/game/server/application/SwgGameServer/src/linux/main.cpp for the canonical example of how a SWG server boots — every other server follows the same shape.

The six core servers (built by servers.mak)

ServerSourceRole
LoginServer src/engine/server/application/LoginServer/ Front-of-house TCP service. Validates account credentials against Sony Station API + login Oracle schema. Returns the list of galaxies and characters and a session token.
CentralServer src/engine/server/application/CentralServer/ Per-galaxy coordinator. Maintains the cluster map, places players on planet servers, brokers chat/commodity routing.
ConnectionServer src/engine/server/application/ConnectionServer/ Holds the player's UDP session. All gameplay UDP packets flow through this process; it forwards them to the correct game server.
PlanetServer src/engine/server/application/PlanetServer/ Owns one shard (a planet or a space zone). Hosts the planetary world snapshot, brokers handoff between game-server children, and routes objects across zone boundaries.
SwgGameServer src/game/server/application/SwgGameServer/ The big one. Runs the game tick: AI, combat, movement, scripting. Hosts the embedded JVM that runs dsrc/ scripts. There are usually multiple per planet, partitioned by region.
SwgDatabaseServer src/game/server/application/SwgDatabaseServer/ Writes object/character state back to Oracle on a fixed cadence. Designed so game servers never block on the DB.

Supporting processes

ServerWhat it provides
ChatServer Persistent chat rooms, in-game mail, friends list. Cross-galaxy via the same backbone. Source: src/engine/server/application/ChatServer/.
CommoditiesServer / CommodityServer The galactic auction house / bazaar. Two implementations exist in the tree — the older CommodityServer and the newer CommoditiesServer.
CustomerServiceServer GM tools / petitions / appeals routing. Talks to SwgCsTool.
TransferServer Character transfer between galaxies. The "paid character transfer" feature.
MetricsServer / LogServer Telemetry sinks. servercommon.cfg shows SharedLog shipping to a network log target on port 44467.
StationPlayersCollector Reports concurrent player counts back to SOE Station infrastructure.
TaskManager Per-host supervisor. Reads _taskmanager.cfg / .rc and spawns, monitors, and restarts the other servers on its host.
LoginPing Lightweight health-check responder used by load balancers.
PhonyApp Stub harness used to bring up just the foundation framework — handy for testing shared libs in isolation.

The combat-engine glue specifically is in src/game/server/application/SwgGameServer/src/shared/combat/CombatEngine.cpp. It dispatches into scripts so designers can override damage formulas, hit rolls, special-attack effects from dsrc/.../script/combat without recompiling C++.

Configuration model

Every server reads a stack of cfg files. For example, exe/linux/loginServer.cfg opens with:

.include "../../exe/shared/servercommon.cfg"

[LoginServer]
databaseProtocol=OCI
DSN=sdswgp5b
databaseUID=login
databasePWD=login_prod8
databaseThreads=4
validateStationKey=1
sessionServer=sdt-session1.station.sony.com:3000
...
These cfg files still contain the original SOE production hostnames, Oracle DSNs, and session servers. They are reference, not runnable. On any local revival you must:
  • set validateStationKey=0 (or replace with custom auth)
  • point DSN at your own Oracle instance
  • strip privateIpMask entries that match SOE's old ranges
  • retarget SharedLog away from SOE log endpoints

Search paths and TreeFile

Each server resolves its content under [SharedFile] via stacked searchPath0/1/2 entries (numbered low-to-high priority). servercommon.cfg wires this to data/sku.0/sys.shared/compiled/game, sys.server/compiled/game, and a number of exported trees. The same abstraction transparently reads from a .tre archive if one is configured — see Data Pipeline.

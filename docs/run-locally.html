Running it locally

Bottom line: the unmodified February 2015 tree will not build, let alone run, on a current machine. To actually play SWG locally you pick one of three established paths below — none of which is "make this repo go". This document is here to set expectations and route you to the right next step.

Path A — Read / explore the codebase

This is what these docs are for. Open index.html and walk through architecture → servers → scripting → data-pipeline. To navigate the C++ specifically, point your editor at src/ and start from SwgGameServer's main.cpp.

Path B — Run an editor / tool from exe/win32/

Most of the *.exe files in exe/win32/ are content-creation tools built against the same engine. They do load on a recent 64-bit Windows (32-bit-compatible). The trick is that they expect to read game assets via the TreeFile system, which means they need .tre archives nearby.

  1. Get a legitimate retail SWG install (CD media or community archive).
  2. Copy its data/, patch/, and profile/ directories into a working folder.
  3. Drop the contents of exe/win32/ from this repo on top.
  4. Edit the matching template_<Tool>.cfg if needed (search paths).
  5. Launch the tool — it should read assets out of the retail tree.

Tools worth playing with first: ViewIff_o.exe (inspect any .iff file), WorldSnapshotViewer_o.exe, ParticleEditor_o.exe, SwgConversationEditor_o.exe. They make few network calls and degrade gracefully when SOE backends are unreachable.

Note: some tools (SwgGodClient_o.exe, the editor mode) want to attach to a running SwgGameServer; they're not standalone.

Path C — Stand up a private server and play

Three serious projects exist; pick by what you actually want to play:

ProjectEraCodebaseWhere to start
SWG-Source / StellaBellum NGE (this leak's era) Direct descendant of this source swg-main wiki — covers VM install, Oracle setup, Ant build, server admin.
SWGEmu Pre-CU (original 2003 design) Independent clean-room C++ rewrite swgemu.com
ProjectSWG / NGECore2 CU/NGE era Java rewrite NGECore2

If you want to run the actual code in this repo (or its modern descendant), the only practical route is SWG-Source's StellaBellum dev VM:

  1. Install Oracle VirtualBox.
  2. Download the StellaBellum dev VM from the SWG-Source wiki.
  3. Import the .vbox file. Default 2 cores / 8 GB RAM is fine; bump if you can.
  4. Inside the VM, clone SWG-Source/swg-main and run: 
    ant update_configs    # one-off site setup
ant create_database   # bootstrap Oracle schemas
ant swg               # full build (C++ + scripts)
./start_cluster.pl    # bring up the cluster
  5. On your Windows host, point a retail SWG client at the VM's IP via a modified login.cfg.
Why not just port this repo? You can — and SWG-Source has been doing it since 2015. Their fork is the result of years of incremental work to get the same code building on current toolchains. Reproducing that effort from the unmodified February 2015 snapshot is multiple months of work, and the finished product is a reimplementation of what already exists publicly. If your goal is a running server, fork SWG-Source. If your goal is to study the original SOE code, this archive plus these docs is the right starting point.

What to not do

  • Don't try to run the linux configs in exe/linux/ against current Sony domains — they're long dead.
  • Don't expect swg.sln to upgrade cleanly in VS 2022. It won't, without significant patching.
  • Don't run start_cluster.pl as-is on a stock Linux VM — it expects production paths under /swg/swg/current/ and an Oracle install at fixed locations.
  • Don't trust the credentials/DSNs in loginServer.cfg — they're SOE production values that no longer point to anything reachable.

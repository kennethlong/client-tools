Data pipeline

Most of SWG's "weight" isn't code, it's content: Maya assets, terrain shaders, schematics, conversation graphs, world snapshots. The whole pipeline ends in .tre archives that ship to the player. The repo includes:

  • Authoring source under dsrc/ (datatables, server scripts, shared object templates).
  • Authoring tool binaries in exe/win32/ + Maya plug-ins in plugin/win32/.
  • publish/*.mft manifests describing the live .tre tree at the time of leak.

It does not include the actual .tre archives.

The file formats

ExtensionWhat it is
.treEA-style chunk archive: container of many small files. Compressed, MD5-hashed, indexed. The runtime opens many of them in priority order — see TRE format breakdown linked below.
.iffInterchange File Format: nested chunk binary. Used for almost every authored asset (meshes, animations, datatables, templates, world snapshots). ViewIff.exe opens any of them.
.tabTab-separated text spreadsheets with a typed header row. Compiled into .iff datatables.
.tpfTemplate source — the human-readable form of an object template. Compiled to .iff by TemplateCompiler.exe.
.stfString tables — localisation. UpdateLocalizedStrings tool round-trips them.
.ddsDirectDraw Surface textures (DXT-compressed).
.scriptSWG scripting source (see Scripting).
.scriptbinCompiled script bytecode for the embedded JVM.
.wsWorld snapshot — placement of static objects in a planet shard.
.mp / .mpbMesh / mesh bone files.
.cfgPlain-text key=value config; supports .include directives.

TreeFile resolution

Both client and server access content through the same sharedFile/TreeFile abstraction (src/engine/shared/library/sharedFile/src/shared/file/TreeFile.cpp). At startup the engine reads servertreefiles.cfg (in exe/shared/servertreefiles.cfg — the server side) or the client's patch tree (data_*.txt, patch_*.txt in publish/) and layers them. When you ask for texture/foo.dds, it walks through the archives in priority order and serves the first hit — which is exactly how the patch tree overrides base content.

The publish/ directory

Files here are not assets, they are manifests of what's in each .tre. A representative line from publish/client_data_tree_files.mft: 

other  0  same  unpatched  47  47  0ff469dc1658dd740ecf6f05cd108dfe
       //depot/swg/test/data/sku.0/sys.shared/compiled/game/abstract/slot/arrangement/arrangement_ammo_cartridge.iff#1
       abstract/slot/arrangement/arrangement_ammo_cartridge.iff

Columns: category, flags, state, which patch tree, compressed size, raw size, md5, perforce depot path, tre-relative path. This is enough to confirm what the live tree looked like. It is not enough to recreate the bytes — for that you need the assets themselves.

Sku layout

"sku" = sales SKU. The dsrc/ tree (and the data/ tree it generates) is split:

  • sku.0 — base game content.
  • sku.1 — Jump to Lightspeed (space) expansion.
  • sku.2 — Rage of the Wookiees.
  • sku.3 — Trials of Obi-Wan.
  • sku.ja — Japanese release.

See servercommon.cfg — the search paths walk all four: 

searchPath0=../../data/sku.0/sys.shared/exported/architecture
searchPath0=../../data/sku.2/sys.shared/exported/architecture
searchPath0=../../data/sku.0/sys.shared/exported/item
searchPath0=../../data/sku.1/sys.shared/exported/space
searchPath0=../../data/sku.0/sys.shared/exported/vehicle
searchPath0=../../data/sku.0/sys.shared/exported/world

Cluster architecture

SWG is not one server — it's a galaxy made up of cooperating Linux processes. A single planet (Tatooine) has its own PlanetServer with one or more SwgGameServer children running gameplay. A single customer connection rides custom UDP to a ConnectionServer, which acts as the front door. CentralServer coordinates the rest. TaskManager is the per-host supervisor that respawns processes if they die.

The networking layer

Client ↔ server uses a proprietary SOE UDP library at src/external/3rd/library/udplibrary — a reliable, ordered protocol on top of UDP with custom acks and fragmentation. This is wrapped by src/engine/shared/library/sharedNetwork on both sides. Inter-server traffic uses TCP through the same sharedNetwork abstraction.

Network messages are typed structures defined under src/engine/shared/library/sharedNetworkMessages and src/game/shared/library/swgSharedNetworkMessages (game-specific). Each message is a class with serialise/deserialise methods. Look at messageTypes.def files for the registry of message IDs.

Heads-up: the UDP library (udplibrary) appears in this repo as headers and source — but the wire protocol is what makes a community client/server pair compatible. If you swap clients (e.g. retail SWG client vs. SWGEmu client), you have to match the UDP protocol bit-for-bit.

Message bus & "object IDs"

Internally, the world is modelled as a big set of game objects (creatures, items, buildings, etc.) each with a 64-bit network id. Servers move these objects around the cluster by controllers in src/engine/shared/library/sharedObject, with messages like CMSG_* / SMSG_* defined in sharedNetworkMessages. The same id identifies an object on the client.

When a player crosses a zone boundary, the object handoff goes: SwgGameServer (source) → PlanetServer → SwgGameServer (target), with database write-back via SwgDatabaseServer. This is one of the trickiest paths in the engine.

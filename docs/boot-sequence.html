Use this document during build work, not as a one-time read. It's structured as 17 phases, each with: entry point, what it does, subsystems initialised, external dependencies, failure modes, and stub potential. The three milestone lines indicate where you can pause incremental builds — most importantly, "login screen renders" is the natural minimum viable boot target.

Phases 1–9 — process to login screen

Phase 1 — Process entry & memory cap

  • Entry: SwgClient/src/win32/WinMain.cpp:124
  • Reads env var: SWGCLIENT_MEMORY_SIZE_MB (optional integer).
  • Default: 75% of physical RAM, clamped to [250, 750] MB in dev or [250, 2000] MB in production.
  • Failure modes: none significant.
  • Stub potential: N/A.

Phase 2 — Thread + debug bootstrap

  • Entry: ClientMain.cpp:137 — SetupSharedThread::install() + SetupSharedDebug::install(4096).
  • Initialises: sharedThread, sharedDebug (4096-entry ring buffer).
  • Failure modes: TlsAlloc failure → FATAL.
  • Stub potential: none required.

Phase 3 — Foundation: config, window, registry

  • Entry: SetupSharedFoundation::install(data) in SetupSharedFoundation.cpp:180, called from ClientMain.cpp:152-167.
  • Sets up SEH handler (writes *-<ver>-<timestamp>.{txt,mdmp,log} minidumps if data.writeMiniDumps).
  • Subsystems (in order):
    1. CommandLine::install + absorbs lpCmdLine.
    2. ConfigFile::install + ConfigFile::loadFile("client.cfg").
    3. FatalInstall + (non-PROD) DebugMonitor::install.
    4. ConfigSharedFoundation::install (frame-rate limit, demo mode, verbose warnings).
    5. MemoryBlockManager::install, ExitChain::install, Report::install, Clock::install, CrashReportInformation::install.
    6. RegistryKey::install — opens / creates HKCU\Software\Sony Online Entertainment\Default and HKLM\...\Default. Cfg key [ConfigSharedFoundation] ProductRegistryPath=... overrides.
    7. PersistentCrcString::install, CrcLowerString::install, WatchedByList::install.
    8. Os::install(hInstance, windowName, normalIcon, smallIcon) — RegisterClassEx + CreateWindow(WS_POPUP, 640×480). Resized later in SetupClientGraphics.
    9. StaticCallbackEntry::install, setFatalVersionString.
  • External deps:
    • File: client.cfg in cwd. If empty / missing → FATAL("Config file not specified").
    • Registry: HKCU\Software\Sony Online Entertainment\Default (created if absent), HKLM\...\Default (read).
    • WinAPI: RegOpenKeyEx, RegCreateKeyEx, RegisterClassEx, CreateWindow, SetUnhandledExceptionFilter.
  • Failure modes: missing/empty client.cfg → fatal dialog. Restrictive HKLM policies could fatal on key creation.
  • Stub potential: override registry path via ProductRegistryPath in client.cfg. No SOE infrastructure touched.

  • Cfg keys read: [Station] gameFeatures, [Station] subscriptionFeatures, [ClientGame] gameBitsToClear, [ClientGame] setJtlRetailIfBetaIsSet.
  • Files: every .tre on the configured search path ([SharedFile] searchPath0/1/2, [Tre]).
  • Failure modes: missing .tre files don't FATAL here — first asset miss is in Phase 6 (misc/override.cfg) or Phase 8 (misc/asynchronous_loader_data_*.iff).
  • Stub potential: ship a tiny private .tre with the bare-minimum boot files. Set [Station] gameFeatures=15 to enable all four SKUs.

Phase 6 — Override config from TRE

Optional file inside any mounted .tre; layered over client.cfg. No-op if missing.

Phase 7 — Audio / Graphics / libMozilla / Input

  1. SetupClientAudio::install — boots Miles via Mss32.dll.
  2. SetupClientGraphics::install({ screenWidth=1024, screenHeight=768, alphaBufferBitDepth=0 }) — resizes window, creates D3D9 device through the gl0X_*.dll plug-in.
  3. libMozilla::init(Os::getWindow(), "<cwd>\\mozilla") — loads XPCOM DLLs. DEBUG_FATAL on failure (DEBUG builds only).
  4. VideoList::install(Audio::getMilesDigitalDriver()) — registers Bink (loaded via LoadLibrary("binkw32.dll")).
  5. SetupClientDirectInput::install(hInstance, Os::getWindow(), DIK_LCONTROL, Graphics::isWindowed).
  6. SetupClientObject, SetupClientAnimation, SetupClientSkeletalAnimation, SetupClientTextureRenderer, SetupClientTerrain, SetupClientParticle.

Required DLLs in cwd: Mss32.dll, binkw32.dll, gl0X_*.dll, dpvs.dll/dpvsd.dll.

Required DLLs in cwd\mozilla\: xul.dll, xpcom.dll, nspr4.dll, plc4.dll, plds4.dll, nss3.dll, nssckbi.dll, smime3.dll, softokn3.dll, ssl3.dll, js3250.dll, gksvggdiplus.dll + chrome\ tree.

Stub potential: libMozilla/Bink/Vivox can each be stubbed cleanly (see middleware inventory).

Phases 10–12 — login flow

Phase 10 — User submits login form

// SwgCuiLoginScreen.cpp:230+
GameNetwork::setUserName(Unicode::wideToNarrow(name));
GameNetwork::setUserPassword(Unicode::wideToNarrow(passwd));

std::vector<std::pair<std::string, uint16> > loginServerList;
int i = 0;
do {
    sprintf(buffer, "loginServerAddress%i", i);
    address = ConfigFile::getKeyString("ClientGame", buffer, 0, 0);
    if (address) {
        sprintf(buffer, "loginServerPort%i", i);
        const uint16 port = static_cast<uint16>(ConfigFile::getKeyInt("ClientGame", buffer, 0, 0));
        if (port != 0)
            loginServerList.push_back(std::make_pair(std::string(address), port));
        ++i;
    }
} while (address);

const int choice = Random::random(loginServerList.size() - 1);
GameNetwork::connectLoginServer(loginServerList[choice].first,
                                loginServerList[choice].second);
  • Cfg keys: [ClientGame] loginServerAddress0..N, loginServerPort0..N.
  • Network: outbound UDP to chosen (addr, port). Note: setupData.useTcp = false — login is UDP, not TCP.
  • Stub action: set loginServerAddress0=127.0.0.1 + loginServerPort0=44453 in your private client.cfg.

Phase 11 — LoginConnection handshake (Station auth divergence)

LoginConnection.cpp:53:

void LoginConnection::onConnectionOpened() {
    GameNetworkConnection::onConnectionOpened();
    CuiLoginManager::setOverrideSessionIdKey(std::string());

    bool sendUserName = true;
    if (CuiLoginManager::getSessionIdKey() && !ConfigClientGame::getEnableAdminLogin())
        sendUserName = false;

    LoginClientId id(sendUserName ? GameNetwork::getUserName() : "",
                     GameNetwork::getUserPassword());
    send(id, true);

#if PRODUCTION != 1
    GenericValueTypeMessage< int > msg("RequestExtendedClusterInfo", 0);
    send(msg, true);
#endif

    static MessageConnectionCallback m("LoginConnectionOpened");
    emitMessage(m);
}

CuiLoginManager::getSessionIdKey() reads [Station] sessionId from client.cfg:

// CuiLoginManager.cpp:1187
const char * CuiLoginManager::getSessionIdKey(bool useOverride) {
    static const char * const SessionIdSection = "Station";
    static const char * const SessionIdKey     = "sessionId";
    const char * sessionIdKey = ConfigFile::getKeyString(SessionIdSection,
                                                        SessionIdKey, NULL);
    if (!sessionIdKey && useOverride && !s_overrideSessionIdKey.empty())
        sessionIdKey = s_overrideSessionIdKey.c_str();
    return sessionIdKey;
}
SOE-specific assumption: if [Station] sessionId is set, the client expects the LoginServer to validate the session against sdt-session*.station.sony.com (dead). For private servers, leave [Station] sessionId unset and rely on plain user/pass. The LoginServer must be configured with validateStationKey=0.

Implication: keep PRODUCTION == 0 while developing — you get the editor entry paths, debug menu, DataLint, video capture, and the RequestExtendedClusterInfo ping. The dev build is not less secure than retail in any meaningful way; there's no anti-cheat in either path.

Closing notes for the build engineer

  • The boot is deeply linear and well-instrumented. Almost every install function calls InstallTimer const installTimer("..."), so with [SharedDebug] installTimer=true in client.cfg you get a profile of every phase by name. Use this religiously during MVB work to find which phase is hanging.
  • client.cfg is the most important file in the project. Treat it as a build artefact: keep it under version control, parametrise hostnames, never check the SOE-shipped one in.
  • The login phase is the only place the client makes assumptions about who the server is. Once a LoginConnection is open, every message is opaque-payload-with-name; matching wire-protocol versions is what makes a private server compatible.
  • "Login screen renders but submit hangs" is almost always: (a) wrong loginServerAddress cfg, (b) firewalled UDP, (c) version mismatch on LoginClientId — in that order.
